第 30 篇里 SignYourName PR 跑通后，工程师的名字进了 RobustMQ 项目的贡献者列表。但那是心理胜利，不是真本事。第一个 PR 证明工程师能在不属于他的代码上交付，第二个 PR 才证明他能在不熟的栈上做出真正的工程贡献。本篇就在第 30 篇那个 Rust 项目上完成两件事：第二个 PR 加一个真实功能 + 第一个 issue 报告一个真实问题。两件事跑完，工程师就正式成为这个项目的 Contributor。

## 2. 整体工作流：用 AI 攻坚不熟悉的栈

<img src="imgs/aicmigr-31-opensource-02-second-pr-rust-tutor/fee52d598d019b06b05bf23c182e1d35_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

### 2.1 为什么"不熟悉的栈"不再是 blocker

Rust 是 AI 时代仍然有学习门槛的语言，所有权、生命周期、async 这些机制不是看一眼就能上手。但有了 AI 当助教，门槛就被压下去：工程师对这门语言不熟没关系，AI 解释机制，工程师做判断。本篇要展示的就是这种工作姿势——不熟的栈，不再是 blocker。

### 2.2 四阶段工作流

#### (1) 第一阶段：摸项目

目标：在不属于工程师的代码上动手前，建立"对当前任务有用"的项目认知地图。关键产出：模块划分、测试组织、贡献热点、适合新人的入口这四维度，但带任务过滤。

#### (2) 第二阶段：筛 issue

目标：从项目 issue 列表中挑出一个能上手、能合并、能 review 的 issue 作为第二个 PR 目标。关键产出：5 个候选 issue + 优先级排序 + 自己最终选定的那一个。

#### (3) 第三阶段：改 + 测 + 提 PR

目标：在不熟悉的栈上，由 AI 跑完从 fork 到 PR 的全流程，工程师审核。关键产出：一个可合并的 PR，带 issue 链接、高层描述、实现要点、测试覆盖、trade-off 说明。

#### (4) 第四阶段：扫漏洞提 issue

目标：让 AI 找一个不复杂的漏洞，工程师把它写成高质量 issue 提交。关键产出：一份带"我可以跟 PR"信号的高质量 issue，价值不低于一个 PR。

### 2.3 与系列其他文章的衔接

本篇工作流并非凭空而来，它建立在系列前几篇建立的方法论基础上，并且可以迁移到任何项目。

| 衔接关系 | 系列前文 | 本篇对应阶段 | 视角变化 |
|---------|---------|-------------|---------|
| 项目摸底 | 第 24 篇四维度框架 | 第一阶段 摸项目 | 从"新建系统"视角转为"找 issue"视角 |
| 项目筛选 | 第 30 篇三道题筛项目 | 第一阶段前置 | 选定本篇要攻坚的项目 |
| 提示词驱动写代码 | 第 27 篇 | 第三阶段 改+测+提 PR | 复用"先 plan 后改"硬规则 |
| 第一个 PR | 第 30 篇 SignYourName | 实战背景 | 心理胜利 → 真本事的过渡 |

这套工作流一旦在一个项目上跑通，工程师在任何项目上都能复刻——本篇用 Rust 项目 RobustMQ 做演示，但载体可以换。

## 3. 摸项目：带任务问的项目地图

<img src="imgs/aicmigr-31-opensource-02-second-pr-rust-tutor/13acb827c56efbc6920361ddc8b6a8ee_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

### 3.1 阶段目标与产出

目标：在不属于工程师的代码上动手之前，建立一份"对当前任务有用"的项目认知地图，避免在不了解项目结构的情况下盲选 issue。产出包含四个维度——模块划分、测试组织、贡献热点、适合新人的入口，但每一维度都要带任务过滤，只回答"对找 issue 有帮助的那部分"。

### 3.2 关键姿势：带任务问的项目地图

#### (1) 任务变了，问 AI 的角度也要变

第 24 篇建立的四维度框架（它是什么、它能做什么、它怎么扩展、它内部怎么跑）可以直接套用，但本篇的任务和第 24 篇不一样。第 24 篇的任务是做一个新系统，本篇的任务是找一个能上手的 issue。任务变了，问 AI 的角度也要变——AI 不知道工程师的目的时，会给一份干巴巴的架构介绍；告诉 AI 目的，它会过滤出对工程师有用的部分。

#### (2) 提示词的关键设计

提示词的关键是"针对找 issue 任务的项目地图"这一句话。它把任务上下文塞给 AI，让 AI 知道要过滤什么、忽略什么。完整的提示词全文见第二部分第 8.2 节。这里只列出关键骨架：

- 显式声明目的：做第二个 PR，目标是提交一个有真实工程价值的小贡献
- 显式声明维度：按四维度给"针对找 issue 任务"的项目地图
- 显式声明约束：不要给完整架构文档，只给对找 issue 有帮助的那部分

### 3.3 四维度框架（针对找 issue 任务）

把第 24 篇的四维度框架套到"找 issue"任务上，每个维度的关注问题和落点如下表：

| 维度 | 关注问题 | 在"找 issue"任务下的具体落点 |
|------|---------|-----------------------------|
| 模块划分 | 这个项目的代码大致分成几块？每块负责什么？ | 知道改一个 issue 大概会落在哪个模块，方便判断改动范围 |
| 测试组织 | 单测放在哪、集成测试放在哪、怎么跑？ | 知道补测试要往哪写、用什么命令跑，避免补测试时摸不着门 |
| 贡献热点 | 最近三个月哪几个模块改动最多？活跃 contributor 主要在哪些模块？ | 找活跃模块更容易被维护者 review 通过，避开冷门死区 |
| 适合新人的入口 | good first issue / help wanted 标签下有没有可以上手的方向？CONTRIBUTING.md 有什么对新人的硬约束？ | 直接定位到待领的小活，知道新增代码必须满足哪些硬约束（如必须带测试） |

### 3.4 操作要点

#### (1) 摸项目阶段的关键操作

##### ① 在仓库目录下的 Claude Code session 里跑

AI 在仓库里有完整的代码访问能力，提示词才能驱动完整的项目分析流程。

##### ② 带任务问，要求 AI 过滤

显式告诉 AI 目的，要求过滤出对当前任务有用的部分。

##### ③ 关注 CONTRIBUTING.md 对新人的硬约束

这是后续 PR 能否被合并的硬门槛，必须在摸底阶段就掌握（例如"所有新增代码必须带测试"）。

##### ④ 跑完后心里要有一张地图

跑完这一段，工程师对项目内部心里要有数：哪几块、最近活跃的是哪块、待领的小活大概在哪、硬约束是什么。地图有了，下一步才是从这张地图上挑一个能干的 issue。

## 4. 筛 issue：三条标准与 AI 协同

<img src="imgs/aicmigr-31-opensource-02-second-pr-rust-tutor/3e186425bffa20c88400b9a611ade3c1_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

### 4.1 阶段目标与产出

目标：从项目 issue 列表中挑出一个"能上手、能合并、能 review"的 issue 作为第二个 PR 目标。关键产出：让 AI 给出 5 个候选 issue + 优先级排序，工程师最终拍板选定那一个。

### 4.2 三条筛选标准

挑 issue 不是凭直觉，是按标准。第二个 PR 不是搞大型重构，目标是做完一件能合并的小事。三条标准如下表：

| 标准编号 | 标准内容 | 为什么这条重要 | 量化指标 |
|---------|---------|---------------|---------|
| 标准一 | 改动量不大 | 第二个 PR 不是搞大型重构，目标是做完一件能合并的小事 | 代码 + 测试加起来控制在 50-200 行 |
| 标准二 | AI 能自己改 | AI 看完代码上下文就能写出来，避开需要深度业务理解或跨多模块协调的 issue | issue 描述清楚 + 改动是局部的（单文件或单模块内） |
| 标准三 | 工程师能 review | AI 写完代码工程师能看懂、能判断对错、能跟维护者讨论 | 工程师看完知道在改什么；签字提 PR 的是工程师，简历上挂这条 commit 的也是工程师，出问题就是工程师的责任 |

### 4.3 让 AI 出候选清单

#### (1) 提示词关键设计

- **复用同一 session**：还在同一个仓库目录下的 Claude Code session 里继续跑，前面那份项目地图 AI 还记得
- **把三条标准丢给 AI**：明确告诉 AI 改动量、描述清晰度、是否需要深度业务理解这三个维度的硬约束
- **要求每个候选给出五项信息**：issue 编号和标题、它在哪个模块、工程师做这个 PR 大概要干什么（一两句话）、为什么它符合三条标准、风险点在哪（比如有没有可能改完发现规模超出预期）
- **要求 5 个候选排优先级**：从最稳到最有挑战

完整的提示词全文见第二部分第 9.1 节。

### 4.4 工程师如何最终拍板

AI 给出候选清单之后，工程师不能直接采纳第一个，需要按下面四个维度做最终判断：

#### (1) 最终拍板的四个判断维度

##### ① 真实工程价值

说清楚这个改动满足什么用户需求。例如查询类接口本来就该用 GET，是 REST 语义的回归——这是真实工程价值。

##### ② 改动是局部的

说清楚在哪个模块、影响范围多大。例如改动只在 admin-server 模块、涉及 4 个文件——这是局部改动。

##### ③ Rust 经验要求适中

说清楚用到哪些 Rust 特性，但不至于硬到看不懂。

##### ④ 工程师能 review

即使对 Rust 不熟，相关模式能看懂。这一点最关键——超出工程师能 review 范围的 issue，不管多有价值都不能选。

### 4.5 操作要点

#### (1) 筛 issue 阶段的关键操作

##### ① 选 issue 不是凭直觉，是按标准

三条标准必须同时满足，缺一不可。

##### ② 不要选自己看不懂的 issue

签字提 PR 的是工程师，简历上挂这条 commit 的也是工程师。出问题就是工程师的责任，AI 不会替工程师背锅。

## 5. 改 + 测 + 提 PR：长提示词驱动工作流

<img src="imgs/aicmigr-31-opensource-02-second-pr-rust-tutor/c8c06d80940b430b1da4623a17116de9_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

### 5.1 阶段目标与产出

目标：在不熟悉的栈上，由 AI 跑完 fork、plan、写代码、补测试、lint、commit、push、PR 全流程，工程师审核。关键产出：一个可合并的 PR，PR 描述里包含 issue 链接、高层描述、实现要点、测试覆盖、trade-off 说明。

### 5.2 长提示词的五个关键设计

挑好 issue 之后，剩下的就是第 30 篇那个工作流的复刻：fork + clone + 一段长提示词，AI 跑完剩下的全部。这一段提示词比 SignYourName 那次长，因为这次是真改代码，不是改 Markdown。五个关键设计如下表：

| 关键设计 | 设计内容 | 为什么这样设计 |
|---------|---------|---------------|
| 设计一：先 plan 后改 | AI 不能拿到 issue 就闷头写，得先把 plan 给工程师审核，工程师同意才动 | 这是第 27 篇提示词驱动写代码那一篇建立的硬规则。AI 自作主张写完代码后再被驳回，浪费时间和上下文 |
| 设计二：关键改动逐段解释 | AI 不只是写代码，边写边教，逐段说"这里在做什么、为什么这么写" | 让工程师能在 review 时真正理解代码，而不是签字画押 |
| 设计三：所有权 / 生命周期 / async 简短解释 | 涉及所有权、生命周期、async 的关键改动处简短解释 Rust 机制 | 这是 AI 当 Rust 助教的具体形态——压低不熟悉栈的门槛 |
| 设计四：不用复杂 idiom | 选最直白的写法；多种合理写法中选项目现有代码最常用的风格 | 防止 AI 写出 Rust 高手风格代码，导致工程师看不懂、review 不了 |
| 设计五：warning 必清不留 allow 绕过 | 所有 warning 都要清掉，不要留 `#[allow(...)]` 绕过 | 开源项目的硬要求，不能用 `#[allow(...)]` 偷懒 |

### 5.3 十步执行流程

#### (1) Step 1：读 issue + 给 plan

读 issue 模块当前实现、依赖 API、上下游关键调用，给工程师一份 3-5 条的"我要改什么"简短 plan。工程师审核 plan 之后才能进下一步。

#### (2) Step 2：建新 branch

plan 工程师同意之后，建一个新 branch，命名跟项目风格对齐（`feature/<跟 issue 主题对应的简短描述>`）。

#### (3) Step 3：按 plan 写代码

Rust 工程师不熟，所以 AI 写完之后：关键改动逐段告诉工程师"这里在做什么、为什么这么写"；任何用到所有权 / 生命周期 / async 的地方，简短解释；不要为了炫技用复杂的 idiom，选最直白的写法；如果有多种合理写法，选项目现有代码里最常用的那种风格。

#### (4) Step 4：补测试

看项目现有测试在哪、怎么组织，跟着同样的风格写。测试要覆盖正常行为、退出场景、参数边界值等关键 case。

#### (5) Step 5：跑测试

如果不知道怎么跑，先看 Makefile / CI 配置 / cargo 配置自己摸清楚。跑通为止。

#### (6) Step 6：跑 lint

跑项目的 lint：`cargo fmt` + `cargo clippy`。所有 warning 都要清掉，不要留 `#[allow(...)]` 绕过。

#### (7) Step 7：写 commit message

写一条 commit message，按 Conventional Commits 规范。然后 `git add + commit`。

#### (8) Step 8：push 到 fork

把 branch push 到工程师自己 fork 的仓库。

#### (9) Step 9：写 PR 描述

PR 描述包括：这个 PR 解决的 issue；改动的高层描述（2-3 句）；实现要点（3-5 条 bullet）；测试覆盖了哪些场景；有任何 trade-off 或后续可改进的点显式说出来。

#### (10) Step 10：提 PR

如果装了 gh CLI，直接 `gh pr create` 提交。没装就把 PR 描述输出给工程师，工程师自己去 GitHub 提。

### 5.4 关键的中间 review 点

#### (1) 步骤 5 与 6 之间必须停一下

在跑测试和跑 lint 之间，工程师必须 review AI 写完的代码。给开源项目提交的代码一定需要 review，这是对开源负责任的体现。

#### (2) 任何报错或不确定，AI 必须停下来问

不要让 AI 自己瞎修。任何报错或不确定的地方，AI 立刻停下来问工程师，由工程师判断怎么处理。

### 5.5 AI 当 Rust 助教的工作姿势

AI 不只是替工程师写代码，是边写边教。逐段解释关键改动、提醒所有权和 async 细节、推荐项目风格 idiom。把不熟悉栈的门槛压下去。**工程师不熟没关系，AI 解释机制，工程师做判断**——这是本篇想留下最重要的工作姿势，也是 Rust 助教这个说法的具体形态。

### 5.6 操作要点

#### (1) 改 + 测 + 提 PR 阶段的关键操作

##### ① 长提示词比 SignYourName 那次长

因为这次是真改代码，不是改 Markdown。提示词长度对应任务复杂度。

##### ② 在 fork 后的本地仓库目录下打开 Claude Code 跑

AI 在仓库里有完整的代码访问能力，提示词才能驱动完整的开发流程。

##### ③ 第 30 篇已经 fork 过的项目，直接在原仓库目录下继续

避免重复 fork、重复 clone，保持 commit history 干净。

##### ④ review 是硬要求，不是可选

给开源项目提交的代码必须 review，没有"信任 AI 跳过 review"这种选项。

## 6. 提高质量 issue：扫漏洞 + 高质量模板

<img src="imgs/aicmigr-31-opensource-02-second-pr-rust-tutor/495d86637ff40414a59ead3ab2e78afa_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

### 6.1 为什么 issue 也是 Contributor 行为

#### (1) 高质量 bug 报告价值不低于 PR

开源项目里很多核心维护者会告诉你：一个写得清楚、可复现、有定位价值的 bug 报告，价值不比一个 PR 小。PR 可能因为方向不对被砍，但 issue 是输入信息，几乎不会被拒。很多 Maintainer 反而更欢迎高质量 issue，因为 PR 代码质量良莠不齐，issue 至少不污染代码库。

#### (2) Contributor 的真实定义

Contributor 不只是改代码的人，是给项目带来价值的人。一个清晰的 bug 报告，就是价值。工程师在 PR 之外主动提 issue，本身就是 Contributor 行为。

### 6.2 扫漏洞：让 AI 找需要上下文判断的真问题

工程师倾向找性能问题，因为 AI 扫静态代码最擅长找这类问题：不必要的 clone、async 处的潜在死锁、热路径上的锁竞争、内存分配的低效模式。

#### (1) 重点扫描方向

##### ① 不必要的 .clone() 或 .to_owned()

特别是热路径上（协议处理、存储读写）的克隆，是性能流失的高发地。

##### ② async 函数里持有 std::sync::Mutex

应该用 `tokio::sync::Mutex` 或者重构掉。在 async 上下文里持有 `std::sync::Mutex` 是潜在死锁源。

##### ③ 锁粒度太粗

把不该锁住的代码包进去了。粗粒度锁在热路径上会显著压低吞吐。

##### ④ 循环里反复分配 Vec/String/HashMap

这种可以预分配的场景在循环里反复分配会制造大量短命对象，给 GC 压力（即便 Rust 没有 GC，频繁分配本身也是开销）。

##### ⑤ Stream / iterator 用法低效

比如 collect 之后又遍历的，可以直接 chain。这种用法低效在编译器层面看不出来。

#### (2) 关键过滤：不要找 clippy 已经查出来的

提示词关键是"不要找 clippy 已经查出来的"。那些是项目自己跑 CI 就能消灭的，提了也没价值。要找需要上下文判断的真问题——这是高质量 issue 和低质量 issue 的分水岭。

### 6.3 AI 候选清单的产出要求

让 AI 给出 5 个候选，每个候选必须按下表给出完整信息，缺一不可：

| 字段 | 内容要求 |
|------|---------|
| 文件路径和行号 | 精确到文件 + 行号区间，方便维护者定位 |
| 问题描述 | 具体哪段代码 + 为什么是问题，要写清问题机制而不是只贴代码 |
| 影响范围 | 只有 corner case 才会触发，还是日常路径上。这决定了 issue 的严重程度 |
| 修复思路 | 不要写代码，讲清楚改动方向就行。把代码留给后续 PR |

### 6.4 高质量 issue 的判断标准

#### (1) 什么样的 issue 值得提

##### ① 在日常路径上

不是 corner case 才能触发，而是规则重算、协议处理、存储读写这种生产环境高频路径上的问题。

##### ② 影响清晰

算法复杂度可量化，例如 O(topics × rules) 在 topic 几万的场景下能算出具体延迟量级。

##### ③ 有具体定位

文件 + 行号 + 函数名齐全，维护者拿到 issue 就能复现。

##### ④ 修复方向明确

讲清思路即可（例如"循环外按 tenant 建分组 map，内层 O(1) 查表"），不用给代码。

#### (2) 什么样的 issue 不值得提

"多余的 collect"、"多余的 clone"那种 1 行就能改的代码风格问题不算高质量 issue——维护者反应通常是"直接 PR 删一行就行，提 issue 干嘛"。算法层面有改进、修复需要讨论方向、需要测试验证的问题，才值得走 issue 流程。

### 6.5 高质量 issue 模板（5 节）

高质量 issue 的正文按下面 5 节组织，让维护者一眼看清问题、复现路径、影响和修复方向：

| 节名 | 写什么 |
|------|-------|
| Description | 问题是什么，3-5 句话。讲清楚复杂度问题（如 O(topics × rules)）的机制 |
| Reproduction | 怎么复现。写清触发条件（如 rules 数量 + topics 数量足够大，触发规则重算路径就能观察到延迟） |
| Impact | 影响什么场景、严重程度。生产环境 topic 数量级是几万的话量化一下延迟 |
| Suggested fix | 可能的修复方向。不要写代码，讲思路（如循环外按 tenant 把 rules 分组成 HashMap，内层改成查表） |
| Environment | 项目版本、Rust 版本、OS |

### 6.6 关键信号：显式说"我可以跟 PR"

#### (1) 这是高质量 issue 的标志

issue 正文最后显式说"我可以在确认方向后跟一个 PR"。这一句话告诉 Maintainer：工程师不是只想刷 contribution 数字，愿意接着把这件事做完。很多时候 Maintainer 看到这一句就会主动来 review 和讨论，因为他们知道这个 issue 不是石沉大海。

#### (2) 为什么算法改进类问题尤其适合走 issue 先讨论

算法改进类问题的修复方向（分组 map vs 索引化 vs 缓存化）有几种合理选择。先在 issue 里对齐方向，再开 PR，效率比直接提 PR 让 Maintainer 选方向高得多——直接提 PR 让 Maintainer 在评审时被迫做架构决策，往往会被打回重做。

### 6.7 操作要点

#### (1) 扫漏洞提 issue 阶段的关键操作

##### ① 在同一 session 跑完最稳

还在仓库目录下的 Claude Code 里继续跑（同一个 session 跑完最稳，AI 对项目代码的上下文还在）。

##### ② 语气是普通工程师报告问题

不是 LinkedIn 帖子，不是问问题，是报告问题。语气直接影响 Maintainer 的第一印象。

##### ③ issue 提交前工程师审核一遍

AI 写完一份 issue，工程师审核一遍再提交到项目仓库。AI 写得再好也不能直接提交，工程师对最终产出负责。

## 8. 实战背景与项目摸底

### 8.1 实战背景

本篇实战在第 30 篇 SignYourName PR 心理胜利的基础上，于同一个 Rust 项目 RobustMQ 上完成两件事——第二个 PR 加一个真实功能、第一个 issue 报告一个真实问题。两件事跑完，工程师正式成为项目 Contributor。

为什么选 Rust 项目？Rust 是 AI 时代仍然有学习门槛的语言，所有权、生命周期、async 这些机制不是看一眼就能上手。但有 AI 当助教，门槛就被压下去——工程师不熟没关系，AI 解释机制，工程师做判断。本篇实战想展示的就是这种工作姿势：不熟的栈，不再是 blocker。

载体可以换。本篇所有动作都在 RobustMQ 上跑，但工程师跟着本篇的方法论，在筛中的任何项目上都能复刻这套流程。

### 8.2 摸项目实战：四维度框架的应用

#### (1) 任务变了，问 AI 的角度也要变

第 24 篇那套四维度框架（它是什么、它能做什么、它怎么扩展、它内部怎么跑）直接套，但本篇的任务和第 24 篇不一样——第 24 篇的任务是做一个新系统，本篇的任务是找一个能上手的 issue。任务变了，问 AI 的角度也要变。工程师用下面这段提示词跑完顶层认知。

#### (2) 提示词全文

```
我在项目仓库目录下。我已经成功提了第一个 PR(SignYourName),
现在要做第二个 PR,目标是给这个项目提交一个有真实工程价值的小贡献。

请你按这四个维度给我一份针对"找 issue"任务的项目地图:

1. 模块划分。这个项目的代码大致分成几块?每块负责什么?
2. 测试组织。单测放在哪、集成测试放在哪、怎么跑?
3. 贡献热点。最近三个月哪几个模块改动最多?活跃 contributor 主要在
   哪些模块?(扫 git log 看)
4. 适合新人的入口。看 issues 里 good first issue / help wanted 标签下,
   有没有可以上手的方向?CONTRIBUTING.md 有什么对新人的硬约束?

不要给我一份完整架构文档,只给我对找 issue 有帮助的那部分。
```

### 8.3 提示词关键设计解读

#### (1) 关键是"针对找 issue 任务的项目地图"

AI 不知道工程师要干嘛的话，会给一份干巴巴的架构介绍。告诉 AI 目的，它会过滤出对工程师有用的部分。这一句话把任务上下文塞给 AI，是整段提示词的核心。

#### (2) 带任务问是第 24 篇建立的姿势

第 24 篇建立了"带任务问"这个姿势，本篇再用一次。同样的四维度框架，因为任务变了，AI 的输出也跟着变——本篇得到的是对"找 issue"有用的项目地图，而不是通用架构文档。

### 8.4 摸底产出

跑完这一段提示词，工程师对项目内部心里有数了：

- 协议层、存储层、metadata、CLI 工具，大概四块
- 最近活跃的是协议层和 CLI
- good first issue 标签下有几个待领的小活
- CONTRIBUTING.md 要求所有新增代码必须带测试

地图有了，接下来就是从这张地图上挑一个工程师能干的 issue。

## 9. 实战：筛 issue

### 9.1 把三条标准丢给 AI

#### (1) 复用同一 session

还在同一个仓库目录下的 Claude Code session 里继续跑，前面那份项目地图 AI 还记得。复用 session 让 AI 保持完整的项目上下文，避免重新建立认知。

#### (2) 提示词全文

```
基于刚才那份项目地图,帮我从这个项目的 issue 列表里筛 5 个候选,
作为我的第二个 PR 目标。

筛选标准:

1. 改动量不大,代码 + 测试加起来 50-200 行
2. 描述清楚,改动是局部的(单文件或单模块内)
3. 不需要深度业务理解,我能 review 得了 AI 写的代码

每个候选告诉我:

- issue 编号和标题
- 它在哪个模块
- 我做这个 PR 大概要干什么(一两句话)
- 为什么它符合上面三条标准
- 风险点在哪(比如有没有可能改完发现规模超出预期)

5 个候选给我排个优先级,从最稳到最有挑战。
```

### 9.2 AI 给出的候选清单

Claude Code 跑完给了一份候选清单。工程师看了一遍，挑了一个改动量合适的 issue：**Issue #1605：把 admin-server 里 shard/list 和 segment/list 两个查询类 API 从 POST 改为 GET**。改 4 个文件，逻辑不变，语义对齐 REST。

### 9.3 为什么挑 Issue #1605

#### (1) 最终拍板的四个判断维度

##### ① 真实工程价值

查询类接口本来就该用 GET，这是 REST 语义的一次回归。改动满足的是 REST 语义对齐这个真实用户需求，不是凭空改造。

##### ② 改动是局部的

只在 admin-server 模块，涉及 4 个文件（router、handler、client 调用方、对应测试）。模块和影响范围清晰，没有跨模块扩散。

##### ③ Rust 经验要求适中

用到 axum 的 Query extractor、axum-extra 的 multi-value 扩展这些 Rust 特性，但不至于硬到看不懂。AI 在写代码时能边写边解释清楚。

##### ④ 工程师能 review

工程师虽然对 Rust 不熟，但相关模式（POST/GET 路由切换、handler 签名变更、测试同步更新）能看懂，能判断对错，能跟维护者讨论。

### 10.1 准备工作：fork + clone + 打开 Claude Code

#### (1) 复用第 30 篇的 fork

### 10.2 长提示词全文

```

`shard/list` 和 `segment/list` 两个查询类端点,从 POST 改为 GET。

具体步骤:

2. plan 我同意之后,建一个新 branch,命名跟项目风格对齐
   (feature/<跟 issue 主题对应的简短描述>)。

3. 按 plan 写代码。Rust 我不熟,所以你写完之后:
   - 关键改动逐段告诉我"这里在做什么、为什么这么写"
   - 任何用到所有权 / 生命周期 / async 的地方,简短解释
   - 不要为了炫技用复杂的 idiom,选最直白的写法
   - 如果有多种合理写法,选项目现有代码里最常用的那种风格

4. 补测试。看项目现有测试在哪、怎么组织,跟着同样的风格写。
   测试要覆盖:正常 watch 行为、Ctrl-C 退出、间隔参数边界值。

5. 跑测试。如果不知道怎么跑,先看 Makefile / CI 配置 / cargo 配置自己摸清楚。跑通为止。

6. 跑项目的 lint:cargo fmt + cargo clippy。所有 warning 都要清掉,不要留 #[allow(...)] 绕过。

7. 写一条 commit message,Conventional Commits 规范。然后 git add + commit。

8. push 到我的 fork。

9. 写一个 PR 描述,包括:
   - 这个 PR 解决的 issue
   - 改动的高层描述(2-3 句)
   - 实现要点(3-5 条 bullet)
   - 测试覆盖了哪些场景
   - 有任何 trade-off 或后续可改进的点,显式说出来

10. 如果装了 gh CLI,直接 gh pr create 提交。没装就把 PR 描述输出给我,我自己去 GitHub 提。
```

### 10.3 提示词关键设计解读

#### (1) 先 plan 后改

AI 不能拿到 issue 就闷头写，得先把 plan 给工程师审核，工程师同意才动。这是第 27 篇提示词驱动写代码那一篇建立的硬规则。

#### (2) 关键改动逐段解释 + 所有权 / async 简短解释

这是 AI 当 Rust 助教的具体形态，它不只是写代码，是边写边教。

#### (3) 不为了炫技用复杂 idiom

这一条是防止 AI 写出 Rust 高手风格的代码，会导致工程师看不懂、review 不了。多种合理写法中选项目现有代码里最常用的那种风格。

#### (4) 所有 warning 清掉不留 allow 绕过

开源项目的硬要求，不能用 `#[allow(...)]` 偷懒。

### 10.4 实战过程复现

#### (1) Step 1：读 issue + 给 plan

Claude Code 先读了 issue + 相关代码，给了一份 4 条 plan：

##### ① router 改造

在 admin-server 的 router 文件里把两个端点的 `.post(...)` 改成 `.get(...)`。

##### ② handler 签名改造

handler 函数签名从 `Json(req)` 改成 `Query(req)`。

##### ③ client 调用方改造

client 端调用方对应改 `.post()` 为 `.get()`，且把请求 body 改成 query string。

##### ④ 跑测试 + 补测试更新

跑现有测试，有失败再补对应的测试更新。

工程师看完同意，AI 开始写代码。

#### (2) Step 2-3：建 branch + 写代码，AI 边写边教

写到 handler 那段，AI 停下来跟工程师解释 axum 的类型驱动 idiom：

- **Json extractor**：把请求 body 反序列化成 struct，只能配 POST/PUT
- **Query extractor**：把 URL query string 反序列化，配 GET
- 这是 axum 的类型驱动 Web 框架 idiom，不是 Rust 语言层面的事，但工程师不知道这一点会卡半天

这一段就是 Rust 助教的价值：工程师自己写不会想到这个细节，AI 写出来还告诉为什么这么写。

写到 ShardListReq 那段，AI 停下来提醒：这个 struct 里有一个 `Vec<String>` 字段，axum 默认的 Query extractor 不支持多值参数（同名 query string 出现多次会被覆盖）。工程师问 AI"那怎么办，要不要把 Vec 改成逗号分隔的 String 然后手动 split"。AI 解释：

- 项目已经引入了 `axum-extra` 这个 crate
- 它的 Query 扩展支持 multi-value
- 直接用 `axum_extra::extract::Query` 替换 `axum::extract::Query` 就行
- 不用动业务代码

工程师学到一个 idiom，而且确认了它符合项目风格。

#### (3) Step 4-5：补测试 + 跑测试

AI 跑现有的集成测试，跑 `cargo nextest run --package admin-server`（项目用 nextest 不是默认 cargo test，这是 Makefile 里发现的项目惯例），有 2 个 case 失败。

AI 停下来给工程师看错误信息，问要不要让它修。工程师看了一眼，是测试还在用 `.post()` 调端点，因为路由改成 GET 了，这两个 case 拿到的是 `405 Method Not Allowed`。AI 把测试里的 `.post(body)` 改成 `.get().query(...)`，重跑，绿了。

#### (4) Step 6：lint

`cargo fmt` + `cargo clippy` 各跑一遍，clippy 报了一个 `needless_borrow` warning，在 client 改造的地方多了一个引用符号。AI 没用 `#[allow]` 绕过，直接改代码消掉。

#### (5) Step 7-10：commit + push + PR

最后 commit + push + 用 gh CLI 提了 PR。整个过程大概 50 分钟，大部分时间在等 `cargo build` 和 `cargo nextest`。

### 10.5 关键的中间 review 点

在步骤 5 到 6 之间，需要停一下，工程师需要 review 代码。这里需要提醒的是，给开源项目提交的代码一定需要 review，这也是对开源负责任的体现。

## 11. 实战：扫漏洞提 issue 与成为 Contributor

<img src="imgs/aicmigr-31-opensource-02-second-pr-rust-tutor/cd6a9676662850a7417d57debf285670_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

### 11.1 让 AI 扫代码找候选漏洞

PR 提了不算结束。本篇要做的另一件事是提一个高质量 issue。工程师倾向找性能问题，因为 AI 扫静态代码最擅长找这类问题：不必要的 clone、async 处的潜在死锁、热路径上的锁竞争、内存分配的低效模式。

还在仓库目录下的 Claude Code 里继续跑（同一个 session 跑完最稳，AI 对项目代码的上下文还在）。

### 11.2 扫漏洞提示词全文

```
帮我扫一遍这个项目的代码,找潜在的性能问题。重点看:

1. 不必要的 .clone() 或 .to_owned(),特别是热路径上(协议处理、存储读写)。

2. async 函数里持有 std::sync::Mutex(应该用 tokio::sync::Mutex 或者重构掉)。

3. 锁的粒度太粗,把不该锁住的代码包进去了。

4. 在循环里反复分配 Vec/String/HashMap 这种可以预分配的场景。

5. Stream / iterator 用法低效,比如 collect 之后又遍历的可以直接 chain。

不要找编译器和 clippy 已经能查出来的(那些项目自己会跑)。

找那种需要看上下文才能发现的真问题。

给我 5 个候选,每个说清楚:

- 文件路径和行号
- 问题描述(具体哪段代码 + 为什么是问题)
- 影响范围(只有 corner case 才会触发,还是日常路径上)
- 修复思路(不要写代码,讲清楚改动方向就行)
```

### 11.3 提示词关键设计

#### (1) 关键是"不要找 clippy 已经查出来的"

提示词关键是"不要找 clippy 已经查出来的"。那些是项目自己跑 CI 就能消灭的，提了也没价值。要找需要上下文判断的真问题——这一句话把候选范围从"代码风格"过滤到"真问题"。

### 11.4 选定候选：topic_rewrite 的 O(n²) 算法问题

#### (1) AI 给的 5 个候选

AI 跑完给了 5 个候选。工程师看完挑了最有算法价值的那一个。

#### (2) 选定问题详情

- **文件**：`mqtt-broker/src/core/topic_rewrite.rs:54-70`
- **函数**：`convert_rewrite_topic`
- **问题**：在外层遍历所有 topic 的循环里，每个 topic 都重新过滤一次全量 rules，生成一个临时 `Vec<&MqttTopicRewriteRule>`
- **算法复杂度**：生产环境 topic 几万的场景下，算法复杂度是 O(topics × rules)，应该降到 O(topics + rules)

#### (3) 为什么这个候选符合高质量 issue 标准

##### ① 在日常路径上

规则重算时遍历全量 topic，不是 corner case 才触发。

##### ② 影响清晰

算法复杂度 O(topics × rules) 可量化，topic 几万、rules 几十的场景下延迟量级可直接算出来。

##### ③ 有具体定位

文件 + 行号（54-70）+ 函数名（`convert_rewrite_topic`）齐全。

##### ④ 修复方向明确

循环外按 tenant 建分组 map，内层 O(1) 查表——讲清思路，不用给代码。

#### (4) 为什么不挑最简单的

AI 推荐的 5 个里有几个是"多余的 collect"、"多余的 clone"那种 1 行就能改的代码风格问题。那种东西维护者反应通常是"直接 PR 删一行就行，提 issue 干嘛"，不算高质量 issue。

算法层面有改进、修复需要讨论方向、需要测试验证的问题，才值得走 issue 流程。topic_rewrite 的 O(n²) 正好满足这三条。

### 11.5 让 AI 写高质量 issue

#### (1) 提示词全文

```
基于上面那个 topic_rewrite O(n²) 过滤的性能问题,帮我写一个
GitHub issue。

要求:

1. 标题简洁,一句话概括问题(<80 字符)

2. 正文按这几节:

   - Description:问题是什么,3-5 句话,讲清楚 O(topics × rules) 这个
     复杂度问题
   - Reproduction:怎么复现(rules 数量 + topics 数量足够大,触发规则
     重算路径就能观察到延迟)
   - Impact:影响什么场景、严重程度,生产环境 topic 数量级是几万的话
     量化一下延迟
   - Suggested fix:可能的修复方向(不要写代码,讲思路:循环外按
     tenant 把 rules 分组成 HashMap,内层改成查表)
   - Environment:项目版本、Rust 版本、OS

3. 语气是普通工程师,不是 LinkedIn 帖子,不是问问题,是报告问题

4. 显式说"我可以在确认方向后跟一个 PR",给维护者一个信号:
   我不只是路过的伸手党
```

#### (2) 关键信号解读

最后那一条很关键。"显式说我可以跟 PR"是高质量 issue 的标志，它告诉 Maintainer 工程师不是只想刷 contribution 数字，愿意接着把这件事做完。很多时候 Maintainer 看到这一句就会主动来 review 和讨论，因为他们知道这个 issue 不是石沉大海。

#### (3) 算法改进类问题尤其适合走 issue 先讨论

算法改进类问题尤其适合走 issue 先讨论，因为修复方向（分组 map vs 索引化 vs 缓存化）有几种合理选择。先在 issue 里对齐方向，再开 PR，效率比直接提 PR 让 Maintainer 选方向高得多。

#### (4) 最终产出

AI 写完一份 issue，工程师审核一遍，提到了项目仓库。

### 11.6 成为 Contributor 的那一刻

#### (1) 贡献列表的真实变化

第二个 PR 合并，第一个 issue 提了。打开工程师的 GitHub 个人页面，那个项目的仓库下，工程师的贡献从一条 SignYourName commit 变成了：1 个有功能价值的 PR + 1 个有定位价值的 issue + 1 条 SignYourName commit。

#### (2) 这不是数字游戏

这是 Contributor 身份的真实开始。简历上工程师可以写：X 项目 contributor，贡献了一个 Y 功能 + 一个性能 issue 报告。这一行简历的分量，远大于"熟练使用 Rust 5 年"。因为它公开可点击验证、不可伪造、AI 写不出来。

#### (3) 流程可复刻

更重要的是，这套流程在一个项目上跑通了，以后任何项目都能复刻，接下来就是选下一个项目继续做。

第一个 PR 是心理胜利，第二个 PR 是真本事。真本事一旦做成第一次，后面就是肌肉记忆。

### 11.7 工作方法小结

#### (1) 让 AI 当 Rust 助教

这是本篇想留下最重要的工作方法。AI 不是替工程师写代码，是边写边教，逐段解释关键改动、提醒所有权和 async 的细节、推荐项目风格的 idiom、把不熟的栈的门槛压下去。工程师不熟没关系，AI 解释机制，工程师做判断。

#### (2) Contributor 不只是改代码的人

一个清晰的 bug 报告，价值不比一个 PR 小。提 issue 是 Contributor 行为，提带"我可以跟 PR"信号的 issue，是更高级的 Contributor 行为。
