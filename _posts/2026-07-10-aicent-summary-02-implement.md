## 1. 核心功能

### 1.1 模型管理

<img src="imgs/aicent-13-fea-model-provider-mgr/cover.jpg" style="display: block; width: 800px;" alt="替换文字">

**完整内容  →   [AI编程方法 13：核心功能 - 模型管理]({% post_url 2026-07-01-aicent-13-fea-model-provider-mgr %})**

**一句话全貌**：一个 AI 业务模块的完整交付 = 咨询想清楚 → 按层拆解 → 前端对接 → 浏览器验收闭环。Provider 是第一个走通全流程的标准模板，后续 Agent / 对话引擎 / MCP 都复用这套流程，区别只在业务复杂度。

#### (1) 四阶段交付流程

| 阶段 | 核心动作 | 完成判据 |
|---|---|---|
| 想清楚 | 咨询式提问，按维度分类问 AI | 带依据的设计决策清单 |
| 拆解 | 按 MVC 分层，每任务只落一层，从底层往上搭 | 每个任务可独立验证 |
| 对接 | 前端 mock 换真实 API | 业务页只关心"调哪个 API、显示哪些字段" |
| 验收 | 浏览器走一遍真实操作 | 数据库到浏览器的完整闭环（curl 跑通不算）|

#### (2) 为什么不能上来就写代码

"没想清楚"会被代码放大成"代码遗漏"——AI 只能按标准模板往下写，遗漏要到联调甚至上线后才暴露，返工成本极高。咨询模式的副产品是提问者借提问把一个陌生领域内化进自己脑子。

#### (3) 咨询式提问：按维度把零散信息结构化

| 提问层次 | 动作 | 产出 |
|---|---|---|
| 广度盘点 | 问业界有哪些玩家 | 一串名单（信息密度低）|
| 维度分类 | 让 AI 按有判别力的维度做分类 | 结构性规律 |
| 翻译依据 | 追问每个维度差异影响哪个设计决策 | 可执行的设计依据 |

关键：显式要求 AI 给"为什么这样分"的理由，答案才会从"名词罗列"变成"因果链"。信息可以查文档补，洞察只能靠提问激发。

#### (4) 两种追问技巧

| 追问 | 触发场景 | 标准话术 |
|---|---|---|
| 轮子评估 | AI 推荐"有现成的库" | "成熟度、覆盖面、概念负担、API 稳定性如何？不用它自己实现工作量多少？性价比成立吗？" |
| 收益追问 | AI 给出"看起来更规范"的设计 | "具体收益是什么（性能/扩展/语义）？不这样做有什么后果？" |

收益追问是过滤过度设计的利器：性能、扩展、语义三个标准都答不上来的"规范"，基本就是过度设计。

#### (5) Provider 三个关键决策

| 决策 | 结论 | 依据 |
|---|---|---|
| 支持哪些供应商 | 一期四种类型：OpenAI / Anthropic / Ollama / OpenAI 兼容 | 按接口兼容性/认证方式/消息格式三维度分类发现"OpenAI 兼容"是巨大阵营，Gemini 放二期 |
| 引不引入大框架 | 不引入 Spring AI / LangChain4j，基于 LlmHttpClient 自己封装 | 大部分供应商兼容 OpenAI 格式，自己封装工作量不大；引入大框架性价比不高 |
| 数据模型 | 三张表：provider / model_config / provider_health | auth_config 用 JSON 存异构鉴权；健康状态独立成表避免锁竞争 |

两个设计亮点：
- **最聪明**：把"OpenAI 兼容"做成一个类型——用户填 baseUrl + Key 即可用，平台零代码改动，**用配置代替代码**。
- **最易被低估**：健康状态独立成 `provider_health` 表——它是高频写字段（每分钟探测更新），放主表会和业务读争锁；拆开后 `provider` 表加 @Cacheable、`provider_health` 表不缓存直接读库，读写路径彻底分离。

#### (6) 为什么能半天交付一个模块

| 支撑 | 作用 |
|---|---|
| 基础设施复用 | BaseEntity / PageHelper / Spring Cache / LlmHttpClient / HifyTable / HifyFormDialog 全已沉淀，每任务只剩业务逻辑 |
| CLAUDE.md 规范 | 接口返回 Result、入参 @Valid、分页用 PageHelper 等约定写进规范，AI 按规范输出，review 成本低 |
| 精力集中 | 脏活交组件、规范交 CLAUDE.md，人只做真正的业务取舍 |

### 1.2 用 Skill 沉淀代码流程

<img src="imgs/aicent-14-fea-consolidate-coding-skills/cover.jpg" style="display: block; width: 800px;" alt="替换文字">

**完整内容  →   [AI编程方法 14：核心功能 - 用 Skill 沉淀编码流程]({% post_url 2026-07-01-aicent-14-fea-consolidate-coding-skills %})**

**一句话全貌**：AI 让执行成本趋零之后，判断的权重反而上升；把每次都要走、每次都差不多的固定流程沉淀成 Skill（Markdown 操作手册），Claude Code 以后引用 Skill 名字就按经验自动走，开发者只在关键决策点拍板。

#### (1) 核心论断：执行趋零，判断升值

AI 写代码极快，但没替开发者做任何取舍——选哪些供应商、鉴权怎么存、健康检查怎么做，都是开发者判断。

| 反向判断 | 代价 |
|---|---|
| 引入 LangChain4j | 多一层重依赖和学习成本 |
| 鉴权用固定列 | 加新供应商就得改表 |
| 健康状态放 Provider 表 | 高频探测与业务读争锁 |

代码快了，错判断会被更快放大。**领域理解的权重不是降低，而是大幅提升**——再快的代码交付，也只是更快地交付了一个更难维护的系统。

#### (2) 领域四问：从外到内建立 70% 认知

| 提问 | 认知目的 | 支撑决策 |
|---|---|---|
| 是什么，解决什么 | 建立认知框架 | 产品定位、商业模式 |
| 用在哪里 | 理解优先级 | 模块优先级、MVP 范围 |
| 由什么组成 | 支撑功能取舍 | 功能裁剪、迭代节奏 |
| 技术架构怎样 | 支撑架构决策 | 技术选型、模块化方式 |

一两小时建立 70% 认知，剩 30% 靠实战补。**不是照抄业界，而是理解 why 后回到自己约束**——Dify 用 Python + Flask + Celery，Hify 选 Java + Spring Boot 模块化单体，是结合团队背景的判断。每一问都显式追问"为什么这样设计"，答案才会从事实陈述变成因果链。

#### (3) Skill 机制：把固定流程变成操作手册

Skill = `.claude/skills/{name}.md`，写在 Markdown 里的操作手册，提到 Skill 名字时让 Claude Code 按流程执行。**本质就是按格式定规范、在 Markdown 里写好规范、让 AI 识别执行**。

| 维度 | CLAUDE.md | Skill |
|---|---|---|
| 作用范围 | 全局规范（什么规矩要遵守） | 具体任务流程（这类任务怎么做） |
| 加载时机 | 每次对话自动加载 | 引用时才生效 |
| 内容粒度 | "Controller 只做参数校验" | "做新模块先梳理需求再拆解再执行" |
| 文件位置 | 项目根目录 | `.claude/skills/{name}.md` |

两者配合：CLAUDE.md 保证代码符合通用规范，Skill 保证特定任务按流程走。

#### (4) Skill 三步法：不用翻文档，问 AI 即可

| 步骤 | 动作 |
|---|---|
| 一 | 让 Claude Code 教你 Skill 是什么、和 CLAUDE.md 区别 |
| 二 | 让 Claude Code 列业界用法（五大类：开发流程 / 质量保障 / 运维部署 / 文档生成 / Git 工作流） |
| 三 | 让 Claude Code 帮你写 Skill |

写 Skill 三要点：产出物明确（"需求分析文档含功能范围+DDL+决策理由"）/ 决策点标注"等待用户确认" / 踩过的坑原样写进去。Skill 是活文档，每做一个模块迭代一次——做到第四五个模块时几乎覆盖所有场景。

#### (5) Skill 思维反向重构代码：if-else → 策略模式

第 13 篇连通性测试的 if-else 是刻意保留的"先跑通再优化"。跑通后用策略模式重构：

```text
ProviderAdapterFactory
  └── Map<ProviderType, ProviderAdapter>
        ├── OpenAiAdapter        (OPENAI / OPENAI_COMPATIBLE / DEEPSEEK)
        ├── AnthropicAdapter     (ANTHROPIC)
        ├── AzureOpenAiAdapter   (AZURE_OPENAI)
        └── OllamaAdapter        (OLLAMA)
```

加新供应商只需两步：写一个 Adapter 类 + 加 `@Component`（Factory 启动时自动扫描注册，**不改任何已有代码**）。这个固定流程本身沉淀成 `provider-adapter.md` 小 Skill。

为什么不一上来就用策略模式？基础设施未稳定时过早抽象，往往让设计模式绑定到尚未成型的业务流程上；先跑通真实场景再重构，抽象才经得起扩展。

#### (6) 大小 Skill 配合 = 团队经验库

| 类型 | 覆盖范围 | 典型例子 | 触发场景 |
|---|---|---|---|
| 大 Skill | 通用模块开发流程 | `module-delivery.md` | 启动任意新模块 |
| 小 Skill | 特定场景操作步骤 | `provider-adapter.md` | 加新供应商 / MCP 工具 / Agent 类型 |

`.claude/skills/` 目录就是团队经验库——个人经验变团队资产，新人引用 Skill 即按团队标准交付，老人离开经验不流失。

#### (7) 方法论升华：脏活累活适合 AI

> 脏活累活，开发者嫌烦、觉得没技术含量的，都是适合 AI 做的。

| AI 做 | 人做 |
|---|---|
| 重复流程、模板化代码、繁琐验证 | 领域理解、关键决策、经验沉淀 |

**不是写更多代码，而是写更少代码、做更多判断、沉淀更多经验**——这才是 AI 编程时代开发者该做的事。

### 1.3 Agent模块及复杂逻辑拆解

<img src="imgs/aicent-15-fea-agent-module/cover.jpg" style="display: block; width: 800px;" alt="替换文字">

**完整内容  →   [AI编程方法 15：核心功能 - Agent模块复杂逻辑拆解]({% post_url 2026-07-02-aicent-15-fea-agent-module %})**

**一句话全貌**：本篇借 Agent 模块演示"现学现卖"——遇到陌生业务概念（审批/支付/数据管道），按"领域四问 → 映射数据结构 → 场景验证 → 拆解执行"四步走，就能从零做到交付。**不是每个领域都要三五年经验才能动手，有方法论就能快速进入。**

#### (1) 领域四问：把陌生概念速理解

**触发**：面对新概念不要直接写代码，先让 Claude Code 扮演领域专家做产品层面梳理。

```text
在 {应用场景} 里，{概念} 是什么？它和 {参照物} 有什么区别？用户创建一个 {概念} 需要配置哪些东西？
```

**产出三层**：核心定义（一句话讲清）→ 差异对比（表格）→ 配置清单（分层组织）。任何一层缺失就追问。

**验证理解的标志**：能把概念映射为存储结构（表/对象模型）才算真懂——映射不出来回继续问。

#### (2) 概念到数据结构的映射法

**三步**：列属性 → 区分实体属性（直接做字段）与关联关系（建关联表）→ 对每个存储决策做对比选型。**AI 擅长对比但不擅长选型，选型能力只能慢慢养成**。

**关键存储决策速判表**：

| 决策点 | 候选方案 | 选型倾向 | 判断依据 |
|-------|---------|---------|---------|
| 固定参数存储 | 字段打散 / JSON / 混合 | 打散 | 参数对所有实体相同、数量少（≤5）、需 SQL 过滤 |
| 异构参数存储 | 字段打散 / JSON | JSON | 字段按供应商类型完全不同（如 auth_config）|
| 多对多关联粒度 | 绑聚合（Server）/ 绑细粒度（Tool）| 绑聚合 | 内部场景，新条目自动生效 |
| 浮点参数类型 | FLOAT / DECIMAL | DECIMAL(3,2) | 避免精度问题 |
| 关联表索引 | 单列 / 联合唯一 | 联合唯一 | 防止重复绑定 |

**反模式警示**：同一技术手段不要到处套——JSON 在 auth_config 是必需，在 Agent 参数（3 个固定字段）就是过度设计。**每个场景单独判断**，"以后可能加参数"也不是用 JSON 的理由。

#### (3) 复杂 CRUD 拆解法

**主线**：不按"增删改查"平铺，以**创建路径为主线**、其他路径对比差异——创建最复杂，拆透它其他都是简化版。

**创建路径标准步骤**：

```text
Controller @Valid 参数校验
  → Service 业务校验（唯一性 + 跨模块存在性，走 Service 接口）
  → @Transactional 事务内写入（主表 + 关联表）
  → @CacheEvict 清缓存
  → 返回详情响应
```

**三个易漏点**：

| 易漏点 | 正确做法 | 原因 |
|-------|---------|------|
| 跨模块校验 | 走 Service 接口，不直接查 Mapper | 模块边界硬规范 |
| 关联表更新 | 数据量小优先全量替换（DELETE + INSERT）| 比增量 diff 简单，性能可忽略 |
| 语义歧义接口 | 必须拆开（基本信息与工具绑定分两个接口）| "toolIds 不传"是清空还是不修改，两种语义都合理 |

**删除策略**：主表逻辑删（deleted=1，历史可追溯）、关联表物理删（无逻辑删意义）、历史会话不处理（agent 删了对话自然报错，接受该行为）。

#### (4) 场景落地：配置不是随便填的

**核心认知**：模型是引擎，Agent 配置是方向盘——同一模型换套配置就是完全不同的应用（智能客服 / 代码审查 / 创意写作 / 数据分析）。

以智能客服为例，每个值背后都有产品思考：

| 配置项 | 值 | 思考 |
|-------|-----|------|
| 模型 | GPT-4o | 客服要准确理解专业问题，3.5 容易理解偏差 |
| temperature | 0.3 | 同一问题答案要稳定，要可靠不要创意（创意写作可设 0.9）|
| max_context_turns | 8 | 客服大多 3-5 轮解决，留余量又不浪费 token |
| System Prompt | 客服指令四要点 | 语气专业友好 / 回答简洁 / 超范围诚实告知（防幻觉）/ 引导联系人工（兜底）|

#### (5) 手动指令 vs Skill 的选型

| 维度 | 手动指令优先 | Skill 优先 |
|------|------------|----------|
| 概念熟悉度 | 第一次接触 | 已熟悉同类模块 |
| 学习目标 | 过程本身就是学习 | 只要交付结果 |
| 流程可复用性 | 一次性 | 模式清晰可沉淀 |
| 决策密度 | 高（每步都要判断）| 低（只在选型点拍板）|

**Skill 启动模板**：

```text
按模块交付 Skill 的流程，帮我做 {模块名}。{一句话定义}。关联 {上游} 和 {下游}（关系：一对多/多对多）。先从第一步开始。
```

本篇手动是因为开发者第一次理解 Agent，过程本身是学习；后续做对话引擎若模式匹配，直接用 Skill。

#### (6) 逐步执行与验证

**Service 提示词要写清每步前置条件和异常情况**——只说"创建 Agent"，Claude Code 大概率漏掉 name 唯一校验或跨模块调 Service（直接查 mapper）。**需求描述的细致程度决定代码的正确程度**。

**两个有意思的发现**：

| 发现 | 启示 |
|------|------|
| 格式混乱的复制文本与规范指令在 Agent 看来是一样的 | 不必纠结指令格式，把信息讲清楚即可 |
| 第一遍代码 6 个接口 curl 全通过、边界用例（2005/3001/3000）都对 | 拆解到位 + 提示词细致，就能一次做对 |

#### (7) 沉淀进 CLAUDE.md 的经验

| 经验 | 价值 |
|------|------|
| 跨模块调用走 Service 接口不走 Mapper | 模块边界硬规范（已有，本篇验证价值）|
| 关联表更新优先全量替换 | 数据量小时比 diff 简单得多 |
| 语义有歧义的接口要拆开 | 基本信息和工具绑定分开更新，消除二义性 |

**真正要带走的能力**：以后遇到任何不懂的业务需求——让 AI 教概念 → 翻译成数据结构 → 用具体场景验证理解 → 拆解执行。

### 1.4 对话引擎（上）：链路和流式技术

<img src="imgs/aicent-16-chat-engine-1/cover.jpg" style="display: block; width: 800px;" alt="替换文字">

**完整内容  →   [AI编程方法 16：核心功能 - 对话引擎（上）链路和流式技术]({% post_url 2026-07-02-aicent-16-chat-engine-1 %})**

**一句话全貌**：对话引擎不是"请求转发器"，是承担会话管理 / 上下文拼装 / 流式推送 / 消息存储四件事的"处理管道"。本篇只做两件事——把链路拆成六步可裁剪流水线、敲定 SSE 流式选型。

#### (1) 对话引擎是什么：管道而非转发器

**为什么不直接让前端调 LLM**：

| 问题 | 风险 |
|---|---|
| API Key 暴露 | 浏览器可被任何人盗取，按调用计费被刷爆 |
| 多供应商格式不一 | 每接一家前端改一套，更换模型要发版 |
| 无上下文管理 | LLM 无状态，前端要自己拼历史、管会话 |

**与普通 HTTP 的本质差异**：请求-响应（一来一回）vs 长连接 + 流式（一来持续回，几十秒到一两分钟）→ 派生三类绕不开的挑战：线程不能一直占、连接不能提前断、前端要边收边渲染。

#### (2) 数据模型：两张表的地基

| 表 | 关键字段 | 决策理由 |
|---|---|---|
| chat_session | status（ACTIVE/ARCHIVED）| 流式中途出错要记状态 |
| chat_session | title | 首条 user 消息前 20 字自动截取，零 token 成本 |
| chat_message | role（user/assistant/system）| 对齐 OpenAI messages 三值，适配层几乎零成本 |
| chat_message | content → LONGTEXT | 代码生成场景避免 TEXT 截断 |
| chat_message | finish_reason（stop/length/error）| 区分正常停止和异常，计费/重试依赖 |
| chat_message | latency_ms | 性能分析、慢响应排查 |
| chat_message | tokens（流结束后写）| 只信 LLM 返回的 usage，不信预估值 |

**关键洞察**：system 消息不入库（Agent prompt 改了立刻对历史会话生效）；chat_message 是全平台增长最快的表，分库归档要提前预判。

#### (3) 六步可裁剪流水线

| 步骤 | 落点 | 要点 |
|---|---|---|
| ① 查配置 | MySQL + Redis | session→Agent→ModelConfig→Provider→历史消息 |
| ② 写用户消息 | MySQL | 调 LLM 前先存，失败也不丢用户输入 |
| ③ 构造请求 | 内存 | system 动态注入 + 历史交替 + 当前消息 |
| ④ 流式调用 LLM | llmExecutor + Resilience4j | **独立线程池 + 熔断器**，稳定性命门 |
| ⑤ SSE 边收边推 | SseEmitter | 本篇技术焦点 |
| ⑥ 流结束存回复 | MySQL + Redis | assistant 消息 + tokens + 更新历史 |

**为什么 ④ 必须独立线程池**：LLM 一次调用几十秒，占 Tomcat 线程（默认 200）几并发就占满，连管理页面一起卡死。线程池隔离 + 熔断 = LLM 调用的两道防线。

#### (4) 流式选型：SSE 的工程默认地位

| 维度 | SSE | WebSocket |
|---|---|---|
| 方向 | 单向（服务端→客户端）| 双向 |
| 协议 | 普通 HTTP | 独立 ws:// |
| 断线重连 | 浏览器原生 | 自行实现 |
| 复杂度 | SseEmitter ~20 行 | 连接池 + 心跳 + 重连，3-5 倍工作量 |

**结论不是偏好选择，是工程默认**：对话场景"客户端发一条、服务端流式回一段"的双向能力用不上；OpenAI / Anthropic / Dify 全选 SSE。数据格式统一为 `data: {json}\n\n`、最后一条带 done 事件。

#### (5) SseEmitter 线程模型与三个坑

**线程切换**：Tomcat 线程创建 SseEmitter → 提交任务给 llmExecutor → 立即返回（释放）；llmExecutor 线程跑 LLM 流式 → 边收 delta 边 `emitter.send()` → `[DONE]` 后存库 + `complete()`。

**三个必须处理的坑**：

| 坑 | 后果 | 解法 |
|---|---|---|
| 默认超时 30s | 长回复被截断成半截 | `new SseEmitter(120_000L)` |
| 客户端断开 | 线程白跑到 LLM 输出完，浪费池资源 | send 的 catch 里停 LLM 调用 |
| Nginx 缓冲 | 等十几秒一次性出现，流式效果失效 | `proxy_buffering off` + `proxy_cache off` + `proxy_read_timeout ≥ SseEmitter` |

**反直觉的事务边界**：`@Transactional` 不要跨 SseEmitter。事务提交时机 ≠ 流结束时机，强行套用要么事务提前提交（流没跑完 DB 写一半），要么长持事务资源到超时。正确做法：主流程不套事务，每段落库用独立 `@Transactional` 方法包裹。

#### (6) 适配层扩展与有状态接口

**ProviderAdapter 加两个方法**：chat（同步，调试用）+ streamChat（流式，靠 `Consumer<String> onChunk` 回调咬合）。关键解耦——适配层只管解析各家格式丢纯文本增量，ChatService 只管接到增量就推前端，两层靠最简单的函数式接口咬合。

**四个 Adapter 的流式格式差异（review 重点）**：

| Adapter | 格式 | 增量字段 | 结束标志 |
|---|---|---|---|
| OpenAiAdapter | SSE | `choices[0].delta.content` | `data: [DONE]` |
| OpenAiCompatibleAdapter | 继承零改动 | 同上 | 同上 |
| AnthropicAdapter | SSE | `delta.text`（事件 content_block_delta）| `event: message_stop` |
| OllamaAdapter | 每行完整 JSON | `message.content` | JSON 内 `"done":true` |

**实现顺序**：先跑通 OpenAiAdapter（最规范），review 通过后其他三家批量复制——风险可控。

**接口选有状态 RESTful**（面向终端用户，前端越简单越好）：

| 路径 | 用途 |
|---|---|
| `POST /api/v1/chat/sessions` | 创建会话 |
| `POST /api/v1/chat/sessions/{id}/messages` | 发消息（请求体只含 content + stream）|
| `GET /api/v1/chat/sessions` | 会话列表 |
| `GET /api/v1/chat/sessions/{id}/messages` | 历史消息 |
| `DELETE /api/v1/chat/sessions/{id}` | 删除会话 |

**done 事件回带 finishReason + latencyMs**：前端据此停 loading 和展示性能。

#### (7) 方法论收获：让 AI 解释自己的代码

**流式 / 多线程 / 回调代码无法靠逐行硬读验证**。高效方法：让写代码的 AI 解释自己的代码，专问三类边界——线程怎么切换、LLM 中途异常如何、客户端断开如何，**听逻辑找自相矛盾之处**（比如"异常时调 completeWithError"但代码里没 catch，就说明有 bug）。

**工程权衡意识**：有状态接口换来前端简单但后端要管会话——不是免费午餐；20-50 人在线规模同步写 MySQL 没压力，不引入 MQ，能省则省。


### 1.5 对话引擎（下）智能客服对话

<img src="imgs/aicent-17-chat-engine-2/cover.jpg" style="display: block; width: 800px;" alt="替换文字">

**完整内容  →   [AI编程方法 17：核心功能 - 对话引擎（下）智能客服对话]({% post_url 2026-07-02-aicent-17-chat-engine-2 %})**

**一句话全貌**：上下文是多轮对话的命门——LLM 本身无记忆，所谓"记得"是每次请求把历史消息重新拼进 messages 数组造成的假象；上下文管理的本质是在有限 token 预算内取舍历史，存储上分三层各司其职（MySQL 真相来源 / Redis 热缓存 / pgvector RAG 知识库），用 Cache-Aside 协作。

#### (1) 上下文本质：破除"LLM 有记忆"的误解

LLM 每次调用无状态，上一轮告诉它的下一轮全忘。"记忆"是对话引擎每次把历史重新拼进 messages 数组造成的工程假象。**上下文 = messages 数组的全部内容**，去掉任何一条语义就断。

三个约束（所有策略的出发点）：

| 约束 | 含义 |
|---|---|
| 窗口有限 | context window 单位是 token（GPT-4o 128K / Claude 3.5 200K / 本地 Llama 3 仅 8K）|
| 超限即错 | 超出窗口直接报错或截断 |
| token 计费 | 历史越长每次调用越贵 |

#### (2) 上下文管理策略：四种对比与选型

| 策略 | 做法 | 优 / 缺 |
|---|---|---|
| 滑动窗口 | 只保留最近 N 轮 | 最简单 / 丢早期历史 |
| Token 预算裁剪 | 按 token 数而非轮数裁剪 | 更精准 / 实现稍复杂 |
| 摘要压缩 | 历史过长先让 LLM 总结再拼装 | 保留信息 / 额外调用成本 |
| 向量召回 | 用相关性而非时序挑选历史 | 适合长对话 / 最复杂 |

**Hify 选滑动窗口**：规模匹配（20-50 人内部）/ 可配置（`maxContextTurns`）/ 实现极简（Redis List `RPUSH`+`LPOP` 十行内）/ 保留升级路径（不过度设计）。

两个 token 实操问题：System Prompt 占 token（须控制长度）/ Token 估算（1 汉字 ≈ 1.5 token，直接用 LLM 返回的 `usage` 记录，不自算）。

#### (3) 对话存储三层分工：真相来源 / 热缓存 / 知识库

| 存储 | 角色 | 存什么 | 谁来读 |
|---|---|---|---|
| MySQL | 持久化真相来源 | 全量对话记录 | 历史查看、数据分析、缓存失效回填 |
| Redis | 工作内存（热缓存）| 最近 N 轮 | 组装 LLM 请求（O(1) 毫秒级，TTL 2h）|
| pgvector | RAG 知识库 | 文档片段向量 | 语义检索（**不存对话历史**）|

**关键认知**：对话历史是结构化数据（MySQL+Redis 够用），pgvector 服务的是"语义检索"另一个维度——两者混在一起，数据增长后架构难以拆分。

#### (4) Cache-Aside：写双写、读旁路、过期回填

| 时机 | 动作 |
|---|---|
| 写消息 | 双写：MySQL 落地 + Redis `RPUSH` |
| 组装上下文 | 只读 Redis |
| Redis 过期 | 从 MySQL 加载最近 N 条回填 |

**铁律**：热缓存只是加速，真相来源才是权威。"只在缓存里、真相来源没有"的数据是隐患——缓存一过期就丢。双写必须在同一条消息的生命周期里完成。

#### (5) sendMessage：项目最复杂的方法

九步链路串联多表查询、Redis 读写、Token 计算、线程切换、流式回调、异常处理。**复杂的不是某行代码，而是多线程、多存储、多回调的交织**——边界条件（超时、断连、调用失败）恰是线上必现的。

指令必须细到每一步：

| 维度 | 必须写清 |
|---|---|
| 线程归属 | 哪些在 Tomcat 线程，哪些在 llmExecutor 线程 |
| 读写时机 | Redis 与 MySQL 的写入顺序 |
| 异常处理 | LLM 超时 `onTimeout` / 客户端断开 catch IOException / send 失败 `completeWithError` |
| 事务边界 | 写消息拆独立方法；返回 `SseEmitter` 的方法**不加** `@Transactional` |

#### (6) 自验证：让 AI 解释自己的代码

让 AI 逐行解释执行流程（哪个线程做什么、读写时机、失败时已写入数据怎么办），人工核对"代码事实 vs AI 自述"。

**判断标准**：AI 说的逻辑，代码里到底有没有。自述里"多出来的逻辑"最易藏 bug——例如 AI 说"LLM 失败会回滚用户消息"，实际代码没有也不需要（用户确实发了这条，AI 没回只是下次重试时它已在历史里）。

#### (7) 三轮验收

| 轮次 | 验证点 | 判断方法 |
|---|---|---|
| 流式回复 | 六步链路 + SSE 落地 | `curl -N` 看到 `data: delta` 逐字输出 |
| 上下文带入 | 历史拼装正确 | 追问不重复背景，看回答是否"记住"了 |
| 历史裁剪 | 滑动窗口生效 | 日志显示"保留最近 N 轮"；`redis-cli LLEN` = `maxContextTurns * 2` |


### 1.6 流式聊天

<img src="imgs/aicent-18-ui-chat-stream/cover.jpg" style="display: block; width: 800px;" alt="替换文字">

**完整内容  →   [AI编程方法 18：核心功能 - 流式聊天]({% post_url 2026-07-02-aicent-18-ui-chat-stream %})**

**一句话全貌**：复杂前端需求向 Claude Code 描述的方法论 = **三层拆解（结构→行为→细节）+ 两套工具（静态四维度 / 动态时间线）+ 两个落地动作（技术约束显式化 / 增量调整 delta）**。方法不绑定技术栈，跨组件库、跨流式方案都可复用。

#### (1) 反例：一次性描述的陷阱

把功能点平铺成一句话、描述层次混在一起，Claude Code 只能按默认理解实现，结果"四处漏风"：

| 硬伤 | 根因 |
|---|---|
| 布局大致对、样式随机 | "气泡长什么样"没定义，模型自由发挥 |
| 流式用定时器假装 | "打字机效果"被理解成整段拿到再逐字显示 |
| 用 EventSource 跑不通 POST | "流式回复"默认套浏览器原生 API（只支持 GET）|
| 调试两小时才发现 | 根因不在代码，在提示词没分层、没技术约束 |

**结论**：复杂页面需要拆解，拆解的核心是**把描述分层**，而不是堆功能点。

#### (2) 第一法则：三层拆解顺序不可逆

| 层级 | 回答 | 典型内容 | 顺序 |
|---|---|---|---|
| 结构层 | 页面长什么样 | 布局骨架、区域划分 | 第一步，必须先对齐 |
| 行为层 | 用户做什么、页面发生什么 | 交互流程、状态变化 | 结构对齐之后 |
| 细节层 | 动画速度、样式、边界 | 打字机间隔、气泡间距、截断 | 跑起来之后打磨 |

乱了顺序返工成本指数上升：结构没对齐就写行为 = 错地基上盖楼；行为写完再调结构 = 前面工作作废；提前打磨细节 = 没定型的骨架做精装修。

> 复杂前端需求第一法则：结构→行为→细节，顺序不可逆。

#### (3) 草稿图对齐 + 验证模式

描述行为前必须先让"页面长什么样"达成共识——文字描述布局信息密度低、容易歧义，**一张草稿图抵几百字**。

| 做法 | 成本 | 适用场景 |
|---|---|---|
| 截参考图（ChatGPT/Claude 对话页）| 极低 | 布局与主流产品一致 |
| 手绘草稿（矩形框+区域名）| 低 | 布局与现有产品有差异 |

**验证模式**：先让 Claude Code 用文字复述它对布局的理解，确认一致后再写代码。此时改一句话成本极低，等写完代码再改整个组件可能要重来——**验证模式把返工风险卡在最便宜的时间点**。

#### (4) 两套描述工具：按交互性质判别

结构对齐后进入行为层，按**交互性质**选描述工具：

| 维度 | 静态四维度 | 动态时间线 |
|---|---|---|
| 适用交互 | 离散（点按钮→弹窗→保存→刷新）| 连续（发送后逐秒状态流转）|
| 描述要素 | 布局 / 数据 / 交互 / 接口 | 触发条件 / 状态变化 / 技术约束 |
| 典型页面 | Provider 管理、Agent 配置、列表页 | AI 对话、实时协作、流式输出 |

判别只问一个问题：**这个页面的交互能不能用"点了什么发生什么"讲完？** 能——四维度；不能——时间线。

四维度对管理页"够用且不冗余"：离散交互状态有限，四格覆盖从看到操作到接口的全链路；缺一格都会在联调或交互环节暴露（漏"二次确认"误删不可逆、漏"接口路径"前后端报 404）。

#### (5) 时间线：天然的技术约束探测器

对话页"发送消息" 8 步时间线里，第 5、6、8 步是技术约束的天然锚点：

| 步骤 | 触发 | UI 变化 | 技术约束 |
|---|---|---|---|
| 5 | 进入流式接收 | 内部建连 | **fetch 手动处理 SSE，不用 EventSource** |
| 6 | 收到 delta chunk | AI 气泡追加 content+滚底 | **新消息滚底，用户上滚不打断** |
| 8 | 失败/error | AI 气泡变红、按钮恢复 | **错误分支显式化** |

三条必踩约束对照（**不说就默认错**）：

| 约束 | 默认做法（错）| 显式说明后（对）| 为什么默认错 |
|---|---|---|---|
| SSE over POST | EventSource（只支持 GET）| fetch 手动解析 SSE 流 | EventSource 不支持 POST body |
| Markdown 渲染 | 纯文本 | marked.js 渲染 HTML+等宽字体 | AI 回复含代码块/列表，不渲染排版混乱 |
| 滚动策略 | 不滚或每次强滚 | 新消息滚底，用户上滚不打断 | 强滚会打断阅读历史 |

**时间线不只是把行为拆步骤，它是"技术约束探测器"**：每写一步就问"Claude Code 默认会怎么做？对不对？不对要显式说什么？"——三个坑就被提前发现，而不是浏览器调试两小时才发现。**显式一句话，胜过调试两小时**。

#### (6) 增量调整 delta 法则

Claude Code 生成代码跑起来，效果通常八九成对，剩下一两成靠 delta 打磨。**关键认知：首次八九成对是常态，剩下的必须靠 delta，而不是推倒重写**。

| 字段 | 含义 | 示例 |
|---|---|---|
| 原来是什么 | 当前状态 | 打字机每字符 50ms |
| 改成什么 | 目标状态 | 改成 30ms |
| 其他不变 | 锁定未改动部分 | 其他保持 |

**反模式：重新描述整个需求**。Claude Code 无法从一整段需求里分辨"这次和上次变了什么"，要么全部重写（引入新问题），要么改了这里破坏那里。**整体需求只在第一次生成时给一次，之后所有改动都走 delta**。

**模型差异的 prompt 锻炼价值**：弱模型首次理解偏差更大，需要多轮 delta 纠正；反复打磨 prompt 的过程本质上是在锻炼"和 AI 说话"的能力——强模型把这件事做得太顺滑，反而掩盖了 prompt 训练价值。

#### (7) 项目 Check List 与延伸思考

| 阶段 | 关键检查点 |
|---|---|
| ① 结构层 | 画草稿图/截参考图，让 Claude Code 先描述理解再写代码 |
| ② 行为层 | 判别离散/连续 → 四维度齐全或时间线覆盖触发→状态→结束 |
| ③ 技术约束 | 流式/POST、渲染、滚动、错误分支是否说明 |
| ④ 细节层 | 跑起来增量调整，每次只改一个维度 |
| ⑤ 验收 | 同接口双验证：curl 验后端 + 浏览器走全流程 |

**三个延伸思考方向**：

| 方向 | 启示 |
|---|---|
| 模式迁移 | "先确认理解再执行"不只对前端布局——数据库 schema、接口契约、组件拆分都可套用 |
| 盲区补齐 | 时间线覆盖正常流程，**异常分支（网络中断、超时、流式中切会话）天然不覆盖**，要单独显式说明 |
| 综合练习 | 给对话页加"重新生成"功能——既要判别用时间线、又要显式新约束（旧回答删不删、上下文算不算）、还要和既有流程衔接 |

**真正要带走的能力**：把模糊意图转成结构化产物时，"先确认理解再执行"+ "结构→行为→细节"+ "显式技术约束"+ "delta 增量调整" 这套组合拳不只适用于前端，可推广到任意复杂需求的描述。

### 1.7 开发过程回顾

<img src="imgs/aicent-19-core-fea-recap/cover.jpg" style="display: block; width: 800px;" alt="替换文字">

**完整内容  →   [AI编程方法 19：核心功能 - 开发过程回顾]({% post_url 2026-07-02-aicent-19-core-fea-recap %})**

**一句话全貌**：与 AI 协作开发业务模块的方法论 = **节奏预判（哪里快哪里慢）+ 想清楚先行（数据模型落字成文）+ 模板复用 / 报错给全量 / 完整验收 + 经验沉淀成 Skill 并真调用**。核心命题：**"想清楚"的时间比"写代码"的时间更值钱**。

#### (1) 快慢分布的反直觉

总时间不重要，**快在哪里、慢在哪里**才重要。"慢"的时间绝大多数是"想清楚"的时间，算前置投入，不算拖延。漏掉一个边界决策，返工时间是前置讨论的 3 倍以上。

| 类型 | 典型内容 | 兜底来源 / 必要性 |
|---|---|---|
| 快项 | 脚手架代码、Skill 一句话启动、前端对接 | 基础设施 / Skill / 基础组件 |
| 慢项 | 数据模型讨论、跨层问题排查（如 Nginx 缓冲）| 决策前置投入，跳过则数倍返工 |

#### (2) 六步方法论骨架

| 步骤 | 动作 | 可观测产物 |
|---|---|---|
| ① 节奏预判 | 判断快慢，分配时间预算 | 快慢预算表 |
| ② 想清楚先行 | 数据模型与边界决策落字成文 | 带 DDL 的设计文档 |
| ③ 模板复用 | 脚手架交给 AI 生成，人只做 review | review 检查项清单 |
| ④ 报错给全量 | 完整栈 + 配置 + 触发步骤喂给 AI | 一轮定位根因 |
| ⑤ 完整验收 | curl 不是终点，走完整链路 | 分层验收清单 |
| ⑥ 经验沉淀复用 | 把流程编码成 Skill 并真的调用 | module-delivery.md |

#### (3) 三个反直觉问题

| 问题 | 答案 | 反面代价 |
|---|---|---|
| "想清楚"能跳过吗？ | 不能 | 跳过直接建表，八九成边界设计会漏，返工 3 倍以上 |
| Skill 真的会被调用吗？ | 习惯养成前需强制提醒 | 写了不用 = 把经验存档却从不读档 |
| curl 验收够吗？ | 不够 | curl 绕过 Nginx，掩盖反向代理缓冲坑 |

#### (4) 三个最值钱的设计决策（Provider 实证）

| 决策 | 价值 |
|---|---|
| `auth_config` 用 JSON 字段 | 不同供应商鉴权差异巨大，未来加新供应商零改表 |
| `provider_health` 独立成表 | 高频写不污染业务表，`provider` 表可放心加 `@Cacheable` |
| `model_config` 分 `name` / `model_id` | 展示名与调用 ID 解耦，Azure deployment name 场景成立 |

#### (5) Skill 闭环：沉淀 → 调用 → 迭代 → 再沉淀

三个动作缺一不可：**只沉淀不调用 = 存档不读档；调用但不迭代 = 第二份模块重复踩第一份的坑**。每做一个模块就回填一条新踩的坑——例如 JSON 字段必须用 `@TableName(autoResultMap=true)` + `JacksonTypeHandler`，否则反序列化静默变 `null`，不报错但数据"消失"。

#### (6) 最低成本的强制动作

在 `CLAUDE.md` 顶部加一行固化提醒：

```text
开始新模块前，先看 .claude/skills/ 目录
```

一行提醒，是让经验真的被读出来的最低成本机制。**Skill 写了不用，等于把经验存档了但从不读档**。

## 2. 高级功能

### 2.1 RAG（上）：向量数据库

<img src="imgs/aicent-20-fea-rag-1/cover.jpg" style="display: block; width: 800px;" alt="替换文字">

**完整内容  →   [AI编程方法 20：高级功能 - RAG（上）向量数据库]({% post_url 2026-07-02-aicent-20-fea-rag-1 %})**

**一句话全貌**：给已跑通的系统加一项完全陌生的技术，方法论比技术本身更值钱——**四步探路法（痛点提问 → 拆组件 → 约束选型 → 最小 Demo）**是一套可迁移的流程；用它走一遍 RAG，发现"看起来很复杂"的 RAG 真正陌生的只有向量数据库一个组件。

#### (1) 元原则：技术在变，探路方法不变

本篇真正交付的不是 RAG，而是接入陌生技术的流程。换成存储引擎、图数据库、搜索引擎，都是同一条路径。

| 可迁移层次 | 含义 |
|---|---|
| 方法可迁移 | 四步探路法原样套用到任何陌生技术 |
| 工具可复用 | 四步都依托 Claude Code，四类提问模板可复用 |
| 决策原则稳定 | 约束驱动、用最简单方案、不为假想未来过度设计 |

#### (2) 四步探路法骨架

| 步骤 | 核心做法 | 产出物 |
|---|---|---|
| ① 痛点提问 | 从业务痛点问"它解决什么"，不问"XXX 是什么" | 一句话心智模型 |
| ② 拆组件 | 问"实现它需要哪些组件"，每个标注陌生度 | 陌生范围缩小清单 |
| ③ 约束选型 | 显式列约束让 AI 对比，由人下判断 | 带理由的选型结论 |
| ④ 最小 Demo | 干净环境跑通陌生组件本身 | 可跑通示例 + 已知坑清单 |

每步都有可观测产出物，不会陷入"读懂了但不知做没做完"的空转。

#### (3) RAG 心智模型：把找答案从模型内部迁到外部

LLM 知识在训练完成那一刻被"冻结"，私有文档根本不在训练集里。

| 路径 | 是否改模型 | 文档更新代价 | 适配客服 |
|---|---|---|---|
| Fine-tuning | 改权重 | 文档更新就得重训 | 极不实际 |
| RAG | 不动模型 | 增量入库即可 | 高度适配 |

RAG 三字精髓：**检索 → 增强 → 生成**。模型没变，变的是它每次看到的输入。

#### (4) 拆解六组件：陌生范围只剩一个

| 阶段 | 组件 | 陌生度 |
|---|---|---|
| 入库管道 | 文档解析器 | 调库 |
| 入库管道 | 文本分块器（chunk 512 / overlap 64）| 纯代码 |
| 入库管道 | Embedding 模型 | 调 API |
| 入库管道 | **向量数据库** | **纯陌生** |
| 检索对话 | 检索器 | 改业务逻辑 |
| 检索对话 | Prompt 组装器 | 改业务逻辑 |

**把整体名词拆开，大部分组件其实已经会**——这是打破恐惧、判断"能不能动手"的第一步。

#### (5) 约束驱动选 pgvector：能不加新依赖就不加

| 选项 | 部署 | Java 友好度 | 一期性能 | 运维负担 |
|---|---|---|---|---|
| **pgvector** | PG 扩展即可 | 高（走 JDBC） | 百万级以下够用 | 零增量 |
| Milvus | 单独集群 + etcd/MinIO | 中 | 最强 | 高 |
| Qdrant | 单二进制 | 中 | 中 | 中 |
| Elasticsearch | JVM 重资源 | 中高 | 8.x 支持 | 极高 |

四条选型理由：① 不引入新组件 ② SQL 可与业务表 JOIN ③ 一期性能够用 ④ 检索层抽象好可平滑替换。**与第 12 篇"不引入 LangChain4j"同构**——优先复用已有基础设施。元原则：**不为假想的未来过度设计**。

#### (6) 最小 Demo 的最大价值是提前踩坑

| 关键点 | 内容 |
|---|---|
| `<=>` 运算符 | pgvector 余弦距离，`1 - 距离` = 相似度 |
| 最大坑 | MyBatis-Plus 不原生支持 `vector` 类型，`float[]` 直接插入报类型转换异常 |
| 解法 | 自定义 TypeHandler 做 `float[]` ↔ 向量字符串互转 |

为什么必须先跑 Demo：这类"类型需特殊处理"的坑文档一笔带过，在 Hify 多层代码里第一次踩中要逐层排查。**干净环境只有"建表、插数据、查询"三件事，坑能立刻定位**。

#### (7) Embedding 选型：同维度零迁移 + 1/5 价格 + 效果更好

| 接口 | 输入 → 输出 | 流式 | 本质 |
|---|---|---|---|
| Chat Completion | messages → 文本 | 支持 | 生成式，有随机性 |
| Embedding | input → 1536 维向量 | 不支持 | 编码式，确定性 |

选 `text-embedding-3-small`：① 同 1536 维表结构零迁移 ② 价格是 `ada-002` 的 1/5 ③ 效果还更好。两个工程关键特性：**`input` 支持数组批量入库**、**同文本同模型向量恒定（可缓存、支持增量入库）**。

常见误区：**Embedding 与 Chat 模型不必配套**——向量只做检索排序，生成只组织语言，二者通过"检索原文拼进 Prompt"衔接，无模型层耦合，可分别按各自最优解挑选。

### 2.2 RAG（下）：客服手册和对话集成

<img src="imgs/aicent-21-fea-rag-2/cover.jpg" style="display: block; width: 800px;" alt="替换文字">

**完整内容  →   [AI编程方法 21：高级功能 - RAG（下）客服手册和对话集成]({% post_url 2026-07-02-aicent-21-fea-rag-2 %})**

**一句话全貌**：在跑通的系统里加新能力的核心心法是"圈范围 → 保旧 → 验新"——找到最小侵入点（只改 `buildMessages` 一个方法）、新能力先独立做完再接入主链路、用精确指令锁死扩散边界。

#### (1) 增量开发三步走：顺序不能乱

| 步骤 | 动作 | 失败时怎么办 |
|---|---|---|
| ① 圈定改动范围 | 找到最小侵入点（改哪个方法、哪一行） | 范围扩散时显式写"不要改什么" |
| ② 先验证旧功能没坏 | 跑原有用例 | 旧功能坏了立即回滚，不在坏状态下继续 |
| ③ 再验证新功能生效 | 跑新用例 | 出错能精确定位是新功能 bug 还是连带影响 |

三步顺序不能反——先测新功能再测旧功能，旧功能出问题时无法判断是这次改动还是原本就有。

#### (2) 最小侵入点：RAG 插在 sendMessage 第 6.5 步

九步链路（加载 session → ... → 加载 Redis 上下文 → 拼 messages → 调 LLM → ...）中，RAG 检索的位置逻辑很清晰：

| 候选位置 | 为什么不行 |
|---|---|
| 第 1-5 步 | 还在加载配置/写用户消息，不知道用户问了什么 |
| **第 6.5 步**（Redis 之后、拼 messages 之前） | **✓ 拿到用户消息，且要在注入 system prompt 之前** |
| 第 7 步之后 | 拼 messages 已完成，注入不进去 |

最终圈定：**只改 `ChatService.buildMessages` 方法**，其余八步不动。改动面小，风险低。

#### (3) 三张表两个数据库：业务数据与向量数据分离

| 数据库 | 表 | 用途 | 访问方式 |
|---|---|---|---|
| MySQL | `knowledge_base` | 知识库容器 | MyBatis-Plus（完整 CRUD） |
| MySQL | `document` | 文档元信息 + 处理状态 | MyBatis-Plus |
| PostgreSQL（pgvector） | `document_chunk` | 分块文本 + 1536 维 embedding | JdbcTemplate（只批量写 + 相似度查） |

`document_chunk` 冗余 `knowledge_base_id`——检索时直接按知识库过滤，不 JOIN document 表。`agent` 表加一列 `knowledge_base_id`，NULL 表示不启用 RAG，**零侵入开关**。

#### (4) 数据管线五环节：每个环节独立方法，管线只串联

| 环节 | 输入 → 输出 | 关键约束 |
|---|---|---|
| ① 状态更新 | document.status = PROCESSING | 驱动前端轮询 |
| ② 解析 | 文件 → 纯文本 | PDF 用 PDFBox，扫描版不支持 |
| ③ 分块 | 文本 → List<ChunkDTO> | chunk_size=512, overlap=64 |
| ④ 向量化 | chunks → 补 embedding | API 返回按 index 排序，不能假设顺序 |
| ⑤ 存储 | chunks → 写 pgvector | batchUpdate 批量写入 |

**接口契约对齐**：先问清楚每环节输入输出，输出格式能直接喂给下一环节。**异步边界**：耗时操作异步执行，接口只接请求、建记录、提交任务。

#### (5) 守边界的 Prompt 设计

```text
{Agent 原始 Prompt}

请基于以下参考资料回答用户问题。
如果资料中没有相关信息，直接说"我没有找到相关资料"，不要编造。

【参考资料】
[1] {chunk1}
[2] {chunk2}
```

| 设计要点 | why |
|---|---|
| 原始 Prompt 保留，参考资料拼接在后 | 拼接非替换，语义层次分明 |
| "没有相关信息就直接说" | 没有这句，LLM 会用训练知识补，且不告诉用户 |
| 没绑知识库的 Agent 走原路径 | 开关在 Agent 维度，零影响 |

RAG 不只让 LLM 能引用文档，也让它**知道自己不知道什么**。

#### (6) 完整验收三维度

| 维度 | 验证方法 | 通过判据 |
|---|---|---|
| ① 链路通 | 绑知识库前后问同一问题 | 回答从"通用猜测"变"引用具体条款"，日志显示命中条数与相似度 |
| ② 边界守住 | 问文档没有的问题 | LLM 诚实告知"没找到"，不编造 |
| ③ 旧功能没坏 | 没绑知识库的 Agent 正常对话 | 行为与之前完全一致，日志无 RAG 输出 |

三个维度都过，功能才算完整交付。


### 2.3 工作流（上）：元数据梳理

<img src="imgs/aicent-22-fea-workflow-1/cover.jpg" style="display: block; width: 800px;" alt="替换文字">

**完整内容  →   [AI编程方法 22：高级功能 - 工作流（上）元数据梳理]({% post_url 2026-07-02-aicent-22-fea-workflow-1 %})**

**一句话全貌**：元数据梳理的核心方法论是"先建立概念，再写代码"——工作流本质就是"节点（做什么）+ 连线（做完去哪）"两张表加一个循环；存储设计有三个关键决策：连线用业务 key 而非自增 id、节点配置用 JSON 字段、类型安全用 `sealed interface + record`。

#### (1) 一句话判断要不要用工作流

| 问题类型 | 答案来源 | 处理方式 |
|---|---|---|
| "退货政策是什么" | LLM 训练知识（脑子里） | 单 Prompt 够用 |
| "我的订单到哪了" | 外部系统（数据库里） | 必须走工作流 |

工作流的本质：把"答案在外部系统"的事，拆成有序步骤去外部取，再交给 LLM 组织语言。

#### (2) 面对陌生子系统：先建立概念，再写代码

人脑会自动用已有经验"硬套"——看到"工作流"就联想到 if-else，方向从一开始就可能跑偏。

| 关键动作 | 反模式 |
|---|---|
| 用具体业务场景提问，不要让 AI 讲理论 | 上来就问"DAG 怎么实现" |
| 先问"是什么、为什么需要"，再问"怎么做" | 顺序反了，方案方向错 |
| 概念没建立前，一行代码都不要写 | 边写边返工，代价远高于多问两轮 |

**提问技巧**：用约束词主动屏蔽学术术语——"不要说 DAG、不要说图论，就告诉我它在代码里是什么数据结构"。把术语挡在门外，AI 的回答会贴近实现。

#### (3) 把概念压成最小数据结构：节点 + 连线

| 概念 | 回答的问题 | 代码表示 |
|---|---|---|
| 节点 | "做什么" | 一条数据库记录：type + config（JSON） |
| 连线 | "做完去哪" | sourceNodeKey → targetNodeKey + condition |

**存库时**：平铺两张表（`workflow_node` / `workflow_edge`）。
**执行时**：加载进内存变成 `Map + while` 循环——不是 DAG，不是图论，就是一个 while 驱动节点依次跑。

#### (4) 数据模型设计的三个关键决策

| 决策 | 选择 | 原因 |
|---|---|---|
| 连线如何引用节点 | `node_key` 字符串而非自增 id | 前端拖拽时不依赖生成顺序，id 没生成也能引用 |
| 节点配置怎么存 | JSON 字段而非拆列 | 不同类型节点配置差异大，拆列每加一种类型就要改表 |
| 类型安全解析 | `sealed interface + record` | switch 漏 case 编译报错，编译期保证穷举 |

配置字段判断标准：**结构稳定 + 需按字段查询 → 拆列；结构多变 + 整体存取 → JSON**。

#### (5) 整体—部分—关系三层：拆表的通用视角

复杂配置拆表前先按三层识别，这和"订单—订单明细—明细关联商品"同构：

| 层 | 表 | 数量 |
|---|---|---|
| 整体（工作流本体） | `workflow` | 主表一张 |
| 部分（节点） | `workflow_node` | 从表多行 |
| 关系（连线） | `workflow_edge` | 连接表多行 |

不要把三层挤进一张宽表，也不要拆成七八张过度细碎的表。

#### (6) CRUD 关键约束：写入即替换，不做 diff

| 约束 | why |
|---|---|
| 三表同事务，任一失败全部回滚 | 防止 workflow 写成功但 nodes 写一半的脏数据 |
| 更新先逻辑删除旧数据再批量插新数据 | diff 逻辑复杂且容易出错，整体替换保证一致性 |
| 删除主记录时关联数据一起逻辑删除 | 防止悬空引用 |
| 提示词逐条写清约束 | 不让 AI 自由发挥，每项约束都是对系统行为的精确控制 |


### 2.3 工作流（下）：执行引擎

<img src="imgs/aicent-23-fea-workflow-2/cover.jpg" style="display: block; width: 800px;" alt="替换文字">

**完整内容  →   [AI编程方法 23：高级功能 - 工作流（下）执行引擎]({% post_url 2026-07-02-aicent-23-fea-workflow-2 %})**

**一句话全貌**：执行引擎本质就是 `while 循环驱动节点依次跑`，骨架几行代码——真正难的是三件事：技术调研排除过度工程化（不引入 MQ / 事件溯源 / 沙箱）、分步实现让错误可定位、异常 review 抓四个边界。

#### (1) 用"食谱与厨师"理解引擎的角色

| 概念 | 角色 | 职责 |
|---|---|---|
| 工作流配置（第 22 篇）| 食谱 | 静态存库，"只管存不管跑" |
| 执行引擎（本篇）| 厨师 | 加载食谱 → 构 nodeMap + edgeMap → while 逐节点执行 |

引擎对外只暴露同步方法 `execute(workflowId, userMessage) → String`——和调 LLM 接口形态一致，`ChatServiceImpl` 一行不改。

#### (2) 技术调研的产物是"排除"，不是"照搬"

| 方案 | 为谁设计 | Hify 是否需要 |
|---|---|---|
| n8n Redis Queue | 多 Worker 水平扩展 | 不需要 |
| Temporal 事件溯源 | 小时/天级长周期工作流 | 不需要 |
| Dify 代码沙箱 | 用户自定义代码节点 | 不需要 |
| Dify VariablePool | 变量在节点间传递 | **借鉴** |

结论：**轻量同步引擎 + VariablePool 模式**，不引入 MQ。调研价值在于知道每个方案为谁设计，才知道哪些复杂度对自己是负担。

#### (3) ExecutionContext 变量池：节点解耦的关键

| 性质 | 实现 |
|---|---|
| key 格式 | `nodeKey.varName`（如 `classify.intent`） |
| 读写规则 | 写只能写自己 nodeKey 下，读可读任意历史节点 |
| 只增不改 | 写入不覆盖历史值，可回放可调试 |

`{{classify.intent}}` 在运行时被 `resolve()` 替换为变量池实际值，这就是节点间传递上下文的全部机制。

#### (4) 异常 review：四个必查边界

正常路径 Claude Code 基本能写对，但这四类边界大概率遗漏：

| 异常场景 | 期望保护 |
|---|---|
| 条件分支所有条件都不匹配 | 走 defaultTarget 或合理报错 |
| `targetNodeKey` 指向不存在的节点 | `nodeMap.get` 返回 null 时抛异常，不空指针 |
| 工作流里有环（A→B→A） | 步数限制（如 50 步）生效 |
| LLM 节点调用失败 | `workflow_node_run` 记 FAILED，`workflow_run` 也 FAILED |

#### (5) 三条方法论的可迁移性

| 方法论 | 反面代价 |
|---|---|
| ① 技术调研先行 | 凭感觉拍方案 / 照搬大厂引入不必要复杂度 |
| ② 分步实现 | 一次写完跑不通，不知哪步错 |
| ③ 异常分支 review | 只测 happy path，上线被边界打穿 |

#### (6) 增量开发铁律：先跑旧用例

接入 `ChatServiceImpl` 是对话引擎**第四次增量开发**：基础链路 → 上下文 → RAG → 工作流。每次都遵循同一节奏——**不改骨架，只在既定位置插入新分支**。先跑不绑工作流的 Agent 确认旧功能没坏，再跑绑工作流的 Agent。


### 2.4 MCP接入（上）：协议和客户端

<img src="imgs/aicent-24-fea-mcp-1/cover.jpg" style="display: block; width: 800px;" alt="替换文字">

**完整内容  →   [AI编程方法 24：高级功能 - MCP接入（上）协议和客户端]({% post_url 2026-07-02-aicent-24-fea-mcp-1 %})**

**一句话全貌**：MCP 把 M×N 套适配代码压成 M+N——本质是"工具自描述（Tool Schema）+ LLM 自决策（Function Calling）"，最关键的两个细节是 description 写法（决定 LLM 选工具准确率）和避开全家桶 SDK（防止两套抽象互相摩擦）。

#### (1) 用"USB"理解 MCP 解决的问题

| 维度 | 直接调 API（M×N） | MCP 标准化（M+N） |
|---|---|---|
| 平台 × 工具连线数 | M×N（每个平台对每个工具写适配） | M+N（平台和工具各自只接协议层） |
| 工具升级影响面 | 所有调用方都要改 | 只改 Server，调用方零改动 |
| 新增工具成本 | 写新适配 + 改 Prompt | 配一个 Server 地址 |

USB 出现之前每个外设一根专用线，出现之后所有外设走同一个口。**MCP 之于工具调用，就是 USB 之于外设**。

#### (2) 三大概念：Server / Client / Tool Schema

| 概念 | 角色 | 职责 |
|---|---|---|
| MCP Server | 工具提供方 | 按 MCP 标准声明能做什么、需要什么参数 |
| MCP Client | 工具调用方（Hify 即 Client） | `tools/list` 发现 + `tools/call` 调用 |
| Tool Schema | 工具标准描述 | name + description + inputSchema |

**Tool Schema 的本质是"工具自描述"**——所有差异化都被 Server 自己消化，调用方走统一接口。

#### (3) Function Calling：LLM 怎么知道要调工具

LLM 只能输入/输出文本，工具调用通过 Function Calling 约定：LLM 输出结构化的"我要调这个函数"指令，真正执行由 Agent 框架（Hify）完成。

| 步骤 | 谁在做 | 干什么 |
|---|---|---|
| 第一次 LLM 调用 | LLM | 判断要不要调、调哪个；返回 `finish_reason=tool_calls` |
| 工具执行 | Hify（MCP Client） | 通过 `tools/call` 调 Server 拿真实数据 |
| 第二次 LLM 调用 | LLM | 基于工具结果生成最终回答，`finish_reason=stop` |

工具定义随每次请求带过去（不是持久记忆）；`description` 决定 LLM 选工具准确率——**这是 MCP 工具开发最重要的地方，没有之一**。

#### (4) description 写"什么情况下调"，不是功能说明

| 写法 | 效果 |
|---|---|
| ❌ "查询订单状态" | LLM 不知道何时该选 |
| ✅ "用户询问订单/物流/快递相关问题时使用。不知道订单号时传空字符串" | 说清触发场景 + 参数语义 |

description 是给 LLM 看的决策依据，不是给人看的注释——要写清触发条件、调用时机、与其他工具的协作关系。

#### (5) Client 选型：避开"全家桶"，优先选只做协议层的轻量 SDK

| SDK | 关键问题 | 结论 |
|---|---|---|
| Spring AI MCP | 强绑定 Spring AI 全家桶（ChatClient/ChatModel/ToolCallback） | 不采用（与 Hify 自研体系两套抽象互相摩擦） |
| Quarkus MCP | 绑定 Quarkus 生态 | 不采用（技术栈不匹配） |
| **官方 Java SDK 1.1.1** | 只做协议层（序列化/反序列化） | **采用** |

**已知坑**：`HttpClientSseClientTransport` 资源泄漏（每次 build 新 HttpClient 不关闭）。规避：**按调用创建 + try-with-resources + 用完关闭**，不长期持有 client 对象。

#### (6) 五条工程约束（增量开发红线）

这是对话引擎**第五次增量开发**：基础链路 → 上下文 → RAG → 工作流 → MCP 工具调用。

| 约束 | why |
|---|---|
| ① 工具列表为空时与原逻辑一行不改 | 回归基准，证明改动没破坏既有功能 |
| ② RAG 与工具调用共存 | 静态知识 + 动态数据天然互补 |
| ③ 工作流分支不进入工具逻辑 | 编排与 LLM 自主决策混在一起行为不可预测 |
| ④ 工具失败回传 LLM 不抛异常 | "软失败 + 自然语言兜底"体验优于硬失败 |
| ⑤ Controller 与 SseEmitter 不动 | 改动面越小、回归风险越低 |


### 2.5 MCP接入（下）：实现服务

<img src="imgs/aicent-25-fea-mcp-2/cover.jpg" style="display: block; width: 800px;" alt="替换文字">

**完整内容  →   [AI编程方法 25：高级功能 - MCP接入（下）实现服务]({% post_url 2026-07-02-aicent-25-fea-mcp-2 %})**

**一句话全貌**：自建 MCP Server 高度模板化（声明 schema + 实现业务 + 注册协议），最核心的方法论是"从用户话术反推工具能力"——错误顺序（先想怎么实现）会导致工具粒度失当；最关键的设计细节是返回值"一鱼两吃"和 description 写"什么情况下调"。

#### (1) 何时自建：一句话决策

**看工具提供方是否已经为你做好了可直接使用的 Server**。

| 维度 | 用现成 | 必须自建 |
|---|---|---|
| 是否有官方 Server | 有（如 Stripe） | 无 |
| 系统是否对外 | 是（公网服务） | 否（内部财务/ERP/工单） |
| 业务规则是否通用 | 通用、标准 | 含公司专属审批/规则 |

退款场景两条都占：财务系统是内部的，退款又要走公司审批流。

#### (2) MCP Server 本质：一个独立的 HTTP Server

| 性质 | 含义 |
|---|---|
| 独立部署 | 与调用方（Hify）完全分开，只通过 HTTP 通信 |
| 黑盒 | 调用方只存 endpoint 地址，不关心 Java/Python、MySQL/Oracle |
| 开发者只关心两件事 | ① 声明工具 schema ② 实现业务逻辑 |
| 协议端点自动暴露 | `GET /sse` + `POST /messages` 由 SDK 处理 |

#### (3) 三步推导法：从用户话术反推工具（**全篇最重要的方法论**）

正确顺序永远是 **场景 → 能力 → 实现**，不能反过来。

| 步骤 | 退款场景的实践 |
|---|---|
| ① 问"用户会说什么" | "我要退款" / "审批了没" / "不退了" / "为什么被拒" |
| ② 问"每类话需要什么能力" | 拆/合判断（敏感操作拆两步；纯查询可合并） |
| ③ 问"怎么实现" | 才有资格谈实现 |

三个常被忽略的判断题：

| 判断题 | 正确做法 |
|---|---|
| 拆还是合 | 涉及"用户确认"的敏感操作要拆（查资格 + 提交申请分两步）；纯查询可合并（拒绝原因并入查状态） |
| 是否都做成 Server | 没标准答案，结合调用频率 / 权限边界 / 复用性权衡 |
| 参数接受多形式 | 同时接受 orderId 或 refundId——Server 做转换比让用户多说一句话合理 |

#### (4) 工具设计原则：消费者是 LLM，不是人

| 原则 | 反面 | 正面 |
|---|---|---|
| 返回值"一鱼两吃" | 只返回 `status=PROCESSING` 让 LLM 自己翻译 | 同时返回 `status`（程序判断）和 `statusLabel="处理中"`（LLM 直读） |
| description 写"什么情况下调" | "查询退款资格" | "用户说'我要退款'时先调此工具确认是否符合条件。**不要跳过此步直接提交**" |
| 错误处理友好 | 抛技术异常中断对话 | 返回 `{"error": "..."}` + `isError=true`，错误原因是 LLM 能转述的自然语言 |

#### (5) 标准开发六步模板（每个 MCP Server 都同一套路）

| 步骤 | 任务 | 验证方式 |
|---|---|---|
| ① 建项目 | 创建独立工程 + MCP SDK 依赖 | 编译通过、能启动 |
| ② 注册 Server Bean | 配置 MCP Server，暴露 SSE 端点 | `curl /sse` 返回 SSE 流 |
| ③ 数据层 | 建业务表 + 数据访问层 | 表自动创建、单测读写正常 |
| ④ 业务逻辑 | 实现工具对应的 Service 方法 | 单元测试覆盖每个方法 |
| ⑤ 注册工具 | 把业务方法注册为 MCP 工具 | `tools/list` 返回所有工具 |
| ⑥ 接入调用方 | 在 Agent 平台注册并绑定 | 端到端对话、日志出现 tool_calls |

每步必须能独立验证——出了问题立刻知道在哪层找。

#### (6) 通用接入工作流：四段式

| 阶段 | 动作 |
|---|---|
| ① 工具自身跑通 | Server 单独运行，curl 验证 schema 与业务逻辑 |
| ② 调试工具验证 | 在 Agent 平台 MCP 管理页用调试工具，不必手拼 curl |
| ③ 接入 Agent | 注册 Server + 测试连通 + 绑定工具到 Agent |
| ④ 端到端测试 | 真实对话触发，验证 LLM 调用时机与返回值表述 |

模板化任务正是 AI 编程助手提效最大的场景——做明白第一个退款 Server，库存 Server、工单 Server 只需换 schema 和业务逻辑。


### 2.6 流程回顾

<img src="imgs/aicent-26-adv-fea-recap/cover.jpg" style="display: block; width: 800px;" alt="替换文字">

**完整内容  →   [AI编程方法 26：高级功能 - 流程回顾]({% post_url 2026-07-02-aicent-26-adv-fea-recap %})**

**一句话全貌**：工作流模块的真正难点不在代码而在概念——「概念先行 / 技术调研 / 分步实现 / 异常 review / 旧用例回归」五步组成一套可迁移到任何陌生子系统的标准流程，核心心法是「正常路径交给 AI，人的精力集中在边界情况和回归保护」。

#### (1) 用大白话讲清"工作流是什么"

判断标准很简单：问"退货政策是什么"单 Prompt 够用（LLM 自己知道）；问"我的订单到哪了"必须走工作流（答案在数据库里，不在 LLM 脑子里）。工作流本质上只有两个元素：

| 元素 | 回答的问题 | 代码表示 |
|---|---|---|
| 节点 | "做什么" | type（LLM/CONDITION/...）+ config（JSON） |
| 连线 | "做完去哪" | sourceKey → targetKey + condition |

**不要从 DAG / 图论入手**——存库时是平铺两张表，执行时加载进内存变成 `Map + while` 循环，就是一个 while 驱动节点依次跑，没什么神秘的。

#### (2) 五条可迁移的核心动作

针对"陌生子系统"的 AI 编程，五条动作按顺序排，每条都有反面代价：

| 动作 | 怎么做 | 反面代价 |
|---|---|---|
| ① 概念先行 | 场景 → 代码表示 → 存储设计，顺序不能乱 | 跳过概念直接写，方向错返工 3 倍 |
| ② 技术调研 | 调研业界方案 + 多维度对比 + 给出选择理由 | 照搬大厂或凭直觉选，引入不必要复杂度 |
| ③ 分步实现 | 复杂系统拆块，每块写完就用单测验证 | 一次写完跑不通，不知哪步错 |
| ④ 异常 review | 正常路径交 AI，人集中查边界情况 | 只测 happy path，上线被边界打穿 |
| ⑤ 旧用例回归 | 接新功能第一件事是确认旧功能没坏 | 只测新功能，旧链路被改坏 |

#### (3) 技术调研的本质是"排除"，不是"照搬"

调研了 Dify / n8n / Coze / Temporal 四个平台，结论是 Hify 都不需要照搬：

| 平台 | 为谁设计 | Hify 是否需要 |
|---|---|---|
| n8n Redis Queue | 多 Worker 水平扩展 | 不需要 |
| Temporal 事件溯源 | 小时/天级长周期工作流 | 不需要 |
| Dify 代码沙箱 | 用户自定义代码节点 | 不需要 |
| Dify VariablePool | 变量在节点间传递 | **借鉴** |

调研的真正产物不是"选哪个"，而是"明确每个方案的适用场景"——只有知道每个方案为谁设计，才知道哪些复杂度对自己是负担。**最终方案是轻量同步引擎 + 变量池，不引入消息队列、不做事件溯源**。

#### (4) 数据模型的三个关键决策

| 决策 | 选择 | 原因 |
|---|---|---|
| 连线如何引用节点 | `node_key` 字符串而非自增 id | 前端拖拽不依赖生成顺序，id 没生成也能引用 |
| 节点配置怎么存 | JSON 字段而非拆列 | 不同类型节点配置差异大，拆列每加一种类型就要改表 |
| 类型安全解析 | `sealed interface + record` | switch 漏写 case 编译报错，编译期保证完整性 |

#### (5) 异常 review：四类必查边界

正常路径 Claude Code 基本能写对，但这四类边界大概率遗漏，必须逐个手动追问：

| 异常场景 | 期望保护 |
|---|---|
| 条件分支所有条件都不匹配 | 有 defaultTarget 或合理报错 |
| `targetNodeKey` 指向不存在的节点 | `nodeMap.get` 返回 null 时抛异常，不空指针 |
| 工作流里有环（A→B→A） | 步数限制（如 50 步）生效，不死循环 |
| LLM 节点调用失败 | `workflow_node_run` 记 FAILED，`workflow_run` 也标记 FAILED |

review 心法：把"如果这一步的输入不是预期的会怎样"逐个追问 AI，**不满足于正常路径跑通**。

#### (6) 工作流相比单 Prompt 的最大收益是"可观测性"

`workflow_node_run` 表让每个节点的输入输出、耗时、状态都清清楚楚：classify ~1200ms / router ~2ms / aftersale ~3400ms。出了问题能精确定位到哪个节点慢或错——单 Prompt Agent 做不到。**工作流的本质收益不是"能跑通"，而是"能看清"**。
