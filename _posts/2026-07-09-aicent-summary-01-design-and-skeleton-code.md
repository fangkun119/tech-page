## 1. 序言、认知构建

### 1.1 序言 - 用 AI 写简 Dify

<img src="imgs/aicent-00-opening/cover.jpg" style="display: block; width: 800px;" alt="替换文字">

**完整内容  →  [AI编程方法 00：序言 - 用 AI 写简 Dify]({% post_url 2026-06-30-aicent-00-opening %})**

#### (1) 核心心法：AI 是放大器，不是发动机

**一句话**：你有方向它放大十倍，你没方向它放大的是零。瓶颈不在 AI 能力，在你的思考质量。

**角色转变**：从"写代码的执行者"变成"做决策的架构师"。Claude Code 是你的工程团队，你是那个下判断的人。

| 必须由人决定 | 可交给 AI 执行 |
|------------|---------------|
| 产品边界（做什么/不做什么） | 实现接口 |
| 架构约束（单体/微服务） | 编写样板代码 |
| 命名与错误处理规范 | 执行单元测试 |
| 拆解粒度 | 给出重构建议（需复核） |

#### (2) 两种协作模式

| 模式 | 适用场景 | 典型句式 |
|------|---------|---------|
| **执行模式** | 已有规范、验收标准明确 | "按 CLAUDE.md 实现 X，异常按 Y 分类" |
| **咨询模式** | 新领域、不确定要考虑什么 | "列出我需要考虑的所有问题，不要给方案" |

**切换判据**：能清晰描述验收标准 → 执行模式；否则先咨询模式梳理维度，再进入执行模式。

#### (3) 三个核心方法

| 方法 | 适用阶段 | 关键动作 |
|------|---------|---------|
| **三问裁剪法** | 项目启动/大功能开始前 | 做什么 / **不做什么** / 做到什么程度 |
| **SDD（CLAUDE.md）** | 规范阶段 | 写定位、技术栈、架构红线、命名、接口、错误处理、数据模型 |
| **三步审查** | 每次 AI 产出后 | 查意图 → 查质量 → 查边界（缺一不可） |

**关键认知**：AI 默认按"通用最佳实践"补全你没说的部分——往往就是过度设计。**"不做什么"和"做什么"同等重要**，等于给 AI 划红线。

#### (4) 全链路开发四步

**拆解（Decompose）→ 规范（Spec）→ 执行（Execute）→ 验收（Verify）**

- 验收通过 → 并入主线
- 验收失败 → 回到"规范"重做

每阶段的配套方法：拆解用三问裁剪法、规范用 CLAUDE.md、执行用两种模式、验收用三步审查。

#### (5) 踩坑复盘：模糊 vs 清晰

第一天踩坑（模糊大需求 → 过度设计）→ 晚上做三件事重塑协作根基 → 第二天质量飞跃。

| 维度 | 模糊大需求 | 清晰小任务 |
|------|----------|----------|
| 任务粒度 | 整个后端架构 | 一个 CRUD 接口 |
| 规范引用 | 无 | 显式引用 CLAUDE.md |
| 验收标准 | 隐式（"看起来专业"） | 显式（异常分三类） |
| 结果 | 引入服务注册、分布式配置、链路追踪 | 一次做对 |

**晚上三件事**（协作根基）：① 三问裁剪法定产品边界（一页纸）② 设计核心数据模型（5~6 张表，覆盖但不展开）③ 写 CLAUDE.md。

**清晰任务四要素**：规范引用 + 任务边界 + 附加功能 + 异常处理。

**核心认知**：不是 AI 变聪明了，是输入变清晰了。

#### (6) 拆解粒度判据

**核心判据**：可独立验收的最小单元——能用一句话描述验收标准、能引用一条 CLAUDE.md 规范。

| 反面 | 后果 |
|------|------|
| 粒度过大（覆盖多模块） | AI 自由发挥、过度设计、无法定位问题 |
| 粒度过小（只改一行） | 上下文割裂、需手动拼接碎片 |

拿不准时宁可拆细，但每个任务至少能引用一条 CLAUDE.md 规范。

#### (7) 跨模块问题：上下文三件套

AI 在跨模块问题上容易跑偏——它默认只看"当前模块"。改了 A，B 的依赖没同步，于是改一轮错一轮。

| 要素 | 作用 |
|------|------|
| 相关模块的规范（A、B 的 CLAUDE.md 章节） | 告诉 AI 两个模块各自的边界 |
| 问题现象 + 复现步骤 | 让 AI 看到真实问题，而不是猜 |
| 已排除的可能 | 避免重复走死路 |

**配套做法**：用咨询模式让 AI 先列假设，你逐个验证——不要让它直接给方案（往往只治标）。根因通常不在"当前模块"，而在"模块之间的契约"。

#### (8) 工具观：源码泄漏事件的启示

51 万行 TypeScript 被暴露，Claude Code 的竞争力一点没少。

| 维度 | 是否会变 | 是否可复制 |
|------|---------|----------|
| 客户端工程实现 | 会变（日更迭代） | 可复制 |
| 模型能力 | 持续投入 | 不可复制 |
| **使用工具的人怎么思考** | **不变** | **不可复制** |

**结论**：工具迭代越快，掌握底层方法的人越值钱。方法论不会过时，工具使用手册会。

### 1.2 认知构建 - 你是架构师AI是你的工程团队

<img src="imgs/aicent-01-engr-and-claude/cover.jpg" style="display: block; width: 800px;" alt="替换文字">

**完整内容  →  [AI编程方法 01：认知构建 - 你是架构师AI是你的工程团队]({% post_url 2026-06-30-aicent-01-engr-and-claude %})**

#### (1) 核心认知：你是架构师，AI 是工程团队

**一句话**：你不写代码，但决定系统长什么样——**定方向、做取舍、立标准、验收成果**。

**输入质量阶梯曲线（关键发现）**：输入和输出**不是线性关系，是阶梯式**的。输入从 60 → 80 分，输出可能从 40 跳到 85。模糊时 AI 需要"猜"，每个猜测都可能错且错误叠加；清晰时它不用猜，输出质变。

| 输入档位 | 描述特征 | 输出特征 |
|---------|---------|---------|
| L1 模糊 | "帮我做个 CRUD" | 过度设计或不够用 |
| L2 半结构化 | "用 Spring Boot 做 CRUD" | 能用但风格混乱 |
| L3 结构化 | "按规范做，错误码、测试列清" | 结构完整可用 |
| L4 全规范 | CLAUDE.md + 接口契约 | 可直接合入 |

**角色转变（精力复利）**：传统 70% 时间在修 bug / review / 测试 / 文档；AI 辅助后 60% 时间在架构决策、30% 在监督验收——终于能把精力放在最该做的事上。

**两个前提认知**：① 必须有技术判断力验证 AI 输出 ② 必须能给出精确任务描述。"AI 帮我做事" ≠ "我可以不懂技术"——技术能力是当好架构师的前提。

#### (2) 实操工具一：三层分工

按"决策权归属"分三层，**越往上越需要判断力**。

| 层级 | 谁做 | 典型任务 | 验收方式 |
|------|------|---------|---------|
| 第一层 | 必须你做 | 产品边界、架构决策、技术取舍、跨模块一致性 | 自己拍板 |
| 第二层 | AI 做，你验收 | 业务代码、接口、页面、测试、文档 | **逐行能说清楚**（硬标准） |
| 第三层 | AI 全权处理 | 格式化、样板代码、简单重构、脚本 | 扫一眼即可 |

**判断标准**：这个决策错了，会不会让整个项目偏离轨道？会 → 第一层。

**边界是活的**——同样的"写代码"，在不同项目里归属不同：

| 项目类型 | 第一层 | 第二层 | 第三层 |
|---------|-------|-------|-------|
| 应用层（如 Hify） | 较小 | **最大** | 中等 |
| 基础设施（分布式、存储） | **最大** | 较小 | 较小 |
| 工具 / 脚本 | 极小 | 较小 | **最大** |

**协作变体**：任务还没成型时，可以让 AI 帮你想思路——**它帮你想，你来判断取舍**。

#### (3) 实操工具二：三步检查法

**顺序不能反**：意图 → 质量 → 边界（意图错了，后面两步白查）。

| 步骤 | 核心问题 | 典型问题 | 处理方式 |
|------|---------|---------|---------|
| 第一步 查意图 | 是不是你要的 | 多了软删除、操作日志等"额外功能" | **保留需要理由，砍掉不需要理由** |
| 第二步 查质量 | 风格规范一致性 | 命名、返回格式不统一 | 同时修代码 + 补 CLAUDE.md |
| 第三步 查边界 | 异常与风险 | 并发、ID 不存在抛 500 | 让 AI 自己 review 自己（稳定不疲劳） |

**警惕放松警惕陷阱**：连续几次输出没问题会不自觉放松——这不是意志力能对抗的。**靠机制代替意志力**：关键模块强制跑测试、CI 自动化检查、Code Review Checklist。不要相信"我会更仔细"，要相信"机制会替我把关"。

#### (4) 实战印证：指令对比

同一个 AI、两种指令，输出天差地别。

| 维度 | 模糊指令（L1-L2） | 结构化指令（L3-L4） |
|------|----------------|------------------|
| 错误码 | 无 / 默认 | 2001 / 2002 / 2003 分场景 |
| 依赖管理 | 缺失 | 完整 |
| 连通性测试 | 无 | 三类异常分别处理 |
| 设计模式 | 可能过度（不必要的工厂模式） | 显式说明"不需要工厂模式" |
| 与 CLAUDE.md 对齐 | 不对齐 | 完全对齐 |
| 成熟度 | 能跑但不像生产代码 | 成熟项目该有的样子 |

**结论**：不是 AI 变聪明了，是指令变具体了。阶梯曲线成立——输入从 L1 跳到 L3，输出从 40 跳到 85。

### 1.3 认知构建 - SDD思想（不是SDD脚手架）

<img src="imgs/aicent-02-sdd-mind-set/cover.jpg" style="display: block; width: 800px;" alt="替换文字">

**完整内容  →  [AI编程方法 02：认知构建 - SDD思想]({% post_url 2026-06-30-aicent-02-sdd-mind-set %})**

#### (1) SDD 是什么：规范驱动开发

**一句话**：先把规范写下来、自动加载，再让 AI 在规范约束下执行。核心立场——**你不告诉它规矩，它就自己编一套**。

**为什么需要（关键认知）**：Claude Code **没有长期记忆**，每次新对话从零开始。它不是"忘了"，是**根本没看到**。单次指令只能解决单次输出，几百次输出的一致性必须靠规范体系。

| 维度 | 单次好指令 | SDD 规范体系 |
|------|-----------|------------|
| 生效范围 | 仅当次对话 | 整个项目所有对话 |
| 一致性 | 依赖每次重写 | 自动加载 |
| 维护成本 | 每次重新构思 | 一次定稿，增量迭代 |

**与写好 Prompt 的本质区别**：写好 Prompt 是一次性技巧（打一枪换一个地方），SDD 是贯穿项目生命周期的方法论（建一座工厂）。

#### (2) SDD 四步闭环

**定规范 → AI 执行 → 人验证 → 迭代规范**（循环）。规范越磨越锋利，AI 越用越顺手。

**第一步 定规范**——覆盖 AI 最容易跑偏的四个地方：

| 维度 | 最小约束 |
|------|---------|
| 命名风格 | 实体大驼峰不加前后缀；字段小驼峰；路径 `/api/v1/{资源复数名}` |
| 返回格式 | 统一 `Result<T>`；空列表返回 `[]` 不返回 `null`；分页 page 从 1 开始 |
| 错误码体系 | 四位数字按模块分段（1000-1999 通用、2000-2999 Provider、3000-3999 Agent…） |
| 设计原则 | 不引入不必要的模式；不过度抽象；不引入技术栈外依赖 |

**第二步 AI 执行**——下达任务时明确引用规范（"按 CLAUDE.md…"）。重点不是告诉它有规范，而是**提醒它去关注那份规范**，而不是按自己的"经验"来。

**第三步 人验证**——有了规范后，review 从主观判断（"感觉不对"）变成**对照清单打勾**的客观核查。

**第四步 迭代规范（最关键、最易忽略）**——每次 AI 跑偏，不要只改代码，停下来三问：① 它为什么跑偏 ② 规范是不是没覆盖 ③ 补上那条规范。**跑偏一次补一条，以后这个点不会再跑偏**。

#### (3) 规范的三条质量标准

| 标准 | 判据 | 反例 → 正例 |
|------|------|------------|
| **具体不模糊** | AI 看到后还需要"猜"吗 | "代码要简洁" → "不引入工厂/策略/观察者等模式" |
| **有优先级** | 反复跑偏的写细，不跑偏的不写 | 该写："Controller 不写业务逻辑"；不该写："变量名要有意义" |
| **带原因** | 工程权衡类必须解释 why | "不要全量删除再插入" + "并发下会瞬间失去所有工具关联" |

**核心规律**：好的 CLAUDE.md 大概两三页，每一条都是因为在某个点跑偏过才补上——**没跑偏过的地方，一条都没有**。

#### (4) 规范的分层组织

| 层级 | 载体 | 覆盖范围 | 改动频率 | 角色 |
|------|------|---------|---------|------|
| 全局规范 | `CLAUDE.md` | 整个项目 | 初期定好，很少改 | 地基 |
| 模块规范 | 模块文档 / 开发前指令 | 单个模块 | 模块开发前定 | 补充 |
| 任务规范 | 任务指令补充 | 单次任务 | 临时性，不落档 | 临场微调 |

**优先级铁律**：任务规范**可覆盖**模块规范（"这次不需要分页"），但**不能违反**全局规范（任何任务都不能绕过 `Result<T>` 统一返回）。

#### (5) 实战印证：Provider 成功 vs Agent 翻车

同一个我、同一个 AI、相隔两天，输出风格分裂——Agent 模块没重复规范（误以为它"记住了"）。

| 规范点 | Provider（写细了） | Agent（只下简短指令） |
|-------|------------------|-------------------|
| 实体命名 | `Provider` | `AgentConfig`（多了后缀） |
| 返回格式 | 统一 `Result<T>` | 混用 `ResponseEntity` |
| 错误码段位 | 2000-2999 | 撞号 2001 |
| 创建方式 | 直接 `new` | Builder 模式 |

**规范生长规律（Hify 的四周补救实录）**：第1周补空值处理 → 第2周补接口契约保护 → 第3周补分层原则 → 第4周补技术栈边界。**没有一条是提前想出来的，全部是开发中一条条补上去的**。

**复利效应**：初期觉得写规范麻烦，到第三周发现 Claude Code 几乎不跑偏了——**不是它变聪明了，是你的规范把所有容易踩的坑都堵上了**。后面每做一个新模块，规范帮你省下的 review 和返工时间，远超当初写规范的投入。

## 2. 顶层设计

### 2.1 定义产品

<img src="imgs/aicent-03-prod-define/cover.jpg" style="display: block; width: 800px;" alt="替换文字">

**完整内容  →  [AI编程方法 03：顶层设计 - 定义产品]({% post_url 2026-07-01-aicent-03-prod-define %})**

#### (1) 核心心法：动手不是写代码

**一句话**：在动第一行代码之前，把产品边界想清楚写进 CLAUDE.md——AI 不会替你拒绝需求，它会"贴心地"把你没说的功能补上。

模糊一句话（如"做个简版 Dify"）会触发两种失败，AI 会把这两种失败放大：

| 失败模式 | 触发条件 | 代价 |
|---------|---------|------|
| 范围膨胀 | 没有显式边界，边做边加 | 工期失控，永远做不完 |
| 推倒重来 | 做到一半才发现方向不对 | 沉没成本高，士气受挫 |

产品定义阶段结束前必须能回答五个问题，答案全部落进 CLAUDE.md：做什么 / 为谁做 / 用什么做 / 跑多快 / 怎么演进。

#### (2) 五步法：从模糊一句话到可执行边界

**调研标杆 → 功能取舍 → 技术选型 → 运维预期 → 写进 CLAUDE.md**（每一步都是"AI 扩展信息面+初筛，人基于约束拍板"的人机协作）。

| 步骤 | AI 做 | 你做 |
|------|-------|------|
| 调研标杆 | 快速梳理标杆功能模块 | 拿官方文档/GitHub 验证修正 |
| 功能取舍 | 三档初筛（必做/降级/砍掉） | 逐项决策，否决不合理建议 |
| 技术选型 | 多维度对比候选方案 | 基于"熟、够用、不分裂"拍板 |
| 运维预期 | 估算 QPS、列缓存策略、查漏 | 识别真正的瓶颈 |
| 写进 CLAUDE.md | 生成初稿 | 检查修改定稿，持续更新 |

#### (3) 三问裁剪法（功能取舍的核心方法）

直接面对"做 / 不做"二选一容易拍脑袋，用三问逐层过滤：

| 层次 | 问题 | 判据 | 归属 |
|------|------|------|------|
| 第一问 | 没有它产品还成立吗 | 砍掉后定位是否站得住 | 核心（必须做满）/ 边缘 |
| 第二问 | 做到什么程度够用 | 当前阶段能用就行还是必须精美 | MVP 版 / 降级最简版 |
| 第三问 | 能不能一句话说清楚它的位置 | 强迫写出功能在定位中的角色 | 写得出→保留 / 写不出→砍掉 |

**关键**：核心功能也要分级（MVP / 可用 / 完整），不是所有核心都要做满。

#### (4) 人机协作判据：什么时候听 AI，什么时候否决

通用原则：**AI 负责拓宽思路，你负责基于自身约束做取舍，每一步的最终判断必须是人做的**。

| 场景 | AI 倾向 | 你的否决判据 | 典型否决 |
|------|---------|-------------|---------|
| 功能取舍 | 建议保留某功能 | 是否击中"为什么要自研"的核心 | 50 人互相认识，权限在系统外约定 |
| 技术选型 | 推荐生态更强的方案 | 一人维护会不会技术栈分裂 | 否决 FastAPI（Java+Python 双栈通信复杂度翻倍） |
| 运维估算 | 给出保守上限 | 真正的瓶颈是不是它指的地方 | 瓶颈在 LLM 长连接，不在并发 |
| 落档 | 生成初稿 | 是否包含显式的"不做什么" | 没写"不做什么"清单 = 不合格 |

#### (5) 技术选型准则：匹配约束，不追新

**技术选型不是选最好的技术，是选最匹配当前约束的技术**。AI 通常按生态强度推荐，你要按"熟、够用、不分裂"三准则否决。

Hify 选 Spring Boot + Vue 的三个理由：① 最熟，开发效率第一 ② AI SDK 的 Java 版够用（向量化调 API 就行，不依赖 LangChain）③ 一人一套技术栈不分裂。

#### (6) 运维预期会反向影响架构

**运维预期不是项目最后才想的事，它从一开始就在影响你的每一个设计决策**。

| 运维判断 | 反向影响的架构决策 |
|---------|-------------------|
| 瓶颈在长连接不在并发 | LLM 调用用独立线程池做隔离 |
| 一期单机够用 | 不需要微服务 |
| 配置变更频率低 | 走 Redis Cache-Aside 而非实时推送 |

SSE 长连接是 AI 应用最容易被忽略的坑（线程阻塞、僵尸连接、Nginx 缓冲三坑），不到生产发现不了。

#### (7) CLAUDE.md 是活的

合格的 CLAUDE.md 项目概述必须包含四块：**做什么 / 不做什么 / 技术栈 / 运维预期**，少一块后续 AI 协作都缺上下文。

CLAUDE.md 不是一次性写好就不动了，随着项目推进持续更新——这和 SDD 的闭环迭代是同一个道理。

### 2.2 应用架构和代码组织

<img src="imgs/aicent-04-arch-code-organize/cover.jpg" style="display: block; width: 800px;" alt="替换文字">

**完整内容  →  **[AI编程方法 04：顶层设计 - 应用架构和代码组织]({% post_url 2026-07-01-aicent-04-arch-code-organize %})

#### (1) 核心心法：AI 给通用最优，你要选项目最优

**一句话**：AI 给的是"技术上最优"的方案，你要基于项目约束选"对你最优"的方案——这两者经常不一样。**决定最终方案的不是 AI 的判断，而是你的项目约束。**

AI 默认推荐微服务、WebFlux 这类"通用最优"，技术上无可挑剔，但你的项目可能一人维护、不想加技术栈，于是一票否决。

#### (2) 四步方法论闭环

| 步骤 | 动作 | 要害 |
|------|------|------|
| ① AI 给方案 | 让 AI 给 2-3 个方案对比 | 不要只问"怎么做"，要问"给我方案对比" |
| ② 列约束清单 | 写下人力、规模、技术栈边界 | 写下来，不是脑子里想 |
| ③ 追问否决 | 成本/兼容/参数三类必问 | 逼 AI 在你的场景下验算一遍 |
| ④ 项目最优 | 得到贴合项目的方案 | 四环缺一环就会跑偏 |

#### (3) 约束驱动：先列清单再问 AI

约束不写下来，很容易被 AI 说服接受"技术上最优但对项目灾难"的方案。

| 维度 | 示例 | 作用 |
|------|------|------|
| 人力 | 1人/3人/10人 | 决定能否承担多服务运维 |
| 当前规模 | 50人/500人 | 决定架构复杂度上限 |
| 技术栈边界 | 不引入 WebFlux/消息队列 | 否决"技术优雅但超界"的方案 |
| 未来扩展点 | 可能拆微服务 | 决定接口设计颗粒度 |

关键动作：**把约束贴在对话开头让 AI 在约束下重给方案**，并写进 CLAUDE.md 让 AI 后续自动遵守。

#### (4) 三类追问挖盲区

AI 不会主动暴露短板，要靠追问挖出来。不是抬杠，是逼 AI 把"通用最优"在"你的场景"下验算一遍。

| 追问类型 | 典型问题 | 专治 |
|---------|---------|------|
| 成本类 | 运维精力能承担吗？排查难度增加多少？ | 看起来漂亮、扛起来要命的方案（微服务、独立网关） |
| 兼容类 | 与现有技术栈冲突吗？学习曲线多陡？ | 为了技术优雅引入新编程模型的诱惑（MVC 够用却上响应式） |
| 参数合理性 | 默认值对场景够用吗？边界情况行为？ | AI 给默认值、你照单全收的懒惰（60 秒超时套 SSE 根本不够） |

#### (5) 规范是写给 AI 看的合约（颠覆传统认知）

这条颠覆传统：代码规范的受众从"团队工程师"变成"AI 代码生成器"——人凭习惯能补全，AI 每次生成都是全新一次理解，颗粒度细才能保证一致性。

| 对比项 | 传统规范 | AI 编程规范 |
|--------|---------|------------|
| 受众 | 团队工程师 | AI 代码生成器 |
| 颗粒度 | 粗（约定俗成） | 细（白纸黑字） |
| 写法准则 | 凭习惯补全 | AI 反复跑偏处写细，不跑偏处不写 |
| 衡量标准 | 工程师读着舒服 | AI 生成的代码符合预期 |

典型例子：AI 喜欢"顺手"在 Controller 加业务逻辑——必须明文写"Controller 只做参数校验和调用 Service，禁止业务逻辑"。**颗粒度的判断标准是"AI 在这块是否反复跑偏"，不是"这条规则多基础"。**

#### (6) 为未来留口：每个决策问"未来 X 改动多大"

架构决策的真正价值不是"现在好用"，而是"未来要改的时候改动可控"。判断决策好坏的标准，是它有没有堵未来的路。

通用 Prompt 骨架：`假设未来要把【某模块】独立出来（拆服务/上多区域/换实现），按当前架构改动量多大？哪些点会卡住？`

这条规范当下不带来收益、只带来约束成本，但价值在未来某次拆分/替换时一次性兑现——**这是"做架构"和"写代码"的本质区别**。

#### (7) 三大主题落地（Hify 实战印证）

三个表面不同的问题（形态怎么选/规范怎么定/方案怎么完善），底层用的是同一套方法论：让 AI 给方案 → 用约束清单 → 追问否决 → 为未来留口。

| 主题 | AI 建议 | 最终决策 | 否决依据 |
|------|---------|---------|---------|
| 应用架构 | 微服务（方便独立扩展） | **模块化单体** | 一人维护六七个服务精力扛不住 |
| 跨模块调用 | 直接引用内部类 | **只走 Service 接口** | 未来拆服务时接口签名都不用变 |
| 外部调用-超时 | 对话接口统一 60 秒 | **SSE 调到 120 秒** | 流式响应 60 秒会切断正常对话 |
| 外部调用-框架 | WebFlux 做异步非阻塞 | **SseEmitter + 独立线程池** | 不为技术优雅引入新编程模型 |

外部调用四维度（线程/容错/超时/重试）必须一次性整体设计，不能东补一块西补一块：

| 维度 | 落地方案 | 解决什么问题 |
|------|---------|------------|
| 线程管理 | 独立线程池，与 Tomcat 业务池隔离 | LLM 慢响应不拖垮管理页面 |
| 容错 | Resilience4j，每供应商独立熔断器 | OpenAI 挂了不影响 Claude |
| 超时 | 分场景：同步 60s / SSE 120s / 连通 10s | 避免误杀长任务或放任卡死 |
| 重试 | 按异常区分：网络抖动重试/认证失败不重试/限流退避 | 认证失败重试百次还是失败 |

#### (8) 沉淀进 CLAUDE.md：让 AI 自动遵守

把架构决策、代码组织、外部调用三段规范写进 CLAUDE.md，让 Claude Code 后续每次生成都自动遵守，不用重复说明。**把合约固化成 AI 的长期记忆，架构决策就不再依赖人的反复提醒。**

如果只能留一条，留"规范写进 CLAUDE.md"——它带来的复利最大。

### 2.3 性能部署与数据设计

<img src="imgs/aicent-05-performance-and-data-design/cover.jpg" style="display: block; width: 800px;" alt="替换文字">

**完整内容  →  [AI编程方法 05：顶层设计 - 部署性能与数据设计]({% post_url 2026-07-01-aicent-05-performance-and-data-design %})**

#### (1) 核心心法：部署形态从第一天就在影响代码设计

**一句话**：缓存放哪、数据库怎么连、服务怎么通信，全都取决于部署形态。不是"后面再说"的事，提前想清楚不返工。系统设计的目标是**用最低成本跑起来、跑得稳、跑得久**——简单可跑、留扩展口。

五个子主题构成一条思考链：

| 子主题 | 核心问题 | 产出 |
|------|---------|------|
| 部署设计 | 当前形态长什么样 | 架构图 + "现在做/以后做"边界 |
| 瓶颈预判 | 系统软肋在哪 | 软肋地图（按严重程度排序） |
| 扩展路径 | 未来怎么演进 | 阶段化路线（触发条件驱动） |
| 数据模型 | 核心表有哪些 | 表名 + 关系（不展开字段） |
| 数据库规范 | 建表查询怎么写 | 通用字段 + 索引 + 分页 + 向量库 |

#### (2) 部署设计：50 人规模的核心诉求是"简单、留口"

50 人规模真实压力极小（峰值并发几十个请求），核心诉求**不是"高可用、高并发"，而是"简单、可观测、留扩展口"**。三条反常识：

| 反常识 | 说明 |
|-------|------|
| K8s ≠ 复杂 | 早期可只把 K8s 当部署壳（单副本、不引入服务网格/多副本），触发条件到再加 |
| 静态资源与 API 必须分流 | Nginx 同时托管静态 + 反代 API，避免后端被静态请求拖慢 |
| 有状态服务独立部署 | DB/缓存/向量库作为外部服务，不和应用混在一起，便于扩展和替换 |

最有价值的产出是 **"现在做/以后做"对比表**——每条"以后做"都有触发条件（用户量/数据量），不是偷懒。

#### (3) 性能瓶颈预判：价值在"地图"而非"解决"

**一句话**：在设计阶段就知道系统软肋在哪，即使一期不处理也要心里有数。出问题时不用猜，查表就行。地图不用自己从零画——让 AI 基于架构分析，你判断它说得对不对。

50 人规模下真正能触发的瓶颈只有 **LLM 调用慢且占线程**——而且这其实是稳定性/成本问题，不是性能问题。

| 一期处理决策 | 内容 |
|------------|------|
| 做 | LLM 调用设超时 + 独立线程池 + 熔断；Nginx 开 gzip |
| 不做 | 其他全部标记"已知但暂不处理"——不是忽略，是时机未到 |

关键区分：性能问题 vs 稳定性/成本问题要分开。LLM API 超时是稳定性问题，费用失控是成本问题，但都可能在早期触发。

#### (4) 扩展路径：触发条件驱动，条件不到不动

**过早优化比性能瓶颈本身的破坏力更大**。架构功力不在于现在做多复杂，在于不堵未来的路。

| 阶段 | 触发条件 | 改什么 | 不改什么 |
|------|---------|-------|---------|
| 50→500 | CPU/内存持续告警 | 应用层扩副本 + MySQL 主从 + 静态资源 CDN | 单体架构、pgvector、Redis 单节点 |
| 500→2000 | LLM 队列堆积、检索 P99>2s | MQ 异步 + Qdrant + 分库 | 单体不拆微服务、Redis 架构 |
| 2000→几千 | 模块发布互相干扰、资源消耗不均 | 微服务拆分 + API Gateway + Redis 集群 | MySQL 主从、Qdrant、K8s 编排层 |

**每阶段只动一个层**（应用层/数据层/架构层），不要一次全动。难点（SSE 连接粘连、前端 breaking change、分布式事务）要在规划阶段就识别，留好接口设计。

#### (5) 数据模型：只定表名和关系，字段后面再定

**现在定太细了后面一定会改**。按功能域分组，每组识别"主表→子表→关联表"。两条注意：

- **配置型表 vs 流水型表要区分**：provider/agent/workflow 是配置（增长慢），message/document_chunk 是流水（增长快），差异化对待
- **向量数据单独存储**：业务库只存元数据和 `vector_id`，向量本身进 pgvector/Qdrant

设计亮点：`message_reference` 记录"AI 回复引用了哪些知识库片段"——很多人做 RAG 不存这个关系。**一张中间表，成本很低，价值不小**：对用户提供可信度依据，对开发者回溯分析检索质量。

#### (6) 数据库规范：看起来是常识，不写 AI 就不遵守

**这些规范对有经验的工程师是常识，对 Claude Code 不是**。不写进 CLAUDE.md，它建表时就不会自动加 `deleted` 进索引、不会考虑分页性能、不会给 pgvector 建 HNSW 索引。

| 维度 | 关键规则 |
|------|---------|
| 通用字段 | 主键自增 BIGINT（禁 UUID）/ 时间戳 DATETIME(3) / 逻辑删除 deleted / 禁 NULL / 枚举用 VARCHAR(32) |
| 索引 | ① 逻辑删除字段必须进组合索引 ② 等值列在前、范围列在后 ③ 多对多关联表两方向都建索引 ④ 唯一约束用 UNIQUE INDEX（不只代码校验）⑤ 禁在 TEXT/BLOB 建索引 |
| 分页 | 默认游标分页（`WHERE id < lastId ORDER BY id DESC LIMIT N`）；OFFSET 限 10000 条；COUNT 只在第一页查 |
| 大表预判 | message 建 (conversation_id, created_at)；document_chunk 只存元数据 |
| pgvector | 必须建 HNSW 索引；检索必须加 LIMIT（禁止全量排序） |

规范的价值是**持续受益**——现在花 10 分钟定好，后面每一次建表、每一次写查询都受益。

#### (7) 沉淀进 CLAUDE.md：让 AI 自动遵守

三大块结构化写入：**部署架构 + 数据库规范 + 扩展路径**。

**检验标准**：Claude Code 后续建表会自动加 `deleted` 进索引、自动用游标分页、自动给 pgvector 建 HNSW 索引——如果不会，说明规范没写够细。

至此 CLAUDE.md 已包含项目概述（系列三）、应用架构（系列四）、部署与数据库规范（本篇），Claude Code 对 Hify 从代码怎么写到系统怎么跑都有了完整认知。


### 2.4 工程规范、CLAUDE.md和Skill

<img src="imgs/aicent-06-standard-claudeme-skill/cover.jpg" style="display: block; width: 800px;" alt="替换文字">

**完整内容  →  [AI编程方法 06：顶层设计 - 工程规范、CLAUDE.md和Skill]({% post_url  2026-07-01-aicent-06-standard-claudeme-skill %})**

#### (1) 本质一句话：把 AI 反复出错的地方用文字钉死

规范的难点从不在"怎么写"，而在"什么时候写、写什么、写多细"。所有工具（CLAUDE.md / Skill / 业界喂入）都是同一本质的不同封装——**把"AI 跑偏的地方"沉淀成可执行指令**。

#### (2) 四个递进抓手（工具 + 元规则）

| 抓手 | 定位 | 何时建立 | 类比 |
|------|------|---------|------|
| 一：CLAUDE.md | 项目级全局规范 | 项目启动时建，渐进完善 | 地基 |
| 二：Skill | 任务级可复用规范 | 指令在重复时创建 | 预制构件 |
| 三：业界喂入 | 专家经验转化器 | 需要补充某领域规范时 | 外购建材 |
| 四：三条思维方式 | 判断"写不写"的元规则 | 每写一条都用 | 施工准则 |

前三个解决"用什么装"，第四个解决"值不值得装、装多细"——两阶段缺一不可。

#### (3) 三条思维方式（贯穿所有规范的元规则）

| 心法 | 规则 | 反例 | 正例 |
|------|------|------|------|
| 从架构决策推导 | 追溯不到决策就不写 | "代码要优雅"（空话） | 模块化单体 → 跨模块走 Service 接口 |
| 颗粒度跟着问题走 | AI 反复跑偏处写细 | "UTF-8 编码"（AI 不会错） | "Controller 不写业务逻辑"（AI 常犯） |
| 规范是长出来的 | SDD 闭环驱动补全 | Day 1 想写"完美版" | 每次跑偏后问"规范没覆盖到？"再补 |

#### (4) CLAUDE.md 五层结构（宏观到微观）

| 层级 | 写什么 | 项目阶段 |
|------|-------|---------|
| 1 项目上下文 | 做什么、**不做什么**、技术栈 | 项目启动 |
| 2 架构规范 | 模块划分、依赖方向、跨模块调用规则 | 架构设计 |
| 3 代码组织 | 分层职责（Controller/Service/Mapper） | 架构设计 |
| 4 部署/数据库 | 部署拓扑、索引规则、分页规范、向量库 | 数据库/部署设计 |
| 5 接口规范 + 行为指令 | 路径风格、响应结构、错误码、**AI 行为模式** | 开发迭代 |

第五层最易被忽略但最有用——**行为指令约束的是 AI 怎么行事，不是代码风格**。三组行为指令：

| 场景 | 关键约束 |
|------|---------|
| 开发功能时 | 简单实现、不引入非技术栈依赖、外部调用必超时、配置外化 |
| 改代码时 | 先理解原设计、不破坏既有契约、改完测试通过 |
| **不确定时** | **架构给 2-3 方案让用户拍板、规范没覆盖先问不编规矩** |

**措辞心法**：用"必须"不用"建议"——AI 会把"建议"当可选参考。

#### (5) Skill vs CLAUDE.md：分工不替代

| 维度 | CLAUDE.md | Skill |
|------|-----------|-------|
| 定位 | 项目级全员规范 | 任务级标准流程 |
| 触发 | 每次会话自动加载 | 执行某类任务时按需匹配 |
| 位置 | 项目根目录 | `.claude/skills/` |
| 例子 | 路径风格用 `/api/v1/{资源}` | 设计新模块接口契约的 7 步流程 |

**Skill 化信号**：同一指令重复 ≥ 2 次。一期不要建太多，跟着重复信号长出来。CLAUDE.md 是"什么算合规"，Skill 是"怎么做一件事"。

#### (6) 业界喂入四步法（让 AI 当专家顾问）

核心：**不是把整本手册塞进 CLAUDE.md**（稀释重点），而是让 AI 提炼适配版。

| 步骤 | 动作 | 关键 |
|------|------|------|
| ① 选标准 | 选业界公认那一份（如阿里巴巴 Java 手册） | 不纠结小差异 |
| ② 喂 Prompt | 项目背景 + 标准来源 + 重点维度 + **"精简到 AI 能直接执行"** | 决定产出形态 |
| ③ AI 提炼 | 产出项目适配版（不是原文照搬） | 不写"详见手册第 X 条" |
| ④ Review 定稿 | 人判断保留/删除/调整措辞 | 你是审稿人不是发明者 |

**Prompt 关键句**："不要照搬原文，要精简到 AI 能直接执行。"——这一句决定产出可用性。

#### (7) 接口契约：介于第五层与代码实现之间的模块级蓝图

CLAUDE.md 接口规范定通用规矩（路径格式、Result 结构、错误码分段），**接口契约定具体模块的每个接口**（HTTP 方法 + 路径 + 入参 + 出参 + 关联检查）。每个模块开发前写那个模块的，不需要一开始全写完。

**给 AI 蓝图的价值**：生成的 Controller 和 DTO 精确匹配设计，不需大面积返工。没蓝图，字段命名、模糊搜索、关联检查这些 AI 都拍脑袋，每次拍得还不一样。

#### (8) 决策树：一条规范该装在哪里

| 问题 | 答案 | 去向 |
|------|------|------|
| 是项目级还是任务级？ | 项目级 → 业界有标准吗？ | 有 → 喂入提炼入 CLAUDE.md；无 → 自己写进 CLAUDE.md |
| 同上 | 任务级 → 指令在重复吗？ | 在重复 → 创建 Skill；没重复 → 临时描述 |

装好后再用三条思维方式校验：能追溯到决策吗？颗粒度合适吗？不合适调整，追溯不到不写。

#### (9) 初稿 review 三处 diff（方法论集中演练）

AI 生成 CLAUDE.md 初稿约 80% 可用，20% 必须人改。三处典型调整：

| diff 调整 | 体现的方法论 |
|----------|-------------|
| "建议使用简单实现" → "每个功能用简单直接的方式实现，不引入不必要的设计模式，除非明确要求" | 措辞要具体强硬，AI 理解"建议"和"必须"的区别 |
| 删掉详细的 logback 配置 | 颗粒度跟着问题走，AI 不会跑偏的不占篇幅 |
| 补全"不做什么"列表（不做多租户/计费/微调/WebApp） | 边界越明确，AI 越不会好心办坏事 |

**核心命题**：让业界标准为基准，逐步补充项目特点。CLAUDE.md 是地基、Skill 是构件、业界喂入是建材、三条思维方式是施工准则。地基先打、构件按需造、建材借外力、准则贯穿始终。

### 2.5 流程回顾

<img src="imgs/aicent-07-design-proc-recap/cover.jpg" style="display: block; width: 800px;" alt="替换文字">

**完整内容  →  [AI编程方法 07：顶层设计 - 流程回顾]({% post_url 2026-07-01-aicent-07-design-proc-recap %})**

#### (1) 核心命题：AI Coding 使用面远比"写代码"宽

**一句话**：Claude Code 是全方位协作者，代码只是其中一环，瓶颈在"你怎么用"。

| 工作类型 | 价值定位 | 关键判据 |
|---------|---------|---------|
| 做页面 | 非前端工程师独立完成官网，效果超预期 | 协作者，不是执行者 |
| 写文档/翻译 | 讨论后整理成文，一句话生成英文版 | 一句话指令即可 |
| 市场调研 | 利用信息储备梳理竞品方案与痛点 | AI 加速信息收集，但需验证 |
| 角色推演 | 多视角（架构师/开发/算法/Agent）发表看法 | 避免技术自嗨 |
| 讨论技术方案 | 频率不高——最多验证，不能纯听 | 方案"看起来对但不那么适合" |
| 协作写代码 | 按项目重要程度决定合作深度 | 区分核心项目与外围项目 |

#### (2) 协作铁律：它建议、你判断

四场景反复演示同一模式：AI 给"通用最优"方案 → 人用**现实约束**否决 → 选"项目最优"。

| 场景 | AI 建议 | 否决依据 | 最终方案 |
|------|---------|---------|---------|
| 技术选型 | FastAPI + Java/Python 混合 | 一个人维护两套栈是灾难 | Spring Boot 统一栈 |
| 应用架构 | 微服务 | 一个人维护六七个服务太重 | 模块化单体 |
| 流式响应 | WebFlux | 不增加技术栈复杂度 | SseEmitter + 独立线程池 + 120s 超时 |
| 部署组件 | MQ + 链路追踪 | 50 人规模过度设计 | 单副本够用，跳过 |

**核心**：现实约束（团队规模、运维能力）是最终裁判权，AI 不掌握你的约束——理由充分 ≠ 必须全做（RAG 保留但降级为只支持 TXT）。

#### (3) 四场景一闭环

| 场景 | 三步骤 | 落地产物 |
|------|--------|---------|
| 产品定义 | 梳理全景 → 功能取舍 → 技术选型 | 做什么/不做什么/技术栈 |
| 应用架构 | 架构选型 → 代码组织 → 外部调用 | 模块划分 + 分层规范 + 线程熔断方案 |
| 部署数据 | 部署架构 → 瓶颈预判 → 数据库规范 | 组件表 + 软肋地图 + 索引规则 |
| 实战技巧 | compact/clear → 业界规范喂入 → 合并 | 完整 CLAUDE.md |

每步都是"它建议、你判断"，每步产物都沉淀进 CLAUDE.md——规范越清楚，后面写代码越快越稳。

#### (4) compact vs clear

| 命令 | 时机 | 作用 |
|------|------|------|
| /compact | 任务中途上下文快满 | 压缩当前对话保留要点 |
| /clear | 切换任务 | 清空上下文，CLAUDE.md 自动加载 |

**最佳实践**：每个任务完成后 commit + clear。CLAUDE.md 自动加载，项目规范不丢。

#### (5) 终极认知

> 没写一行业务代码，但所有决策都已经落成文字。

到这一步 Claude Code 对项目的全部认知就绪：模块怎么分、代码放哪里、接口什么格式、索引怎么建、什么不能做。这就是 SDD 的威力——**人定规范、AI 写代码**在生产环境得到真实验证。

## 3. 工程搭建 

### 3.1 后端骨架和基础设施

<img src="imgs/aicent-08-backend-foundation/cover.jpg" style="display: block; width: 800px;" alt="替换文字">

**完整内容  →   [AI编程方法 08：工程搭建 - 后端骨架和基础设施]({% post_url 2026-07-01-aicent-08-backend-foundation %})**

一句话讲透：让 AI 搭后端骨架，难的不是"让它做"，而是"控制它每次做多少、做完怎么验、跑偏了怎么堵"。本篇把这件事提炼成四条方法论，核心目标是三件事——**可控、可复现、可迭代**。

#### (1) 为什么要拆任务？

一次性让 AI 生成几十个文件，它会出错：父 pom 与目录不符、版本号冲突、错误码格式不一致。更麻烦的是——出错了你 review 不过来。

拆的本质收益不是"省事"，而是**错误可定位、规范可迭代**。这是"道"不是"术"，即便模型更强也依然要拆。

| 拆的两条判据 | 含义 |
|-------------|------|
| 生成量超一次 review 范围 | 十几行能看，几十文件看不过来 |
| 步骤间有依赖 | 第二步依赖第一步结果 |

满足任一就拆。拆的顺序按依赖关系排：**先地基、后框架、最后验收**。

#### (2) 四条方法论（整篇精华）

| 方法论 | 一句话 | 关键动作 |
|--------|--------|---------|
| 大任务拆解 | 把大任务切成"可review、可验收、可定位"的小块 | 指令末尾声明范围边界（如"只建pom，不写Java"） |
| 影响范围 Review | 不逐行看，按影响半径排优先级 | P0结构→P1公共模块→P2配置→P3空壳 |
| 每步验收 | 验收前置，别等全搭完再验 | 每步做完就验，拿"到这里为止是对的"信心 |
| SDD 闭环 | 让规范成为可持续迭代资产 | AI 每跑偏一次，CLAUDE.md 补一条 |

这四条是本篇的全部骨架，下面逐条说清"怎么做"。

#### (3) 影响范围 Review：为什么 P0 最重要？

结构错了（模块依赖、pom 声明、package 路径），后面所有东西都建在错误地基上——影响半径是"全盘"。空壳类几乎不会错，扫一眼即可。

逻辑很简单：**影响范围越大的问题越先查**。这个思路不只适用于初始化，后面每次 AI 批量生成都能复用。

#### (4) SDD 闭环怎么转起来？

用一个真实例子讲清：

全局异常处理器兜底 Exception 时，AI 硬编码了 `code:500, message:"系统繁忙"`，没走 ErrorCode 枚举。

正确做法不止是改成 `Result.fail(ErrorCode.INTERNAL_ERROR)`，更要在 CLAUDE.md 补一条规范："异常处理必须用 ErrorCode 枚举，禁止硬编码错误码"。

| 步骤 | 动作 |
|------|------|
| 1 定规范 | 写进 CLAUDE.md |
| 2 AI 执行 | 生成代码 |
| 3 发现偏差 | 与规范不符 |
| 4 迭代规范 | 补一条可执行指令，回到步骤 2 |

这就是 SDD（Spec-Driven Development）闭环。它不是宏大流程，而是**写代码时随手做的事**。

#### (5) 实战拆解：Hify 后端四步

按"地基→框架→验收"，每步独立验收：

| 步骤 | 做什么 | 验收标准 |
|------|--------|---------|
| 1 Maven 骨架 | 父 pom + 子 pom + 目录，不写 Java | `mvn install` 通过 |
| 2 hify-common | Result/异常处理/MyBatis-Plus/Redis 配置 | 编译无错、符合 CLAUDE.md 规范 |
| 3 业务模块空壳 | package 结构 + 占位类 | 结构一致、可编译 |
| 4 总验收 | 启动项目 | `spring-boot:run` 起来、健康检查返 200 |

hify-common 又拆成五个小任务（Result→ErrorCode/BizException→全局异常处理→MyBatis-Plus 配置→Redis 配置），前三个有依赖链。原因同上：拆开后每个任务上下文更聚焦，AI 输出质量更高，出问题易定位。

#### (6) 三个反直觉的洞察

1. **"只做什么"比"做什么"更重要**：指令不加范围声明，AI 会顺手生成 Application 类、配置文件、示例代码，和后续步骤冲突。
2. **0 代码也能完成初始化**：本篇全程没写一行代码，很顺，原因是模型强 + 基础搭得好 + **任务拆得细**。手工做至少一天。
3. **小验收避免大回溯**：不验证就往前冲，错误会累积到结尾，无法定位是哪一步出的。

#### (7) Check List（拿走就能用）

工程初始化阶段，逐项打勾：

- [ ] 任务是否按生成量与依赖关系拆成有序子任务？
- [ ] 每条指令是否声明了范围边界（只做什么、不做什么）？
- [ ] 子任务顺序是否按依赖关系排好（地基→框架→验收）？
- [ ] Maven 骨架：父 pom `<modules>` 与目录一致？版本号只在 `<dependencyManagement>`？
- [ ] 公共模块：Result/异常处理/配置类是否符合 CLAUDE.md 规范？
- [ ] 每步是否做了小验收？
- [ ] 发现的偏差是否回写进 CLAUDE.md？

记住核心模式：**拆任务 → 按依赖排序 → 每步验收 → 跑偏补规范**。这套打法在后续每个模块都通用。

### 3.2 前端工程初始化

<img src="imgs/aicent-09-frontend-init/cover.jpg" style="display: block; width: 800px;" alt="替换文字">

**完整内容  →   [AI编程方法 09：工程搭建 - 前端工程初始化]({% post_url 2026-07-01-aicent-09-frontend-init %})**

#### (1) 核心要义

**一句话**：一份规范（CLAUDE.md）驱动前后端两端 → 提示词讲清"要什么"不靠熟练度 → 每层用最小链路验证 → 启动脚本交给 AI 全权处理。

承接第 8 篇后端骨架补齐前端。完成后 `./start.sh` 一键拉起前后端，浏览器看到 Hify 页面 + 绿色"后端已连接"。本篇四条方法论全部围绕"让 AI 产出可控、可对齐、可验收"。

#### (2) 规范两端对齐：一份规范驱动两端

后端定统一响应 `Result<T> = {code, message, data}`，前端 axios 拦截器若不在此处自动解包，业务代码每个接口都要重复判断 `code`、手动取 `data`。封装到位后，业务直接拿到业务数据——几十个接口累积下来，代码干净很多。

| 规范项 | 后端落点 | 前端落点 |
|--------|---------|---------|
| 响应格式 `Result<T>` | Controller 返回 `Result.ok(data)` | 拦截器自动解包 `data` |
| 路径规则 `/api/v1/...` | `@RequestMapping` | baseURL + 相对路径 |
| 错误码 / message | ErrorCode + 全局异常处理 | 非 200 用 ElMessage 提示并 reject |

**关键动作**：前端 axios 封装与后端 Result 实现**必须在同一次工程初始化里完成**——否则中间版本漂移，联调时才发现字段对不上。这就是为什么第 6 篇要把接口规范写进 CLAUDE.md：后端在用，前端也对着同一份做。**一份规范，两端对齐**。

#### (3) 提示词四要素：不懂也能产出

作者实话："我一点不懂前端。"但靠一条交代清楚四要素的指令，就拿到能跑的工程。说明对 AI 编程而言，提示词的**完整性 > 领域熟练度**——你不需要会写 Vue，只需要会把"要什么"讲明白。

| 要素 | 说明 | 本篇示例 |
|------|------|---------|
| 技术栈 | 明确框架与版本取向 | Vue 3 + TS + Vite + Element Plus |
| 目录结构来源 | 引用既有规范，不临场发挥 | "按 CLAUDE.md 中定义的前端结构来" |
| 关键配置 | 跨进程/跨域细节显式写出 | Vite 代理：`/api` → `localhost:8080` |
| 范围边界 | 只做什么、不做什么 | 本例只做初始化，后续分指令给 |

**反例**："帮我建个前端项目"——技术栈、目录、代理全靠 AI 猜，结果与项目规范对不上。**凡是能引用既有规范的，就引用，不要在指令里重复一遍规范细节。**

#### (4) 最小可验证链路：一条线验四件事

前端页面加载时调 `getHealth()`：成功绿色"后端已连接"、失败红色。一条最小链路同时验证四件事——axios 封装、Vite 代理、后端接口、Result 格式前端解析。

| 失败现象 | 排查路径 |
|---------|---------|
| 页面显示红色 | 打开控制台 → 复制错误 → 贴给 Claude Code |
| 根因：后端没起 | 启动后端 |
| 根因：代理错 | 检查 Vite 代理配置 |
| 根因：格式不对 | 核对 Result 与拦截器 |

**最常见的根因往往最朴素**——比如"后端服务被关了"。先确认这类低级原因，再深入。**不要人去读源码，把错误信息直接给 Claude Code 让它排查。**

#### (5) 第三层自动化：脚本交给 AI

启动脚本、Makefile 是典型的"第三层"——AI 全权处理，人只验收结果。

**为什么交给 AI**：脚本逻辑不难，但手写易漏工程细节——进程管理（PID 文件）、日志分流、端口检查（MySQL/Redis 就绪）、优雅停止（先 SIGTERM 再等待，超时 SIGKILL）。这些琐碎但关键，交给 Claude Code 特别合适。

**验收标准**：`start.sh` 能起、`stop.sh` 能干净停掉就过了。**不逐行 review shell 语法——第三层的验收就是看结果。**

#### (6) 实战三步拆解

按依赖关系排序，每步独立验收后再进下一步。

| 顺序 | 步骤 | 做什么 | 为什么这个序 |
|------|------|--------|-----------|
| 1 | 项目骨架 | hify-web 容器 + Vite 代理 | 承接所有前端代码 |
| 2 | axios 请求层 | request.ts 拦截器自动解包 | 前端与规范体系的接合点（1.1 节落点） |
| 3 | 路由页面空壳 | 菜单 + router-view | 有了骨架和请求层才能组织 |

三步之外还有前后端联通（验收链路）、启动脚本（工程化收尾）、最终验收。**两篇合起来，从零到能跑的完整闭环交付完成。** 后面就是往这个骨架里填功能。

#### (7) Check List（拿走就能用）

- [ ] 提示词含技术栈、目录结构来源、关键配置（代理/端口）？
- [ ] Vite 代理把 `/api` 正确转发到后端端口？
- [ ] axios 封装与 Result 对齐（自动解包 `data` + 非 200 提示并 reject）？
- [ ] 用最小示例接口（health）验证封装、代理、路径三者都通？
- [ ] 路由与页面空壳生成完整导航结构（菜单 + router-view）？
- [ ] 启动脚本做依赖检查（MySQL/Redis 就绪）、健康轮询、错误中断？
- [ ] 停止脚本优雅停止（SIGTERM → 等待 → 超时 SIGKILL）？
- [ ] 最终：`sh start.sh` 一键拉起，三菜单 + 绿色连接？

**核心模式**：**规范两端对齐 → 提示词四要素 → 每层最小验证 → 脚本交给 AI**。这套打法让"我不懂前端"也能交付能跑的工程——重点不在你会不会写 Vue，在你能不能把"要什么"讲明白。

### 3.3 后端业务基础设施

<img src="imgs/aicent-10-be-business-infra/cover.jpg" style="display: block; width: 800px;" alt="替换文字">

**完整内容  →   [AI编程方法 10：工程搭建 - 后端业务基础设施]({% post_url 2026-07-01-aicent-10-be-business-infra %})**

前两篇搭出能跑的空项目，本篇补齐"业务开发前必须就绪"的基础组件层——schema、MapperScan、线程池、BaseEntity、分页、校验、时间序列化、Cache、HTTP 客户端、熔断、结构化日志。同时引入一种新协作模式：**咨询模式（先问再做）**。

**核心打法**：咨询模式梳理全景（查漏）→ 按"不做会怎样"分档 → 按依赖排序逐档执行 → 参数从需求推导 → 三层隔离让一处故障不拖垮全局。

#### (1) 咨询模式：先问再做（查漏）

你想到了线程池、熔断这些有设计感的东西，却容易漏掉最基础的——表还没建、MapperScan 路径不对。脑子里都是"高大上"，反而漏了地基。这时不是让它替你决策，而是把它当资深架构师来咨询。

| 维度 | 执行模式 | 咨询模式 |
|------|---------|---------|
| 适用 | 你对任务很清楚 | 知道大方向但不确定细节 |
| 流程 | 你想清楚 → AI 做 → 你验收 | 你问"该考虑什么" → AI 梳理全景 → 你取舍 → 再执行 |
| 价值 | 高效落地 | 遗漏检查（AI 见过的项目比你多） |

**关键**：AI 的建议不能照搬，但用来对照自己的清单非常好——**它帮作者发现了自己会漏掉的 schema.sql 和 @MapperScan**。咨询模式不能当甩手掌柜，应**在每个阶段切换时反复用**。

#### (2) 三档优先级：按"不做会怎样"分档

咨询模式给的清单很全，但不是所有东西都要现在做。判据不是"重不重要"，而是"不做会怎样"。

| 档位 | 判据 | 组件 | 不做的后果 |
|------|------|------|-----------|
| 第一档 必须先做 | 不做业务跑不起来 | schema.sql、@MapperScan、llmExecutor/asyncExecutor | 启动报错、Mapper 注入失败 |
| 第二档 同步补 | 第一个列表接口之前搞定 | BaseEntity、分页、校验、时间序列化、Spring Cache、CRUD 标准 | 每实体重复写字段、前端解析不了时间 |
| 第三档 后补 | 不影响功能开发 | HTTP 客户端、Resilience4j 熔断、结构化日志 | 联调时才发现 LLM 没隔离、日志查不到链路 |

**反直觉**：看似平淡的 schema.sql 才是第一档；看起来有设计感的 BaseEntity、熔断往往不是。**不分会陷入两种极端**：要么全做完才写业务（进度卡住），要么随手做随手漏（启动报错才发现）。

#### (3) 按依赖排序与最小验证

三档有依赖：没 schema.sql，Mapper 跑不起来；没 BaseEntity，CRUD 验证跑不全。乱序会出现"搭完一半发现依赖没就绪"的返工。**每档做完用一个最小验证确认就绪，再进下一档**——不积压到全搭完才联调。

| 顺序 | 步骤 | 最小验证 |
|------|------|---------|
| 1 | 第一档：schema.sql → MapperScan → 线程池 | `mvn spring-boot:run` 启动成功 |
| 2 | 第二档：BaseEntity → 分页 → 校验 → 时间 → Cache → DemoItem CRUD | 五个 curl 全通过 |
| 3 | 第三档：HTTP 客户端 → 熔断 → 结构化日志 | 日志有 traceId / llm- / async- 前缀 |

**DemoItem CRUD 不是终点**：用最简 CRUD 把第二档所有组件串起来跑通，留作后续模块的参考模板。

#### (4) 参数从需求推导（程序员不可替代的价值）

AI 不知道 GPT-4 一次响应要 10-20 秒，会按"普通 HTTP 调用"给 5s 超时——结果 LLM 调用全部超时。**这是程序员的价值：理解需求、从宏观推导参数——AI 替代不了的能力。**

| 参数 | 需求场景 | 推导 | 取值 |
|------|---------|------|------|
| llmExecutor 核心 | 20-50 人在线，一半同时对话 | 10-25 个并发 | 10 |
| llmExecutor 最大/队列 | 留余量 + 缓冲超核心请求 | 峰值 50 / 缓冲 100 | 50 / 100 |
| llmExecutor 拒绝策略 | LLM 调用不能丢 | 回退到调用线程执行 | CallerRunsPolicy |
| asyncExecutor | 日志异步写入等非关键任务 | 比关键任务小，失败可接受 | 5 / 20 / 200 / AbortPolicy |
| RestTemplate 读超时 | 同步 LLM 请求 | GPT-4 响应 10-20s | 60s |
| OkHttpClient 读超时 | 流式 SSE | 可能持续 1-2min | 120s |
| 熔断阈值 / 开路时间 | Provider 健康判定 / 恢复等待 | 50% 失败即异常 / 30s 后半开试探 | 50% / 30s |

**操作要点**：先写需求场景（多少人在线、响应多久、能否丢失），再推参数；一期按经验给值，后面根据监控调整。

#### (5) 隔离思想：一处故障不拖垮全局

三层隔离是生产级项目的底层能力——**资源隔离、故障隔离、链路隔离**。线程池解决"一个 LLM 卡了不拖累其他功能"；熔断解决"Provider 挂了系统还在白等超时"；traceId 解决"一个请求十几条日志大海捞针"。

| 隔离点 | 实现 | 隔离什么 |
|--------|------|---------|
| 线程池隔离 | llmExecutor 与 asyncExecutor 分开 | LLM 卡住不阻塞管理页面（命名前缀 llm-/async- 便于看日志） |
| Provider 独立熔断 | 按 providerName 各自一个 CircuitBreaker | OpenAI 挂了不拦住 Claude |
| traceId 链路隔离 | 拦截器生成 traceId 放入 MDC | Controller→Service→LLM→DB 的日志用一个 ID 串起 |

**重试也要按异常隔离**：认证失败重试一百次也不会成功，只有网络超时、限流这种"暂时性"失败才值得重试。

#### (6) Check List（拿走就能用）

**阶段一 第一档**：
- [ ] 用咨询模式问过"业务开发前还需要哪些基础组件"？
- [ ] schema.sql 含所有业务表 DDL（小写下划线、bigint 主键自增、utf8mb4）？

### 3.4 端 UI 设计和基础组件

<img src="imgs/aicent-11-fe-base-ui/cover.jpg" style="display: block; width: 800px;" alt="替换文字">

**完整内容  →   [AI编程方法 11：工程搭建 - 前端UI设计和基础组件]({% post_url 2026-07-01-aicent-11-fe-base-ui %})**

第十篇补齐后端基础设施，本篇补齐前端 UI 层——设计系统、侧边栏、布局、五个公共组件。**核心价值是让 AI 当设计师、用人话指挥 CSS。**

**核心打法**：设计指令五要素（不让 AI 猜）→ CSS 变量化统一视觉（一处定义全局生效）→ 一条指令批量封装组件（业务页变填配置）→ 看→说→改分类分轮打磨 → AI 出初稿、人做审美判断。

#### (1) 设计指令五要素：让 AI 当设计师

"帮我设计一个好看的界面"是最常见的废指令——等于让 AI 猜。01 篇"思考质量决定输出质量"在 UI 场景同样适用：输入越具体，产出越接近设计师水准。

| 要素 | 给什么 | 缺失后果 |
|------|--------|---------|
| 上下文 | 产品是什么（AI Agent 平台/技术团队内部用） | 不知是管理后台还是营销页 |
| 约束 | 用户是谁（开发者/技术管理者） | 信息密度无参照，可能误选花哨风格 |
| 方向 | 风格方向（浅底 + 科技感点缀，侧边栏深色） | 深浅乱搭、长时间看眼累 |
| 细节 | 色调偏好（蓝紫主色/青色辅色） | 配色无体系、状态色与品牌色冲突 |
| 锚点 | 参考标杆（Linear、Supabase） | 退回通用审美、千篇一律 |

**操作要点**：先用咨询模式给上下文，再下设计指令；明确产物形态（一套 CSS 变量）；拿到结果 review 大方向，不逐个色值较真；第一版需要调整是协作过程，不是 AI 能力不够。

#### (2) 设计系统 CSS 变量化：一处定义全局生效

设计系统如果不变成 CSS 变量，就是一份"配色说明书"——改一处要全局搜索替换。变量化后，换主色只动一处，侧边栏/布局/组件自动跟随，这也是后续"看→说→改"能快速生效的前提（你的指令改的是变量，不是散落的硬编码）。

| 维度 | 变量族 | 用途 |
|------|--------|------|
| 颜色-主色 | `--color-primary`（蓝紫系） | 按钮渐变、选中态竖线、品牌文字 |
| 颜色-辅色 | `--color-secondary`（青/薄荷绿） | 数据指示、状态点缀 |
| 颜色-背景 | `--color-bg` / `-secondary` / `-dark` | 浅底主背景/浅灰内容区/深色侧边栏 |
| 颜色-文字 | `--color-text` / `-secondary` / `-light` | 主/辅助/占位三级层次 |
| 形态 | `--radius-*` / `--shadow-*` | 统一圆角与阴影语言，制造层次感 |
| 动效 | `--transition-*` | hover/弹窗/折叠统一过渡时长曲线 |

**操作要点**：禁止任何位置硬编码 `#xxxxxx`；变量按主色/辅色/背景/文字族命名（禁 `--c1` `--blue`）；review 三件事——搭配刺不刺眼、对比度够不够、调性对不对；大方向通过即用，细节留给"看→说→改"。

#### (3) 一条指令批量封装五个公共组件

后端工程师做前端最怕不是写不出来，是写得慢——每页重复处理表格分页、弹窗开关、loading、删除确认。封装后业务页只关心"展示什么、调什么接口"。一条指令批量生成而非逐个写，是因为五个组件的接口可一次性描述清楚，统一产出统一验收，远快于来回五次。

| 组件 | 类型 | 解决什么 |
|------|------|---------|
| `HifyTable.vue` | 通用列表 | 自动管理 loading/分页/请求，暴露 `refresh()`，业务页只配列和接口 |
| `HifyFormDialog.vue` | 通用表单弹窗 | 新增/编辑同一套，关闭自动重置，提交触发 `submit` 事件 |
| `useConfirm.ts` | 删除确认 composable | 一行完成"确认→调接口→提示成功"全流程 |
| `useRequest.ts` | 请求状态 composable | 自动管理 `{ data, loading, error, execute }` 三态 |
| `notify.ts` | 统一通知封装 | 导出三方法，改一处全局生效 |

**验收三件事**：TypeScript 类型（含泛型）对不对、接口合不合理、和 Element Plus 集成有没有问题。**不逐行 review 实现细节——前端组件封装不是后端工程师的核心能力区，能用就行，有问题后面调。**

#### (4) 看→说→改：UI 迭代的真实节奏

UI 初稿永远不完美——这不是 AI 能力不够，而是 UI 细节本就需要人来判断。真正的价值不在你会不会写 CSS，在你能看着屏幕说清"哪里不对、应该怎样"。ProviderList 页面跑了五轮：

| 轮次 | 维度 | 典型指令（落到具体数值与元素） |
|------|------|-------------------------------|
| 1 | 布局 | 间距 24→16px；行高 52px；操作列两按钮间留 8px |
| 2 | 色调 | 启用用 `success` 绿；禁用用 `info` 灰——禁用不是错误 |
| 3 | 表单弹窗 | 宽度 600→520px；API Key 改 `password` + 显示切换；label 右对齐 |
| 4 | 响应式 | <1200px 侧边栏自动折叠为窄模式；窄屏隐藏 Base URL 和创建时间列 |
| 5 | 整体微调 | 表头换浅灰；分页器居右加分割线；行 hover 变色；编辑/删除按钮改 `text` 类型蓝/红 |

**操作要点**：每轮只针对一类问题，不堆叠五件事；指令具体到像素与元素（"小一点""紧凑些"等于没说）；必须实际启动看效果，很多 UI 问题（间距、对齐、hover）只有眼睛能发现；UI 打磨本质上就是来回五六次的过程，期望初稿完美只会让双方挫败。

#### (5) AI 出初稿、人做审美判断（三层分工第二层）

以"禁用状态用红还是灰"为例：AI 第一版把禁用设成红色（语义上判为负面状态），但管理后台里禁用是日常运营的正常状态（某 Provider 暂时下线），用红色 danger 让人误以为出错了，用灰色 info 才符合中性语义。**这种场景化语义判断 AI 给不了，只有看着实际场景的人能给。**

| 职责 | AI 负责 | 人负责 |
|------|---------|--------|
| 设计系统初稿 | 输出主色/色阶/圆角/阴影/动效 | review 搭配是否刺眼、对比度、调性 |
| 组件实现 | 生成 TypeScript 类型与 Element Plus 集成 | 验证类型/接口/能不能用 |
| UI 细节 | 把"间距 16px、禁用用灰"翻译成 CSS | 看屏幕判断"哪里不对、应该怎样" |
| 翻译 | 自然语言判断 → 代码改 | 用人话描述判断，不需要写 CSS |

**操作要点**：审美判断要敢下结论（禁用就是灰不是红、间距就是 16 不是 24）；不需要会写 CSS，只需要能说清"哪里不对"；判断用人话表达即可。**这种判断力可训练——做几个页面下来对 UI 的感觉会明显变好，后面每个前端页面都复用同一套节奏。**

#### (1) 核心心法：工程搭建 = 喂约束的循环

**一句话**：搭建工程的本质不是写代码，是把团队约束一点点喂给 AI——每跑偏一次就堵一个口子（SDD 闭环），CLAUDE.md 就这样"长厚"。

整条链路是一个循环：**对齐 → 拆解 → 验证 → 沉淀**。每个锚点都把"对的"固化下来，问题只在锚点之后的范围里找。

#### (2) 六条方法论：每条都在堵一个口子

| 方法论 | 解决的痛点 | 一句话本质 |
|--------|----------|-----------|
| 认知对齐先于执行 | AI 不知约束，按通用模板生成 | 让 AI 复述 CLAUDE.md，模糊点就是规范漏洞 |
| 拆到"一次能验证" | 任务太大无法 review | 每步对应一条可跑命令 |
| 边界约束（只做+不做） | AI 倾向生成"完整方案" | 一行"不需要 Java 代码"省掉一堆返工 |
| 异常处理给完整上下文 | AI 凭最后一行报错盲猜 | 完整日志 + 相关代码 + 项目结构 |
| SDD 闭环 | 改完就忘、同类问题反复 | 跑偏即补 CLAUDE.md |
| UI "位置+问题+期望" | 后端看出问题但说不清 | 判断力归人、CSS 归 AI |

#### (3) 颗粒度判据：一次能验证

| 颗粒度 | 特征 | 后果 |
|--------|------|------|
| 太粗 | 一个任务跨骨架+组件+业务 | review 困难、问题混在大输出里 |
| 太细 | 每个类单独成任务 | 频繁切换上下文 |
| **合适** | **跑一条命令就能确认对错** | **锚点串行、定位快** |

#### (4) 异常处理：差 vs 好

**反例**："报错了帮我修复"——AI 凭空猜，准确率几乎为零。

**正例三要素**：完整错误日志 + 相关代码/项目结构 + 明确问"根本原因"。

| 失真类型 | 表现 | 根因 |
|---------|------|------|
| 表面修复 | 报错消失、换位置又报 | 只抹掉报错那一行 |
| 方向跑偏 | 修改和架构无关 | 缺模块依赖/版本/扫描路径约束 |

**止损信号**：两三轮没收敛 → 停下来自己看，往往是 Bean 注入失败、`@MapperScan` 漏写、依赖版本冲突。

#### (5) UI 句式：位置 + 问题 + 期望

**核心**：不需要懂 CSS，只需要判断力。AI 负责把判断翻译成 CSS/组件 props。

| 示例 | 位置 | 问题 | 期望 |
|------|------|------|------|
| 删除按钮 | 操作列 | danger 类型带背景 | text 类型红色文字 |
| 侧边栏选中态 | 菜单项 | 不明显 | 左侧 3px 蓝紫竖线 |
| 弹窗 loading | 提交按钮 | 无反馈 | 加转圈动画 |

**颜色语义要点**：禁用是正常业务状态用灰色，不是错误——别让 Element Plus 默认渲染成 danger 红色。

#### (6) 验收锚点速查表

| 锚点 | 验证命令 | 通过标志 |
|------|---------|---------|
| Maven 骨架 | `mvn compile` | BUILD SUCCESS |

#### (7) 四个最终产出物

| 产出 | 内容 |
|------|------|
| 前后端骨架 | Spring Boot 多模块 + Vue 工程，健康检查联通 |
| 后端基础三档 | Result/PageResult、ErrorCode/BizException、GlobalExceptionHandler + MyBatis-Plus/Redis 配置 |
| 前端品牌界面 | CSS 变量设计系统 + 深色侧边栏 + 渐变品牌名 |
| 前端公共组件 | HifyTable / HifyFormDialog / useConfirm / useRequest / notify |
