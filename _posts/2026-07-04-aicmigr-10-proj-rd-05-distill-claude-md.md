## 1. 导读地图：本篇怎么读

<img src="imgs/aicmigr-10-proj-rd-05-distill-claude-md/026864d8dbdd82886123d64d7598a042_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

本篇是「了解项目」阶段的第五篇，对应八步心法的 **步 6：让 AI 每次启动都带着项目共识进来**。前四篇让 AI 读了 README 和核心链路、画出了三张俯视图（架构图、模块图、依赖图）、产出了一份接口清单和一份核心数据模型。到上一篇结束时，`docs/` 目录里已经住了五份资产。

但有一件事还没做：**让 AI 在每次启动时就带着这些共识进来**。

本篇就做这件事——把五份资产凝成一份 `CLAUDE.md`，放进项目根目录。下次任何人打开这个项目，Claude Code 启动时会自动读它，AI 一上手就知道"这是什么项目、核心架构是什么、哪些地方不能动"。

这是「了解项目」阶段的关键节点：从"工程师理解了项目"到"AI 也能理解项目"。

两类读者的推荐读法：

- **初学 AI 编程工程师**：建议通读全篇。第一部分建立"CLAUDE.md = 索引 + 常识"的认知骨架；第二部分用 Spring AI Alibaba Admin 把骨架落到真实企业级项目上，把提示词、产出效果、三个常见坑完整走一遍，最后落到"禁区和历史包袱"这两节必须手写的硬核内容。
- **熟练 AI 编程工程师**：可只看第一部分的 Check List 速查；接到一个老项目时，按清单填提示词、按 review 表走一遍、再把禁区/历史包袱两节补齐即可。

## 2. 为什么老项目的 CLAUDE.md 不能凭空写

<img src="imgs/aicmigr-10-proj-rd-05-distill-claude-md/1305d428d6b857e49e8447ab8825411b_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

写 CLAUDE.md 这件事，新项目和老项目完全是两回事。理解了这一点，就理解了本篇为什么要把"提炼 CLAUDE.md"单独拿出来讲。

### 2.1 新项目写 CLAUDE.md：经验就够了

新项目写 CLAUDE.md 不难。比如有个技巧，就是参考阿里巴巴 Java 规范生成 CLAUDE.md 的约束。**因为工程师就是作者，代码是自己写的，规矩是自己定的**，CLAUDE.md 凭经验就能写出来。

### 2.2 老项目写 CLAUDE.md：经验会骗人

老项目不一样。工程师是接手者，项目不是自己写的，很多设计决策背后的原因自己都没搞清楚。这种情况下凭什么写 CLAUDE.md？凭五天前刚读的代码？凭零散的印象？

写出来的东西通常会落入三种陷阱：

| 陷阱 | 表现 | 后果 |
|-----|-----|-----|
| 空洞 | "这是一个 Spring Boot 项目" | AI 等于没读 |
| 错误 | 把猜的当成事实 | AI 被误导，给出错误建议 |
| 遗漏 | 没写出最关键的那几条禁区 | 改造时踩雷 |

### 2.3 正确姿势：从资产里提炼，不是从零写

老项目写 CLAUDE.md 的正确姿势不是从零写，**是从已有资产里提炼**。

这就是为什么系列第 8 篇要画三张图、系列第 9 篇要做两份清单。**这五份资产不只是给自己看的笔记，它们是 CLAUDE.md 的前置条件**。有了它们，CLAUDE.md 才有得提炼；没有它们，CLAUDE.md 就是空中楼阁。

本篇要做的事，其实就是把前两篇的积累，凝成一份 AI 每次启动都能看到的项目常识。

## 3. 方法论：放什么、不放什么、怎么生成

<img src="imgs/aicmigr-10-proj-rd-05-distill-claude-md/1b84b349c804ca3652f6d60aa1d65896_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

第二部分实战之前，先把方法论提炼出来。本节回答四个问题：CLAUDE.md 的定位是什么？放什么、不放什么？怎么让 AI 生成初稿？写完之后怎么 review？

### 3.1 定位：索引 + 常识，不是约束堆

#### (1) 新项目和老项目的定位差异

对于新项目来说，规矩越多，AI 约束得越好，代码质量越高。

而在老项目中，如果手动给 CLAUDE.md 塞太多内容，**就可能会和原先的规范形成冲突**（说实话，不是可能，基本是百分百）。

#### (2) 老项目最常见的写法错误：写得太多

老项目改造中最常见的写法错误是写得太多：

- 把架构图的内容用文字重抄一遍
- 把接口清单全塞进去
- 把数据模型每张表每个字段都列出来

结果 CLAUDE.md 几千行，AI 每次启动都加载一大堆 context，掉进 Dumb Zone（系列第 5 篇讲过的那个东西）。

#### (3) 正确的定位

在老项目中，**CLAUDE.md 的定位是索引 + 常识，而不是大量的约束**：

| 角色 | 职责 |
|-----|-----|
| 索引 | 指向已提炼出来的详细内容（docs/ 下的资产） |
| 常识 | 让 AI 一启动就知道关键信息 |

**超过 300 行就是写多了**。

### 3.2 放什么、不放什么

放进来的东西，按实战经验有这几类：

| 类别 | 写法 | 示例 |
|-----|-----|-----|
| 项目定位 | 一句话说清楚 | "Spring AI Alibaba Admin 是阿里巴巴开源的 Agent 管理平台，提供 Prompt 管理、Dataset、Evaluator、Trace 观测等能力" |
| 核心架构 | 一段话 + 指向 `docs/architecture.svg` 的链接 | 不要把图里的内容文字化重写 |
| 关键模块 | 一个小表，每个模块一句话职责 | 详细依赖关系在 `module-deps.svg` 里 |
| 关键约定 | 硬规则，不展开理由 | "所有 REST 接口响应统一包装 Result"、"数据库字段一律用 snake_case，Java 字段用 camelCase" |
| 怎么跑 | 一句话 + 指向 `docs/` 运行文档的链接 | 系列第 13 篇会单独做 |
| 禁区 | 老项目的灵魂一节 | 第 7 章单独讲 |
| 历史包袱 | 老项目的灵魂另一节 | 第 7 章单独讲 |

后两节是老项目 CLAUDE.md 最重要的部分，下一节单独讲。

不放的东西：

| 不放 | 原因 |
|-----|-----|
| 完整架构细节 | 那是 `architecture.svg` 的事 |
| 完整接口清单 | `api-list.md` 就在 `docs/` 里，不要重复 |
| 完整数据模型 | `data-model.md` 就在 `docs/` 里，不要重复 |
| 通用代码规范 | 阿里 Java 开发手册这种东西不是项目特有的，进 CLAUDE.md 只会稀释重点 |
| 背景故事 | AI 不需要读一篇项目诞生史才能工作 |

#### (1) 核心原则一句话

**CLAUDE.md 里的每一条，要么是"AI 启动就必须知道的常识"，要么是"指向 docs/ 的入口"。不满足这两条的，删掉。**

### 3.3 怎么让 AI 生成初稿

有了 `docs/` 里的五份资产，生成 CLAUDE.md 就简单了。

提示词：

```
读 docs/ 下的所有资产，给我生成一份 CLAUDE.md 初稿。

精简：项目定位、核心架构、关键模块、关键约定、怎么跑，

外加两节空着的：禁区、历史包袱。

架构图、接口清单、数据模型的详细内容不要复制进来，

用链接指向 docs/ 就好。保存到项目根目录的 CLAUDE.md。
```

#### (1) 关键点一：让 AI 基于资产提炼，不是凭空写

"读 `docs/` 下的所有资产"让 AI 基于已有的产出提炼，不是凭空写。

#### (2) 关键点二：用链接而不是复制

"用链接指向 `docs/` 就好"防止 AI 把架构图文字化、把清单复制进来。

#### (3) 关键点三：禁区/历史包袱两节留空——最关键的一招

"两节空着的：禁区、历史包袱"这一条是最关键的一招。**让 AI 留出这两节的位置，但不要让它填**。因为 AI 填不出这两节的真实内容，它会瞎编或者写得很泛。留空让工程师自己填。

### 3.4 写完之后：review 的三个检查点

CLAUDE.md 写完，要 review 一下。三个检查点：

| 检查点 | 通过标准 |
|-------|---------|
| 有没有禁区和历史包袱 | 至少列一两条，后面遇到再补 |
| 是不是太长 | 不超过 300 行，超了就下放到 `docs/` |
| 有没有重复 docs/ | "核心架构"应是引导而非替代（详见第 8 章） |

通过这三个检查，CLAUDE.md 才算定稿。详细实战视角见第 8 章。

## 4. 第一部分 Check List：提炼 CLAUDE.md 速查

<img src="imgs/aicmigr-10-proj-rd-05-distill-claude-md/99a1e188fe6eaf4468d1451ffd72207a_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

本节提供一份可裁剪、可勾选的 Check List，供工程师在提炼 CLAUDE.md 的不同阶段快速查阅。读者可以按项目实际情况裁剪后贴到自己的工作笔记里。

### 4.1 启动前的 Check List

启动 AI 提炼 CLAUDE.md 之前，逐条自检：

- [ ] `docs/` 目录下是否已经集齐五份资产（架构图、模块图、依赖图、接口清单、数据模型）？
- [ ] 是否想清楚 CLAUDE.md 的定位是"索引 + 常识"，而不是"约束堆"？
- [ ] 是否约定好 CLAUDE.md 总长度上限（建议 300 行）？
- [ ] 是否想清楚禁区和历史包袱这两节由人工填、不让 AI 填？

### 4.2 生成初稿的 Check List

提示词和初稿生成环节逐条自检：

- [ ] 提示词里是否明确"读 `docs/` 下的所有资产"，让 AI 基于资产提炼？
- [ ] 是否明确要求"用链接指向 `docs/`，不要把架构图/接口清单/数据模型文字化重写"？
- [ ] 是否要求"禁区、历史包袱两节留空"，由人工填？
- [ ] 第一版出来后，是否检查"核心架构"一节没有被 architecture.svg 的内容文字化？
- [ ] "关键约定"一节是否写的是项目特有的硬规则，而不是通用开发手册？

### 4.3 禁区和历史包袱的 Check List

这两节手写之后，逐条自检：

- [ ] 禁区是否覆盖了"哪些代码动不得、哪些字段有对接方依赖、哪些配置改了会出事"三类？
- [ ] 历史包袱是否覆盖了"看起来奇怪但有原因"的代码/结构？
- [ ] 每一条是否都给了"为什么不能动/为什么这么写"的简短说明（让 AI 自己推断规则）？
- [ ] 这两节写完后，是否回头检查"AI 是否能在改造时自动避开"？

### 4.4 review 收尾的 Check List

CLAUDE.md 定稿前，逐条自检：

- [ ] 是否有禁区和历史包袱两节（没有就是漏了）？
- [ ] 总长度是否在 300 行以内（超了就下放）？
- [ ] 是否有把 `docs/` 资产内容文字化重写的段落（有就改成"一句话 + 链接"）？
- [ ] 每一条是否满足"AI 启动必须知道的常识"或"指向 docs/ 的入口"两条之一（不满足就删）？

## 5. 实战主线延续：Spring AI Alibaba Admin

<img src="imgs/aicmigr-10-proj-rd-05-distill-claude-md/786fe0baf8d130972fd64cd4ca29814f_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

说完方法论，进入第二部分实战。本篇延续系列前几篇的主线项目——**Spring AI Alibaba Admin**，所有出现的提示词、产出的 CLAUDE.md、写出的禁区/历史包袱都围绕它展开。

### 5.1 主线延续：从五份资产到 CLAUDE.md

前四篇走完后，Spring AI Alibaba Admin 的 `docs/` 目录已经住了五份资产：

```
docs/
├── architecture.svg          ← 架构图（系列第 8 篇产出）
├── module-deps.svg           ← 模块图（系列第 8 篇产出）
├── external-deps.svg         ← 依赖图（系列第 8 篇产出）

```

**俯视图和清单告诉工程师和 AI"项目长什么样"，但还没告诉 AI"每次启动就要带着这些共识进来"**，而这就是本篇要做的事。

### 5.2 本篇产出物：项目根目录的 CLAUDE.md

提炼完本篇会拿到一份新的资产：**一份住进项目根目录的 `CLAUDE.md`**。这份文件被 Claude Code 启动时自动读取，让 AI 每次都能带着"项目是什么、核心架构是什么、哪些地方不能动"的共识进来。

### 5.3 本篇只读资产，不跑项目

延续前几篇的约定：本篇仍然只让 AI 读 `docs/` 下的资产、产出 `CLAUDE.md`。把"让项目跑起来"这件事留到系列第 13 篇（构建护栏阶段）一次性讲透。

## 6. 让 AI 生成 CLAUDE.md 初稿

把方法论的提示词套到 Spring AI Alibaba Admin 上，让 AI 一次性产出 CLAUDE.md 初稿。本节给出提示词（code block 原文不改写）、三个关键点、产出效果、三类常见坑。读者可以直接复制提示词到自己的项目运行。

### 6.1 提示词

提示词：

```
读 docs/ 下的所有资产，给我生成一份 CLAUDE.md 初稿。

精简：项目定位、核心架构、关键模块、关键约定、怎么跑，

外加两节空着的：禁区、历史包袱。

架构图、接口清单、数据模型的详细内容不要复制进来，

用链接指向 docs/ 就好。保存到项目根目录的 CLAUDE.md。
```

### 6.2 三个关键点

#### (1) 关键点一：让 AI 基于资产提炼

"读 `docs/` 下的所有资产"让 AI 基于已有的产出提炼，不是凭空写。**这一句话决定了 CLAUDE.md 的可信度**。

#### (2) 关键点二：用链接而不是复制

"用链接指向 `docs/` 就好"防止 AI 把架构图文字化、把清单复制进来。**这一句话决定了 CLAUDE.md 的精简度**。

#### (3) 关键点三：禁区/历史包袱两节留空

"两节空着的：禁区、历史包袱"是最关键的一招。让 AI 留出这两节的位置，但不要让它填。**因为 AI 填不出这两节的真实内容，它会瞎编或者写得很泛**。留空让工程师自己填——这是老项目 CLAUDE.md 区别于新项目的根本所在。

### 6.4 三个常见坑

#### (1) 第一个坑：AI 把架构图文字化

AI 第一次生成会忍不住把 `architecture.svg` 的内容用文字描述一遍，塞进"核心架构"那节。**初稿出来检查一下，如果看到长段的模块描述，直接说"这一段太长了，压成一句话 + 链接"**。

#### (2) 第二个坑：关键约定写得太通用

"关键约定"这一节 AI 容易写通用的（比如"代码要有注释"），不写项目特有的。**如果出现这种情况，直接让 AI 从代码风格里反推项目真实的硬规则**，而不是抄通用开发手册。

#### (3) 第三个坑：AI 自作主张填禁区/历史包袱

AI 可能贴心地把两节空着的也填上。**这时候直接让它恢复空白**："禁区和历史包袱这两节留给我自己写，别帮我猜"。

## 7. 禁区和历史包袱：两节必须手写

<img src="imgs/aicmigr-10-proj-rd-05-distill-claude-md/48719a3bde28aa2f16568214597a527b_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

AI 生成的 CLAUDE.md 里一定没有真正有价值的禁区和历史包袱，**因为这两节的信息不在代码里、不在 `docs/` 里、只在工程师脑子里**。这两节必须手写，它们是老项目 CLAUDE.md 区别于新项目的根本所在，是本篇最硬的部分。

### 7.1 禁区写什么

禁区写什么：**哪些代码动不得、哪些字段有对接方依赖、哪些配置改了会出事**。

示例（不是真实的内容，每个项目的禁区是不一样的）：

```
## 禁区

- `server-core/PromptEntity` 的 `external_key` 字段：某 SDK 客户依赖
  此字段做缓存键，删掉或改名会导致该 SDK 直接报错。不要重构。

- `application.yml` 里 `nacos.server-addr` 的默认值：部分企业用户依赖
  这个默认值做灰度发布，改动需要发公告。

- `POST /api/prompts/search` 接口路径：曾经公开给过社区，更改路径
  会造成外部调用失败。新增同义接口可以，删除原接口不行。
```

### 7.2 历史包袱写什么

历史包袱写什么：**项目里看起来奇怪但有原因的东西**。

示例：

```
## 历史包袱

- `server-core/Dataset` 和 `DatasetItem` 的表结构看起来冗余，是 2024 年某个
  实验性功能留下的。功能已下线但数据保留中，勿删。

- 前端 `PromptTemplate.vue` 用的是 Vue 不是 React，是早期遗留。
  整个 admin 其他地方都用 React，这里例外。不要"顺手重构"统一。

- `LegacyEvaluatorAdapter` 这个类是 v0.x 时代的兼容层，看起来很乱，
  是因为要同时支持三种老 API。v1.0 之后新代码一律走 EvaluatorV1。
```

### 7.3 为什么这两节价值百倍

#### (1) 每一条都是 AI 永远猜不出来的

看出来了吗？**这两节每一条都是 AI 永远猜不出来的东西**。只有接手者和项目原作者聊过、或者自己踩过坑之后才知道。但这两节每一条都价值百倍：

- AI 改造时一旦看到"禁区"里某条，**会自动避开**
- 有了"历史包袱"的说明，**AI 不会"好心"把 Vue 组件改成 React**

#### (2) 老项目 CLAUDE.md 写得好不好，就看这两节

老项目 CLAUDE.md 写得好不好，就看这两节写得深不深。其他内容 AI 都能帮工程师生成，**这两节必须自己投入时间**。

#### (3) 列不出就是信号：理解还不够深

如果工程师现在还列不出几条禁区、几条历史包袱，那是一个信号：**对这个项目的理解还不够深**。再回去挖一挖，和老同事聊一聊，翻翻 git log 里那些奇怪的 commit。**花时间填这两节，是给工程师自己、给团队、给 AI 一起做的最值得的一件事**。

## 8. review 的三个检查点（实战视角）

<img src="imgs/aicmigr-10-proj-rd-05-distill-claude-md/56673fe0824c8929b5b67f21af1f47d5_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

CLAUDE.md 写完，要 review 一下。三个检查点。

### 8.1 检查点一：有没有禁区和历史包袱

没有就是漏了。**每个老项目都有，只是可能还没挖出来**。至少现在列一两条，后面遇到了再补。

### 8.2 检查点二：是不是太长

超过 300 行就是塞了太多详细内容。**扫一下看哪些段落可以下放到 `docs/`，CLAUDE.md 里只保留一句话 + 链接**。

### 8.3 检查点三：有没有重复 docs/

如果 CLAUDE.md 里把 `architecture.svg` 的内容用文字重写了一遍，**那是在复制，不是在索引**。

#### (1) 引导 vs 替代

工程师遇到"核心架构"这一节应该看到的是：

| 写法 | 类型 | 是否合格 |
|-----|-----|---------|
| 架构分为前端、后端、数据库三层，详见 `docs/architecture.svg` | 引导 | 合格 |
| 把架构图的每个模块都文字化描述一遍 | 替代 | 不合格 |

通过这三个检查，CLAUDE.md 才算定稿。

## 9. 与后续篇目的衔接

<img src="imgs/aicmigr-10-proj-rd-05-distill-claude-md/bcc5603c10b711fafa89a4b941ca3e6d_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

CLAUDE.md 定稿之后，项目的"脑图"基本成型。但还有一类东西没落地——这是下一篇要解决的问题。

### 9.1 本篇结束时的资产清单

写完本篇，`docs/` 里有五份资产，**项目根目录多了一份 CLAUDE.md**。这是老项目的"脑图"基本成型的时刻。

```
项目根/
├── CLAUDE.md                  ← 本篇产出（启动时被 Claude Code 自动读取）
└── docs/
    ├── architecture.svg       ← 架构图
    ├── module-deps.svg        ← 模块图
    ├── external-deps.svg      ← 依赖图
    ├── api-list.md            ← 接口清单
    └── data-model.md + data-model-er.svg  ← 数据模型
```

### 9.2 CLAUDE.md 在后续每一篇的作用

CLAUDE.md 不会摆在根目录吃灰，**它是后面每一篇改造工作的 AI 入口**：

| 后续篇目 | CLAUDE.md 的作用 |
|---------|----------------|
| 系列第 13 篇（构建护栏） | AI 启动时读 CLAUDE.md，知道哪些地方是禁区，建测试时自动避开 |
| 系列第 14 篇（建护栏） | AI 读 CLAUDE.md，知道哪些接口要加集成测试、哪些字段不能改 |
| 第四部分（做需求改造） | AI 读 CLAUDE.md，知道改造的硬约束，避免踩雷 |

### 9.3 还差一类资产：可复用的操作流程

但还有一类东西没落地：**反复要做的操作流程**。比如：

- 改造前要跑的体检
- PR 前要过的 checklist
- 资产更新后要做的校对

这些不是常识，是操作。放进 CLAUDE.md 会让它臃肿，每次启动都加载没必要，**它们应该住到另一个地方——下一篇要讲的 SKILL**。下一篇讲怎么从老项目里挖出这些重复流程，写成工程师自己的第一个 SKILL。

## 10. 小结与思考

<img src="imgs/aicmigr-10-proj-rd-05-distill-claude-md/325a76385aa21f1d07ba3ffd5095186d_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

### 10.1 小结

本篇做了一件事：**把 `docs/` 里的五份资产凝成一份 CLAUDE.md，放进项目根目录**。

#### (1) 核心原则：索引 + 常识

CLAUDE.md 是索引 + 常识，不是 `docs/` 的复制粘贴。**里面放项目定位、核心架构、关键模块、关键约定、怎么跑、禁区、历史包袱**。不放完整架构、完整接口清单、完整数据模型、通用代码规范、背景故事。

#### (2) 生成路径：AI 写前五节，人工写后两节

前 5 节让 AI 基于 `docs/` 自动生成，**禁区和历史包袱这两节自己手写**。AI 写不出这两节的真实内容，必须工程师自己投入时间。

#### (3) 区别根本：禁区 + 历史包袱

老项目 CLAUDE.md 区别于新项目的根本，就在"禁区"和"历史包袱"这两节。**写得深不深，决定了这份 CLAUDE.md 有没有用**。

#### (4) 脑图成型

写完本篇，`docs/` 里有五份资产，项目根目录多了一份 CLAUDE.md。**这是老项目的"脑图"基本成型的时刻**。

### 10.2 思考

#### (1) 关于禁区/历史包袱的自检

读者手上的项目如果现在让写 CLAUDE.md 的"禁区"和"历史包袱"两节，能列出几条？**列不出的原因是真的没有这些坑，还是自己还没搞清楚？**

如果是后者，这些信息目前存在哪里——同事脑子里？Jira 的某个老 ticket 里？还是没人知道？

#### (2) 关于团队现状的检验

想想团队正在运行的老项目，有没有一个类似 CLAUDE.md 的东西（比如项目的 README 或者内部 wiki）？**如果有，里面写的是"索引 + 常识"，还是变成了什么"都塞的大杂烩"？**

如果是后者，问题出在哪？
