## 1. 测试部署

### 1.1 质量保证体系

<img src="imgs/aicent-27-quality-assurance/cover.jpg" style="display: block; width: 800px;" alt="替换文字">

**完整内容: [AI编程方法 27：测试部署 - 质量保证体系]({% post_url 2026-07-02-aicent-27-quality-assurance %})**

#### (1) 一句话讲清楚

> 质量靠体系，不靠 AI——80-90% 代码可放手给 AI，前提是知道剩下的 10-20% 在哪、自己把关。错方向比错代码危险。

#### (2) 两个命题的对立

| 命题 | 现实性 |
|---|---|
| AI 帮你保证质量 | 不现实。AI 写代码专注功能实现，不质疑安全模型、不想到权限漏洞、不考虑并发一致性 |
| 体系保证质量，AI 协助执行 | 现实。工程师识别核心链路、划定测试边界、制定 review 策略，AI 在边界里高效执行 |

#### (3) 质量四层模型

| 层次 | 目标 | 谁主导 | 何时介入 |
|---|---|---|---|
| 核心链路识别 | 判断哪些是核心路径、风险在哪 | 工程师判断，AI 辅助分析 | 架构期，写进 CLAUDE.md |
| 单元测试 | 覆盖纯逻辑，不依赖外部 | 工程师划定边界，AI 执行 | 编码期 |
| 集成测试 | 端到端验证，mock 外部 API 保留真实 DB | 工程师定 mock 策略，AI 执行 | 编码后期至联调期 |
| 混沌测试 | 故障注入验证韧性 | 工程师设计场景 | 维护期，但架构期就要预留韧性设计 |

前一层做错了，后一层再努力都是在错误的地方使劲。

#### (4) 代码分工判断

| 维度 | 适合给 AI | 不适合给 AI |
|---|---|---|
| 特征 | 有规范、有模式、有标准答案 | 需要深度判断、没有标准答案 |
| 典型 | CRUD、适配器、DTO、工具类、测试代码 | 架构设计、数据模型取舍、安全边界、性能瓶颈核心逻辑 |
| 风险 | 错代码会被测试发现 | 错方向表面看不出，一路堆代码 |

#### (5) 单元测试判断三问

| 问题 | 判断 |
|---|---|
| 有没有 IO（DB/HTTP/Redis/文件）？ | 有 → 走集成测试 |
| 去掉外部依赖，剩下的逻辑有复杂度吗？ | 没有 → 不值得测；有 → 可单测 |
| 改动频率高？出错影响大？ | 是 → 优先做；否 → 优先级放低 |

AI 偏逻辑完备、倾向覆盖所有路径，必须用规范把"不值得测"的范围写死，避免在错误的地方写大量无价值测试。

#### (6) 双模型 Review

| 维度 | 说明 |
|---|---|
| 为什么 | 两个模型训练方式不同、盲点不同；写代码的视角 ≠ 找问题的视角 |
| 提示词维度 | 安全性 / 静默失败 / 边界条件 / 性能隐患 |
| 必加句 | "不需要夸代码写得好"——避免模型讨好倾向先说一堆好话 |
| Hify 实例 | ChatGPT 找出 4 个高危：越权访问他人会话 / SQL 取最旧消息 / MCP 静默返回 mock / Redis read-modify-write 无锁 |

代码能跑、功能正常，但安全有漏洞、并发会丢数据——单看 Claude Code 写的代码本身看不出。

#### (7) 测试驱动修复：红绿循环

| 节奏 | 锁住什么 |
|---|---|
| 错误做法 | 修完 bug 再补测试——测试永远通过，锁不住任何东西 |
| 正确做法 | 先写失败测试（红）→ 修 bug → 测试变绿 |
| Hify 实例 | SQL `ORDER BY created_at ASC LIMIT` 取最旧消息，先写测试暴露，再改 SQL |

红 = 复现 bug，绿 = 修复证据。

#### (8) 五步流水线 + 四样产出物

| 步骤 | 核心动作 | 产出物 |
|---|---|---|
| Step 0 | AI 读完项目输出核心链路清单与风险地图 | 核心链路地图 + 风险地图（写入 CLAUDE.md） |
| Step 1 | 基于地图生成项目专属单测规范 | 单测规范（写入 CLAUDE.md） |
| Step 2 | 对照规范输出值得/不值得测清单 | 待测模块清单（27 个，17 值得 / 10 不值得） |
| Step 3 | 单测实操，跑通后固化 | 单测 SKILL（`/单测`） |
| Step 4 | 集成测试三场景递进，跑通后固化 | 集成测试 SKILL（`/集成测试`） |

四样产出物层层递进：**方法论 → CLAUDE.md 共识 → 两个 SKILL 一键自动化**。


### 1.2 容器化

<img src="imgs/aicent-28-containerize/cover.jpg" style="display: block; width: 800px;" alt="替换文字">

**完整内容: [AI编程方法 28：测试部署 - 容器化]({% post_url 2026-07-02-aicent-28-containerize %})**

#### (1) 一句话讲清楚

> 交付 = 把"只能在你机器上跑"变成"任何环境都能部署"；AI 是协作者不是代码生成器——咨询先行、执行前读上下文、基于实际做决策。

#### (2) 三种交付形态对比

| 形态 | 适用环境 | 运维要求 | 典型产物 |
|---|---|---|---|
| tar.gz 本地 | 客户内网、无 Docker | 低 | jar + dist + 启停脚本 + 配置模板 |
| Docker | 技术型团队 | 中 | Dockerfile + docker-compose + .env |
| K8s | 追求高可用 | 高 | Deployment/Service/ConfigMap/Secret |

投入产出比：**Docker 最高**。推进顺序：Docker → tar.gz → K8s（复用镜像，增量成本最低）。

#### (3) 咨询模式三段式

| 阶段 | 做什么 | 价值 |
|---|---|---|
| 讨论形态 | 让 AI 分析应支持哪些部署形态 | 写代码前对齐目标，避免返工 |
| 发现遗漏 | 让 AI 主动提确认问题 | AI 见过大量项目，能补盲区 |
| 动手执行 | 边界确定后再生成配置 | 基于明确上下文，输出更准 |

操作要领：提示词结尾加"不要写代码，先给我分析"；要求 AI"提出你需要确认的问题"。成功标志：**AI 至少提出一个工程师没想到的必要确认项**（本例是健康检查接口）。

#### (4) 通用前置件：健康检查接口

一个 `GET /api/v1/health` 同时服务三态，是典型的"高杠杆前置件"：

| 部署形态 | 用途 |
|---|---|
| tar.gz | 启动后 curl 确认就绪 |
| Docker | `HEALTHCHECK` 指令 |
| K8s | livenessProbe / readinessProbe |

先识别共用件、优先补齐，避免每种形态做到一半才发现没东西可探。

#### (5) 配置隔离三条铁律

| 铁律 | 处理方式 |
|---|---|
| 敏感配置不进代码/镜像/git | `.env` 文件 / K8s Secret |
| 外部服务连接信息不硬编码 | 环境变量注入 `${ENV_VAR:default}` |
| 镜像保持环境无关 | 同一镜像跑开发/测试/生产 |

外部 application.yml 优先级高于内置，不重新打包即可调参。

#### (6) 三个易踩的工程坑

| 坑 | 后果 | 解决 |
|---|---|---|
| Nginx 未关 `proxy_buffering` | SSE 事件攒批发出，打字机效果消失 | `proxy_buffering off` |
| 容器内用 localhost 连外部服务 | localhost 指容器自身，连接失败 | 用宿主机真实 IP 或服务名 |
| Secret 直接 `kubectl create` | CI/CD 第二次执行报错 | `--dry-run=client -o yaml \| apply -f -` 幂等更新 |

#### (7) AI 协作的关键节奏

| 步骤 | AI 做什么 | 对应方法论 |
|---|---|---|
| 先讨论 | 主动提确认问题 | 咨询模式 |
| 再扫描 | 读已有文件（HealthController、pom、vite.config 等） | 执行前读上下文 |
| 提供选项 | 发现不一致不擅自改全局 | 把决策留给工程师 |
| 最小改动 | 只改必要部分（不动全局 Result 类） | 影响面可控 |

工程师全程只做三件事：**用自然语言描述需求、检查 AI 生成结果的关键点、按顺序执行命令**。

### 1.3 监控设施和拍错

<img src="imgs/aicent-29-observability/cover.jpg" style="display: block; width: 800px;" alt="替换文字">

**完整内容: [AI编程方法 29：测试部署 - 监控设施和排错]({% post_url 2026-07-02-aicent-29-observability %})**

#### (1) 一句话讲清楚

> 可观测性 = 让黑盒变透明：日志定位单次请求、指标看整体趋势、健康检查决定流量去留；动手前先用咨询模式让 AI 暴露架构决策。

#### (2) 为什么要做：从猜到定位

| 状态 | "客服回复慢"的排查路径 | 耗时 |
|---|---|---|
| 黑盒 | 靠猜，挨个排查 LLM？数据库？熔断？Redis？ | 半天 |
| 透明 | Grafana 看 P95 趋势 → traceId 搜日志 → `durationMs=3768 providerId=2` | 两分钟 |

#### (3) 三件套各管什么

| 设施 | 回答的问题 | 关键设计 |
|---|---|---|
| 结构化日志 | 某次请求发生了什么 | JSON + traceId 贯穿（含跨线程 MDC 恢复） |
| 指标埋点 | 系统整体状况如何 | Actuator + Micrometer，统一前缀 `hify_` |
| 健康检查 | 流量该不该进来 | DOWN 返回 503，K8s 自动摘流量 |

#### (4) 四条方法论模式

| 模式 | 怎么做 | 价值 |
|---|---|---|
| 咨询先行 | 提示词禁止写代码，问"做什么/不做什么/哪些决策趁早定" | AI 主动读码发现你没说的问题 |
| 架构决策前置 | "今天加一行、明天改一片"的决策写进 CLAUDE.md | 避免后期伤筋动骨 |
| AI 自检闭环 | 产出后要求对照需求逐条自检 | 执行前发现矛盾、执行后发现遗漏 |
| 业务驱动分级 | 先分析业务背景再给阈值 | 内部工具阈值比 C 端宽松 |

#### (5) 三个趁早定的架构决策

| 决策 | 不前置的代价 |
|---|---|
| Actuator 端口隔离（业务 8080、监控 8081） | 后续要改 K8s Service 和 Nginx |
| traceId 提前进 MDC | 接 OTel 时要改所有日志打点 |
| 日志字段规范写进 CLAUDE.md | 字段进生产后所有 grep/告警规则依赖它 |

#### (6) AI 自检的两个典型时机

| 时机 | 发现的问题 | 不发现的后果 |
|---|---|---|
| 执行前 | 指标名与实际埋点不符（`_seconds` vs `_ms`） | Grafana 面板空白 |
| 执行后 | DistributionSummary 默认不生成 `_bucket` | `histogram_quantile()` 查询空白 |

两个问题提示词都没提，AI 一个在执行前发现、一个在执行后发现——没有这两步，导入的 Dashboard 全是空白。

#### (7) 业务驱动的告警分级

| 级别 | 典型触发 | 响应时限 |
|---|---|---|
| P0 | scrape 失败 / health 503 / 所有 provider 熔断 | 立即 |
| P1 | LLM 错误率 >30% / 单 provider 熔断 10min | 30 分钟 |
| P2 | LLM P95 >15s / MCP 失败率 >20% | 次日 |

"不建议告警"清单与正面清单同等重要：HALF_OPEN 是正常恢复流程，告警只会让人误判；告警太多会产生疲劳，让人忽略真正重要的告警。

#### (8) 可观测性建设 Check List

| 阶段 | 要做的事 | 注意点 |
|---|---|---|
| 方案讨论 | 先咨询再动手 | 让 AI 暴露架构决策 |
| 日志规范 | JSON + traceId 贯穿 | 字段名写进 CLAUDE.md |
| 日志跨线程 | 子线程恢复父线程 MDC | 否则 traceId 在异步链路丢失 |
| 指标埋点 | Actuator + Micrometer | 自建实例需手动注册 Gauge |
| 指标暴露 | Actuator 独立端口（8081） | 抓取端口要和实际配置一致 |
| 健康检查 | DOWN 返回 503 | K8s 探针看状态码，200 会让流量打进故障 Pod |
| Dashboard | 给指标列表让 AI 生成 JSON | 指标名以实际埋点为准 |
| 告警策略 | 按 P0/P1/P2 分级 | 附"不建议告警"清单 |
| 验收 | 真实产生数据后看大盘 | 用 traceId 验日志链 |


### 1.4 流程回顾

<img src="imgs/aicent-30-test-deployment-recap/cover.jpg" style="display: block; width: 800px;" alt="替换文字">

**完整内容: [AI编程方法 30：测试部署 - 流程回顾]({% post_url 2026-07-02-aicent-30-test-deployment-recap %})**

#### (1) 一句话讲清楚

> 测试部署 = 质量体系 → 可交付形态 → 可观测性，三件事层层递进；用"咨询先行 + 先读代码 + 执行后自检 + 红绿循环"四条动作把每一步做扎实。

#### (2) 三件事为什么有先后

| 顺序 | 解决什么 | 没有它的后果 |
|---|---|---|
| 质量体系 | 怎么知道代码是对的 | 交付产物不可信 |
| 可交付形态 | 怎么把能跑的代码变成能部署的产物 | 质量再好也只在开发机上跑 |
| 可观测性 | 上线后怎么知道它在正常跑 | 系统是黑盒，出问题无法定位 |

#### (3) 四条核心动作

| 动作 | 怎么做 | 价值 |
|---|---|---|
| 先咨询，再执行 | 高风险任务用"不要写代码，先给分析"结尾 | Claude Code 主动暴露前置依赖（如健康检查接口） |
| 先读代码再动手 | 执行提示词以"先扫描项目"开头 | 凭真实代码修正假设，避免"能跑但有缺陷" |
| 执行后自检 | 关键产出对照需求逐项核对 | 发现提示词没覆盖的问题（如面板会空白） |
| 先写失败测试，再修 bug | 红 → 修 → 绿，顺序固定 | 红 = 复现 bug，绿 = 修复证据 |

#### (4) 咨询模式 vs 执行模式

| 模式 | Claude Code 的行为 | 用途 |
|---|---|---|
| 咨询 | 追问、提醒、主动暴露前置依赖 | 高风险任务前置（测试计划、架构、部署形态） |
| 执行 | 只完成指令，不主动暴露遗漏 | 落地代码 |

混用会丢掉咨询阶段的信息——两种节奏分开用，效果差很多。

#### (5) 必须显式给出的约束

不说，Claude Code 不会自己猜：

| 约束 | 不说的后果 |
|---|---|
| SSE 流式 → Nginx `proxy_buffering off` | 事件攒批一次性发出，打字机效果消失 |
| LLM 慢调用 → 超时时间要够长 | 慢响应被切断 |
| 三个架构决策趁早定 | Actuator 独立端口 / traceId 进 MDC / 日志字段规范，改晚了伤筋动骨 |

#### (6) 四样可迁移的交付物

| 交付物 | 内容 |
|---|---|
| 核心链路地图 | 写进 CLAUDE.md，标明不能动的关键路径 |
| 两个测试 SKILL | 单测 / 集成测试的可复用流程 |
| 三种部署形态 | 本地 tar / Docker / K8s，三态分离 |
| Grafana 大盘 + 告警策略 | JVM/HTTP/业务三层大盘 + 含"不建议告警"清单 |


## 2. 复盘进阶

### 2.1 项目回顾

<img src="imgs/aicent-31-project-retro-advanced/cover.jpg" style="display: block; width: 800px;" alt="替换文字">

**完整内容: [AI编程方法 31：复盘进阶 - 项目回顾]({% post_url 2026-07-02-aicent-31-project-retro-advanced %})**

#### (1) 一句话讲清楚

> SDD = 先写规范，再让 AI 执行；工具按作用域分层、按团队规模选型——工具只是手段，解决问题才是目的。

#### (2) 规范为什么要分四层

把所有规范堆进一个文件 = 混乱的根源（加载冲突、读者错位、衰减加速）。

| 层次 | 管什么 | 加载方式 |
|---|---|---|
| CLAUDE.md | 项目全局规则（风格、约束） | 每次启动全量读入 |
| Skills | 某类任务怎么做（单测、迁移） | `/命令` 按需加载 |
| Spec-Kit | 结构化规范 + 版本管理（做什么） | 开源工具，与 Skill 对接 |
| Kiro | 强制 spec→design→tasks→implement | 合规大团队专用 |

核心判别：**这句话是"全局规范"还是"任务流程"？** 去掉任务名词还成立 → 全局规范（CLAUDE.md）；不成立 → 任务流程（Skills）。

#### (3) 工具选型：按团队规模，够用就好

| 团队规模 | 首选 | 代价 |
|---|---|---|
| ≤3 人 | CLAUDE.md + Skills | 能力上限有限 |
| 3–15 人 | Spec-Kit（补"做什么"） | 需投入维护 |
| >15 人 + 合规 | Kiro（强制流程） | 审批拖慢节奏 |
| >20 人 + 高频部署 | Harness（平台编排） | 平台费用 |

原则：停在最小够用档，不主动越级。

#### (4) Spec Coding：先规范后执行

| 维度 | 直接执行 | Spec Coding |
|---|---|---|
| 假设透明度 | 低，AI 默默假设 | 高，规范里逐项写清 |
| 修改轮次 | 多，逐个错假设返工 | 少，规范阶段一次对齐 |
| 适用场景 | 简单、单文件 | 复杂、跨文件、跨界 |

三步：写规范 → review 规范（不是 review 代码）→ 执行。满足任一即用：改动 >3 文件、需求不确定、涉及外部系统集成；简单 CRUD 不需要。

#### (5) 老项目改造三原则（顺序不能反）

| 原则 | 动作 | 为什么 |
|---|---|---|
| 摸底先于动手 | 让 AI 输出具体到行号的问题清单 | "感觉很乱"无法指导改造 |
| 先补安全网再改造（最重要）| 先写集成测试锁住现状，再重构 | 顺序反了，无法区分新 bug 和老问题 |
| 童子军规则 | 每次动文件顺手对齐规范 | 改造成本摊到日常，不搞大重构 |

节奏：5 年项目 → 覆盖率 80% 约 3–6 个月，新功能正常交付不停。

#### (6) 选工具从痛点出发

先问：**是"现有工具解决不了"，还是"用得不够好"？**

- 代码生成质量不好 → 90% 是 CLAUDE.md 不清晰，不是换工具
- 不追工具，追问题（代码生成质量 / 规范同步 / 交付保障，三类问题稳定不变）
- Claude Code + Cursor ≈ 40 美元/月，投入产出比最高
- AI 应用上线同时接 LangSmith/Langfuse，不要等出问题再补

### 2.2 核心能力聚焦

<img src="imgs/aicent-32-core-capability/cover.jpg" style="display: block; width: 800px;" alt="替换文字">

**完整内容: [AI编程方法 32：复盘进阶 - 核心能力聚焦]({% post_url 2026-07-03-aicent-32-core-capability %})**

#### (1) 一句话讲清楚

> AI 编程的瓶颈不在工具，而在人——工具是放大器，放大的是你已有的东西：能力或缺陷。

#### (2) 提示词质量的真相

| 误区 | 真相 |
| --- | --- |
| 好提示词靠技巧 | 靠认知深度 × 经验 |
| 问不出好问题是 AI 不行 | 是经验没到 |

能问出"用 Redis 还是 MySQL 做缓存"，背后是对两种存储特性的理解，不是模板。

#### (3) 三个能带走的能力

| 能力 | 通俗解释 | 依赖工具吗 |
| --- | --- | --- |
| 领域理解 | 拿到陌生需求，快速搞懂它是什么、做什么 | 否 |
| 技术判断 | 多个方案摆在面前，判断哪个适合自己 | 否，靠积累 |
| 问对问题 | 把模糊大问题，拆成一系列明确小问题 | 否 |

工具会迭代、会被替代，这三个能力不会。

#### (4) 谁在驾驶谁

| 状态 | 典型表现 | 结果 |
| --- | --- | --- |
| 无积累 | 不会选方案、看不出隐患、只会说"你觉得哪个好" | AI 驾驶你 |
| 有积累 | 能给限定条件、能预判踩坑、能把跑偏的 AI 拉回来 | 你驾驭 AI |

Vibe Coding 的本质是驾驭，不是依赖。

#### (5) 把项目当主干

知识点不能飘着，要挂在项目主干的具体位置上：

| 知识点 | 挂载位置（举例） |
| --- | --- |
| 数据库事务 | Redis 的 read-modify-write 并发问题 |
| HTTP 协议 | SSE 流式响应 |

遇到没想透的点就"往下看一层"；用 AI 解释概念、做最小 Demo、出题考自己，形成学习闭环。

#### (6) 复盘四问

| 问题 | 判断标准 |
| --- | --- |
| 哪些决策真懂了，哪些只是看懂了？ | 能否说清"为什么这么做、不那么做" |
| 哪些环节最依赖 AI、没自己判断？ | 那些地方就是知识盲区 |
| 哪个选型当时跟着 AI 走、现在还说不清？ | 把"接受建议"变成"理解为什么对" |
| 知识盲区清单是什么？ | 列出来，就是专属学习路线图 |

#### (7) 快与慢

| 类型 | 内容 | AI 能加速吗 |
| --- | --- | --- |
| 快 | 工具的执行速度 | 能 |
| 慢 | 领域理解、架构判断、方法论内化 | 只有自己能积累 |
