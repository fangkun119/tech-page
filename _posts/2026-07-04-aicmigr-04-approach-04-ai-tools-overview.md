第一部分是方法论参考手册（第 1-4 章），不深入具体技术栈，目标是让工程师在任何阶段都能快速查阅"这件武器解决哪类问题、归到三层骨架的哪一层、该不该学"。第二部分把方法论放回真实场景做实战演示（第 5-7 章），让工程师看到五大典型场景的武器组合怎么配、四件重度工具怎么落、为什么"够用就停"反而是节奏感最强的选型策略。

## 1. 导读地图与系列定位

<img src="imgs/aicmigr-04-approach-04-ai-tools-overview/1291e4273b801760d4d5d24e0336a2d3_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

### 1.1 系列位置与本文目标

本文位于系列方法论基础的第四篇。前三篇分别回答了"做什么"（九步链路）、"AI 做多少"（三档分工图）、"怎么让 AI 不乱来"（三层控制）。本篇回答最后一个偏选型的问题：工具那么多，到底该学哪几件。读完本篇，工程师应获得三件东西：

#### (1) 一张可校准新工具的武器库地图

按解决的问题把市面上主流 AI 编程工具分成五类。新工具冒出来，先看它属于哪类。

#### (2) 一组按场景选武器的判断动作

主武器永远是 AI 编码 Agent，辅武器按场景复杂度加减。

#### (3) 一个反直觉的选型心法

工具不是学得越多越好，是"够用就停"。

### 1.2 武器库地图一览

<img src="imgs/aicmigr-04-approach-04-ai-tools-overview/cc7b0cb0d403c5c6f76e66a1ed7e8f6d_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

下面这张总览表把五类工具的核心定位、对应到三层骨架的哪一层、关键代表摆在一起，是全篇的速查入口。

| 类别 | 一句话定义 | 对应三层骨架 | 代表工具 |
|------|----------|------------|---------|
| AI 编码 Agent | 跟 AI 打交道的主入口 | 三层都在用 | Claude Code、Cursor、Copilot、Cline、Aider |
| 上下文管理 | 把上下文从"临时说一遍"变成"常驻 AI 自动读" | 理解层 + 约束层 | CLAUDE.md、AGENTS.md、.cursorrules、SKILL.md、各类 Memory Bank |
| 能力扩展 | 让 AI 从"只能看代码"升级到"能看代码之外的东西" | 理解层 | MCP、Sourcegraph、各类 MCP Server |
| 规范驱动开发（SDD） | 先写规范再写代码 | 约束层 | Spec-Kit、Kiro、Tessl、OpenSpec |
| 工程辅助 | 给 AI 的产出兜底 | 验证层 | SonarQube、测试框架、CI/CD、Cross-provider review |

## 2. 武器库的五类划分

### 2.1 第一类：AI 编码 Agent（主武器）

这是工程师和 AI 打交道的主入口。

#### (1) 形态分两种

| 形态 | 代表工具 | 工作方式 |
|------|---------|---------|
| IDE 内置型 | Cursor、Copilot、Cline | 在编辑器里直接对话、直接改代码 |
| 命令行型 | Claude Code、Aider | 在终端里开一个会话，用自然语言指挥 |

#### (2) 选型原则：选一个就够

这类工具选一个就够。本系列用 Claude Code，不是因为它唯一，是因为它在老项目改造这个场景下最顺手。用其他工具（例如 CodeX）效果基本一样——主武器形态不同，方法论可以平移。

### 2.2 第二类：上下文管理（老项目改造最核心的辅助）

光有 AI Agent 不够。工程师得告诉 AI 这个项目是什么、哪些地方不能动、哪些规矩要守。这一类是老项目改造最核心的辅助武器：第 03 篇三层骨架里的理解层和约束层，大部分动作都落在这里。

#### (1) 几种常见形式

| 形式 | 归属工具/工具族 | 角色 |
|------|--------------|------|
| CLAUDE.md | Claude Code | 项目级上下文文件 |
| AGENTS.md | 其他 Agent 工具 | 类似的上下文约定 |
| .cursorrules | Cursor | 规则文件 |
| SKILL.md | Claude Code 等 | 专项技能文件（本系列重度使用） |
| 各类记忆系统 | Cline Memory Bank、MemGPT、Zep 等 | 跨会话持久化记忆 |

> 共同特点：把上下文从"每次对话临时说一遍"变成"常驻在项目里 AI 自动读"。一次写好，后面每次会话都自动加载，不用重复讲。

### 2.3 第三类：能力扩展（有用但不是必需）

AI 默认只能读代码。让它能做别的事，需要扩展。

#### (1) 典型扩展点

| 扩展方向 | 代表工具 | 让 AI 多看见什么 |
|---------|---------|---------------|
| 通用协议 | MCP（Model Context Protocol） | 给 AI 接外部能力的标准协议 |
| 代码搜索 | Sourcegraph | 跨仓库代码搜索、依赖分析 |
| 协作历史 | GitHub / GitLab MCP Server | PR、issue、commit 历史 |
| 数据查询 | 数据库 MCP Server | schema、跑查询 |
| 内部系统 | 企业自建的 MCP Server | 企业自有的指标、配置、监控 |

> 共同特点：让 AI 从"只能看代码"升级到"能看到代码之外的东西"——历史、依赖、数据库、线上指标都能接进来。老项目改造里这一类有用但不是必需：零 MCP 也能跑完整改造，但接上关键几个 MCP，很多事会快很多。

### 2.4 第四类：规范驱动开发 SDD（本系列不主线讲）

近一年冒出来的一类。

#### (1) 代表工具与共同特点

| 工具 | 出身 | 定位 |
|------|------|------|
| Spec-Kit | GitHub 开源 | spec-driven 开发框架 |
| Kiro | AWS | Agentic IDE，主打规范驱动 |
| Tessl | 商业产品 | 商业 spec-driven |
| OpenSpec | 社区组件 | 相对较轻，适合 brownfield/改造 |

> 共同特点：先写规范再写代码。先定义 spec、验收标准、接口契约，再让 AI 基于 spec 生成代码。

#### (2) 本系列为什么不在主线讲 SDD

| 不主线讲的原因 | 具体说明 |
|--------------|---------|
| 老项目本来就没有清晰 spec | 给一段老代码强行"补 spec"意义不大 |
| 核心思想已含在第 03 篇 | SDD 的"先约束、再动手"已经在三层控制里，不需要再套流程化的壳 |

知道有这么一类工具就够。将来做新项目，可以正经学一下。

### 2.5 第五类：工程辅助（验证层的主要载体）

最后一类是传统工程工具，不是 AI 时代特有的，但在 AI 协作下重要性被放大了。

#### (1) 几件代表性工程辅助

| 工具类型 | 代表 | 在 AI 协作中的角色 |
|---------|------|------------------|
| 静态代码分析 | SonarQube / Sonar | 自动发现 AI 产出里的代码异味 |
| 测试框架 | Jest、JUnit、pytest 等 | 本来就有，AI 时代更关键 |
| CI/CD 集成 | 各类 CI 系统 | 让 AI 产出必须跑过 CI 才能合入 |
| Cross-provider review | Codex ↔ Claude Code | 让不同模型互相 review 对方产出 |

> 为什么 AI 协作下这一类更重要：AI 写代码的速度远快于人的 review 速度，传统的人肉把关不够了，工程工具必须进来兜底。三层骨架里的验证层，很多动作就落在这一类。

## 3. 选武器的心法

<img src="imgs/aicmigr-04-approach-04-ai-tools-overview/c946610ae667e396950483d95c379d28_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

武器库铺开了，下一个问题是：场景来了，拿哪一件。本章给出一条贯穿所有场景的选型心法。读完本章，工程师再遇到任何改造任务，都能照着这条心法定主武器、配辅武器。

### 3.1 一条贯穿所有场景的选型公式

下面这张表把场景复杂度从低到高排开，标出主武器和辅武器怎么配。

| 场景复杂度 | 主武器 | 辅武器配置 |
|----------|--------|----------|
| 极低（改一个小 bug） | AI 编码 Agent | 无 |
| 中（跨模块的中等改造） | AI 编码 Agent + CLAUDE.md + SKILL.md | 测试框架 + CI |
| 高（接手陌生项目第一天） | AI 编码 Agent | 上下文管理 + 能力扩展（Sourcegraph MCP） |
| 更高（陌生语言 + 陌生项目） | AI 编码 Agent + CLAUDE.md | MCP（GitHub / Sourcegraph）+ 语言学习助手模式 |
| 最高（高风险改造：安全、跨核心服务） | AI 编码 Agent + SKILL.md（专项技能） | Sourcegraph（找全调用路径）+ 独立 review（攻击者视角等多角度） |

#### (1) 关键认知：主武器永远不变，辅武器随复杂度加减

> 不管场景多复杂，主武器永远是 AI 编码 Agent。其他工具都是辅助，按场景复杂度增加。这也回答了开头那个困惑：为什么工程师学了一堆工具，真坐下来干活的时候还是只用了那一个 AI 编码 Agent——这不是工具学得不够，是场景配比本来就如此。

### 3.2 主辅之外的几类旁支

除了主线工具，还有几类"特定场景提到但不是主线"的旁支工具。下表列出它们的定位，帮工程师判断要不要花时间。

| 旁支工具 | 在本系列中的定位 | 建议 |
|---------|---------------|------|
| Sourcegraph、SonarQube、Cross-provider review | 特定场景提到 | 知道有，遇到对应场景再用 |
| Spec-Kit、Kiro 等 SDD 工具 | 不用 | 跟老项目改造主线不对路，先不学 |
| 各种新冒出来的工具 | 用武器库地图校准 | 先问属于哪类、解决哪层、相比现用工具有何增量 |

#### (1) 拿到新工具先问三个问题

| 问题 | 判断 |
|------|------|
| 它属于武器库哪一类？ | 五类里对号入座 |
| 解决理解、约束、验证哪一层的问题？ | 跟三层骨架对位 |
| 相比现在用的工具有什么增量价值？ | 想清楚再决定要不要花时间学 |

> 不对路的工具，知道有就够。这是地图的核心价值：不是为了多学，是为了不被新工具淹没。

## 4. 工具地图与项目阶段 Check List

### 4.1 收窄到四件重度工具

<img src="imgs/aicmigr-04-approach-04-ai-tools-overview/e1827a3ac6ee3ba52b437d265a88d5d2_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

把地图再收窄一点，本系列会重度使用的就四件。下表把这四件工具的角色和三层骨架对位关系摆出来。

| 工具 | 角色 | 三层骨架对应 |
|------|------|------------|
| Claude Code | AI 编码 Agent（主武器） | 三层都在用 |
| CLAUDE.md | 项目级上下文 | 理解层 + 约束层 |
| SKILL.md | 专项技能文件 | 约束层 |
| MCP | 按需接入外部能力 | 理解层（用得相对较少，主要是前三件） |

> 老项目改造的核心不是工具数量，是把这几件用熟。CLAUDE.md 写出来能让 AI 稳定工作，比学会 10 个新工具重要得多。

### 4.2 项目阶段 Check List

<img src="imgs/aicmigr-04-approach-04-ai-tools-overview/7ec5b49cb7ded6133cf3296d28f2d9c5_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

本节把前三章的方法论浓缩成一份可裁剪的速查清单。工程师在接手任何一个老项目改造任务时，可以照着这份清单逐项打勾。条目按"主武器—上下文—能力扩展—工程辅助"四类组织，每类末尾给出层间切换提示。

#### (1) 主武器选定 Check List

判定准则：是否为本次改造选定了一个 AI 编码 Agent 并坚持用熟。

##### ① 选一个就够

在 Claude Code、Cursor、Copilot、Cline、Aider 里选一个，本系列推荐 Claude Code。

##### ② 不要频繁换工具

老项目改造是长跑，主武器的肌肉记忆比花样重要。

> 层间切换提示：主武器选定后，下一件事是把项目的上下文喂给 AI，进入上下文管理。

#### (2) 上下文管理 Check List

判定准则：是否把项目级上下文写到了 AI 自动加载的位置。

##### ① 写 CLAUDE.md

把业务场景、架构组织、接口边界、数据模型、隐性约定沉淀进 CLAUDE.md。

##### ② 识别专项技能写 SKILL.md

对反复出现的复杂操作（例如"按九步链路画架构"），沉淀成 SKILL.md。

##### ③ 一次写好长期复用

上下文管理是静态投资，写一次后面每次会话自动加载，不要每次临时说一遍。

> 层间切换提示：上下文管理写扎实后，再考虑给 AI 接外部能力，进入能力扩展。

#### (3) 能力扩展 Check List

判定准则：是否在不破坏主线简洁性的前提下按需接入 MCP。

##### ① 评估是否真需要 MCP

零 MCP 也能跑完一个完整改造，先问"接入这个 MCP 带来的提速值不值得新增的复杂度"。

##### ② 优先接高价值的几个

代码搜索（Sourcegraph）、协作历史（GitHub MCP）通常 ROI 最高，优先接。

##### ③ 不接企业内部低价值 MCP

不是每个企业自建 MCP 都值得接，能用人工查的就先人工查。

> 层间切换提示：能力扩展到位后，进入工程辅助兜底验证层。

#### (4) 工程辅助 Check List

判定准则：是否给 AI 的产出搭起了独立于 AI 的验证网。

##### ① 测试框架先搭起来

第 03 篇验证层的"集成测试 + Characterization Test"都要靠测试框架承载。

##### ② CI 兜底

让 AI 产出必须跑过 CI 才能合入，把人肉把关替换成自动门禁。

##### ③ 关键节点加独立 review

高风险改造加 Cross-provider review，让另一个模型从攻击者视角审一遍。

> 层间切换提示：四件主线工具加上工程辅助兜底，构成本系列完整的工具栈。第二部分把这套配置放回真实场景做实战演示。

## 5. 五大场景的武器组合

第 4 章给出了主线工具配置。本章把方法论放回真实场景，看五个典型场景下武器组合具体怎么配。核心结论是：主武器永远是 AI 编码 Agent，辅武器按场景复杂度加减。

### 5.1 五个场景的武器组合总览

下表把五个场景的"目标—主武器—辅武器—当日产出"摆在一起。

| 场景 | 目标 | 主武器 | 辅武器 | 当日产出 |
|------|------|--------|--------|---------|
| 接手陌生项目第一天 | 让 AI 帮你扫仓库、画架构、梳接口、列数据表 | AI 编码 Agent（如 Claude Code） | 上下文管理（准备接收项目信息）+ 能力扩展（项目大就上 Sourcegraph MCP） | 沉淀到 CLAUDE.md |
| 改一个小 bug | 快速修复，不动其他 | AI 编码 Agent | 无 | 给 AI 上下文，改完 review |
| 跨模块的中等改造 | 本系列大部分实战场景 | AI 编码 Agent + CLAUDE.md + SKILL.md | 测试框架 + CI | 改造代码 + 测试 |
| 陌生项目 + 陌生语言 | 让 AI 既帮你理解项目又帮你学语言 | AI 编码 Agent + CLAUDE.md | MCP（GitHub / Sourcegraph）+ 语言学习助手模式 | 项目理解 + 语言习得 |
| 高风险改造（安全、跨核心服务） | 容错空间小，必须找全调用路径 | AI 编码 Agent + SKILL.md（专项技能） | Sourcegraph（找全调用路径）+ 独立 review（攻击者视角） | 改造 + 多角度审查 |

### 5.2 场景三：本系列的主战场

五个场景里，第三个——跨模块的中等改造——是本系列大部分实战的主战场。下面把这一场景的武器组合拆开来看。

| 三层骨架 | 对应工具 | 在这一场景里做什么 |
|---------|---------|----------------|
| 理解层 | CLAUDE.md | 沉淀前几次改造中积累的项目理解 |
| 约束层 | SKILL.md + 明确的提示词 | 写明本次改造的边界、不能动什么、改成什么样 |
| 验证层 | 测试框架 + CI | 改完跑测试、跑 CI，独立校验 AI 的产出 |

> 这是第 03 篇三层骨架在工具层的落地：理解层靠 CLAUDE.md，约束层靠 SKILL.md，验证层靠测试框架 + CI。武器组合的配置，本质是三层骨架在工具栈上的投影。

### 6.2 为什么是这四件

下表把这四件工具和"老项目改造的核心矛盾"对位起来，说明为什么主线就这四件。

| 工具 | 解决的老项目核心矛盾 | 不可替代性 |
|------|------------------|----------|
| Claude Code | 老项目复杂度 vs 工程师注意力带宽 | 主武器，AI 协作的入口 |
| CLAUDE.md | 上下文每次都要重讲 vs 项目知识应常驻 | 把项目理解固化到 AI 自动加载的位置 |
| SKILL.md | 反复出现的复杂操作 vs 每次重新描述 | 把操作流程沉淀成可复用的专项技能 |
| MCP | AI 只能看代码 vs 老项目依赖历史和跨仓库信息 | 按需把"代码之外的东西"接进来 |

> 四件工具里前三件（Claude Code、CLAUDE.md、SKILL.md）必须吃透，MCP 用得相对较少。其他工具——Sourcegraph、SonarQube、Cross-provider review——会在特定场景提到，但不是主线。

## 7. 够用就停的反向校准

<img src="imgs/aicmigr-04-approach-04-ai-tools-overview/b0faa9ebbf946fdb6cc5a06f7adf8487_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

第 6 章解释了为什么是这四件。本章反过来回答一个常见质疑：为什么不用更多工具，比如 Spec-Kit 和 OpenSpec？这一章是"够用就停"心法的实战展开。

### 7.1 为什么 Spec-Kit 和 OpenSpec 不进主线

在 2026 年的 SDD 生态里，Spec-Kit 和 OpenSpec 是两个核心工具。下表把它们的定位和本系列不主线用的原因摆清楚。

| 工具 | 定位 | 不进主线的原因 |
|------|------|--------------|
| Spec-Kit | GitHub 官方开源，流程重 | 适合从 0 到 1 做新项目，老项目改造强行套用流程重 |
| OpenSpec | 社区组件，相对 Spec-Kit 较轻 | 适合 brownfield / 改造，但本系列想让读者深入细节、自己拆解项目 |

> 本系列刻意尽量不用成熟工具，是为了让读者深入细节、亲手拆解项目。这不是否定成熟工具的价值，是在"学方法论"和"用现成工具"之间优先选了前者。

### 7.2 多学一个工具的真实代价

"够用就停"是个反直觉的心法。下面这张表把"多学一个新工具"的真实代价拆开来看。

| 代价类型 | 具体表现 |
|---------|---------|
| 认知成本上升 | 学的时候觉得懂了，用的时候又发现模糊的地方 |
| 反复回查 | 真要用的时候记不清细节，反复查文档 |
| 注意力切碎 | 注意力被工具切碎，真正花在项目上的思考反而少了 |
| 肌肉记忆被稀释 | 老工具的熟练度被新工具的初学状态稀释 |

#### (1) 关键认知：熟练度才代表能力

> 工具数量不代表能力，熟练度才代表能力。CLAUDE.md 闭着眼都能写、SKILL.md 看一眼就知道哪些该加、MCP 知道什么时候该接什么时候不该接——这些熟到不用思考，工程师的脑子才能留给项目本身。

### 7.3 "够用就停"呼应"熬过冷启动"

"够用就停"和第 01 篇讲的"熬过冷启动"是同一个心法的两面。下表把这两个心法的呼应关系摆出来。

| 心法 | 出现位置 | 共同的本质 |
|------|---------|----------|
| 熬过冷启动 | 第 01 篇九步链路 | 老项目改造是长跑，前几步最慢，不要急 |
| 够用就停 | 本篇工具选型 | 老项目改造是长跑，节奏比花样重要 |

> 老项目改造是个长跑，节奏比花样重要。把那四件工具吃透之前，不要着急学更多。学完之后觉得不够，再去扩展——这是健康的学习路径。

## 8. 小结与思考

<img src="imgs/aicmigr-04-approach-04-ai-tools-overview/db4bd9b5a0d6e85082b3542feb772665_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

### 8.1 小结

本篇给了一张 AI 编程工具的武器库地图。

#### (1) 五类武器库

按解决的问题分五类：AI 编码 Agent（主武器）、上下文管理、能力扩展、规范驱动开发、工程辅助（四类辅助武器）。

#### (2) 按场景选武器

主武器永远是 AI 编码 Agent，辅武器根据场景复杂度加减。本系列主线工具是 Claude Code + CLAUDE.md + SKILL.md + MCP 四件，前三件必须吃透。

#### (3) 五类对应三层骨架

| 五类工具 | 三层骨架对应 |
|---------|------------|
| AI 编码 Agent | 三层都在用 |
| 上下文管理 | 理解层 + 约束层 |
| 能力扩展 | 理解层 |
| 规范驱动开发 | 约束层 |
| 工程辅助 | 验证层 |

#### (4) 够用就停

工具不是学得越多越好，是够用就停。工具数量不代表能力，熟练度才代表能力。

#### (5) 心法呼应

"够用就停"呼应第 01 篇的"熬过冷启动"——老项目改造是长跑，节奏比花样重要。

读到这儿，方法论基础四篇就讲完了。第 01 篇九步链路回答"做什么"，第 02 篇三档分工回答"AI 做多少"，第 03 篇三层控制回答"怎么让 AI 不乱来"，本篇武器库地图回答"用哪些工具"。从下一篇开始，系列进入具体的项目阶段，把这四篇方法论放回老项目的真实改造里逐一展开。

### 8.2 思考

#### (1) 思考一

工程师现在经常用的 AI 编程工具有几件？按本篇的五类划分，它们分别属于哪类？有没有哪一类工具完全没用过、但可能应该用起来？

#### (2) 思考二

回想最近学过但没用起来的某个 AI 工具，为什么没用起来？是场景不匹配，还是其实不需要？如果用本篇的武器库地图来判断，当时该不该学它？
