## 1. AI 编程方法论全景：四步探路法

### 1.1 核心问题：遇到陌生技术怎么做

<img src="imgs/aicent-20-fea-rag-1/d981332edc4f41a18b5755acdf5677b1_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

基础篇做完了 Hify 的核心链路，但智能客服的回答只能依赖模型自身的通用知识。当用户问"你们的退换货政策是什么"，客服给不出一段能引用公司真实条款的回答——这不是 Agent 配置或 Prompt 写得不够好的问题，而是模型根本不知道私有数据。

要解决这个问题，需要引入一项对开发者而言完全陌生的技术：RAG。于是本篇真正要回答的核心问题，并不是"RAG 是什么"，而是更高一层的那个问题：**当要给一个已经跑通的系统加入一项完全不了解的技术能力，应该怎么做？**

答案不是去读官方文档、也不是去搜教程，而是用开发者手头已有的工具 Claude Code，一步一步把陌生的东西变熟悉、把熟悉的东西变可动手。这一路径可以提炼成"四步探路法"，遇到任何陌生技术都能复用。

### 1.2 四步探路法总览

<img src="imgs/aicent-20-fea-rag-1/4de0a44374b76742c4b9a2a8c9aca57b_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

四步探路法把"接入陌生技术"这件看起来无从下手的事，拆成四个有明确产出物的阶段。每一步都对应一个可观测的目标，做完一步就拿到一份具体的产出物，不会陷入"读懂了但不知道做没做完"的空转。四步的顺序与产出物如下图所示。

```mermaid
flowchart LR
    S1["第一步·痛点提问"]
    S2["第二步·拆组件"]
    S3["第三步·约束选型"]
    S4["第四步·最小 Demo"]
    S1 -->|"产出：一句话心智模型"| S2
    S2 -->|"产出：陌生范围缩小清单"| S3
    S3 -->|"产出：带理由的选型结论"| S4
    S4 -->|"产出：可跑通的最小示例 + 已知坑清单"| End["进入工程集成"]
```

四步的含义可以概括为下表：

| 步骤 | 核心做法 | 产出物 |
|---|---|---|
| **第一步 痛点提问** | 从业务痛点出发问"它解决什么问题"，而不是问"XXX 是什么" | **一句话心智模型** |
| **第二步 拆组件** | 问"实现它需要哪些技术组件"，对每个组件标注"已会 / 调库 / 调 API / 纯陌生" | **陌生范围缩小清单**（往往只剩一两个真正陌生的组件） |
| **第三步 约束选型** | 把技术栈、已有基础设施、数据量、性能要求等约束显式列出，让 Claude Code 做对比、由人下判断 | **带理由的选型结论** |
| **第四步 最小 Demo** | 单独建表、插数据、查询，在干净环境里跑通最小示例 | **可跑通的最小示例 + 已知坑清单**（如类型兼容、SDK 差异等集成期问题提前暴露） |

这四步是本篇后续 7 章的统领：第 2 章按这四步分步详解，第 3 章把四步做成可裁剪的 Check List，第 4～8 章则依次演示如何在 Hify + pgvector 上把每一步落地。

### 1.3 元原则：技术在变，探路的方法不变

四步探路法的价值不止于接入 RAG 这一次。原文在总结这一过程时点出了一条贯穿始终的元原则：<span style="color: red; font-weight: bold;">换成任何陌生技术，比如存储引擎、图数据库、搜索引擎，都是同样的路径；技术在变，探路的方法不变。</span>

这条元原则落到可迁移性上有三层含义：

- **方法可迁移**：无论是接入一个新的存储引擎（如从关系型库换到时序数据库）、一个图数据库（如 Neo4j）、还是一个搜索引擎（如 Elasticsearch），开发者都可以复用"痛点提问 → 拆组件 → 约束选型 → 最小 Demo"这四步，不必为每一项新技术重新摸索路径。
- **工具可复用**：四步的执行都依托 Claude Code 这一工具——问场景、问组件清单、问选型对比、问最小示例，四类提问模板可以原样套用到任何陌生技术上。
- **决策原则稳定**：贯穿四步的判断标准是"约束驱动决策、用能解决当前问题的最简单方案、不为假想的未来过度设计"。这一原则不随具体技术变化，是开发者面对任何陌生选型时都可以回到的锚点。

正是这条元原则，使得本篇虽然以 RAG 和 pgvector 为载体，但真正交付给读者的是一套可复用的陌生技术接入流程。后续章节会先在第 2、3 章把这套流程展开成手册和 Check List，再在第 4～8 章用 Hify 项目把它完整跑一遍。

## 2. 四步探路法分步详解

第 1 章给出了四步探路法的总览，本章把它拆成四步逐一展开。每一步统一用"目标 / 关键提问模板 / 操作要领 / 常见误区 / 判断标准"五元组结构呈现，便于速查、便于照着做。本章只讲方法本身，不展开 pgvector、Embedding 等具体技术细节——那些是第二部分实战演示的职责；每一步里最多用一行极简例子点题。

### 2.1 第一步：从业务痛点出发——问清楚它解决什么问题

<img src="imgs/aicent-20-fea-rag-1/688453b673a8c78f215e64493081128f_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

#### (1) 目标

建立"该技术在自己场景里到底解决什么业务问题"的一句话心智模型，而不是背一段教科书式的宽泛定义。

#### (2) 关键提问模板

以下是当时使用的提问模板，原文保留：

```
我的智能客服 Agent 现在只能靠模型的通用知识回答问题，无法引用公司的产品手册和政策文档。我听说 RAG 可以解决这个问题。帮我解释：RAG 的核心思路是什么？它是怎么让 LLM 能引用私有文档的？
```

#### (3) 操作要领

- 提问必须包含场景与痛点要素：显式写出"我的场景是……""我的痛点是……""我听说 X 可以解决，请解释 X 的核心思路"，让回答直接对应要做的事。
- 让 Claude Code 解释该技术的核心思路（它如何作用、为什么能解决问题），而不是给出一段名词定义。
- 建立一句话心智模型作为这一步的收束：例如"RAG 是把找答案这件事从模型内部迁到外部系统"，能向同事讲清即可，不必涉及实现细节。
- 痛点描述越具体，得到的答案越有针对性；可先把现状、限制、期望分别列一两句再提问。
- 若回答仍偏定义，可追加一句"请结合我的场景说明它解决了其中哪个环节"，把回答拉回场景。

#### (4) 常见误区

- 问"XXX 是什么"，得到一大段通用定义，读完仍不知道和自己的场景有什么关系。
- 不带场景提问，得到中立的百科式解释，无法直接落到动作。
- 试图在第一步就搞懂所有实现细节，把心智模型阶段变成通读文档阶段，反而拖慢节奏。

#### (5) 判断标准

能用一句话说清该技术在自己场景里解决的具体问题，并把这句话写进笔记或文档；该技术"解决什么、不解决什么"的边界已经清楚，可以进入下一步拆组件。

### 2.2 第二步：拆解技术组件——缩小陌生范围

<img src="imgs/aicent-20-fea-rag-1/2b209b1f493485969d8072c7bf32d6b7_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

#### (1) 目标

把整体看起来复杂的技术名词拆成一张组件清单，识别出真正陌生的范围，把心理压力和后续精力聚焦到那一两个陌生组件上。

#### (2) 关键提问模板

以下是当时使用的提问模板，原文保留：

```
实现一个完整的 RAG 功能，需要哪些技术组件？每个组件的作用是什么？
```

#### (3) 操作要领

- 问"实现 X 需要哪些技术组件、每个组件的作用是什么"，按数据流向拿到一份组件清单（如 RAG 的两阶段六组件：文档解析器、文本分块器、Embedding 模型、向量数据库、检索器、Prompt 组装器）。
- 对清单上的每个组件人工标注陌生度，分四档：已会、调库即可、调 API 即可、纯陌生。
- 识别真正陌生的范围：把已会和调库/调 API 的组件排除后，剩下那一两个才是接下来要重点投入的点。
- 用一句话收束陌生范围（如"六个组件里真正陌生的只有向量数据库"），打破面对整体名词时的恐惧。
- <span style="color: red; font-weight: bold;">拆开后大部分组件其实已经会——这是缩小陌生范围、打破恐惧的第一步，也是判断"这事儿能不能动手"的依据。</span>

#### (4) 常见误区

- 把整个技术名词当成一个不可分割的整体来评估难度，被表面复杂度吓退。
- 只拿组件清单却不标注陌生度，无法识别该把精力放在哪里。
- 误把"调库/调 API"的组件也当成陌生组件，导致选型与学习范围被人为放大。

#### (5) 判断标准

拿到一张带陌生度标注的组件清单，并能指出其中真正陌生的组件（通常只有一两个）；清单可直接作为后续选型和学习任务的输入。

### 2.3 第三步：给约束做选型——AI 对比、人判断

<img src="imgs/aicent-20-fea-rag-1/44bfd34bb0b8c2f5e62f1831584d9f0b_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

#### (1) 目标

在真实约束下让 Claude Code 给出有立场的方案对比，由人基于约束和对比结果下选型判断并记录理由，而不是停留在"各有优缺点"。

#### (2) 关键提问模板

以下是当时使用的提问模板，原文保留：

```
向量数据库有哪些主流选项？我的约束是：Java 技术栈、已有 PostgreSQL、一期数据量不大（几千到几万条分块）、基本的相似度检索够用。帮我对比 pgvector、Milvus、Qdrant、Elasticsearch 的优缺点。
```

#### (3) 操作要领

- 在提问里显式列出约束清单，至少覆盖四类：技术栈、已有基础设施、数据量、性能要求。
- 要求 AI 给出对比表加优缺点，而不是直接要一个"推荐方案"。
- 由人下最终判断：结合团队运维能力、是否引入新组件、是否复用已有基础设施、一期是否够用，选一个当前约束下最简单的方案。
- 把选型理由逐条写下来（不引入新依赖、SQL 可与现有表 JOIN、一期性能够用、检索层抽象好可平滑替换），作为后续回溯和团队对齐的依据。
- 遵守"用能解决当前问题的最简单方案、不为假想未来过度设计"的原则——后续约束真变化时再换，成本可控。

#### (4) 常见误区

- 不给约束直接问"哪个好"，答案只能是"各有优缺点"，无法支撑决策。
- 把选型完全交给 AI，没有结合自身运维能力和已有基础设施做人的判断。
- 为假想的未来过度设计，一期就引入重量级组件，徒增运维负担和集成复杂度。

#### (5) 判断标准

产出一份带约束的方案对比表，以及一条明确的选型结论加逐条理由；结论可向团队讲清"为什么是这个、为什么不是别的"，并能回溯到具体约束。

### 2.4 第四步：最小 Demo 优先——跑通再说

<img src="imgs/aicent-20-fea-rag-1/faadb9c2893bdd5f63f8a3410cc1993c_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

#### (1) 目标

在干净环境里用一个最小可运行示例把陌生组件跑通，建立手感、暴露集成期可能的坑，再进入业务代码集成。

#### (2) 关键提问模板

以下是当时使用的提问模板，原文保留：

```
我选了 pgvector。帮我从零上手：怎么安装、怎么建表、怎么插入向量、怎么做相似度查询？给我一个能跑通的最小示例。
```

#### (3) 操作要领

- 单独建表、单独插数据、单独查询，不混进业务代码；这一步的目标是跑通陌生组件本身。
- 确认通后再进业务代码：Demo 跑通、手感建立之后，再把组件集成进既有系统（如 Hify 的 Java 服务）。
- 在干净环境里主动暴露集成期可能的坑（类型需特殊处理、SDK 不原生支持、运算符不兼容等），并就地解决，而不是留到业务代码里排查。
- 节奏上不是"读完文档再动手"，而是"直接问怎么用 → 拿最小可运行示例 → 跑通建立手感"。
- 跑通后简要记录本次 Demo 踩到的坑与解法，作为工程集成阶段的避坑清单。

#### (4) 常见误区

- 跳过 Demo 直接集成进业务系统，第一次踩坑发生在业务代码里，排查成本高、干扰因素多。
- 把 Demo 写成功能完整的小项目，偏离"最小可运行"的初衷，反而拖慢节奏。
- Demo 跑通却不记录踩到的坑，到了集成阶段同样的问题再踩一遍。

#### (5) 判断标准

在干净环境里跑通陌生组件的最小示例（能建表/插数据/查询或等价动作），并记录下至少一条集成期需要注意的坑与解法；手感已建立，可以进入业务代码集成阶段。

## 3. 项目接入阶段 Check List

<img src="imgs/aicent-20-fea-rag-1/37939282f56b0260a64962c5180a9e05_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

### 3.1 如何使用这份 Check List

本 Check List 把四步探路法落成一份可直接拷到项目里逐条勾选的动作清单。读者可按项目实际裁剪、增删：每一条都设计成独立可勾选的动作，删掉任一条不影响其它条的语义，也不破坏四步的整体流程。建议在项目启动会时由技术负责人带着团队过一遍，确认本阶段每一阶段的检查项完成后，再进入下一阶段，避免在后续工程集成时回头补探路的功课。检查项前用 `[ ]` 模拟勾选框，完成一项划掉一项；备注栏可填入本次项目的具体产出物位置或链接，便于回溯。

## 4. 实战背景：Hify 智能客服为什么需要 RAG

<img src="imgs/aicent-20-fea-rag-1/0d1cae38e5710f540a367b748299d85d_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

### 4.1 项目状态与业务天花板

本系列基础篇已经做完了 Hify 智能客服的核心链路：模型管理、Agent 配置、流式对话、前端界面，整条主链路都跑通了。但与智能客服聊几轮就会撞到一个业务天花板——它的回答全部依赖模型自身的通用知识。当用户问"你们的退换货政策是什么"，客服只能给出一段泛泛的通用回答，引用不了公司真实的退换货条款；同样，产品手册、内部 FAQ 这些关键的私有文档，模型一条都用不上。

这不是 Agent 配置的问题，也不是 Prompt 写得不够好，而是模型根本不知道这些私有数据。这条天花板横在那里，意味着基础篇的链路再顺，也无法让客服真正"懂公司的业务"。

要突破这条天花板，需要的就是 RAG。

### 4.2 本篇真正的方法论主题

在开始做之前，不妨假设开发者对 RAG 的了解仅限于"听说过这个词"。但这一篇真正的主题并不是 RAG 本身，而是更高一层的命题：**当要给一个已经跑通的系统加入一个完全陌生的技术能力，应该怎么做？**

答案不是去读文档、去找教程，而是用现有工具 Claude Code，一步一步把陌生的东西变熟悉。这就是第一部分第 1 章提炼出的四步探路法——这套方法论以后遇到任何陌生技术都能复用。本章接下来就按照四步探路法的第一步展开实战：从业务痛点出发提问。

### 4.3 第一步实战·痛点提问

第一步从业务痛点出发，用自己的场景去提问，而不是去查"XXX 是什么"。以下是当时使用的提问模板，原文保留：

```
我的智能客服 Agent 现在只能靠模型的通用知识回答问题，无法引用公司的产品手册和政策文档。我听说 RAG 可以解决这个问题。帮我解释：RAG 的核心思路是什么？它是怎么让 LLM 能引用私有文档的？
```

### 4.4 深挖 why：为什么这样提问

碰到陌生技术，很多人的第一反应是去查"XXX 是什么"。这个问题太宽泛，得到的答案往往是一大段定义，读完还是不知道和自己的场景有什么关系。带场景的提问则完全不同——问题里有具体的业务痛点，Claude Code 给出的答案会直接对应要做的事，而不是通用教科书式的解释。痛点描述得越清楚，回答就越有针对性。

#### (1) 问定义得到宽泛答案

如果直接问"RAG 是什么"，得到的是一段标准定义：检索增强生成（Retrieval-Augmented Generation）是一种结合检索与生成的技术……这类答案信息量大，但读完依然不知道它到底怎么解决"客服引用不了私有文档"这件事。

#### (2) 带场景提问得到针对性答案

换成上面的提问模板，问题里就包含了三层信息：

- **业务场景**：智能客服 Agent、模型管理已跑通；
- **具体痛点**：只能靠模型通用知识回答，无法引用产品手册和政策文档；
- **想验证的方向**：听说 RAG 可以解决，请解释其核心思路。

Claude Code 据此给出的答案会直接落到"RAG 如何让 LLM 引用私有文档"这个具体问题上，而不是泛泛地讲技术名词。这就是第一步提问的关键所在。

### 4.5 建立 RAG 心智模型

Claude Code 对上面提问的回答，帮助建立起第一个关键认知：<span style="color: red; font-weight: bold;">RAG 解决的根本问题，是 LLM 的知识是"冻结"的。</span> 可以从三个层次把这个心智模型立起来。

#### (1) LLM 知识"冻结"的本质

LLM 的知识来自训练数据，训练完成的那一刻就被固化了。公司的产品手册、退换货政策、内部 FAQ 这些文档根本不在训练集里，模型物理上不知道这些内容的存在。无论把 Prompt 写得多好、把 Agent 配置得多精细，模型都"变"不出它没见过的私有数据。

#### (2) 两条解决路径对比：Fine-tuning vs RAG

要让模型能用上私有数据，有两条路径：

| 路径 | 是否改动模型 | 更新文档的代价 | 实时性 | 适配智能客服场景 |
|---|---|---|---|---|
| **Fine-tuning（微调）** | 改变模型权重，把文档塞进训练过程 | 文档一旦更新就得重新训练 | 慢、贵 | 极不实际 |
| **RAG（检索增强生成）** | 不动模型，每次对话时实时把相关文档片段塞进 Prompt | 增量入库即可，无需重训 | 实时检索 | 高度适配 |

Fine-tuning 又贵又慢，而且文档一旦更新就得重新训练，完全跟不上业务节奏。RAG 选的是第二条路——<span style="color: red; font-weight: bold;">模型本身没变，变的是它每次看到的输入。</span>这就把"模型不知道私有数据"的难题，转化成了"如何在每次提问时把相关文档实时送进 Prompt"。

#### (3) 检索→增强→生成落到智能客服场景

RAG 的全称 Retireval-Augmented Generation，思路正是三个字：**检索 → 增强 → 生成**。数据流是：用户提问 → 去文档库找最相关的几段内容 → 把这几段原文拼进 Prompt → LLM 基于这些内容回答。

把它落到 Hify 智能客服的具体场景里就非常直观：

1. 管理员上传退换货政策，系统把它切成小块并向量化存好；
2. 用户问"支持七天无理由退货吗"；
3. 系统先在文档库里检索，找到原文"自签收之日起七天内，未拆封商品支持无理由退货"这一段；
4. 把这段原文塞给 LLM，并要求它基于这段资料回答；
5. 客服就能引用真实条款给出准确回答，而不是靠模型猜。

这就是 RAG 让智能客服"开口能引用公司文档"的完整机理。

### 4.6 一句话收束

第一步的目标不是搞懂所有细节，而是建立一个清晰的心智模型：<span style="color: red; font-weight: bold;">RAG 是把"去文档里找答案"这件事，从 LLM 内部迁移到外部系统来完成。</span> 有了这个心智模型，下一步就可以去拆解这个"外部系统"到底包含哪些技术组件了。

## 5. 第一步实战：拆解技术组件，缩小陌生范围

<img src="imgs/aicent-20-fea-rag-1/d1e57a23129e4c20497e9759393bb368_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

### 5.1 提问模板

知道 RAG 是什么、解决什么问题之后，下一步是搞清楚它涉及哪些技术组件。这一步的提问要尽量直接，让 Claude Code 按数据流向把组件拆出来：

```
实现一个完整的 RAG 功能，需要哪些技术组件？每个组件的作用是什么？
```

RAG 这个词作为整体看起来很复杂，但拆开之后会发现大部分组件其实已经会了，真正陌生的可能只有一两个点。把陌生范围缩小，是打破恐惧的第一步。

### 5.2 两阶段六组件清单（表格）

Claude Code 按数据流向把 RAG 拆成两个阶段、六个组件。下表把六个组件的作用、Hify 的选择、陌生度一次列清，后续 5.3～5.5 再围绕关键组件逐层深挖。

| 阶段 | 组件 | 作用 | Hify 的选择 | 陌生度 |
|---|---|---|---|---|
| 阶段一·文档入库管道 | 文档解析器 | 把 TXT、PDF、Word 等格式的文件转成纯文本 | 一期只支持 TXT 和 PDF 文字版，PDF 走文字版解析 | 调库 |
| 阶段一·文档入库管道 | 文本分块器 | 把长文档切成适合检索的小块，最影响 RAG 效果 | 递归分割，chunk_size 512 token，overlap 64 token | 纯代码逻辑 |
| 阶段一·文档入库管道 | Embedding 模型 | 把文本转成向量，语义相近的文本向量在空间里也相近 | OpenAI `text-embedding-3-small`（第 8 章详述） | 调 API |
| 阶段一·文档入库管道 | 向量数据库 | 存向量，支持相似度检索，找与查询向量距离最近的几条 | pgvector（第 6 章选型、第 7 章最小 Demo） | 纯陌生 |
| 阶段二·检索对话链路 | 检索器 | 用户提问时把问题也转成向量，在向量库里找 Top-K 最相关块 | 基于 pgvector `<=>` 距离的查询逻辑 | 改业务逻辑 |
| 阶段二·检索对话链路 | Prompt 组装器 | 把检索到的原文块拼进上下文，告诉 LLM 基于以下资料回答 | 在 ChatService 里拼装 Prompt | 改业务逻辑 |

### 5.3 深挖 why：文本分块为什么最影响效果

六个组件里，文本分块器是最容易被忽略、却最影响最终效果的环节。Embedding 与检索质量都建立在"块"的粒度合理之上。

#### (1) 块太大的问题

- Embedding 的语义被稀释：一段上千 token 的长文混合了多个主题，向量化后语义模糊。
- 检索不准：查询向量很难与这种"主题混杂"的块精准匹配。
- 占 Prompt token：塞进 LLM 上下文时，相关信号被噪声淹没，还要付出更多 token 成本。

#### (2) 块太小的问题

- 上下文丢失：只切出一个孤立的句子，缺乏前后的语境支撑。
- 答案不完整：检索命中的块本身信息量不足，LLM 难以给出完整回答。

#### (3) 递归分割策略与 chunk_size/overlap 权衡

Hify 采取的策略是递归分割：先按段落切，段落仍太长再按句子切，避免简单按固定字节硬切造成的语义断裂。关键参数是两个：

- `chunk_size 512 token`：兼顾"语义完整"和"检索精度"的中间值，既能装下一个完整的小主题，又不至于稀释向量。
- `overlap 64 token`：相邻块之间重叠一段，保证跨块边界的语义不会被硬切断，是缓解"块太小丢上下文"的廉价补救。

这两个数字是权衡出来的，不是公式推出来的；不同语料需要微调，但 512/64 是一个稳妥的起点。

### 5.4 陌生度盘点

对照 5.2 表格的陌生度列盘点，可以得出一个清晰的判断：六个组件里，真正陌生的只有向量数据库。

- 文档解析器：调库（PDF/Word 解析有成熟库）。
- 文本分块器：纯代码逻辑，参数和策略可调。
- Embedding 模型：调 API，和第 14 篇调 Chat Completion 是同一类调用。
- 向量数据库：纯陌生，是本篇之后的核心攻坚点。
- 检索器：改 ChatService 业务逻辑。
- Prompt 组装器：改 ChatService 业务逻辑。

把陌生范围从整个 RAG 缩小到向量数据库这一个点，心理压力小了很多，也清楚地知道下一步该把精力放在哪里。

### 5.5 深挖 why：向量搜索 vs 关键词搜索

一个容易被忽略的判断是：为什么 RAG 检索要用向量搜索，而不是关键词搜索（SQL 的 LIKE）？

#### (1) 关键词搜索只能匹配字面

用户问"东西坏了怎么办"，文档里写的是"产品质量问题处理流程"。两者语义是同一件事，但关键词一个都不重叠，LIKE 查询根本命中不到。

#### (2) 向量搜索基于语义距离

向量搜索的关键是：语义相近的文本，向量距离就近。不管用词怎么不同，只要意思相近就能被检索到。Embedding 把"东西坏了怎么办"和"产品质量问题处理流程"映射到空间里相近的位置，Top-K 自然就能召回。

这就是 RAG 检索质量高于关键词搜索的根本原因，也是 5.2 表格里"向量数据库"这一陌生组件必须啃下来的原因。

## 6. 第二步实战：约束驱动选型——为什么是 pgvector

<img src="imgs/aicent-20-fea-rag-1/4663fe7bedbe6dee19db48ee0cc14c1c_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

### 6.1 提问模板（含三条约束）

向量数据库是唯一陌生的点，也是本篇最重要的技术决策。仍然采用第三步探路法"给约束让 Claude Code 分析、人来判断"的范式，把约束显式写进提问：

```
向量数据库有哪些主流选项？我的约束是：Java 技术栈、已有 PostgreSQL、一期数据量不大（几千到几万条分块）、基本的相似度检索够用。帮我对比 pgvector、Milvus、Qdrant、Elasticsearch 的优缺点。
```

三条约束（Java 技术栈、已有 PostgreSQL、一期数据量不大）是这次选型的关键变量，决定了对比结果的立场。

### 6.2 深挖 why：为什么提问要列约束

#### (1) 无约束的选型只能是"各有优缺点"

没有约束时，AI 给出的答案只能是中立百科全书式的：每个方案都列出优缺点，最后留一句"按需选择"。这种回答听起来全面，但对决策毫无帮助。

#### (2) 有约束才能得到有立场的判断

把实际约束给出来，AI 才能在"这些约束下谁更合适"的坐标系里下判断。同样是 Milvus，在"已有 PostgreSQL、一期几千条数据"的约束下是过度工程；在"十亿级向量、毫秒级检索"的约束下就是正确选择。约束变了，结论会变；不列约束，等于把判断权又交回给百科全书。

这也是 5.1 的"业务痛点提问"原则在选型场景的延伸——让答案直接对应自己要做的事。

### 6.3 四选项对比表

Claude Code 基于上述约束给出了四选项的对比，整理成表格如下。五个维度对应提问里的三条约束（Java 友好度、适配已有基础设施、性能对应数据量）再加上部署复杂度和运维负担两个隐含成本项。

### 6.4 决策：选 pgvector 及理由

结论先行：最终选 pgvector。理由逐条展开如下。

#### (1) 不引入新组件

Hify 已经在用 PostgreSQL，pgvector 只是一个扩展插件，装上即可，不增加任何新的服务依赖，运维负担为零增量。

#### (2) SQL 可与 document/knowledge_base 表 JOIN

向量与业务数据同库，检索结果可以直接通过 SQL 与 `document`、`knowledge_base` 等业务表 JOIN，权限、归属、元数据过滤都用现有逻辑，业务实现简单。

#### (3) 一期性能够用

一期数据量是几千到几万条分块，pgvector 在百万级以下性能完全够用，没有性能瓶颈。

#### (4) 检索层抽象好可平滑替换

只要把检索逻辑抽象成一层接口，未来数据量真大到 pgvector 扛不住，替换到 Milvus/Qdrant 的成本也只在存储和检索层，不会向上扩散到业务代码。先把当前问题解决，把切换的口子留好。

### 6.5 与第 12 篇"不引入 LangChain4j"同构对照

这次选 pgvector 与本系列第 12 篇"不引入 LangChain4j"是同一个决策逻辑：能不加新依赖就不加。

- 第 12 篇：Java 生态已能直接调 OpenAI 接口，引入 LangChain4j 只会增加一层抽象和依赖风险，所以不加。
- 本篇：已有 PostgreSQL，pgvector 能以扩展形式满足需求，引入 Milvus/ES 只会增加一套独立集群和运维成本，所以不加。

两次决策遵循同一原则——<span style="color: red; font-weight: bold;">优先复用已有基础设施，而非为单一功能引入新组件。</span>

### 6.6 元原则：不为假想未来过度设计

这是贯穿整个系列的技术决策原则：<span style="color: red; font-weight: bold;">用能解决当前问题的最简单方案，不为假想的未来过度设计。</span>

具体到向量数据库选型：一期数据量在 pgvector 的舒适区，就先用 pgvector；后续如果数据量真的大到扛不住，第 6.4 (4) 的检索层抽象已经留好了切换口子，按需升级即可，成本可控。先把当前路径跑通，再谈升级，比一开始就背上 Milvus/ES 的运维包袱更划算。

### 7.3 核心建表与查询 SQL（Code Block，不改写）

Claude Code 给了建表和查询的核心 SQL，原文如下：

```sql
CREATE EXTENSION IF NOT EXISTS vector;

CREATE TABLE document_chunk (
    id BIGSERIAL PRIMARY KEY,
    content TEXT NOT NULL,
    embedding vector(1536)
);

SELECT content,
       1 - (embedding <=> '[0.011, -0.033,...]') AS similarity
FROM document_chunk
ORDER BY embedding <=> '[0.011, -0.033,...]'
LIMIT 3;
```

### 7.4 深挖 <=> 余弦距离运算符

`<=>` 是 pgvector 的余弦距离运算符。用 1 - 距离 转成相似度分数，越接近 1 越相似。这个运算符是 pgvector 扩展引入的，标准 SQL 里没有。

#### (1) 为什么是 1 - 距离 转相似度

余弦距离的取值范围是 0 到 2：0 表示方向完全一致（最相似），2 表示方向完全相反。距离越小越相似这个语义不直观，所以实践中常用 `1 - 距离` 把它映射到相似度分数——分数越接近 1 越相似，符合人对"相似度"的直觉认知。

#### (2) 为什么标准 SQL 里没有这个算符

标准 SQL 面向的是标量类型（整数、字符串、日期等），没有向量类型，自然也没有向量距离运算符。pgvector 作为 PostgreSQL 的扩展，注册了 `vector` 类型和 `<=>`（余弦距离）、`<->`（L2 距离）、`<#>`（负内积）三个运算符，把向量检索能力以 SQL 语法暴露出来，让 Java 生态可以直接用 JDBC 调用，零额外学习成本。

若读者对 pgvector 不熟，也不知道 `<=>` 是什么，没关系。可按本篇的探路思路自行深入 pgvector，本篇不展开。

### 7.5 Java/MyBatis-Plus 集成的坑

![[imgs/aicent-20-fea-rag-1/9eaa651399a2aa1ef155ab8f722424e4_MD5.jpg]]

跑通 SQL 之后，下一步是 Java 集成。提问模板原文如下：

```
pgvector 在 Java 里怎么用？MyBatis-Plus 能直接操作 vector 类型吗？
```

Claude Code 的回答点出了一个容易被忽略的坑：MyBatis-Plus 不原生支持 vector 类型，直接用会报类型转换错误。

#### (1) 坑的表象

MyBatis-Plus 在处理实体字段时，没有内置 `vector` 类型的 JDBC 映射。把 Java 实体里声明为 `float[]` 的字段直接插入 pgvector 的 `vector(1536)` 列，会触发类型转换异常，因为 MyBatis 默认会把 `float[]` 当作数组或二进制处理，而 PostgreSQL 侧期望的是形如 `[0.011,-0.033,...]` 的向量字符串字面量。

#### (2) 解法：自定义 TypeHandler

解法是自定义 TypeHandler，告诉 MyBatis 怎么把 Java 的 `float[]` 和 pgvector 的 `vector` 类型互转：

- 插入时：`float[]` 转向量字符串（拼成 `[v1,v2,...,v1536]` 形式）。
- 查询时：向量字符串转 `float[]`（按逗号切分、解析浮点数）。

TypeHandler 只需写一次，注册后在所有涉及 vector 字段的 Mapper 上通用，后续业务代码完全无感。

### 7.6 深挖 why：坑为什么是 Demo 的最大价值

这类坑——"这个类型需要特殊处理"——是新技术里最容易在集成阶段踩到的。

#### (1) 类型需特殊处理是集成期最易踩的坑

新技术的坑通常不在于"它能不能做"，而在于"它和现有工程栈怎么接"。类型系统、序列化协议、SDK 兼容性这类问题，文档里往往一笔带过甚至不提，但真正集成时卡住人的恰恰是这些点。pgvector 的 vector 类型与 MyBatis-Plus 的不兼容，就是典型的"类型需特殊处理"坑。

#### (2) 最小 Demo 把坑提前暴露在干净环境里

先跑最小 Demo 的价值之一，就是把这些坑提前暴露出来，在一个干净的环境里解决，而不是混在业务代码里排查。<span style="color: red; font-weight: bold;">干净环境里只有"建表、插数据、查询"三件事，一旦报错能立刻定位是 TypeHandler 的问题</span>；而如果在 Hify 的 ChatService、DocumentService、KnowledgeBaseService 交织的代码里第一次遇到，光是确认"是不是 Agent 配置出了问题"就要绕一大圈。

### 7.7 探路节奏总结

从完全没用过 pgvector，到 Java 代码能插入和查询向量，这就是探路陌生技术的节奏。

这个节奏不是"把文档从头读完再动手"，而是：

1. 直接问怎么用——带着上手目标发问，让 Claude Code 给出最小可运行示例。
2. 拿最小可运行示例——建表、插数据、查询三段 SQL，不掺杂任何业务逻辑。
3. 跑通建立手感——亲手确认向量怎么存、距离算符怎么读、Java 怎么接。

跑通就有手感了。这一步的产出物是"Java 代码能插入和查询向量"这个可观测结果，它意味着 pgvector 这块陌生区域已经被消化，可以进入 Hify 的工程集成。

## 8. 第四步实战：Embedding 模型选型与接入

<img src="imgs/aicent-20-fea-rag-1/d86b9c3cc60c627822b1e35859af8bfe_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

### 8.1 提问模板

向量数据库搞定了。接下来是 Embedding 模型，它是把文本变成向量的工具。提问模板原文如下：

```
我要调用 OpenAI 的 Embedding API。接口怎么调？和 Chat Completion 有什么区别？有哪些模型可以选？有什么限制？
```

### 8.2 深挖 why：接口差异

问这个问题是因为：Embedding 接口虽然也是调 OpenAI，但它和 Chat Completion 是完全不同的接口，返回结构也不一样——Chat Completion 返回文本，Embedding 返回向量数组。把差异搞清楚，才不会用错。

#### (1) 调用本质不同决定接口形态不同

Chat Completion 的本质是"文本→文本"：输入一段 prompt，输出一段模型生成的回答。Embedding 的本质是"文本→向量"：输入一段文本，输出该文本在语义空间里的坐标。前者需要模型进行生成式推理（带随机性、可流式），后者只需要模型做一次前向编码（确定性、一次性返回）。两种本质差异决定了接口路径、入参、返回结构都不同。

#### (2) 搞清差异才不会用错

如果不搞清差异，常见的误用包括：把 Embedding 接口当流式调用（会报参数错误）、把返回的向量数组当字符串处理（会丢失精度）、把 Embedding 模型名填到 Chat Completion 的 model 字段（会返回 404）。提前问清差异，能在接入阶段省掉大量调试时间。

### 8.3 Chat Completion vs Embedding 接口对比表

Claude Code 给了清晰的对比。下表把两个接口的关键差异列出来：

| 接口 | 输入 | 输出 | 是否流式 | 调用本质 |
|---|---|---|---|---|
| Chat Completion | `{model, messages}`，messages 为消息数组 | `{choices:[{message:{content}}]}`，一段文本 | 支持流式（`stream: true`） | 文本 → 文本（生成式，有随机性） |
| Embedding | `{model, input}`，input 为字符串或字符串数组 | `{data:[{embedding}]}`，embedding 为长度 1536 的 float 数组 | 不支持流式，同步返回 | 文本 → 向量（编码式，确定性） |

接口是 `POST /v1/embeddings`，入参 `{model, input}`，返回 `data[0].embedding`，一个长度 1536 的 float 数组。和 Chat Completion 最大的区别是没有流式响应，同步返回，等向量算完一次性返回。

基于上图对比，最终选 `text-embedding-3-small`。理由有三条：

1. 同样 1536 维，pgvector 建表不需要改——第 7 章的 `embedding vector(1536)` 定义直接复用，零迁移成本。
2. 价格是 `ada-002` 的五分之一——入库量越大省得越多，长期成本优势明显。
3. 效果还更好——新模型在评测基准上优于老模型，不是用精度换价格。

### 8.5 两个关键特性：批量与确定性

另外 Claude Code 提醒了两个重要特性，这两个特性直接决定了入库管线的实现方式。

#### (1) 批量处理：input 支持数组

`input` 字段支持数组，可以一次传多个文本块。文档入库时一次传多个文本块，比循环单条调用省时间也省钱：

- 省时间：减少 HTTP 往返次数，网络延迟只付一次。
- 省钱：批量调用在计费上与单条调用按 token 总量一致，但不会因多次请求产生额外开销。

这个特性让"把一份 100 页的产品手册切成几百个块一次性入库"成为可行操作，而不是要等几十分钟的单条循环。

#### (2) 确定性：同文本同模型向量恒定

<span style="color: red; font-weight: bold;">同一段文本，向量永远相同。</span>Embedding 是确定性的，不像 Chat Completion 有随机性。同一模型同一文本每次返回完全一样的向量。

这个特性的工程价值在于：

- 可以缓存——文本块的向量算过一次就存进 pgvector，不需要重复调用。
- 无需幂等处理——重跑入库脚本不会因为向量漂移导致检索结果不稳定。
- 支持增量入库——新文档只对新切块调 Embedding，老切块的向量直接复用。

### 8.6 接入实现（复用第 14 篇 LlmHttpClient）

实现接入很简单，复用第 14 篇封装的 `LlmHttpClient`，加一个 `embed` 方法。传入文本字符串，拿回 `float[]`。

实现逻辑和 Chat Completion 类似，只是接口路径和返回字段不同：

| 维度 | Chat Completion 实现 | Embedding 实现 |
|---|---|---|
| 接口路径 | `/v1/chat/completions` | `/v1/embeddings` |
| 请求体 | `{model, messages}` | `{model, input}` |
| 返回字段解析 | `choices[0].message.content` | `data[0].embedding` |
| 返回类型 | `String` | `float[]` |

因为两套实现在 HTTP 调用、鉴权、错误处理、重试上是同构的，所以共用 `LlmHttpClient` 的底层设施，只在最外层加一个 `embed` 方法封装路径和返回字段差异即可。这种"复用基础设施、差异化端点"的写法，是接入第三方 API 时的标准做法。

### 8.7 常见误区：Embedding 与 Chat 模型不必配套

还有一个常见误区值得说清楚：Embedding 模型和 Chat 模型不需要配套。

Embedding 只管把文本转向量，和后面用什么 LLM 生成回答完全无关。Hify 里 Embedding Provider 和 Chat Provider 可以独立配置：

- 可以用 OpenAI 做 Embedding，用 Anthropic Claude 生成回答。
- 也可以用本地 BGE 模型做 Embedding，用 OpenAI 生成回答。
- 甚至可以用一个厂商的老 Embedding 模型 + 另一个厂商的新 Chat 模型。

这个误区的来源是把"同厂商"等同于"必须配套"，实际上向量空间和生成能力是两套独立的系统：<span style="color: red; font-weight: bold;">向量只用来做检索排序，生成只用来组织语言，二者通过"把检索到的原文块拼进 Prompt"这个动作衔接，不存在模型层面的耦合。</span>理解了这一点，在选型时就可以分别按各自的最优解挑选，而不是被"必须用同一家"的假约束绑住。

1. pgvector 维度：

### 8.9 拓展思考（原文三个思考，第三人称、练习风格）

本篇结尾保留原文留下的三个思考，改成练习风格，供读者作为延伸训练。

#### (1) 约束变化下 pgvector 选型是否依然成立

本篇选 pgvector 的核心理由是"不引入新依赖"。如果场景变成：数据量千万级、需要毫秒级检索、团队有专职运维，请读者用同样的约束驱动选型方法重新判断——pgvector 是否依然是最佳选择？还是应该切换到 Milvus、Qdrant 等专业向量数据库？建议把新的约束清单列出来，与 Claude Code 讨论一遍，得出有立场的选型结论。

#### (2) 数据隐私场景下的本地开源 Embedding 替代方案

本篇 Embedding 模型选了 OpenAI 的 `text-embedding-3-small`。如果出于数据隐私考虑，不想把公司文档内容发给 OpenAI，有哪些本地部署的开源替代方案？建议读者用本篇的四步探路法调研一遍（痛点提问 → 拆组件 → 约束选型 → 最小 Demo），重点考察 BGE、M3E、E5 等主流开源 Embedding 模型，得出符合自身约束（本地化、中文友好、部署复杂度、效果）的选型结论。

#### (3) pgvector 的 IVFFlat 与 HNSW 索引机制

pgvector 的相似度查询在数据量大时会变慢，这时需要建索引。请读者让 Claude Code 帮忙了解 pgvector 的两种索引机制——IVFFlat 和 HNSW——各自的原理、适用场景和建索引时机。重点搞清楚三个问题：

- 两种索引的工作原理分别是什么（IVFFlat 基于倒排、HNSW 基于分层图）。
- 各自适合什么场景（数据量规模、召回率要求、内存预算）。
- 什么时候需要建、怎么建（建索引的时机和参数调优）。

这是把 pgvector 从"能用"推进到"好用"的关键一步，也是第 21 篇数据管线落地后必然会遇到的性能优化问题。
