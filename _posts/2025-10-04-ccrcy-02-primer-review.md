## 2. 并发编程的价值

**并发编程**与**性能优化**密切相关，合理运用并发编程技术可以带来以下价值：

### 2.1 加快响应时间

#### (1) 基本原理

多线程并发能有效缩短任务执行时间。以迅雷下载为例，用户通常会选择多线程下载而非单线程，原因很直观：多个线程同时下载，速度更快。

#### (2) 应用场景

| 场景类型 | 说明 | 性能优势 |
| --- | --- | --- |
| **文件下载** | 迅雷等多线程下载工具 | 多线程同时下载，显著提升下载速度 |
| **Web应用优化** | 互联网应用的页面响应优化 | 响应时间优化 1 秒，在高流量场景下即可显著提升转化率 |
| **静态资源加载** | Web前端性能优化技术 | 将静态资源分散到多个子域名加载，每个子域名让浏览器开启更多线程并发加载页面资源，从而提升页面加载速度 |

### 2.2 代码模块化与异步化

**基本原理**

并发编程能够实现**业务模块解耦**和**异步化处理**。

**应用示例：电商系统**

| 处理方式 | 操作流程 | 说明 |
| --- | --- | --- |
| **同步方式** | 下单 → 发送短信 → 发送邮件 → 返回成功 | 所有操作串行执行，用户等待时间长 |
| **异步方式** | 下单 → 独立线程处理短信和邮件 | 将通知类任务独立为单独模块，交给其他线程异步执行 |

**设计优势**

| 优势维度 | 说明 |
| --- | --- |
| **响应提速** | 主流程无需等待耗时操作完成，可立即响应用户 |
| **吞吐量提升** | 系统吞吐量提高，能够处理更多并发请求 |
| **模块化** | 业务逻辑清晰分离，各模块职责单一 |
| **可读性增强** | 代码结构更加清晰，易于理解和维护 |
| **复杂度降低** | 便于后续扩展和重构 |

### 2.3 充分利用CPU资源

#### (1) 多核CPU场景

**基本原理**

当前主流 **CPU** 均采用**多核架构**。若采用单线程技术思路，无法充分利用 **CPU** 的**多核特性**。

| 技术维度 | 说明 |
| --- | --- |
| **多线程设计** | 在多个 **CPU** 核心上同时运行多个线程 |
| **资源利用** | 充分利用 **CPU** 资源，减少 **CPU** 空闲时间 |
| **性能提升** | 发挥运算能力，提高并发处理能力 |

**生活类比：时间复用**

这如同日常生活中的**时间复用**：很多人乘坐长线地铁时会利用碎片时间阅读，而非将通勤时间和阅读时间分开安排。这种做法相当于将 1 小时当作 2 小时用。

同样的道理也体现在工作中：有些人擅长并发处理多项任务，充分利用时间；有些人则串行处理，总感觉时间不够用。**CPU** 资源利用也遵循这个道理。

#### (2) 单核CPU场景

对于**单核 CPU**，是否也能利用并发编程的优势？答案是可以的。

**应用示例：QQ聊天程序**

| 并发任务 | 说明 |
| --- | --- |
| **接收键盘输入** | 用户在输入框中输入文字 |
| **通过网络发送消息** | 将用户输入的消息发送到服务器 |
| **接收网络消息** | 从服务器接收对方发送的消息 |
| **在屏幕显示消息** | 将接收到的消息显示在聊天窗口 |

**处理方式对比**

| 处理方式 | 执行流程 | 用户体验 |
| --- | --- | --- |
| **串行方式** | 一问一答，发送完成后才能接收 | 交互受限，体验差 |
| **并发方式** | 多项任务**同时进行**，互不阻塞 | 流畅自然，体验好 |

> **价值体现**：即使只有单核 CPU，并发编程也能实现任务间的**协作与配合**，通过快速切换给用户"同时执行"的体验，而不是让某个任务独占 CPU 导致其他任务无法响应。

## 3. 并发基础概念

在正式学习Java的并发编程之前，还有几个并发编程的基础概念我们需要熟悉和学习。

### 3.1 进程与线程的概念

#### (1) 进程

**基本定义**

应用程序由**指令**和**数据**组成。程序未运行时，代码以**二进制形式**存储在**磁盘**（如 U 盘、远程网络存储等）中。程序启动后，需要将**指令**加载至 CPU，将**数据**加载至内存，运行过程中还需使用**磁盘**、**网络**等设备。从这个角度看，**进程**就是用于加载指令、管理内存、管理 I/O 的运行实体。

**进程创建**

当一个程序从磁盘加载至内存并运行时，就创建了一个进程。

**进程实例化特性**

| 程序类型 | 实例支持 | 典型示例 |
| --- | --- | --- |
| **多实例程序** | 可同时运行多个进程 | 记事本、画图、浏览器 |
| **单实例程序** | 只能启动一个进程 | 网易云音乐、360 安全卫士 |

**程序与进程的区别：**

| 对比维度 | 程序 | 进程 |
| --- | --- | --- |
| **状态** | 静态 | 动态 |
| **存在形式** | 磁盘上的二进制代码 | 内存中的运行实体 |
| **生命周期** | 持久存储 | 运行时创建与销毁 |

**进程分类**

| 进程类型 | 说明 | 示例 |
| --- | --- | --- |
| **系统进程** | 完成操作系统功能的进程 | 操作系统内核进程 |
| **用户进程** | 由用户启动的进程 | 应用程序进程 |

**资源分配单位**

从操作系统角度，进程是资源分配的最小单位（以内存为主）。

#### (2) 线程

**基本原理**

一台计算机同时运行多个程序，但 **CPU 数量有限**。为让有限 CPU 执行多个程序，需要一种机制协调程序之间的执行，即 **CPU 调度**。线程是 CPU 调度的最小单位。

**核心定义**

| 特性维度 | 说明 |
| --- | --- |
| **依赖关系** | 线程必须依附于进程存在，是进程中的执行实体 |
| **调度角色** | 是 CPU 调度和分派的基本单位 |
| **资源占用** | 仅拥有运行时必不可少的资源（程序计数器、一组寄存器和栈） |
| **资源共享** | 与同属一个进程的其他线程共享进程所拥有的全部资源 |
| **数量关系** | 一个进程可以拥有多个线程，一个线程必须有一个父进程 |

**线程与轻量级进程**

线程有时也被称为 **轻量级进程**（Lightweight Process，LWP）。

| 维度 | 说明 |
| --- | --- |
| **概念来源** | 早期 Linux 的线程实现几乎完全复用了进程机制 |
| **技术演进** | 后来才独立出专属 API（如 NPTL） |
| **名称含义** | "轻量级"体现在资源占用少、创建销毁快 |

#### (3) 进程与线程的区别

进程与线程在多个维度存在差异：

##### ① 独立性差异

| 对比维度     | 进程             | 线程             |
| -------- | -------------- | -------------- |
| **基本特性** | 相互独立，拥有完整的资源空间 | 从属于进程，是进程的执行单元 |
| **依赖关系** | 独立运行，不依赖其他进程   | 必须依附于进程存在      |

##### ② 资源共享机制

| 对比维度 | 进程 | 线程 |
| --- | --- | --- |
| **资源归属** | 拥有独立的内存空间和系统资源 | 共享所属进程的全部资源（内存、文件句柄等） |
| **隔离性** | 进程间相互隔离，无法直接访问彼此资源 | 同一进程内的线程可直接访问共享资源 |

##### ③ 通信复杂度

**进程间通信（IPC）**

| 维度 | 说明 |
| --- | --- |
| **定义** | 同一台计算机的进程间通信称为 **IPC（Inter-Process Communication）** |
| **跨机通信** | 不同计算机之间的进程通信需要通过网络，并遵守共同的协议（如 **HTTP**） |
| **复杂度** | 进程间通信相对复杂，因为进程拥有独立的内存空间，需要通过特定的 IPC 机制 |

**线程间通信**

| 维度 | 说明 |
| --- | --- |
| **基本原理** | 线程间通信相对简单，因为它们共享进程内的内存 |
| **访问方式** | 多个线程可以直接访问同一个共享变量，无需复杂的通信机制 |
| **注意事项** | 仍需注意同步问题，避免数据竞争 |

**通信方式对比**

| 对比维度 | 进程间通信 | 线程间通信 |
| --- | --- | --- |
| **通信方式** | 需要 IPC 机制（管道、消息队列、共享内存等） | 直接访问共享变量 |
| **复杂度** | 较高，需要额外的同步机制 | 较低，但仍需注意同步问题 |
| **开销** | 较大 | 较小 |

##### ④ 性能开销

线程被称为**轻量级进程**，其上下文切换成本通常低于进程：

| 对比维度 | 进程 | 线程 |
| --- | --- | --- |
| **资源占用** | 较大（独立的内存空间） | 较小（共享进程资源） |
| **上下文切换** | 开销较高（需切换完整的地址空间） | 开销较低（仅切换线程栈和寄存器） |
| **创建速度** | 较慢（需分配独立资源） | 较快（复用进程资源） |
| **典型应用** | 适合隔离性要求高的场景 | 适合并发处理和任务协作 |

### 3.2 CPU核心数与线程数

#### (1) 基本概念

目前主流 **CPU** 均采用**多核架构**。理解 **CPU** 核心数与线程数的关系，有助于合理规划并发策略。

| 核心特性 | 说明 |
| --- | --- |
| **执行单元** | 线程是 **CPU** 调度的基本单位 |
| **1:1 关系** | 同一时刻，一个 **CPU** 核心只能执行一个线程 |
| **物理上限** | **CPU** 物理核心数决定了同时运行线程数的上限 |
| **典型示例** | **8 核 CPU** 可同时执行 **8 个线程** |

#### (2) 超线程技术

**Intel** 引入**超线程技术**后，单个物理核心可以对应多个**逻辑处理器**。

| 对比维度 | 物理核心 | 逻辑处理器 |
| --- | --- | --- |
| **数量关系** | 1 | 2（启用超线程后） |
| **Windows任务管理器示例** | 内核数为 **6** | 逻辑处理器数为 **12** |

**技术特点**：

| 特性维度 | 说明 |
| --- | --- |
| **映射关系** | 启用超线程后，物理核心与逻辑处理器呈 **1:2** 的关系 |
| **工作方式** | 通过复用 CPU 资源，在单个核心上模拟多个执行单元 |

#### (3) Java获取CPU核心数

**Java** 提供了获取 **CPU** 核心数的 **API**：

```java
Runtime.getRuntime().availableProcessors()
```

**使用说明**：

| 注意事项 | 说明 |
| --- | --- |
| **返回值** | 该方法返回的是**逻辑处理器**数量，而非物理核心数 |
| **应用场景** | 在**并发编程**中，获取 **CPU** 核心数有助于优化线程池配置 |
| **优化参考** | 制定**性能优化**策略时，需要根据 **CPU** 核心数来调整线程数量 |

### 3.3 上下文切换

#### (1) 上下文概念

**基本原理**

操作系统需要在多个进程（线程）之间进行调度。每个线程执行时会使用 CPU 寄存器和程序计数器等资源。为保证线程在调度前后能正常执行，操作系统引入了**上下文切换**机制。

**核心定义**

| 概念 | 定义 |
| --- | --- |
| **上下文** | CPU 寄存器和程序计数器在某一时刻的内容快照 |
| **寄存器** | CPU 内部的小容量高速存储器（速度高于 CPU 缓存和主内存 RAM），通过快速访问常用数据来加速程序执行 |
| **程序计数器** | 专用寄存器，用于指示 CPU 在指令序列中的执行位置，保存当前或下一条将要执行的指令地址 |

<img src="/imgs/ccrcy-02-primer-review/c29bbf4ce0b49502b2a89bf6c7b331ed_MD5.jpg" style="display: block; width: 650px;" alt="c29bbf4ce0b49502b2a89bf6c7b331ed_MD5">

**多视角理解**

| 视角 | 说明 |
| --- | --- |
| **程序员视角** | 方法调用过程中的局部变量与资源 |
| **线程视角** | 方法调用栈中存储的各类信息 |

#### (2) 切换过程

**定义**

上下文切换是**操作系统内核**在 CPU 上切换进程（包括线程）的完整流程。

**执行步骤**

| 步骤 | 操作 | 说明 |
| --- | --- | --- |
| **步骤 1** | **保存当前进程状态** | 暂停当前进程执行，将其 **CPU 状态**（即上下文）保存到内存。包括：<br>• **寄存器**中的数据<br>• **程序计数器**的值（指向当前执行指令的地址） |
| **步骤 2** | **加载下一个进程状态** | 从内存读取下一个进程的上下文信息，并将其恢复到 **CPU 寄存器**，使 CPU 能够从上次中断的位置继续执行 |
| **步骤 3** | **恢复进程执行** | 跳转到**程序计数器**指示的指令地址（即进程被中断时的代码行），恢复该进程的正常执行 |

#### (3) 性能影响

**触发场景**

上下文切换在以下情况下发生：

| 触发类型 | 说明 |
| --- | --- |
| **线程调度** | CPU 时间片用尽、优先级调整 |
| **进程切换** | 多进程并发执行时的调度 |
| **系统调用** | 用户态与内核态切换 |

**性能开销**

上下文切换是**计算密集型操作**，主要性能消耗包括：

| 开销类型 | 具体消耗 |
| --- | --- |
| **寄存器数据拷贝** | 保存和恢复 CPU 寄存器状态 |
| **缓存失效** | 切换后 CPU 缓存命中率下降 |
| **指令流水线重置** | 打断 CPU 指令流水线执行 |

**时钟周期对比**

| 操作类型 | 时钟周期数 | 对比关系 |
| --- | --- | --- |
| **简单指令** | 5 ~ 15 个 | 基准单位 |
| **上下文切换** | 5000 ~ 20000 个 | 是简单指令的 **300 ~ 1300 倍** |

> 从 CPU 时间成本来看，一次上下文切换的开销相当于执行 **300 ~ 1300 条简单指令**。在高并发场景下，频繁的上下文切换会**降低系统整体性能**。

### 3.4 并发与并行的区别

#### (1) CPU 与高速公路的类比

为理解并发与并行，先通过 CPU 与高速公路的类比建立直观认知：

| 类比维度 | 高速公路系统 | CPU 系统 |
| --- | --- | --- |
| 基本单元 | 单条高速公路 | 单个 CPU |
| 并行能力 | 8 条车道可并行 8 辆车 | 多个核心/线程可同时执行多个任务 |
| 扩展方式 | 多条高速公路并排 | 多 CPU 系统 |

从硬件角度看，并行的物理上限由 CPU 核心数决定。
#### (2) 并发

**定义**：并发指应用在单位时间内交替执行多个任务的能力。

**本质特征**：通过快速任务切换实现"同时执行"的效果。

**单核场景下的执行机制**：

| 执行特征 | 具体说明 |
| --- | --- |
| 实际执行 | 线程轮流使用 CPU 时间片 |
| 执行方式 | 在极短时间内交替切换（毫秒甚至微秒级） |
| 用户感知 | 因切换速度极快，用户无法察觉到交替过程 |

**总结**：微观串行，宏观并行。

#### (3) 并行

**定义**：并行指应用在同一时刻同时执行多个任务的能力。

**硬件要求**：必须依赖多核 CPU 支持。

**多核场景下的执行机制**：

| 执行特征 | 具体说明 |
| --- | --- |
| 实际执行 | 每个 CPU 核心独立调度一个线程 |
| 执行方式 | 多个任务真正同时进行，无需切换 |

<img src="/imgs/ccrcy-02-primer-review/620dc4c34593f0fd528667c0cfaa11d0_MD5.jpg" style="display: block; width: 500px;" alt="620dc4c34593f0fd528667c0cfaa11d0_MD5">

#### (4) 两者的区别

通过上述概念辨析，可以总结出两者的主要差异：

| 对比维度 | 并发 | 并行 |
| --- | --- | --- |
| 执行方式 | 交替执行，通过快速切换实现"同时"的效果 | 同时执行，多个任务真正同步进行 |
| 时间维度 | 单位时间内处理多个任务 | 同一时刻执行多个任务 |
| 硬件要求 | 单核 CPU 即可实现 | 需要多核 CPU 支持 |
| 本质特征 | 微观串行，宏观并行 | 微观并行 |

从执行时机来看：并发强调的是交替执行的能力，并行强调的是同时执行的能力。

## 4. Java线程入门

### 4.1 Java程序的线程特性

Java 程序从 `main()` 方法开始执行，按照代码逻辑顺序运行。看起来程序似乎只有主线程在工作，但实际上 **Java 程序天生就是多线程的**，执行 `main()` 方法的只是其中一个名为 `main` 的线程。

**获取 JVM 线程信息**

可以通过 `ThreadMXBean` 查看当前 JVM 中的所有线程：

```java
public class OnlyMain {
    public static void main(String[] args){
        // Java 虚拟机线程系统的管理接口
        ThreadMXBean threadMXBean = ManagementFactory.getThreadMXBean();
        // 不需要获取同步的 monitor 和 synchronizer 信息，仅获取线程和线程堆栈信息
        ThreadInfo[] threadInfos = threadMXBean.dumpAllThreads(false, false);
        // 遍历线程信息，仅打印线程 ID 和线程名称信息
        for (ThreadInfo threadInfo : threadInfos) {
            System.out.println("[" + threadInfo.getThreadId() + "] " + threadInfo.getThreadName());
        }
    }
}
```

**JVM 默认启动的线程**

即使没有手动创建线程，JVM 也会自行启动多个系统线程：

```bash
[6] Monitor Ctrl-Break    // 监控 Ctrl-Break 中断信号
[5] Attach Listener       // 内存 dump、线程 dump、类信息统计、获取系统属性等
[4] Signal Dispatcher     // 分发处理发送给 JVM 信号的线程
[3] Finalizer             // 调用对象 finalize 方法的线程
[2] Reference Handler     // 清除 Reference 的线程
[1] main                  // main 线程，用户程序入口
```

这些线程的数量和名称会根据 **JDK 版本**和**运行环境**有所差异，但都验证了 Java 程序的多线程特性。

### 4.2 线程创建与启动

#### (1) 概述

前面介绍的线程都是 **JVM 自动启动的系统线程**。实际开发中，我们更关注如何**自主创建和管理线程**，以便在程序中灵活运用并发编程。

#### (2) 三种方式

Java 中创建线程主要有 **三种方式**，分别适用于不同的使用场景。

##### ① 使用 Thread 类或继承 Thread 类

**基本用法：**

```java
// 创建线程对象
Thread t = new Thread() {
    public void run() {
        // 要执行的任务
    }
};
// 启动线程
t.start();
```

**实际示例：**

```java
// 构造方法的参数是给线程指定名字，推荐
Thread t1 = new Thread("t1") {
    @Override
    // run 方法内实现了要执行的任务
    public void run() {
        log.debug("Hello Thread");
    }
};
t1.start();
```

**方式特点：**

| 特点 | 说明 |
| --- | --- |
| **简单直接** | 适合简单的线程创建场景 |
| **耦合度高** | 任务逻辑与线程定义绑定在一起 |
| **扩展性差** | 由于 Java 不支持多继承，继承 Thread 后无法继承其他类 |

##### ② 实现 Runnable 接口配合 Thread

这种方式将 **线程** 与 **任务** 分离，体现了更好的设计理念。

**概念区分：**

| 概念 | 说明 |
| --- | --- |
| **Thread** | 代表线程本身 |
| **Runnable** | 代表可运行的任务（线程要执行的代码） |

**基本用法：**

```java
Runnable runnable = new Runnable() {
    public void run(){
        // 要执行的任务
    }
};
// 创建线程对象
Thread t = new Thread( runnable );
// 启动线程
t.start();
```

**实际示例：**

```java
// 创建任务对象
Runnable task2 = new Runnable() {
    @Override
    public void run() {
        log.debug("hello");
    }
};
// 参数1 是任务对象; 参数2 是线程名字，推荐
Thread t2 = new Thread(task2, "t2");
t2.start();
```

**Java 8 优化：**

Java 8 引入 lambda 表达式后，代码更加精简：

```java
// 创建任务对象
Runnable task2 = () -> log.debug("hello");
// 参数1 是任务对象; 参数2 是线程名字，推荐
Thread t2 = new Thread(task2, "t2");
t2.start();
```

**核心优势：**

| 优势 | 说明 |
| --- | --- |
| **职责分离** | 将线程与任务分开，符合单一职责原则 |
| **灵活扩展** | 任务类可以继承其他类，避免单继承限制 |
| **易于复用** | 同一个任务可以交给不同线程执行 |
| **便于维护** | 代码结构更清晰，易于与线程池等高级 API 配合 |

**方式对比：**

| 对比维度 | 方式 ①（继承 Thread） | 方式 ②（实现 Runnable） |
| --- | --- | --- |
| **抽象层次** | Thread 是线程的唯一抽象 | Runnable 是任务的抽象 |
| **耦合程度** | 线程与任务合并在一起 | 线程与任务分离 |
| **灵活性** | 较低，受单继承限制 | 较高，可以继承其他类 |
| **适用场景** | 简单场景 | 推荐使用，更符合设计原则 |

> **实践建议**：优先使用 **方式 ②**，让任务类脱离 Thread 继承体系，提升代码灵活性和可维护性。

##### ③ 使用 FutureTask 配合 Thread

**适用场景：**

前两种方式都无法获取线程执行的 **返回结果**，`FutureTask` 则专门用于处理需要返回值的任务。

**基本用法：**

```java
// 创建任务对象
FutureTask<Integer> task3 = new FutureTask<>(() -> {
    log.debug("hello");
    return 100;
});
// 参数1 是任务对象; 参数2 是线程名字，推荐
new Thread(task3, "t3").start();
// 主线程阻塞，同步等待 task 执行完毕的结果
Integer result = task3.get();
log.debug("结果是:{}", result);
```

**接口对比：**

| 接口 | 所在包 | 方法 | 返回值 |
| --- | --- | --- | --- |
| **Runnable** | java.lang | run() | void（无返回值） |
| **Callable** | java.util.concurrent | call() | V（泛型返回值） |

**Future 接口的作用：**

`Future` 接口用于对 `Runnable` 或 `Callable` 任务的执行结果进行控制：

| 功能 | 说明 |
| --- | --- |
| **取消任务** | 可以取消任务的执行 |
| **查询状态** | 查询任务是否完成 |
| **获取结果** | 通过 `get()` 方法获取执行结果，会阻塞直到任务完成 |

##### ② run 和 start 的区别

**Thread 类的作用**

Thread 类是 Java 对线程概念的抽象。通过 `new Thread()` 只是创建了一个 Thread 实例，此时还未与操作系统中的真正线程建立关联。只有调用 `start()` 方法后，才真正启动了操作系统层面的线程。

从 Thread 源码可以看到，`start()` 方法内部调用了 `start0()` 方法，而 `start0()` 是一个 **native 方法**，这说明 `Thread#start()` 与操作系统密切相关。

**核心区别对比：**

| 对比维度 | start() 方法 | run() 方法 |
| --- | --- | --- |
| **功能** | 启动线程，让线程进入就绪队列等待 CPU 分配 | 执行业务逻辑的代码 |
| **执行时机** | 调用后线程进入就绪状态，获得 CPU 后才执行 run() | 可被单独调用，或由线程调度器调用 |
| **调用次数** | **只能调用一次**，重复调用会抛出 `IllegalThreadStateException` | 可以**重复执行**或被单独调用 |
| **线程创建** | 会创建操作系统线程并与之建立映射 | 不会创建新线程，仅执行方法代码 |
| **本质** | 启动线程的入口方法 | 普通的成员方法，与其他类的方法无本质区别 |

> **注意**：多次调用一个线程的 `start()` 方法会抛出 `IllegalThreadStateException` 异常。

**工作流程：**

```
调用 start() → 线程进入就绪状态 → 等待 CPU 分配 → 获得 CPU → 执行 run() 方法
```

## 5. Java线程核心机制

### 5.1 线程的状态与生命周期

#### (1) 操作系统层面的五种状态

从操作系统层面，线程生命周期包含 **五种基本状态**：

<img src="/imgs/ccrcy-02-primer-review/9a4f03274e03acdad2e26121504d08d8_MD5.jpg" style="display: block; width: 500px;" alt="9a4f03274e03acdad2e26121504d08d8_MD5">

| 状态            | 说明                          | 转换条件                    |
| ------------- | --------------------------- | ----------------------- |
| **初始状态**      | 在语言层面创建线程对象，尚未与操作系统线程关联     | 调用 `start()` 方法后进入可运行状态 |
| **可运行状态**（就绪） | 线程已创建并与操作系统线程关联，可被 CPU 调度执行 | 获得时间片后进入运行状态            |
| **运行状态**      | 获取 CPU 时间片，正在执行指令           | 时间片用完后返回可运行状态           |
| **阻塞状态**      | 调用阻塞 API（如 BIO 读写），不占用 CPU  | 操作系统唤醒后返回可运行状态          |
| **终止状态**      | 线程执行完毕，生命周期结束               | 不可转换为其他状态               |

**状态转换要点：**

- **上下文切换**：**运行状态**与**可运行状态**之间的切换，以及**可运行状态**与**阻塞状态**之间的切换，都会导致线程上下文切换
- **阻塞与就绪的区别**：**阻塞状态**的线程即使 CPU 空闲也不会被调度，必须等待操作系统唤醒；而**可运行状态**的线程可以随时被 CPU 调度执行

#### (2) Java API层面的六种状态

Java 从 **API 层面**定义了线程的六种状态，这些状态通过 **Thread.State** 枚举表示：

| 状态 | 说明 |
| --- | --- |
| **初始 (NEW)** | 线程对象已创建，但尚未调用 `start()` 方法 |
| **运行 (RUNNABLE)** | 包含**就绪 (ready)** 和**运行中 (running)** 两种情况：线程位于可运行线程池中，等待被调度选中获取 CPU 使用权（就绪），或在获得 CPU 时间片后正在执行（运行中） |
| **阻塞 (BLOCKED)** | 线程因等待**锁**而阻塞 |
| **等待 (WAITING)** | 线程等待其他线程执行**特定动作**（通知或中断）才能返回 |
| **超时等待 (TIMED_WAITING)** | 与 WAITING 不同，可在**指定时间后**自行返回 |
| **终止 (TERMINATED)** | 线程已执行完毕 |

> **注意**：RUNNABLE 状态合并了传统操作系统中的**就绪**和**运行中**两种状态。

#### (2) sleep与yield

##### ① sleep方法

**基本特性：**

| 特性 | 说明 |
| --- | --- |
| **状态转换** | 当前线程从 Running 进入 **Timed Waiting** 状态（阻塞） |
| **锁处理** | **不会释放对象锁** |
| **中断响应** | 可通过 `interrupt()` 方法打断，抛出 `InterruptedException` |
| **执行时机** | 睡眠结束后线程**未必立即执行**，需等待 CPU 调度 |
| **特殊用法** | 传入参数为 0 时，效果等同于 `yield()` |

> **最佳实践**：建议使用 `TimeUnit.sleep()` 代替 `Thread.sleep()`，提升代码可读性。

**代码示例：**

```java
Thread t1 = new Thread(new Runnable() {
    @Override
    public void run() {
        try {
            Thread.sleep(3000);
        } catch (InterruptedException e) {
            throw new RuntimeException(e);
        }
        log.debug("执行完成");
    }
},"t1");
t1.start();
log.debug("线程t1的状态："+t1.getState());
try {
    Thread.sleep(500);
} catch (InterruptedException e) {
    throw new RuntimeException(e);
}
log.debug("线程t1的状态："+t1.getState());
//t1.interrupt();
```

**应用场景：**

在没有利用 CPU 进行计算时，避免 `while(true)` 空转浪费 CPU 资源：

```java
while(true) {
    try {
        Thread.sleep(50);
    } catch (InterruptedException e) {
        e.printStackTrace();
    }
}
```

**与其他等待方式对比：**

| 等待方式 | 是否需要加锁 | 是否需要唤醒 | 适用场景 |
| --- | --- | --- | --- |
| **sleep** | ❌ 否 | ❌ 否 | **无需锁同步**的场景 |
| **wait/条件变量** | ✅ 是 | ✅ 是 | 需要**同步协作**的场景 |

> **选择建议**：`sleep` 适用于简单的时间等待场景，无需考虑锁同步；`wait` 和条件变量适用于需要线程间精确协作的场景。

##### ② yield方法

**基本特性：**

| 特性 | 说明 |
| --- | --- |
| **状态转换** | 当前线程从 Running 进入 **Runnable** 状态（就绪） |
| **CPU 释放** | **释放 CPU 资源**，让优先级相同或更高的线程获得执行机会 |
| **锁处理** | **不会释放对象锁** |
| **调度行为** | 具体实现**依赖操作系统的任务调度器** |

> **注意事项**：如果当前进程只有 main 线程，调用 `yield()` 后 main 线程会继续运行，因为没有更高优先级的线程。

**实际应用案例：**

`ConcurrentHashMap#initTable` 方法中的 `yield()` 应用：

**应用场景分析：**

| 场景特征 | 问题分析 | yield 解决方案 |
| --- | --- | --- |
| **多线程初始化** | 多个线程同时初始化 table，但只允许一个线程执行 | 让其他线程调用 `yield()` 让出 CPU |
| **操作耗时短** | 初始化操作执行很快，阻塞/等待的上下文切换开销较大 | 避免阻塞和等待带来的上下文切换 |
| **性能优化** | Doug Lea 大师的设计，减少不必要的调度开销 | 让执行初始化的线程更快完成 |

> **设计思想**：对于**极短时间的等待**，使用 `yield()` 比阻塞/等待更高效，避免了上下文切换的开销。

#### (3) 线程优先级

**基本原理**

线程优先级是向**线程调度器**提供的**调度建议**，调度器可以选择忽略。其作用体现在两个维度：

| CPU 资源状态 | 优先级作用 |
| ---- | ---- |
| **资源紧张** | 优先级较高的线程倾向于获得**更多时间片** |
| **资源充足** | 优先级的作用**不明显** |

> **重要说明**：线程优先级仅为**建议性质**，具体调度行为取决于**操作系统**的实现策略。

**Java 实现机制**

| 属性 | 说明 |
| ---- | ---- |
| **成员变量** | `priority`（整型） |
| **取值范围** | 1 ~ 10 |
| **默认值** | 5 |
| **设置方法** | `setPriority(int)` |

**优先级设置建议**

根据线程特性设置合理的优先级，有助于优化系统整体性能：

| 线程类型 | 推荐优先级 | 设置原因 |
| ---- | ---- | ---- |
| **频繁阻塞型**（I/O 操作、休眠等） | **较高** | 减少因频繁阻塞导致的**响应延迟** |
| **计算密集型**（占用大量 CPU 时间） | **较低** | 避免**独占处理器资源** |

**平台差异说明**

不同 **JVM 实现**和**操作系统**对线程优先级的处理策略存在差异：

- **部分操作系统**会忽略优先级设置
- **相同优先级**在不同平台可能产生**不同的调度效果**

> **使用建议**：线程优先级适合作为**调优辅助手段**，不宜作为**严格的行为控制机制**。

**代码示例**

```java

    int count = 0;
    for (;;) {
        System.out.println("t1---->" + count++);
    }
};
Runnable task2 = () -> {

        // Thread.yield();
        System.out.println("t2---->" + count++);
    }
```

#### (4) join方法

##### ① 为什么需要join

**join方法**的作用是**等待调用join方法的线程结束之后，程序再继续执行**，一般用于**等待异步线程执行完结果之后才能继续运行**的场景。

> **核心价值**：解决多线程执行顺序问题，确保主线程能获取到子线程的执行结果。

**问题引出**：下面的代码执行后，打印的 count 值是多少？

```java
private static int count = 0;

public static void main(String[] args) throws InterruptedException {
    log.debug("开始执行");
    Thread t1 = new Thread(() -> {
        log.debug("开始执行");
        SleepTools.second(1);
        count = 5;
        log.debug("执行完成");
    },"t1");
    t1.start();
    log.debug("结果为:{}", count);
    log.debug("执行完成");
}
```

**输出结果**：

```bash
19:30:09.614 [main] DEBUG com.tuling.learnjuc.base.ThreadJoinDemo - 开始执行 19:30:09.660 [t1] DEBUG com.tuling.learnjuc.base.ThreadJoinDemo - 开始执行 19:30:09.660 [main] DEBUG com.tuling.learnjuc.base.ThreadJoinDemo - 结果为:0 19:30:09.662 [main] DEBUG com.tuling.learnjuc.base.ThreadJoinDemo - 执行完成 19:30:10.673 [t1] DEBUG com.tuling.learnjuc.base.ThreadJoinDemo - 执行完成
```

**原因分析**：

| 分析维度 | 具体说明 |
| --- | --- |
| **执行方式** | 主线程和线程 t1 是**并行执行**的 |
| **时间差异** | t1 线程需要 **1 秒**之后才能算出 count=5 |
| **打印时机** | 主线程**立即打印** count 结果，此时 t1 尚未执行完成 |
| **最终结果** | 只能打印出 **count=0** |

**解决方案对比**：

| 方案 | 可行性 | 问题分析 |
| --- | --- | --- |
| **使用 sleep** | 不推荐 | 无法准确预估线程执行时间，存在等待不足或过度等待的风险 |
| **使用 join** | **推荐** | **精确等待**目标线程执行完成，无需估算时间 |

> **最佳实践**：在 `t1.start()` 之后调用 `t1.join()`，即可确保主线程等待 t1 执行完成后再继续执行。

##### ② 实现同步

**同步与异步的区分**（以调用方角度）：

| 调用方式 | 定义 | 特点 |
| --- | --- | --- |
| **同步** | **需要等待结果返回**，才能继续运行 | 阻塞式调用，必须获取结果 |
| **异步** | **不需要等待结果返回**，就能继续运行 | 非阻塞式调用，通过回调或轮询获取结果 |

> **join方法实现的是同步调用**：主线程调用 `t1.join()` 后，会**阻塞等待** t1 线程执行完成，确保能够获取到 t1 的执行结果。

**代码示例**（使用join实现同步）：

```java

public static void main(String[] args) throws InterruptedException {
    log.debug("开始执行");
    Thread t1 = new Thread(() -> {
        log.debug("开始执行");
        SleepTools.second(1);
        count = 5;
        log.debug("执行完成");
    },"t1");
    t1.start();
    //SleepTools.second(1);
    t1.join();
    log.debug("结果为:{}", count);
    log.debug("执行完成");
}
```


| --- | --- | --- |

| **结果获取** | 无法获取t1的计算结果 | **成功获取**t1的计算结果 |

##### ③ 应用实例

**问题场景**：现在有 T1、T2、T3 三个线程，如何保证 **T2 在 T1 执行完后执行，T3 在 T2 执行完后执行**？

> **解决思路**：利用 `join()` 方法实现**线程顺序执行**。将指定的线程加入到当前线程，可以将两个**交替执行的线程**合并为**顺序执行**。

**实现原理**：

| 线程 | join调用 | 执行顺序 |
| --- | --- | --- |
| **T2** | 调用 `t1.join()` | **等待 T1 执行完毕**后，才执行 T2 剩余代码 |
| **T3** | 调用 `t2.join()` | **等待 T2 执行完毕**后，才执行 T3 剩余代码 |

> **关键点**：join 方法会使**当前线程**进入**等待状态**，直到**目标线程**执行完成。

```java
Thread t1 = new Thread(new Runnable() {
    @Override
    public void run() {
        log.debug("线程t1执行完成");
    }
},"t1");
Thread t2 = new Thread(new Runnable() {
    @Override
    public void run() {
        try {
            t1.join();
        } catch (InterruptedException e) {
            throw new RuntimeException(e);
        }
        log.debug("线程t2执行完成");
    }
},"t2");
Thread t3 = new Thread(new Runnable() {
    @Override
    public void run() {
        try {
            t2.join();
        } catch (InterruptedException e) {
            throw new RuntimeException(e);
        }
        log.debug("线程t3执行完成");
    }
},"t3");
t1.start();
t2.start();
t3.start();
```

#### (5) 守护线程

##### ① 介绍

**默认情况下**，Java 进程需要等待所有线程执行完毕才会退出。**守护线程**是一种特殊的线程，当所有**非守护线程**结束时，JVM 会强制退出，即使守护线程的代码尚未执行完毕。

> **核心特征**：守护线程的**生命周期**依赖于非守护线程，不会阻止 JVM 进程退出。

##### ② 例子

**代码示例：**

```java
log.debug("开始运行...");
Thread t1 = new Thread(() -> {
    log.debug("开始运行...");
    SleepTools.second(3);
    log.debug("运行结束...");
}, "t1");
// 设置t1线程为守护线程
t1.setDaemon(true);
t1.start();
SleepTools.second(1);
log.debug("运行结束...");
```

**输出结果：**

```bash
21:21:58.104 [main] DEBUG com.tuling.learnjuc.base.DaemonThreadDemo - 开始运行...
21:21:58.143 [t1] DEBUG com.tuling.learnjuc.base.DaemonThreadDemo - 开始运行...
21:21:59.158 [main] DEBUG com.tuling.learnjuc.base.DaemonThreadDemo - 运行结束...
```

从输出可以看到，主线程结束后，守护线程 `t1` 虽然未执行完毕（应该打印"运行结束..."但未出现），但也被强制终止。

##### ③ 应用场景

守护线程在实际开发中有着重要的应用价值：

| 应用场景 | 说明 |
| --------- | ---- |
| **垃圾回收器** | JVM 中的 **GC 线程**是守护线程，当没有用户线程时，程序不会产生垃圾，GC 也无需工作 |
| **心跳检测** | 中间件中的**心跳检测任务**采用守护线程，当进程退出时，检测任务自动结束 |
| **事件监听** | **异步事件监听器**使用守护线程，后台持续监听但不阻止进程退出 |
| **定时任务** | 某些**后台定时任务**适合使用守护线程，进程退出时任务自动终止 |

> **使用原则**：对于**不希望阻止 JVM 退出**的后台任务，应将其设置为守护线程。

### 5.3 线程终止机制

#### (1) 线程自然终止

线程的自然终止有两种情形：

| 终止情形 | 说明 |
| ---- | ---- |
| **正常执行完成** | `run()` 方法执行完毕，线程正常退出 |
| **异常导致终止** | 抛出未处理的异常，导致线程提前结束 |

> **注意**：自然终止是被动的终止方式，无法主动控制线程的终止时机。如果需要主动终止正在运行的线程，需要使用其他机制（见下文）。

#### (2) stop方法（不要使用）

`stop()` 方法**已被 JDK 废弃**。调用该方法会强制终止线程，无论 `run()` 方法中的逻辑是否执行完毕，都会立即释放 CPU 资源和锁资源。这会导致**严重的线程安全问题**。

**stop() 方法的两大问题：**

| 问题类型 | 具体表现 | 潜在风险 |
| --- | --- | --- |
| **异常抛出** | 立即抛出 `ThreadDeath` 异常，`run()` 方法中**任何执行指令**都可能抛出 | 业务逻辑无法正常完成 |
| **锁释放失控** | 释放当前线程持有的**所有锁**，且释放过程**不可控** | 破坏数据一致性 |

**转账场景示例：**

假设线程 A 执行转账逻辑（获得锁 → 1号账户减少100元 → 2号账户增加100元 → 释放锁），若在执行完"1号账户减少100元"后被调用 `stop()` 方法，将导致：

| 问题类型 | 具体说明 |
| --- | --- |
| **锁资源被强制释放** | 线程持有的所有锁被立即释放 |
| **转账逻辑未完成** | 2号账户未增加 |
| **数据不一致** | 1号账户平白减少100元 |

<img src="/imgs/ccrcy-02-primer-review/3a7610d31d74417132b504a27b99fbd3_MD5.jpg" style="display: block; width: 100%;" alt="3a7610d31d74417132b504a27b99fbd3_MD5">

> **重要提示**：这种不可控的锁释放会造成数据不一致，在金融交易、库存扣减等场景下将产生**严重的业务错误**。

```java
public class ThreadStopDemo {
    private static final Object lock = new Object();
    private static int account1 = 1000;
    private static int account2 = 0;

    public static void main(String[] args) {
        Thread threadA = new Thread(new TransferTask(),"threadA");
        threadA.start();
        // 等待线程A开始执行
        SleepTools.ms(50);
        // 假设在转账过程中，我们强制停止了线程A
        threadA.stop();
        //验证锁是否释放
        // synchronized (lock){
        //     System.out.println("主线程加锁成功");
        // }
    }

    static class TransferTask implements Runnable {
        @Override
        public void run() {
            synchronized (lock) {
                try{
                    System.out.println("开始转账...");
                    // 1号账户减少100元
                    account1 -= 100;
                    // 休眠100ms
                    SleepTools.ms(50);
                    // 假设在这里线程被stop了，那么2号账户将不会增加，且锁会被异常释放
                    System.out.println("1号账户余额: " + account1);
                    account2 += 100; // 2号账户增加100元
                    System.out.println("2号账户余额: " + account2);
                    System.out.println("转账结束...");
                }catch (Throwable t){
                    System.out.println("线程A结束执行");
                    t.printStackTrace();
                }
            }
        }
    }
}
```

#### (3) 中断机制

##### ① 中断原理

**中断机制**是一种线程协作方式，一个线程通过调用目标线程的 `interrupt()` 方法请求其停止运行。

**核心特点：**

| 特点 | 说明 |
| --- | --- |
| **协作式** | 中断只是**请求**，不会强制线程立即停止 |
| **可忽略** | 线程可以选择**不响应**中断请求 |
| **标志位检测** | 线程通过检查**中断标志位**判断是否被中断 |

**中断检测方法：**

| 方法 | 作用 | 特点 |
| --- | --- | --- |
| `isInterrupted()` | 判断线程是否被中断 | **不影响**中断标志位 |
| `Thread.interrupted()` | 判断**当前线程**是否被中断 | 会**清除**中断标志位（置为 false） |

**阻塞状态下的中断响应：**

当线程处于阻塞状态（调用 `Thread.sleep()`、`Thread.join()`、`Object.wait()` 等方法）时，如果检测到中断请求，会在**阻塞方法处抛出 `InterruptedException` 异常**，并**立即清除中断标志位**。

> **重要提示**：处于**死锁状态**的线程无法被中断。

**为什么推荐使用中断而非自定义标志位：**

| 对比维度 | 中断机制 | 自定义标志位 |
| --- | --- | --- |
| **阻塞响应** | 阻塞方法**原生支持**中断检测 | 需等待阻塞方法返回才能检查 |
| **资源消耗** | 复用线程内置标志位 | 需额外声明标志变量 |
| **代码简洁** | 无需额外声明和检查 | 增加样板代码 |

因此，**使用中断机制更加高效和简洁**。

##### ② 中断正常运行的线程

中断**正常运行**的线程，**不会清除**中断标志位。

**示例代码：**

```java
Thread t1 = new Thread(()->{
    while(true) {
        Thread current = Thread.currentThread();
        boolean interrupted = current.isInterrupted();
        if(interrupted) {
            log.debug(" 中断状态: {}", interrupted);
            break;
        }
    }
}, "t1");
t1.start();
//中断线程t1
t1.interrupt();
log.debug("中断状态：{}",t1.isInterrupted());
```

**输出结果：**

```bash
20:45:17.596 [t1] DEBUG com.tuling.learnjuc.base.ThreadInterruptDemo - 中断状态: true 20:45:17.596 [main] DEBUG com.tuling.learnjuc.base.ThreadInterruptDemo - 中断状态：true
```

从输出可见，中断标志位**保持为 `true`**，未被清除。

##### ③ 中断阻塞状态的线程

当线程因调用 `Thread.sleep()`、`Thread.join()`、`Object.wait()` 等方法而处于阻塞状态时，中断操作会**抛出 `InterruptedException` 异常**并**清除中断标志位**。

**示例代码（以 `sleep` 为例）：**

```java
Thread t1 = new Thread(()->{
    while(true) {
        try {
            Thread.sleep(2000);
        } catch (InterruptedException e) {
            e.printStackTrace();
        }
    }
}, "t1");
t1.start();
Thread.sleep(100);
//中断线程t1
t1.interrupt();
log.debug("中断状态：{}",t1.isInterrupted());
```

**关键区别对比：**

| 线程状态 | 中断标志位处理 | 说明 |
| --- | --- | --- |
| **正常运行** | **保持为 `true`** | 标志位不被清除 |
| **阻塞状态** | **清除为 `false`** | 抛出异常后自动清除 |

> **注意**：由于阻塞状态下中断会清除标志位，如果需要在捕获 `InterruptedException` 后继续传递中断状态，应**再次调用 `interrupt()` 方法**恢复中断标志位。

### 5.4 线程的调度机制

#### (1) 调度方式概述

**线程调度**是指系统为线程分配 **CPU 使用权**的过程。线程调度方式决定了线程何时执行、执行多久以及何时切换，直接影响程序的性能与响应能力。

现代操作系统主要采用以下两种调度方式：

| 调度方式 | 英文名称 | 特点 |
| --- | --- | --- |
| **协同式调度** | Cooperative Threads-Scheduling | 线程主动让出 CPU 控制权 |
| **抢占式调度** | Preemptive Threads-Scheduling | 系统强制分配 CPU 执行时间 |

#### (2) 协同式线程调度

**基本原理：**

使用协同式线程调度的系统，线程执行时间由**线程自身控制**。线程完成当前任务后，需**主动通知**系统切换到其他线程。

**特点分析：**

| 特性 | 说明 | 影响 |
| --- | --- | --- |
| **实现简单** | 线程完成任务后才通知系统切换 | 无需复杂的线程同步机制 |
| **无同步问题** | 线程主动让出控制权 | 不会出现资源竞争导致的同步问题 |
| **系统稳定性差** | 某个线程异常会导致整个进程阻塞 | 系统容错能力较弱 |

> **适用场景**：协同式调度适用于简单的应用场景，现代操作系统普遍采用抢占式调度以提升系统稳定性。

#### (3) 抢占式线程调度

使用抢占式线程调度的多线程系统，**每个线程执行的时间以及是否切换都由系统决定**。线程无法主动控制执行时长，系统会在适当时机强制切换线程，因此不会出现「一个线程导致整个进程阻塞」的问题。

> **Java 线程采用抢占式调度**。原因分析见下节「Java 线程的实现模型」。

**Java 线程的调度控制：**

| 控制手段 | 作用 | 说明 |
| --- | --- | --- |
| **Thread.yield()** | 让出 CPU 执行时间 | 线程主动放弃当前时间片，回到 **Ready** 状态 |
| **线程优先级** | 影响 CPU 获取概率 | Java 提供 **10 个级别**（1-10），优先级越高越容易被系统选择执行 |

**注意事项：**
- 线程**无法主动获取 CPU 执行时间**，只能通过设置优先级影响被调度的概率
- 当多个线程同时处于 **Ready** 状态时，优先级高的线程更容易被系统选择执行
- 优先级只是**建议**，具体调度策略由操作系统决定

### 5.5 Java线程的实现模型

前面提到 Java 线程采用抢占式调度，这需要从 Java 线程的实现模式说起。

线程本质上是操作系统层面的实体，那么 Java 线程是如何与操作系统线程建立映射关系的呢？

#### (1) 线程实现的三种模式

<img src="/imgs/ccrcy-02-primer-review/4252f4e781b8f35c6c46cda9ca8837f4_MD5.jpg" style="display: block; width: 100%;" alt="4252f4e781b8f35c6c46cda9ca8837f4_MD5">

不同编程语言实现线程机制时，主要有三种映射模式：

| 实现模式       | 映射关系 | 说明                      |
| ---------- | ---- | ----------------------- |
| **内核线程实现** | 1:1  | 每个 Java 线程映射到一个操作系统内核线程 |
| **用户线程实现** | 1:N  | 多个 Java 线程映射到一个内核线程     |
| **混合实现**   | N:M  | 多个 Java 线程映射到多个内核线程     |

接下来逐一分析这三种实现方式。

#### (2) 内核线程实现（1:1）

**基本概念**

内核线程（Kernel-Level Thread，KLT）是由操作系统内核直接支持的线程。内核通过**调度器**完成线程切换，并将线程任务映射到各个处理器上。这种方式也被称为 **1:1 实现**，即每个用户线程对应一个内核线程。

**工作特点**

| 特点 | 说明 |
| --- | --- |
| **独立调度单元** | 每个线程都成为独立的调度单元，由操作系统直接管理 |
| **阻塞隔离** | 某个线程在系统调用中被阻塞时，不会影响整个进程继续工作 |
| **自动调度** | 线程调度工作由操作系统自动处理，无需程序干预 |

**局限性**

内核线程实现存在以下限制：

| 限制 | 说明 | 影响 |
| --- | --- | --- |
| **系统调用开销大** | 线程创建、销毁及同步等操作都需要系统调用，在用户态和内核态之间切换 | 性能开销较高 |
| **资源消耗** | 每个用户线程需要一个内核线程支持，消耗内核资源（如栈空间） | 系统能支持的线程数量有限 |

**CPU 的两种工作模式**

理解内核线程实现，需要了解现代 CPU 的两种工作模式：

| 模式 | 访问权限 | 说明 | 崩溃影响 |
| --- | --- | --- | --- |
| **内核模式**<br>(Kernel Mode) | 完全访问底层硬件和任意内存地址 | 为操作系统核心功能保留 | 灾难性，会导致整个系统瘫痪 |
| **用户模式**<br>(User Mode) | 受限访问，需通过系统 API | 应用程序通常在此模式运行 | 可恢复，仅影响当前程序 |

<img src="/imgs/ccrcy-02-primer-review/2bdcdc5cd1e059e483cf99df8983d691_MD5.jpg" style="display: block; width: 500px;" alt="2bdcdc5cd1e059e483cf99df8983d691_MD5">

> **用户态与内核态切换**：当应用程序需要进行系统调用（如创建线程、文件 I/O）时，CPU 会从用户模式切换到内核模式。这种切换需要保存当前上下文、切换权限级别，因此开销较大。

**Linux 系统中的线程实现**

Linux 系统的线程实现经历了两个阶段：

| 实现方式 | 时期 | 说明 |
| --- | --- | --- |
| **LinuxThreads** | glibc 2.3.2 之前 | 早期的线程实现 |
| **NPTL**<br>(Native POSIX Thread Library) | glibc 2.3.2 之后 | 符合 POSIX 标准的现代线程库，性能更优 |

**POSIX 线程库**

POSIX 标准定义了一套线程操作函数库 **pthread**，让程序员能够更方便地操作和管理线程。`pthread_create` 是类 Unix 操作系统（Unix、Linux、Mac OS X 等）中创建线程的函数。

查看当前系统使用的线程库：

```bash
[root@node02 ~] getconf GNU_LIBPTHREAD_VERSION
NPTL 2.17
```

#### (3) 用户线程实现（1:N）

**基本定义**

严格意义上的用户线程指的是**完全建立在用户空间的线程库**上，系统内核**无法感知**用户线程的存在及实现方式。

| 特征维度 | 说明 |
| --- | --- |
| **运行空间** | 完全在用户态运行 |
| **管理方式** | 建立、同步、销毁、调度均由用户程序完成 |
| **内核参与** | 无需内核帮助 |
| **切换开销** | 无需切换到内核态，开销极低 |
| **线程规模** | 可支持更大规模的线程数量 |

> **应用场景**：部分高性能数据库中的多线程采用用户线程实现。

**优势与挑战**

| 维度 | 优势 | 挑战 |
| --- | --- | --- |
| **性能** | 操作快速且低消耗 | 需要用户程序自行处理所有线程操作 |
| **独立性** | 不依赖系统内核支持 | 线程创建、销毁、切换、调度均需手动实现 |
| **灵活性** | 可根据应用场景定制调度策略 | 操作系统仅分配处理器资源到进程 |
| **复杂度** | - | 阻塞处理、多处理器映射等问题解决困难 |

> **实践现状**：由于实现复杂度较高，一般应用程序较少使用用户线程。**Java** 早期曾使用用户线程后放弃，但近年来**Golang** 等高并发编程语言又普遍支持用户线程。

#### (4) 混合实现（N:M）

**基本概念**

除了完全依赖内核线程和完全由用户程序实现之外，还存在一种将内核线程与用户线程结合使用的方式，称为 **N:M 实现**。这种混合模式同时使用用户线程和内核线程。

**工作特点**

| 特性维度 | 具体说明 |
| --- | --- |
| **用户线程管理** | 用户线程完全建立在用户空间中，创建、切换、销毁等操作成本低廉，支持大规模用户线程并发 |
| **内核调度利用** | 可以使用内核提供的线程调度功能及处理器映射，用户线程的系统调用通过内核线程完成 |
| **数量关系** | 用户线程与内核线程的数量比例不定，呈现 **N:M** 的映射关系 |

> **混合实现的权衡**：N:M 模式试图在性能与灵活性之间取得平衡，既保留用户线程的轻量级特性，又能利用内核的调度能力。

#### (5) Java线程的实现

<img src="/imgs/ccrcy-02-primer-review/3328f1b8c0052b7302ea02931010cd20_MD5.jpg" style="display: block; width: 800px;" alt="3328f1b8c0052b7302ea02931010cd20_MD5">

##### ① 平台线程的实现

**历史演变**

Java 线程的实现方式经历了重要的技术变迁：

| 时期 | 实现方式 | 特点 |
| --- | --- | --- |
| **JDK 1.2 之前** | 用户线程实现 | 基于 Classic 虚拟机 |
| **JDK 1.3 起** | **操作系统原生线程** | 主流商用 JVM 普遍采用 **1:1 线程模型** |

**HotSpot 虚拟机的实现机制**

以 **HotSpot** 虚拟机为例，其线程实现具有以下特征：

| 特征维度 | 具体说明 |
| --- | --- |
| **映射关系** | 每个 **Java 线程**直接映射到一个 **操作系统原生线程** |
| **中间结构** | 无额外间接结构，映射关系简洁直接 |
| **调度主体** | **HotSpot 不干涉线程调度**，全权交由操作系统处理 |

> 由于线程调度完全由操作系统控制，因此 **Java 线程采用抢占式调度**。

**线程优先级的映射限制**

Java 线程优先级通过映射到操作系统原生线程实现，但存在以下限制：

| 问题 | 原因 |
| --- | --- |
| **优先级不一定可靠** | 操作系统线程优先级与 Java 线程优先级**无法完全一一对应** |
| **调度依赖操作系统** | 线程调度最终由操作系统决定，Java 层无法完全控制 |

##### ② Thread#start()源码分析

理解 `Thread#start()` 的工作原理，有助于我们理解 Java 线程与操作系统线程的映射关系。

**方法调用链路：**

| 调用层次 | 方法 | 职责 |
| --- | --- | --- |
| Java 层 | `Thread#start()` | 启动线程的入口方法 |
| JVM 层 | `Thread#start0()` | Native 方法，调用 JVM 实现 |
| JVM 实现 | `JVM_StartThread` | 创建操作系统原生线程 |
| 系统调用 | `pthread_create` | Linux 内核线程创建 |

`Thread#start()` 源码体现了线程启动的关键步骤：

```java
public synchronized void start() {
    // 1. 状态检查：线程只能启动一次
    if (threadStatus != 0)
        throw new IllegalThreadStateException();

    // 2. 线程组注册
    group.add(this);

    boolean started = false;
    try {
        // 3. Native 调用启动线程
        start0();
        started = true;
    } finally {
        try {
            if (!started) {
                group.threadStartFailed(this);
            }
        } catch (Throwable ignore) {
            // 4. 启动失败处理
        }
    }
}

// Native 方法，由 JVM 实现
private native void start0();
```

**执行步骤分析：**

| 步骤 | 关键操作 | 设计目的 |
| --- | --- | --- |
| 状态检查 | 验证 `threadStatus != 0` | 防止线程重复启动 |
| 线程组注册 | `group.add(this)` | 纳入线程组生命周期管理 |
| Native 调用 | `start0()` | 建立 Java 线程与 OS 线程映射 |
| 异常处理 | `threadStartFailed()` | 失败时从线程组移除 |

> **synchronized 修饰的意义**：
> - 保证状态检查与线程组注册的原子性
> - 防止多线程同时调用同一线程的 `start()` 方法
> - 避免竞态条件导致的线程状态不一致

**JVM 与操作系统的交互：**

`start0()` 调用后，JVM 执行以下流程：

| JVM 层操作 | 系统层对应 | 效果 |
| --- | --- | --- |
| 分配线程栈 | `pthread_create()` | 创建内核线程及其栈空间 |
| 设置线程属性 | 调度策略、优先级配置 | Java 属性映射到 OS 线程 |
| 启动线程执行 | 线程进入就绪队列 | 等待 OS 调度器分配时间片 |
| 回调 `run()` | 获得 CPU 时间片 | 开始执行 Java 代码 |

> **注意**：`start()` 调用后线程**不会立即执行** `run()`，而是进入 `RUNNABLE` 状态，由操作系统调度器决定何时分配 CPU 时间片。

##### ③ 虚拟线程

Java 21 引入了**虚拟线程**（Virtual Threads），这是一种由 JVM 管理的**用户级线程**。与传统平台线程不同，虚拟线程通过在少量操作系统线程上复用大量虚拟线程，显著降低了并发编程的资源消耗。

> **设计目标**：虚拟线程不提升单个任务的执行速度，而是通过大幅增加并发任务数量，提升系统整体吞吐量。

<img src="/imgs/ccrcy-02-primer-review/6a407d8dbc5df095858162a49914e9cb_MD5.jpg" style="display: block; width: 100%;" alt="6a407d8dbc5df095858162a49914e9cb_MD5">

**平台线程与虚拟线程对比：**

| 对比维度 | 平台线程（Platform Thread） | 虚拟线程（Virtual Thread） |
| --- | --- | --- |
| **实现方式** | 1:1 映射到操作系统线程 | M:N 映射，多个虚拟线程复用少量操作系统线程 |
| **资源消耗** | 较高，每个线程需要独立的栈内存（约 1MB） | 极低，虚拟线程栈内存可以更小 |
| **创建成本** | 高，涉及系统调用 | 低，JVM 内部创建 |
| **适用数量** | 数千个已达到上限 | 可轻松创建数十万甚至数百万个 |
| **阻塞行为** | 阻塞时占用操作系统线程 | 阻塞时自动挂起，释放底层操作系统线程 |
| **是否需要池化** | 需要线程池复用线程 | 不需要池化，用完即抛 |

**适用场景分析：**

| 场景类型 | 适用性 | 说明 |
| --- | --- | --- |
| **I/O 密集型任务** | ✅ 高度适用 | 数据库查询、HTTP 请求、文件读写等阻塞操作 |
| **CPU 密集型任务** | ❌ 不适用 | 虚拟线程不会提高计算速度，反而增加调度开销 |
| **非阻塞异步任务** | ❌ 不适用 | 使用传统异步编程模型更合适 |
| **高并发服务** | ✅ 高度适用 | Web 服务器处理大量并发请求 |

> **使用建议**：
> 
> - 虚拟线程适合**数量庞大**且**频繁阻塞**的任务
> - 不适合**计算密集**或**纯非阻塞**的任务
> - 虚拟线程创建和销毁成本低，无需池化
> - 主流框架已支持虚拟线程，包括 Tomcat、Jetty、Netty、Spring Boot 等

**代码示例：**

```java
public class VTDemo {
    public static void main(String[] args) throws InterruptedException {
        // 创建平台线程
        Thread.ofPlatform().start(new Runnable() {
            @Override
            public void run() {
                System.out.println(Thread.currentThread());
            }
        });

        // 创建虚拟线程
        Thread vt = Thread.ofVirtual().start(new Runnable() {
            @Override
            public void run() {
                System.out.println(Thread.currentThread());
            }
        });

        // 等待虚拟线程执行完毕
        vt.join();
    }
}
```

**输出结果：**

```
Thread[#22,Thread-0,5,main]
VirtualThread[#23]/runnable@ForkJoinPool-1-worker-1
```

**关键点说明：**

- 平台线程名称格式：`Thread[编号,名称,优先级,父线程]`
- 虚拟线程名称格式：`VirtualThread[编号]/状态@调度线程`
- 虚拟线程默认由 ForkJoinPool 调度

## 6. 线程通信机制

### 6.1 线程通信概述

在实际开发中，单个线程独立完成任务的场景较少，更多情况需要多个线程协同工作。线程协同离不开**信息传递**、**状态同步**和**任务协调**。

**为什么需要线程通信**

| 需求维度 | 说明 | 典型场景 |
| --- | --- | --- |
| **数据传递** | 线程间需要传递数据或共享计算结果 | 生产者-消费者模式 |
| **状态同步** | 线程需要感知其他线程的执行状态 | 等待线程结束、获取执行结果 |
| **任务协调** | 多个线程按特定顺序或条件协同执行 | 按序执行、条件等待 |

**本章介绍的通信方式**

| 通信方式 | 主要用途 | 适用场景 |
| --- | --- | --- |
| **管道输入输出流** | 线程间数据传输 | 流式数据传递 |
| **volatile 关键字** | 保证变量可见性 | 状态标志位传递 |
| **Thread.join** | 等待线程结束 | 获取线程执行结果 |
| **等待/通知机制** | 线程间协作与协调 | 生产者-消费者模式 |
| **LockSupport** | 线程阻塞与唤醒 | AQS 等并发工具基础 |

### 6.2 管道输入输出流

管道输入输出流是 Java 提供的线程间数据传输机制，数据通过内存缓冲区传递，无需借助磁盘等外部存储。

#### (1) 应用场景

**文件生成与上传**

当需要将 Java 应用生成的文件上传到云端时，可以采用不同的处理方式：

| 处理方式 | 操作步骤 | 性能特点 |
| --- | --- | --- |
| **传统方式** | 生成文件 → 写入磁盘 → 从磁盘读取 → 上传云端 | 磁盘 I/O 开销大，处理耗时较长 |
| **管道方式** | 生成文件 → 直接通过管道传输 → 上传云端 | 内存传输，避免磁盘 I/O，性能更优 |

使用管道输入输出流可以跳过"写入磁盘"这一步骤，直接在内存中完成线程间的数据传递，从而减少 I/O 开销。

#### (2) 实现类

Java 提供了 **4 种**管道输入输出流实现：

| 类型 | 输入流 | 输出流 | 数据单位 | 适用场景 |
| --- | --- | --- | --- | --- |
| **字节流** | PipedInputStream | PipedOutputStream | byte | 二进制数据传输 |
| **字符流** | PipedReader | PipedWriter | char | 文本数据传输 |

**关键特性：**

| 特性 | 说明 |
| --- | --- |
| **传输媒介** | 内存缓冲区 |
| **连接要求** | 输入流与输出流必须通过 `connect()` 方法连接后才能使用 |
| **单向传输** | 数据只能从输出流向输入流单向流动 |
| **线程配对** | 通常用于生产者线程和消费者线程之间的数据传递 |

#### (3) 使用示例

下面的示例展示了如何使用 `PipedWriter` 和 `PipedReader` 在两个线程间传递字符数据：

```java
public class Piped {
    public static void main(String[] args) throws Exception {
        PipedWriter out = new PipedWriter();
        PipedReader in = new PipedReader();

        // 将输出流和输入流进行连接，否则在使用时会抛出IOException
        out.connect(in);

        Thread printThread = new Thread(new Print(in), "PrintThread");
        printThread.start();

        int receive = 0;
        try {
            while ((receive = System.in.read()) != -1) {
                out.write(receive);
            }
        } finally {
            out.close();
        }
    }

    static class Print implements Runnable {
        private PipedReader in;

        public Print(PipedReader in) {
            this.in = in;
        }

        @Override
        public void run() {
            int receive = 0;
            try {
                while ((receive = in.read()) != -1) {
                    System.out.print((char) receive);
                }
            } catch (IOException ex) {
            }
        }
    }
}
```

**代码说明：**

| 线程 | 职责 | 操作 |
| --- | --- | --- |
| **main 线程** | 生产者 | 从标准输入读取数据，写入 PipedWriter |
| **Print 线程** | 消费者 | 从 PipedReader 读取数据，输出到控制台 |

> **使用注意**：管道流适用于小数据量的线程间通信。如果数据量较大或需要高吞吐量的场景，建议使用 `java.util.concurrent` 包中的阻塞队列（如 `ArrayBlockingQueue`）。

### 6.3 volatile 关键字

#### (1) 基本概念

volatile 是 Java 虚拟机提供的**轻量级同步机制**，用于确保共享变量在多线程间的**可见性**。当一个线程修改了 volatile 变量的值，这个新值对其他线程来说**立即可见**。

**核心特性：**

| 特性 | 说明 | 注意事项 |
| --- | --- | --- |
| **可见性** | 保证变量的读写操作直接作用于主内存 | 不保证原子性 |
| **有序性** | 禁止指令重排序优化 | 不替代 synchronized |
| **轻量级** | 无需加锁，性能开销较小 | 适用场景有限 |

**工作机制：**volatile 变量的读写操作会直接与主内存交互，而不是使用线程的本地内存缓存，从而确保所有线程都能看到最新的值。

#### (2) Java 内存模型（JMM）

理解 volatile 需要先了解 Java 内存模型（Java Memory Model，JMM）。

**JMM 抽象结构：**

| 组成部分 | 说明 | 作用 |
| --- | --- | --- |
| **主内存** | 所有线程共享的内存区域 | 存储共享变量 |
| **本地内存** | 每个线程独立的内存区域 | 缓存主内存变量的副本 |
| **交互机制** | 线程通过本地内存与主内存交互 | 实现数据读写 |

**JMM 规定：**

- 线程对共享变量的所有操作都必须在自己的本地内存中进行
- 不能直接读取或修改主内存中的数据
- 不同线程之间无法直接访问对方本地内存的数据

**volatile 如何工作：**

| 操作 | 普通变量 | volatile 变量 |
| --- | --- | --- |
| **写操作** | 修改本地内存副本 | 直接写入主内存，并使其他线程的缓存失效 |
| **读操作** | 读取本地内存副本 | 直接从主内存读取最新值 |

#### (3) 使用示例

**问题场景：**主线程修改标志位，子线程需要感知变化。

```java
public class VolatileDemo {
    private static volatile boolean stop = false;

    public static void main(String[] args) throws InterruptedException {
        Thread t1 = new Thread(new Runnable() {
            @Override
            public void run() {
                System.out.println("线程t1开始执行");
                int i = 0;
                while (!stop) {
                    i++;
                }
                System.out.println("跳出循环");
            }
        }, "t1");

        t1.start();
        Thread.sleep(1000);
        stop = true;
        System.out.println("主线程修改stop=true");
    }
}
```

**对比分析：**

| 情况 | 子线程行为 | 原因 |
| --- | --- | --- |
| **不加 volatile** | 无法感知 stop 值变化，陷入死循环 | 读取本地内存缓存，看不到主内存的修改 |
| **加 volatile** | 立即感知 stop 值变化，退出循环 | 直接读取主内存，可见主内存的最新值 |

#### (4) 适用场景与限制

**适用场景：**

| 场景 | 说明 | 示例 |
| --- | --- | --- |
| **状态标志位** | 一个线程写，多个线程读 | 控制线程的启动、停止 |
| **单例双重检查** | 配合实例初始化使用 | DCL（Double-Checked Locking） |
| **独立观察** | 被观察值无需参与其他不变性条件 | 定期发布观察结果 |

**volatile 最适用的场景：** 一个线程写，多个线程读。

**局限性：**

| 属性 | volatile | synchronized / Lock |
| --- | --- | --- |
| **原子性** | ❌ 不保证复合操作的原子性 | ✅ 保证 |
| **可见性** | ✅ 保证 | ✅ 保证 |
| **有序性** | ✅ 禁止特定重排序 | ✅ 保证 |
| **适用范围** | 单一读写操作 | 复杂同步场景 |

**不能保证原子性的示例：**

```java
volatile int count = 0;
count++;  // 不是原子操作，包含：读取、加1、写入三步
```

即使使用 volatile，多个线程同时执行 `count++` 仍会出现数据竞争。

**使用建议：**

- 当只需要保证可见性，且操作本身是原子性时，优先使用 volatile
- 当需要保证复合操作的原子性时，应使用 synchronized 或 Lock
- volatile 不能替代锁机制，但可以作为锁的补充手段

### 6.4 Thread.join

#### (1) 基本概念

Thread.join 用于让当前线程等待另一个线程执行完毕。常见应用场景包括：

| 场景 | 说明 |
| --- | --- |
| **等待结果** | 主线程需要等待子线程计算完成后获取结果 |
| **顺序控制** | 多个线程需要按特定顺序依次执行 |
| **任务协调** | 某个线程需要在其他线程完成后再开始执行 |

#### (2) 工作原理

当线程 A 调用线程 B 的 join() 方法时，执行流程如下：

| 步骤 | 说明 |
| --- | --- |
| 1 | 线程 A 进入 WAITING 状态，释放 CPU 执行权 |
| 2 | 线程 B 继续执行 |
| 3 | 线程 B 执行完毕后，线程 A 被唤醒并继续执行 |

从同步与异步的角度来看：

| 类型 | 定义 | 场景示例 |
| --- | --- | --- |
| **同步** | 需要等待结果返回才能继续执行 | 主线程调用 join() 等待子线程结果 |
| **异步** | 不需要等待结果返回就能继续执行 | 主线程启动子线程后立即继续执行 |

join() 方法的底层实现基于等待/通知机制（wait/notify）：

```java
public final void join() throws InterruptedException {
    join(0);
}

public final synchronized void join(long millis) throws InterruptedException {
    // 省略参数校验代码
    if (millis == 0) {
        while (isAlive()) {
            wait(0);  // 调用 Object 的 wait() 方法等待
        }
    }
    // 省略超时等待代码
}
```

当调用 thread.join() 时，当前线程会在 thread 对象上调用 wait() 进入等待状态，直到 thread 线程执行完毕时，JVM 会自动调用 notifyAll() 唤醒所有等待的线程。

### 6.5 等待/通知机制

#### (1) 基本原理

##### ① 应用场景

线程间协作在实际开发中十分常见。例如：一个线程修改了某个对象的值（生产者），另一个线程需要感知到这个变化并做出相应的处理（消费者）。这种模式将"做什么"（what）和"怎么做"（how）进行了分离。

##### ② 轮询方式及问题

实现上述场景的简单办法是让消费者线程不断循环检查变量是否符合预期：

```java
while (条件不满足) {
    Thread.sleep(轮询间隔);
}
```

在 `while` 循环中设置不满足的条件，如果条件满足则退出循环。这种方式虽然简单，但存在以下问题：

| 问题 | 说明 |
| --- | --- |
| **及时性差** | 消费者线程无法立即感知条件变化，存在延迟 |
| **资源浪费** | 缩短休眠时间（如休眠1毫秒）可以提高响应速度，但会消耗更多处理器资源 |

##### ③ 等待/通知机制

等待/通知机制能够有效避免轮询方式的问题：

| 对比维度 | 轮询方式 | 等待/通知机制 |
| --- | --- | --- |
| **CPU占用** | 持续消耗CPU资源 | 等待期间释放CPU资源 |
| **响应及时性** | 存在延迟，取决于轮询间隔 | 条件满足后立即响应 |
| **资源利用** | 低效，大量无效轮询 | 高效，线程按需执行 |

#### (2) Object 类的等待通知方法

等待通知机制是 Java 中实现线程间协作的基础方式。一个线程调用锁对象的 `wait()` 方法后会进入等待队列，直到其他线程通过 `notify()` 或 `notifyAll()` 将其唤醒。

##### ① 方法说明

Java 在 `Object` 类中提供了等待通知相关的方法：

| 方法 | 功能说明 | 关键特性 |
| --- | --- | --- |
| **wait()** | 线程进入 WAITING 状态，等待通知或中断 | 调用后释放对象锁 |
| **wait(long)** | 超时等待指定时间（毫秒） | 超时后自动返回，无需通知 |
| **wait(long, int)** | 超时等待（毫秒 + 纳秒） | 提供纳秒级精度的超时控制 |
| **notify()** | 唤醒对象上等待的**一个**线程 | 被唤醒线程需重新获取锁 |
| **notifyAll()** | 唤醒对象上等待的**所有**线程 | 所有线程竞争获取锁 |

> **使用建议**：优先使用 `notifyAll()`，谨慎使用 `notify()`。`notify()` 只唤醒一个线程，无法确保唤醒的是目标线程，可能导致信号丢失。

##### ② 使用范式

等待通知机制的使用需要遵循严格的范式，以保证线程安全。

**等待方（消费者）范式：**

```java
synchronized(对象) {
    while (条件不满足) {
        对象.wait();
    }
    // 条件满足后执行逻辑
}
```

| 步骤 | 说明 |
| --- | --- |
| 1. 获取对象锁 | 使用 synchronized 进入同步代码块 |
| 2. 检查条件 | 使用 while 循环检查条件（防止虚假唤醒） |
| 3. 等待通知 | 条件不满足时调用 wait() 释放锁并等待 |
| 4. 重新检查 | 被唤醒后必须重新检查条件 |
| 5. 执行逻辑 | 条件满足后执行对应业务逻辑 |

**通知方（生产者）范式：**

```java
synchronized(对象) {
    改变条件;
    对象.notifyAll();
}
```

| 步骤 | 说明 |
| --- | --- |
| 1. 获取对象锁 | 使用 synchronized 进入同步代码块 |
| 2. 改变条件 | 修改共享变量状态 |
| 3. 发送通知 | 调用 notifyAll() 通知所有等待线程 |

> **关键要点**：
> - wait()、notify()、notifyAll() 必须在 synchronized 代码块中调用
> - 等待方使用 while 循环而非 if 判断，防止虚假唤醒
> - 通知方应先修改条件再通知，保证顺序性

##### ③ 示例代码

```java
public class WaitDemo {
    public static void main(String[] args) throws InterruptedException {
        Object locker = new Object();

        // 等待线程
        Thread t1 = new Thread(() -> {
            try {
                System.out.println("wait开始");
                synchronized (locker) {
                    locker.wait();
                }
                System.out.println("wait结束");
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
        });
        t1.start();

        // 确保 t1 先启动并执行 wait()
        Thread.sleep(1000);

        // 通知线程
        Thread t2 = new Thread(() -> {
            synchronized (locker) {
                System.out.println("notify开始");
                locker.notifyAll();
                System.out.println("notify结束");
            }
        });
        t2.start();
    }
}
```

**执行流程说明：**

| 阶段 | 线程 t1 | 线程 t2 |
| --- | --- | --- |
| 1. 启动 | 获取 locker 锁，调用 wait() 释放锁并等待 | - |
| 2. 等待 | 进入 WAITING 状态，释放 locker 锁 | - |
| 3. 通知 | - | 获取 locker 锁，调用 notifyAll() |
| 4. 唤醒 | 被唤醒，重新竞争获取 locker 锁 | 释放 locker 锁 |
| 5. 执行 | 获取锁后从 wait() 返回，继续执行 | 结束 |

#### (3) LockSupport 的 park/unpark

##### ① 基本概念

LockSupport 是 JDK 中用于线程阻塞和唤醒的工具类，其核心机制是"许可"（permit）：

| 方法 | 功能 | 许可状态 |
| --- | --- | --- |
| **park()** | 阻塞当前线程 | 消耗许可，若无许可则阻塞 |
| **unpark(Thread)** | 唤醒指定线程 | 提供许可 |

LockSupport 类似于二元信号量（只有 1 个许可可供使用）：
- 许可未被占用时：当前线程获取许可并继续执行
- 许可已被占用时：当前线程阻塞，等待获取许可

> **特点**：unpark 可以先于 park 调用，这是与 wait/notify 机制的重要区别。

##### ② 与 AQS 的关系

Java 并发包中许多锁和同步器的实现都依赖于 LockSupport。例如：

**AbstractQueuedSynchronizer（AQS）**：通过调用 LockSupport.park() 和 LockSupport.unpark() 实现线程的阻塞和唤醒，是 Java 并发工具的基础框架。

##### ③ 使用示例

```java
public class LockSupportDemo {
    public static void main(String[] args) throws InterruptedException {
        Thread parkThread = new Thread(new Runnable() {
            @Override
            public void run() {
                System.out.println("ParkThread开始执行");
                // 当没有『许可』时，当前线程暂停运行；
                // 有『许可』时，用掉这个『许可』，当前线程恢复运行
                LockSupport.park();
                System.out.println("ParkThread执行完成");
            }
        });

        parkThread.start();

        Thread.sleep(1000);
        System.out.println("唤醒parkThread");
        // 给线程 parkThread 发放『许可』
        // （多次连续调用 unpark 只会发放一个『许可』）
        LockSupport.unpark(parkThread);
    }
}
```

##### ④ 与 wait/notify 的对比

| 对比维度 | LockSupport | wait/notify |
| --- | --- | --- |
| **调用位置** | 可在任何地方调用 | 必须在 synchronized 代码块中 |
| **唤醒对象** | 可指定任意线程唤醒 | 只能唤醒同一锁对象上的等待线程 |
| **调用顺序** | unpark 可先于 park 调用 | notify 必须在 wait 之后调用 |
| **所属类** | java.util.concurrent.locks.LockSupport | java.lang.Object |

> **关键区别**：如果先调用 unpark(thread2)，然后 thread2 再调用 park()，thread2 不会阻塞。但如果先调用 notify，然后线程再调用 wait，线程会被阻塞。

> **注意事项**：连续多次调用 unpark 的效果与一次调用相同，只会发放一个许可。
