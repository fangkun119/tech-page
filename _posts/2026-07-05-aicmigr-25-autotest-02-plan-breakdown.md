## 1. 开篇导读

<img src="imgs/aicmigr-25-autotest-02-plan-breakdown/5407cfda9b2b551bced339df5a824359_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

本篇是"自动测试"系列的第二篇。上一篇把 Hermes 摸透了，知道它能提供什么、自己要写什么；但摸 Hermes 不是终点，而是手段。本篇要做的事是给 RobustMQ 加一套 7×24 不间断的自动化混沌测试，并把这件事从一句话需求翻译成可以动手的技术方案。

选 RobustMQ 作为本篇载体，有两个理由：一是它是基础软件项目，质量要求达到生产级，自动化测试是真实需要；二是社区已有 issue 在讨论多语言 SDK 兼容性、长时间运行稳定性这类问题，光靠人工测试覆盖不到，确实需要一套 7×24 跑的自动化测试。参考资料：

- https://robustmq.com/zh/Blogs/84
- https://robustmq.com/zh/Blogs/95
- https://github.com/robustmq/robustmq

本篇的真正主角不是 RobustMQ，而是把"一句话需求"翻译成"可开干方案"的**两次翻译**打法。整篇文档分为两部分：第一部分是方法论提炼（参考手册风，不深入技术栈），第二部分是结合 RobustMQ/Hermes 的实战演示（讲清 why）。下面这张导读地图帮助两类读者快速定位。

## 2. 第一部分 · 方法论提炼：两次翻译

### 2.1 为什么一句话需求危险

<img src="imgs/aicmigr-25-autotest-02-plan-breakdown/a54a8b09bd192b392114fa08b0536814_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

一句话需求最大的风险在于：**它藏的事比说的多得多**。一句听起来很具体的需求，背后通常隐藏着 5~10 个未明确的决策点。如果不先把它们拍清楚就动手写代码，会出现两种典型后果：

#### (1) 两种典型后果

##### ① 写到一半发现假设不对
某个未明确的选择（比如"测试报告对内还是对外"）在不同段落里被默认成不同答案，方案自相矛盾。

##### ② 改代码远比改文档贵
不拍清楚就动手，等于把代码当成需求的探索工具。代码改一次的代价是文档改一次的数倍到数十倍。

**结论**：接到一句话需求，第一件事不是拆方案，而是先把需求本身拆完整。

### 2.2 两次翻译总览

<img src="imgs/aicmigr-25-autotest-02-plan-breakdown/ad5b7907c55bc48542f69a47f577a91a_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

把一句话需求翻译成可开干方案，分两次完成。两次产物合起来构成后续开发的**单一事实源**。

<img src="imgs/aicmigr-25-autotest-02-plan-breakdown/7fc80d3035066d15b2dc99b21f180cc5_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

- **第一次翻译**：从一句话到完整设计文档。产物与技术栈无关，只讲"为什么做、做什么、不做什么、按什么思路、覆盖什么场景、产出什么"。
- **第二次翻译**：从完整设计文档到完整方案文档。产物落到具体技术栈，包含 Skill 设计、Tool 实现、场景库、触发配置等可以动手的东西。
- **关键差异**：第二次翻译开始前必须先反问"路径选对了吗"，质疑第一次翻译里 AI 顺手给的执行路径。

### 2.3 第一次翻译：方法论 Check List

<img src="imgs/aicmigr-25-autotest-02-plan-breakdown/17b1e172db3bade04cb3aec5a0e82390_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

第一次翻译的目标是让 AI 帮你穷举该拍的事，由人拍方向。操作步骤如下。

#### (1) 操作步骤

##### ① 抛展开提示词
提示词不需要长，四五句话即可。好的提示词不是清单，是**预设 + 余地**：预设你已经想到的（目标、视角、问题域和技术栈分开），余地留给 AI 去补。

##### ② 在提示词里预设三件事
- "**先别谈具体技术框架**"：把问题域和方案分开，这是第一次翻译的核心动作。不强调的话，AI 立刻就会往具体框架上跳。
- "**几个我拍不准的选择，列出来给我**"：让 AI 列选项、人来拍。AI 不知道项目定位、社区接受度，这些只有人能拍。
- "**不要复述，要展开**"：堵掉 AI 把一句话改写成几句话就交差的偷懒回答。

##### ③ 拿"完整设计该长什么样"的预期去比对初版
看完 AI 初版，拿一份自己对完整设计文档的预期去比对，差什么就反问什么。预期从工作经验里来。

##### ④ 反问（不是追问）
差什么补什么。具体含义见下方对照表。

##### ⑤ 整合成可对外评审的完整文档
最后一步给 AI 一个具体写作对象（如"能拿出去给社区评审"），让它整合所有讨论。AI 默认从最近对话回忆，不点关键产物它会漏。

#### (2) 反问 vs 追问

| 维度 | 反问 | 追问 |
|---|---|---|
| 触发原因 | 拿预期比对发现漏了 | 想多听一遍前文 |
| 典型表达 | "你这里漏了，补上" | "你前面那段再讲讲" |
| 对产物的影响 | 把完整度补齐 | 增加篇幅但不补洞 |
| 谁主导 | 人（凭经验判断漏点） | AI（凭上下文继续展开） |

**关键判断**：AI 不会主动告诉你它漏了什么。能不能反问到位，取决于人脑里有没有"完整设计该长什么样"的预期。

#### (3) 好提示词的判定标准

| 标准 | 说明 |
|---|---|
| 长度 | 四五句话即可，模板化的二十行反而限制 AI |
| 预设 | 把你已经想清楚的边界写进去（如"先别谈技术框架"） |
| 余地 | 把你拍不准的明确交给 AI 列选项 |
| 防偷懒 | 显式写"不要复述，要展开" |
| 写作对象 | 给一个具体场景（如"能拿出去给社区评审"） |

### 2.4 第二次翻译：方法论 Check List

<img src="imgs/aicmigr-25-autotest-02-plan-breakdown/6d75ed9b1840d3049a49cff8ff76f2b9_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

第二次翻译的目标是把完整设计文档落到具体技术栈，跑出能开干的方案文档。最容易忽视的环节是开始前的关键反问。

#### (1) 关键反问：路径选对了吗

第一次翻译产物里的执行路径，往往是 AI 顺着你的提问**随手给的**，不是认真比较过的最优路径。第二次翻译开始前，必须先反问：

> "第一次翻译产物里选的路径，真的是最优的吗？我最近摸过的工具里，有没有已经把 80% 工程做好的？"

这一步反问，就是把之前建立的**工具认知**拿出来用。你刚摸过的开源项目，可能已经把方案里大部分工程都做了，没必要自己撸。

#### (2) 三个动作

##### ① 拿已有工具能力清单去比对路径
列一张对照表，把生产级 AI Agent 系统需要的能力（工具函数、Agent 框架、调度、上下文、触发、通道、安全沙箱、持久化）逐项列出，标注自研需要做什么、已有工具提供什么。工具函数只是冰山一角，框架/调度/通道/沙箱/持久化加起来比工具函数大好几倍。

##### ② 演进收益评估
不止看当下工程量，还要看未来演进空间。基于已有平台做的系统，未来可能演进成"AI 助理"，这是自研脚本给不出的可能性。

##### ③ 给方案文档硬约束
路径选定后，给方案文档加几条硬约束：
- **结构化数据用代码生成，JSON 别交给 LLM 写**（这是容易跑偏的硬规则）。
- 列出方案文档最低限度要包含的章节（如"至少 8 节"）。
- 明确已有工具的能力边界（白嫖 / 自己开发的分割线）。

#### (3) 收敛式反问

第二次翻译已到方案阶段，不能再发散。每次反问最后必须加一句"**别又冒新决策，把这版收紧**"。否则 AI 一边补、一边给你冒新决策点，方案永远定不下来。

| 翻译阶段 | 反问模式 | 典型提示词尾句 |
|---|---|---|
| 第一次翻译 | 展开式 | "给我两三个选项和影响，我来拍" |
| 第二次翻译 | 收敛式 | "别又冒新决策，把这版收紧" |

### 2.5 单一事实源与项目阶段 Check List

<img src="imgs/aicmigr-25-autotest-02-plan-breakdown/83bee77a3e853f07d0856435e66cf537_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

两份文档（完整设计文档 + 完整方案文档）之间有清晰因果链。后续每一行代码、每一个配置、每一个判断，都能追溯到上游某条需求决策。这两份文档就是开发的**单一事实源**——不是开发完成后补的文档，而是开发过程中的硬约束。任何代码偏离文档，要么改代码，要么改文档，然后回来重新对齐。

#### (1) 两份文档的因果链示例

设计文档里的需求决策 → 直接决定方案文档里的工程实现：

- "协议兼容性是核心命题" → 决定 run_client Tool 必须支持多语言 SDK 矩阵。
- "测试记录完全公开建立社区信任" → 决定 push_report 提交 GitHub 公开仓库 + 静态 Dashboard。
- "P0/P1/P2 场景分级" → 决定 cron.yml 的多条不同频率触发配置。

这条因果链的实际用法：改协议兼容性策略时，立刻知道下游哪些 Tool 会受影响；新人问"为什么 run_client 要支持 SDK 版本切换"，答案是"设计文档里这么定的"，而不是"我觉得这样好"。

#### (2) 项目阶段 Check List（可裁剪）

| 项目阶段 | 关键动作 | 产物 | 反问点 |
|---|---|---|---|
| 接到一句话需求 | 不立刻拆方案；先列出需求里藏的决策点 | 隐藏决策清单 | 这句话里哪些事没说清？ |
| 第一次翻译 · 抛提示词 | 预设 + 余地；先别谈技术框架 | 展开提示词 | 提示词是否堵住了 AI 偷懒？ |
| 第一次翻译 · 比对初版 | 拿完整设计预期去比对 | 漏点清单 | 哪些维度 AI 没列？ |
| 第一次翻译 · 反问补齐 | 反问（不是追问） | 补齐后的设计文档 | 还能继续反问吗？ |
| 第一次翻译完成 | 整合成可对外评审文档 | 完整设计文档 | 文档能让外部工程师读懂吗？ |
| 第二次翻译 · 路径反问 | "路径选对了吗" | 路径对照表 | 已有工具能复用多少？ |
| 第二次翻译 · 演进评估 | 评估未来演进空间 | 演进收益分析 | 自研路径的天花板在哪？ |
| 第二次翻译 · 收紧反问 | "别又冒新决策" | 完整方案文档 | 还有哪些点没收紧？ |
| 方案落地后 | 文档即硬约束；偏离则改代码或改文档 | 单一事实源 | 每行代码能否追溯到上游需求？ |
| 动手前夜 | 放下文档一晚，第二天再看 | 修订后的两份文档 | 哪个决策当时拍得太草率？ |

## 3. 第二部分 · 实战演示：RobustMQ 混沌测试的两次翻译

### 3.1 项目背景与一句话需求

<img src="imgs/aicmigr-25-autotest-02-plan-breakdown/b1efe8d514e36a9281b32e7a4558894f_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

#### (1) RobustMQ 与现有测试盲区

RobustMQ 是一个多协议消息中间件，支持主流的 MQTT 协议和创新的 mq9 协议，定位生产级使用。作为一个开源消息中间件项目，用户使用时的第一个问题不是功能够不够，而是**敢不敢用**：消息会不会丢、节点挂了能不能恢复、客户端 SDK 换个版本还能不能正常连。

这些问题靠单元测试和常规集成测试覆盖不到，必须在真实的混乱条件下持续验证。现有测试盲区集中在四点：

- 协议兼容性没有持续验证机制。
- 故障场景没有系统化覆盖。
- 长时间运行问题暴露不出来。
- 组合场景几乎没碰。

问题发现周期以月计，等用户踩坑反馈再修，节奏太慢，跟不上早期项目的迭代速度。所以需要一套自动化的 7×24 不间断混沌测试。

#### (2) 一句话需求

```
用 AI Agent 给 RobustMQ 实现一套 7×24 不间断跑混沌测试的系统，自动起集群、自动注入故障、自动调多语言 SDK 跑测试、自动收集日志、自动出报告。
```

这句话听起来已经很具体：5 个功能动词、1 个时间约束、1 个技术取向。但接到这种需求当天就开始写代码的人，没一个交付得好。这句话里藏的事比说的多得多，至少包括：

- [ ] 测什么协议、覆盖哪些 SDK 语言和版本？
- [ ] 故障注入到底是 kill 进程，还是网络延迟，还是磁盘填满？
- [ ] 测试报告给谁看？对内还是对外公开？
- [ ] 多语言 SDK 是预先写好的脚本，还是让 AI 现写？
- [ ] 频率怎么定？所有场景都跑一样频率，还是分级？

每一件都有几种合理选择，每种选择都会影响后面的技术方案。不先拍清楚就进技术设计，会出现"写到一半发现某个假设不对、回头改方案、改完又发现另一处不对"的循环。所以接到一句话需求，第一件事不是拆方案，是先把需求本身拆完整——这就是两次翻译要做的事。

### 3.2 第一次翻译：从一句话到完整设计

#### (1) 抛展开提示词

打开 Claude Code，给它这样一段提示词：

```
帮我把这个一句话需求展开成完整的技术设计文档：用 AI Agent 给 RobustMQ
实现 7×24 不间断跑混沌测试的系统，自动起集群、注入故障、跑多语言 SDK、
收日志、出报告。

这一步先别谈具体技术框架，把"为什么要做、做什么、不做什么、用什么思路、
覆盖什么场景、产出什么"讲清楚。

有几个我自己一时拍不准的关键选择，你列出来给我看，每个给两三个选项和
影响，我来拍。

不要复述，要展开。
```

提示词里三处关键设计：

##### ① "先别谈具体技术框架"
把问题域和方案分开，这是本篇的核心动作。不强调的话，AI 立刻就会往 Hermes 上跳。

##### ② "几个我拍不准的选择，列出来给我"

#### (5) 关键 Tool 设计要点

##### ① cluster_start：不引入 Docker
直接 spawn 多个 RobustMQ 进程。每个节点用独立端口和数据目录，数据目录用临时目录，`cluster_stop` 时清理。这一条决策极简，单机环境跑混沌测试，Docker 反而增加复杂度。

##### ② inject_fault：Chaosd 主，tc/kill 补
以 Chaosd（Chaos Mesh 单机版）为主，tc / kill 等系统命令为补充。Chaosd 提供 HTTP API，函数封装调用，参数对齐 Chaosd schema。

##### ③ run_client：本地版本管理工具切换
服务器上预装多语言运行时和版本管理工具（pyenv / gvm / rustup / sdkman / nvm），Skill 里通过版本管理工具切换，不用 Docker。单机环境足够，启动快。

##### ④ push_report：git push 到 GitHub 公开仓库
直接 `git clone` / `commit` / `push` 到 GitHub 公开仓库，不调 GitHub API，比 API 调用更简单可控。

##### ① Skill 边界原则
一个 Skill 套件等于一个独立的工作流。chaos-test 是"跑混沌测试"这个工作流；未来 Claude Code 深度分析是"分析失败报告"这个独立工作流，两者通过共享报告仓库联动，但 Skill 内部隔离，不共享 tools 目录。这一条让未来扩展有了明确的切线。

##### ② 故障注入选型标准
- **Chaosd**：用于所有需要精确控制的故障（进程级 kill 信号、网络精确延迟分布、磁盘 I/O 限速、时钟穿越）。
- **tc / kill**：用于极简快验证场景，临时拉起验证一下逻辑就行。

判断标准是"**故障是否需要被精确测量和回放**"——需要就用 Chaosd，只是验证可达性就用系统命令。

##### ③ GitHub 凭据
用 Deploy Key，只授写权限到 test-reports 单一仓库。私钥放在服务器 `~/.ssh/test-reports-deploy`，Skill 里通过 `GIT_SSH_COMMAND` 环境变量指定使用这个 key 做 git push，不进 Hermes 的 `.env`，也不进版本库。这一条接住了上一篇那条硬规则：**7×24 无人值守最容易翻车的就是凭据和 allowlist**。

完整方案文档如下： https://robustmq.com/zh/Blogs/95 （因为内容篇幅有限，就不全贴进来了）。

到这里，第二次翻译做完了。手上拿着的不再是"在 Hermes 上做一个混沌测试系统"这种模糊方向，是一份可以分配给具体工程师按依赖顺序开干的方案。

### 3.4 两份文档之间的因果链

<img src="imgs/aicmigr-25-autotest-02-plan-breakdown/a77f6a23d03bd2daef95d28dedcde1e8_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

走完两次翻译，手上有两份真实的文档：

- **完整设计文档**：讲清楚为什么做、做什么、按什么思路。
- **完整方案文档**：讲清楚怎么具体落到 Hermes 上。

这两份文档之间有一条清晰的因果链：

| 设计文档决策 | 方案文档落地 |
|---|---|
| 协议兼容性是核心命题 | `run_client` Tool 的多语言 SDK 矩阵设计 |
| 测试记录完全公开建立社区信任 | `push_report` 提交到 GitHub 公开仓库 + Quality Dashboard 静态读取 |
| P0 / P1 / P2 场景分级 | `cron.yml` 那 4 条不同频率的触发配置 |

这条因果链是这两份文档真正的价值。不是文档本身有多漂亮，而是后续每一行代码、每一个配置、每一个判断，都能追溯到上游某条需求决策。

- 开发过程中如果有人问"为什么 run_client 要支持 SDK 版本切换"，答案不是"我觉得这样好"，而是"因为设计文档里的协议兼容性目标这么定的"。
- 如果某天需要改协议兼容性策略，也能立刻知道下游有哪些 Tool 会受影响。

接下来开发的时候，这两份文档就是**单一事实源**。不是开发完成后再补的文档，而是开发过程中的硬约束。任何代码偏离了文档的决定，要么改代码，要么改文档，然后回来重新对齐。

## 4. 小结与思考

<img src="imgs/aicmigr-25-autotest-02-plan-breakdown/a55015bdb23a6c0fb7d16ad8ee2f9394_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

### 4.1 两次翻译口诀

把本篇方法论浓缩为口诀，按项目时间线依次执行：

```
一句话需求藏的事，比说的多。
接到先翻译两次，再动手。

第一次翻译：让 AI 穷举该拍的事，人来拍方向。
          产物与技术栈无关，讲清为什么做、做什么、不做什么。
第二次翻译：先反问"路径选对了吗"，再落到技术栈。
          产物可分配开干，包含 Skill/Tool/场景/触发配置。

每次翻译都反问到收敛为止。
AI 的价值是穷举，人的价值是拍方向。
最后留一晚沉淀，第二天再回头改。
```

为什么接到需求当天就开始写代码的人没一个交付得好？理由不是"先写文档显得专业"，而是**写文档的过程本身就是把需求里所有藏着的事拍清楚的过程**。不拍清楚就动手，等于把代码当成需求的探索工具，代价是改代码远比改文档贵。

### 4.2 放下的经验

写完两份文档，第一件事其实是**放下**。

第二天早上再看一遍，常常会发现某个决策当时拍得太草率，趁还没动手赶紧改回来。真正动手前，留一晚给方案沉淀——这条经验比方法论本身值钱。

放下的原因在于：第一次翻译和第二次翻译的每一次反问，都在消耗"当时的判断力"。判断力疲劳时拍出的决策，第二天看往往能立刻发现偏差。一晚上的间隔让大脑从"穷举模式"切回"审视模式"，再回头看方案，往往能挑出前一天挑不出来的问题。
