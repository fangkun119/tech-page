### 1.1 本篇定位与产出物（结论先行）

<img src="imgs/aicent-19-core-fea-recap/e21dff4b9d866d3432c50aa7658c3405_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

系列第 13 到 18 篇，完成了 Hify 从"能跑"到"能用"的全部核心功能——模型供应商（Provider）管理、适配层、智能体（Agent）配置、对话引擎、前端界面。本篇是这条主线的实操回顾，不引入新技术栈，而是把开发过程中散落各处的隐性经验，提炼成一份显性的**"与 AI 协作把一个业务模块从 0 跑到 1"的节奏管理与经验复用方法论**。

本篇的产出物有两份：

- 一份可裁剪的**协作开发节奏方法论**（第 1 章），讲清何时该快、何时该慢、何时该停下来想清楚、何时该完整验收、经验如何沉淀复用；
- 一份**模块交付 Check List**（第 2 章），在新模块启动前、交付中、交付后三个时点逐项勾选。

### 1.3 核心功能篇协作开发节奏方法论六步骨架

<img src="imgs/aicent-19-core-fea-recap/3b64e163bf3c1a44930c0dea78c7c5cd_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

下面六步是本篇的纲领，也是"与 AI 协作开发一个业务模块"在任何技术栈里都成立的通用动作序列。每步用「方法论动作 + 可观测产物」表述，动词开头，能落到具体产物上才算闭环。

#### (1) 节奏预判：先判断这个模块哪里快、哪里慢，再分配时间预算

##### ① 动作

模块启动前，先对开发节奏做一次预判：哪些部分有基础设施或 Skill 兜底、几乎不占时间（脚手架、模板代码、前端基础组件对接），哪些部分必须慢下来（供应商选型、数据模型、边界决策、跨层联调）。把时间预算优先投到"慢"的部分，而不是平均用力。

##### ② 可观测产物

一份该模块的"快慢时间预算表"，至少列出两类条目：快项（标注兜底来源，如"继承 BaseEntity/BaseMapper""模块交付 Skill""HifyTable 组件"）与慢项（标注必须慢的原因，如"数据模型边界决策""跨层问题排查"）。

##### ③ 必须挂住的事实

了解快在哪里、慢在哪里，比一个总时间更有参考价值。"慢"的时间绝大多数是"想清楚"的时间，这不是拖延，是前置投入——漏掉一个边界决策，后面返工的代价是数倍。

#### (2) 想清楚先行：数据模型与边界决策落字成文，再让 AI 动手

##### ① 动作

每个模块开始前，用"咨询模式"与 AI 反复讨论供应商分析、数据模型、边界问题，直到产出一份包含明确字段定义与决策记录的设计文档，再更新建表脚本、再让 AI 写业务代码。讨论环节主动追问"还有哪些边界没考虑到"，而不是直接下令建表。

##### ② 可观测产物

一份可评审的数据模型设计文档，含三类显性产物：表结构 DDL、每个关键字段的取舍理由（如"鉴权配置为什么用 JSON""健康状态为什么独立成表"）、边界决策清单（逐条落字，不靠记忆）。

##### ③ 必须挂住的事实

"想清楚"这一步不可跳过。跳过讨论直接让 AI 建表，七八成的边界设计会漏掉，到下游依赖这些字段时再回头改表、改接口、改前端，代价是前置讨论时间的数倍。把这部分时间当成必要投入，不算慢，算前置。

#### (3) 模板复用：把脚手架交给 AI 一键生成，人只做 review

##### ① 动作

凡是能由项目基础设施兜底的模板代码（Entity 继承基类、Mapper 继承基类、Controller 套统一返回结构、前端页面把数据源从 mock 换成 API），用一条提示词让 AI 生成，人只做针对性 review（如基类注解有没有加、JSON 字段的类型处理器有没有对上），不逐行手写。

##### ② 可观测产物

一份"模板代码 review 检查项"清单，列明每种模板代码生成后必看的几处（如 Entity 看 @TableName(autoResultMap=true)、Controller 看入参校验注解、前端页面看数据源替换是否彻底）。

##### ③ 必须挂住的事实

脚手架不占时间，前提是前面真的把基础设施搭好了。基础组件的回报，在用它的时候才真正感受到——封装得好，每个新管理页面的改动量极小。

#### (4) 报错给全量：把完整日志与相关配置一起喂给 AI

##### ① 动作

遇到报错时，不要只把最后一行错误贴给 AI。把完整调用栈、相关配置文件、触发报错的操作步骤一起提供，让 AI 在完整上下文里定位根本原因，而不是根据一行报错去猜。

##### ② 可观测产物

一次报错排查的提问，包含四要素：完整错误栈、触发操作的步骤、相关配置或代码片段、已尝试的排查方向。通常一轮就能定位根因。

##### ③ 必须挂住的事实

只给最后一行，AI 在猜；给完整栈和相关配置，它能定位根本原因。这条规则对任何调试场景都成立。

#### (5) 完整验收：curl 通了不算交付，要走完整链路

##### ① 动作

模块交付的验收标准是"走完整链路"，不是"接口测通"。后端接口用命令行验证只是必要条件；必须按真实请求路径走完每一层（浏览器操作 → 反向代理 → 后端处理 → 数据推流 → 前端渲染），任何一层都可能有自己的坑（如反向代理对流式响应的缓冲）。

##### ② 可观测产物

一份分层验收清单，按真实链路逐层勾选，每层标注"已验证通过"。命令行验证只勾掉其中一层，不能替代整条链路。

##### ③ 必须挂住的事实

命令行验证会绕过反向代理等中间层，掩盖某些环境相关的问题。换了一层环境就暴露的问题，AI 不会主动提醒，靠的是开发者自己的经验——但只要知道往哪个方向找，排查下来时间不长。

#### (6) 经验沉淀复用：把这次流程编码成 Skill，并在下次真的调用

##### ① 动作

每完成一个模块，把这次的交付流程（步骤、产出物、验证方式、踩过的坑、需要人类拍板的决策点）沉淀成一个 Skill 文件。更关键的是：每开始一个新模块，先停一秒问"有没有对应的 Skill"，主动调用它。Skill 每用一次就迭代一次，把新发现的坑补进去。

##### ② 可观测产物

一份 Skill 文件（如 module-delivery.md），结构包含：分步流程、每步产出物与验证方式、标注"等待用户确认"的决策点、踩坑注意事项。外加一条在项目规范文件顶部固化下来的提醒："开始新模块前，先看 .claude/skills/ 目录。"

##### ③ 必须挂住的事实

经验编码化之后还要真的被调用，才有价值。Skill 写了不用，等于把经验存档了但从不读档。习惯养成之前，可以用一句固化提醒强制自己想起来。

## 2. 核心功能篇模块交付 Check List（可裁剪速查）

<img src="imgs/aicent-19-core-fea-recap/d77419730a87dcf7e251d8cb46f296c2_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

把第 1 章的六步方法论转成项目阶段的勾选项，供新模块启动前、交付中、交付后三个时点快速查阅。照抄进团队 Wiki 即可。

**启动前：判断节奏、想清楚**

- [ ] 是否判断了本模块的快项与慢项，并据此分配时间预算？（对应 1.3-(1)）
- [ ] 供应商选型 / 数据模型 / 边界问题是否已与 AI 讨论清楚，产出落字成文的设计文档？（对应 1.3-(2)）
- [ ] 每个关键字段是否有明确的取舍理由（为什么用这个类型、为什么独立成表）？（对应 1.3-(2)）
- [ ] 数据模型是否更新到建表脚本，并用独立配置启动验证建表成功？（对应 1.3-(2)）
- [ ] 开始前是否停一秒，确认有没有对应的 Skill 可调用？（对应 1.3-(6)）

**交付中：模板复用、给全量报错**

- [ ] 模板代码（Entity/Mapper/Controller/前端数据源替换）是否交给 AI 一键生成？（对应 1.3-(3)）
- [ ] 生成后是否按 review 检查项逐项核对（基类注解、类型处理器、入参校验等）？（对应 1.3-(3)）
- [ ] 遇到报错时，是否提供了完整栈 + 触发步骤 + 相关配置，而不是只给最后一行？（对应 1.3-(4)）
- [ ] 每一步是否编译或命令行验证通过再进下一步？（对应 1.3-(3)/(5)）
- [ ] 关键决策点是否由人类拍板，AI 只在确认后才继续？（对应 1.3-(2)/(6)）

**交付后：完整验收、经验沉淀**

- [ ] 后端接口是否用命令行走通了基础验证？（对应 1.3-(5)）
- [ ] 是否按真实链路逐层验收（浏览器 → 反向代理 → 后端 → 推流 → 前端渲染）？（对应 1.3-(5)）
- [ ] 是否把本次流程沉淀 / 迭代进了 Skill，把新踩的坑补进去？（对应 1.3-(6)）
- [ ] 是否在项目规范顶部固化了"开始新模块前先看 .claude/skills/"的提醒？（对应 1.3-(6)）

## 3. 实战背景：核心功能篇的真实节奏（实战）

本章是第二部分（实战）的开篇，用核心功能篇的真实开发数据，印证第 1 章方法论骨架中的 **1.3-(1) 节奏预判、1.3-(2) 想清楚先行、1.3-(5) 完整验收、1.3-(6) 经验沉淀复用** 四条动作，并自然带出三个反直觉问题——这三个问题正是后续三个场景（第 4—6 章）要逐个回答的。

### 3.1 总节奏：比预期快，但快慢分布不均

<img src="imgs/aicent-19-core-fea-recap/8777e288c5a97240a04451bf5703e82c_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

工程搭建篇（系列第 8—12 篇）结束之后，进入核心功能开发时，开发者普遍会有一点忐忑：骨架搭好、组件封装好、UI 出来了，但真正的业务逻辑才是重头戏——外部 API 调用、流式推送、上下文管理，这些内容以前在项目里摸过，但每次都要花相当多时间。

实际跑下来，比预期快很多。三个核心模块的真实时长如下：

- **Provider 模块**：前后端全部交付，大概**半天**。
- **Agent 模块**：因为有了模块交付 Skill，比 Provider **还快**。
- **对话引擎**：复杂一些，两个完整的**工作日**；但其中大半时间花在了"想清楚"上，真正让 Claude Code 写代码的时间很短。

这正是 **1.3-(1) 节奏预判** 要表达的核心：了解快在哪里、慢在哪里，比一个总时间更有参考价值。下面把"快"和"慢"分别摊开。

### 3.2 快在哪里：基础设施、Skill、前端组件三重兜底

#### (1) 脚手架几乎不占时间——基础设施的回报

有了系列第 8—12 篇的基础设施，每个业务模块的"脚手架"几乎不占时间。Entity 继承 BaseEntity、Mapper 继承 BaseMapper、Controller 套 Result + PageHelper，这些模板性的东西，一条提示词生成，review 一遍，基本不用改。

这一条对应 **1.3-(3) 模板复用**：脚手架交给 AI 生成，人只做 review。

#### (2) Skill 带来的节奏变化——第二个模块开始一句话启动

Skill 带来的节奏变化更明显。系列第 13 篇手动做 Provider，第 14 篇沉淀了模块交付 Skill，到第 15 篇做 Agent 时，一句提示词就启动了整个流程：

```text
按模块交付 Skill 的流程，帮我做 Agent 模块
```

Claude Code 自动按流程走，在数据模型确认环节停下来等开发者拍板，不多做也不漏做。做第三个模块、第四个模块，这个节奏感越来越稳。这一条印证 **1.3-(6) 经验沉淀复用**——经验编码成 Skill 之后被真的调用，才会形成复利。

#### (3) 前端对接改动量极小——基础组件的回报

前端对接也比以前快很多。系列第 11 篇封装的 HifyTable 和 HifyFormDialog，让每个管理页面的改动量非常小：API 文件是新增的，页面代码基本只是把数据源从 mock 函数换成 API 函数，分页、loading、空状态、确认弹窗全都不用重写。基础组件的回报，在用的时候才真正感受到。

### 3.3 慢在哪里：想清楚的时间，与跨层问题的排查

慢的情况分两类，两类都不是"慢"，而是必要投入。

#### (1) 第一类："想清楚"的时间——前置投入，不是拖延

每个模块开始前，供应商分析、数据模型讨论、边界决策，这些和 Claude Code 来回讨论的时间不短。Provider 模块将近**一个小时**，对话引擎的设计讨论甚至超过了写代码的时间。

但这个时间花得值：这些决策如果漏掉，后面返工的代价是几倍。有经验的开发者会把这部分时间当成必要投入——不算慢，算前置。这一条直接印证 **1.3-(2) 想清楚先行**。

#### (2) 第二类：跨层问题的排查——换一层环境就暴露的问题

流式响应在 curl 里验证是对的，上了浏览器却发现打字机效果失效，根因是 Nginx 缓冲——curl 绕过了 Nginx，所以发现不了。

这类"换了一层环境就暴露"的问题，Claude Code 不会主动提醒，靠的是开发者自己的经验。排查下来时间不长，但前提是知道往哪个方向找。这一条印证 **1.3-(5) 完整验收**：curl 不是终点，要走完整链路。

### 3.4 最让人惊喜的：咨询模式下"边问边做"的两件事

两件事都体现了"咨询模式"的价值——不需要先懂再做，可以边问边做，把认知直接翻译成数据结构落地。

#### (1) 供应商分类——五分钟替代一两小时的调研

系列第 13 篇里，让 Claude Code 分析供应商分类的那一步。它按接口兼容性做了分类，指出 DeepSeek、Moonshot 这些国产模型接口和 OpenAI 完全兼容，只需要换 baseUrl。

这个洞察直接影响了架构决策——不是每接一个供应商就写一套适配代码，而是设计一个"OpenAI 兼容"类型让用户自己配 baseUrl。以前这类调研要自己搜文档、找对比文章，花一两个小时还不一定全面；现在一条提示词五分钟，而且直接给出设计建议。

#### (2) Agent 三层配置——主动给结构，而不是问出来的

系列第 15 篇里，让 Claude Code 分析 Agent 概念的那一步。开发者表示自己对 Agent 的理解很模糊，它回了一张普通对话 vs Agent 的对比表，还主动区分了三层配置：**身份定义、能力绑定、运行参数**。

这三层不是问出来的，是它主动给的结构。这让人真正体会到系列第 14 篇说的"让 Claude Code 教你"是什么感觉——边问边做，一两个小时建立七成认知，然后把理解翻译成数据结构落地。

### 3.5 三个反直觉问题：想清楚 / Skill / curl

<img src="imgs/aicent-19-core-fea-recap/bb76843b72cc98fdb411dab0c4780b48_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

下面三个问题，是大家在核心功能篇里比较关心的，也是最容易踩反直觉的地方。每个问题的回答都会回头指认第 1 章方法论骨架的编号。

#### (1) "想清楚"这一步可以跳过吗？

很多人急着动手，觉得先问概念是浪费时间，直接让 Claude Code 写代码更快。

答案是不能跳。系列第 13 篇数据模型讨论将近一个小时，但产出很具体：

- `auth_config` 用 JSON 存——后来加 Anthropic 和 Azure 时，一行表结构都没改；
- 健康状态独立成表（`provider_health`）——定时探测的高频写，不影响 `provider` 表的缓存；
- `model_config` 分展示名（`name`）和调用 ID（`model_id`）——Azure 的 deployment name 和展示名不同，这个字段设计直接解决了问题。

跳过讨论直接让 Claude Code 建表，这三个设计八九成会漏掉。到后面对话引擎依赖这些字段时，要回来改表、改接口、改前端，代价是这一小时的三倍以上。

想清楚不是在拖慢进度，是在前面花一点时间，避免后面花五倍时间返工。这一条印证 **1.3-(2) 想清楚先行**：数据模型与边界决策落字成文，想清楚不可跳过。

#### (2) Skill 写完了之后，真的会用吗？

系列第 14 篇写了模块交付 Skill，第 15 篇第一次用，但有一个现象值得警惕：Skill 的存在感比 CLAUDE.md 低，发提示词时不一定会想起来。

一个可操作的做法是：每开始一个新模块，先停一秒问自己"有没有对应的 Skill？"习惯养成之前，可以在 CLAUDE.md 顶部加一行固化提醒：

```text
开始新模块前，先看 .claude/skills/ 目录
```

Skill 写了不用，等于把经验存档了但从不读档。本系列从第 14 篇开始一直在说"经验可以编码化"，但编码化之后还要真的被调用，才有价值。这一条印证 **1.3-(6) 经验沉淀复用**：编码成 Skill 并真的调用、Skill 迭代。

#### (3) 对话引擎验收，curl 够了吗？

不够。系列第 17 篇用 curl 验证了流式输出是对的，但第 18 篇上了前端之后，打字机效果在浏览器里失效了。这是 Nginx 缓冲的问题，curl 绕过了 Nginx 所以发现不了。

后端接口测通是必要条件，不是充分条件。完整验收必须走完下面这条完整链路，每一层都可能有自己的坑，curl 只覆盖了其中一段：

```text
浏览器操作 → Nginx 转发 → 后端处理 → SSE 推流 → 前端渲染
```

这一条印证 **1.3-(5) 完整验收**：curl 不是终点，走完整链路。

### 3.6 方法论指认

本章用核心功能篇的真实时长数据（Provider 半天、Agent 更快、对话引擎两个工作日、数据模型讨论近一小时）支撑了 **1.3-(1) 节奏预判**——快慢分布比总时间更有参考价值；用"想清楚不可跳过、跳过返工代价三倍以上"支撑了 **1.3-(2) 想清楚先行**；用 curl 绕过 Nginx 的反例支撑了 **1.3-(5) 完整验收**；用"Skill 写了要真的调用、否则等于存档不读档"支撑了 **1.3-(6) 经验沉淀复用**。

本章末尾提出的三个反直觉问题——想清楚能不能跳过、Skill 真的会用吗、curl 够不够——正是后续三个场景要回答的：第 4 章场景一（后端从 Entity 到 Controller）回答"想清楚怎么落到数据模型"，第 5 章场景二（前端对接与浏览器验收）回答"curl 之后还要走多远"，第 6 章场景三（把经验变成 Skill）回答"Skill 怎么沉淀又怎么被调用"。这三个问题，后三个场景分别给出答案。

## 4. 场景一：后端从 Entity 到 Controller（实战）

<img src="imgs/aicent-19-core-fea-recap/c01f28abf4be56c9f6c588d53604bd8e_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

本章印证本篇第 1 章方法论骨架中的 **1.3-(2) 想清楚先行**（数据模型讨论）、**1.3-(3) 模板复用**（Entity/Mapper/Controller 一键生成 + review）、**1.3-(4) 报错给全量**（curl 联调报错把完整堆栈喂给 AI 自愈）。目标：用 Provider 模块后端的四步交付，把"从底层往上搭、每步可独立验证"的节奏演示出来，最后用 curl 验证全部接口。

### 4.1 四步交付流程

#### (1) 确认数据模型，更新 schema.sql

动手写代码前，先把数据模型落字成文——这是 1.3-(2) "想清楚不可跳过"在本步的直接兑现。提示词如下：

```text
按照以下数据模型，在 schema.sql 中新增三张表：
provider（含 auth_config JSON 字段）、
model_config（含 name 和 model_id 分开存）、
provider_health（独立表，含 fail_count、latency_ms、last_success_at）。
```

##### ① 三张表的核心设计决策

提示词虽然短，背后却锁定了三个不能含糊的设计决策，每一个都决定了后续接口与前端怎么搭：

- **auth_config 用 JSON 字段**：不同供应商的鉴权结构差异巨大（OpenAI 用 `apiKey`、Anthropic 还要 `anthropicVersion`、Ollama 完全无鉴权、Azure 需要 `resourceName`/`apiVersion`）。给每种鉴权方式定固定列行不通，用 JSON 按 `type` 存不同结构，未来加新供应商零改表。
- **provider_health 独立成表**：健康状态是高频写——定时探测每分钟更新一次，每次 LLM 调用也可能更新。如果挂在 `provider` 表里，高频写会和业务读竞争行锁。独立成表后，`provider` 表写少读多可以放心加 `@Cacheable` 缓存，`provider_health` 表不缓存直接读库。
- **model_config 分 name 和 model_id**：`name` 是给人看的展示名（如 GPT-4o），`model_id` 是调用时实际传给 API 的值（如 `gpt-4o`）；Azure 场景下 `model_id` 就是 deployment name，和展示名不一样，分开存才能既好看又好调。

##### ② 建表验证

更新完 schema.sql，用 mock profile 启动一次，确认三张表建表成功再往下走——"想清楚"的产出必须能用，不能停留在纸面。

#### (2) Entity + Mapper

数据模型落定后，第一步交给 AI 生成的是最干净的脚手架层，对应 1.3-(3) 模板复用。提示词如下：

```text
按照 CLAUDE.md 规范和 schema.sql 的表结构，在 hify-provider 中创建
Provider、ModelConfig、ProviderHealth 的 Entity 和 Mapper。
Entity 继承 BaseEntity（ProviderHealth 除外），Mapper 继承 BaseMapper。
auth_config 和 extra_params 字段用 JacksonTypeHandler。
```

##### ① 任务干净度判断

这个任务很干净，Claude Code 基本不出错——因为它完全是模板套用：继承关系、注解、TypeHandler 都是 CLAUDE.md 规范里写死的模式，没有业务判断空间。人只做 review，不参与生成。

##### ② review 重点

正因为任务干净，review 反而要盯死两个易漏点：

- **`@TableName(autoResultMap = true)` 有没有加**：这是让 JacksonTypeHandler 生效的前提，漏了会导致 JSON 字段读出来是原始字符串而不是反序列化后的对象。
- **JSON 字段的 TypeHandler 有没有对上**：`auth_config` 和 `extra_params` 必须在字段上显式声明 `typeHandler = JacksonTypeHandler.class`，并且实体类上的 `@TableName` 要打开 `autoResultMap`，二者缺一不可。

#### (3) Service：连通性测试

连通性测试是本步的重点演示任务，因为它是后端四步里唯一带"分发逻辑"的环节。提示词如下：

```text
在 hify-provider 中实现连通性测试。根据 provider.type 分发：
OPENAI 和 OPENAI_COMPATIBLE 调 GET /v1/models（Bearer Token），
OLLAMA 调 GET /api/tags（无认证）。
统一返回 ConnectionTestResult（success、latencyMs、modelCount、errorMessage）。
使用 LlmHttpClient，超时 10 秒。
```

##### ① 按 provider.type 分发到不同探测接口，统一返回结构

连通性测试的核心是按 `provider.type` 分发到不同的探测接口：OPENAI 和 OPENAI_COMPATIBLE 走 `GET /v1/models`（Bearer Token 鉴权），OLLAMA 走 `GET /api/tags`（无认证）；不同分支内部各自处理鉴权方式、接口路径和响应解析差异，但对外统一返回 `ConnectionTestResult`（success、latencyMs、modelCount、errorMessage），上层调用方无需关心底层供应商差异。HTTP 调用复用前面封装的 `LlmHttpClient`，统一超时 10 秒。

#### (4) Controller + curl 验收

最后一步把 Service 收口到对外接口，并用 curl 做端到端验证。提示词如下：

```text
在 hify-provider 中创建 ProviderController，实现所有接口：
POST 创建、GET 列表（分页）、GET 详情（含 modelConfig 和 health）、
PUT 更新、DELETE 删除、POST /{id}/test-connection。
所有接口返回 Result<T>，入参加 @Valid 校验。
```

Controller 是模板复用回报最直观的一层：只做参数校验和调 Service，统一返回 `Result<T>`，分页用 PageHelper，入参加 `@Valid`——和前面跑通的标准流程一模一样，只是字段和业务规则不同。

curl 验收原文如下，行尾续行符原样保留：

```bash
# 创建 OpenAI Provider
curl -X POST http://localhost:8080/api/v1/providers \
  -H "Content-Type: application/json" \
  -d '{
    "name": "OpenAI",
    "type": "OPENAI",
    "baseUrl": "https://api.openai.com",
    "authConfig": {"apiKey": "sk-xxx"}
  }'

# 连通性测试
curl -X POST http://localhost:8080/api/v1/providers/1/test-connection

# 查看详情（含模型列表和健康状态）
curl http://localhost:8080/api/v1/providers/1

# 分页列表
curl "http://localhost:8080/api/v1/providers?page=1&pageSize=10"
```

运行 curl 的过程中通常会触发一连串报错（字段映射、JSON 解析、参数校验等）。处理方式是 1.3-(4) 报错给全量的标准动作：把完整报错堆栈喂给 Claude Code，由它定位、改代码、重新跑，直到四条接口全部跑通。验收时要看到三件事真的成立：

- **test-connection 响应**：`latencyMs` 有值、`modelCount` 是真实拉取到的模型数量，而不是写死的占位值。
- **详情接口**：`modelConfigs` 列表里有从供应商真实拉取的模型，`providerHealth.status` 变成 UP。
- **分页列表**：`PageResult` 结构正确，`total` 和 `list` 两个字段都在。

### 4.2 方法论指认

Provider 后端这四步是标准交付流程在实战中的首次端到端验证，每一步都精确对位到第 1 章方法论骨架的一条：(1) 更新 schema.sql 印证 **1.3-(2) 想清楚先行**——auth_config 用 JSON、provider_health 独立成表、model_config 分 name/model_id 三个决策，必须在动手写 Entity 之前落字成文，想清楚不可跳过；(2) Entity + Mapper 与 (4) Controller 印证 **1.3-(3) 模板复用**——脚手架交给 AI 生成，人只做 review，盯 `@TableName(autoResultMap = true)` 和 JSON TypeHandler 是否对上；(4) 的 curl 联调印证 **1.3-(4) 报错给全量**——把完整堆栈喂给 AI 自愈，直到 test-connection 的 latencyMs/modelCount 真实、详情接口的 providerHealth.status 变 UP、分页 PageResult 结构正确。注意 curl 跑通只是后端接口可达，并不是交付闭环——完整的"数据库到浏览器"闭环要等前端对接完成才算交付。

### 5.3 方法论指认

本章两步对应两条方法论：

- **1.3-(3) 模板复用**：第二步把 ProviderList.vue 的 mock 数据源换成 API 时，HifyTable 只改了一行 api prop，分页、loading、空状态全部自动处理。这是系列第 11 篇封装前端基础组件的回报兑现——封装得好，每个新管理页面改动量极小。脚手架不占时间的前提，是前面真的把基础设施搭好了。
- **1.3-(5) 完整验收**：第三步的浏览器验收不是重复劳动，而是补充场景一 curl 验证绕过的链路段——Nginx 代理、前端渲染、状态更新。命令行只覆盖了后端一段，反向代理的缓冲坑只有在走完整链路时才暴露。验收的标准是"整条链路通"，不是"接口测通"。

## 6. 场景三：把经验沉淀成 Skill（实战）

本章印证第 1 章方法论的 1.3-(6) 经验沉淀复用（核心），并连带印证 1.3-(2) 想清楚先行（Skill 标注"等待用户确认"决策点）与 1.3-(3) 模板复用（Skill 让一份流程在多个模块之间稳定复现）。

**本章定位**：承接场景一（后端交付）与场景二（前端对接）。Provider 模块已经从前到后跑完一个完整闭环，本场景不写新功能代码，而是把这次走过的流程教给 Claude Code，让它在下一个模块上自动按这套流程走。这一步是 1.3-(6) 的兑现环节——经验不只停在"这次做完了"，而是落到 `.claude/skills/` 里，并在 Agent 模块上真的被调用、真的被迭代。

**本章演示的三步**：先让 Claude Code 自己解释 Skill 机制（咨询模式的再一次应用）；再把 Provider 模块的交付流程沉淀成 `module-delivery.md`；最后用一句话启动 Agent 模块，验证 Skill 的复用效果。

### 6.1 第一步：让 Claude Code 教你 Skill 是什么

<img src="imgs/aicent-19-core-fea-recap/a5d0ccb67197c8bc176a3ce4f6e1162c_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

进入沉淀环节之前，先确认一个前提：开发者手上要有一份对 Skill 机制的最基本认知——它是什么、怎么创建、和 CLAUDE.md 有什么区别。这一步不必去翻官方文档，直接问 Claude Code 自己，这是咨询模式在"机制理解"场景下的又一个应用。

提示词（原样保留）：

```text
Claude Code 的 Skill 机制是什么？
怎么创建、怎么使用、和 CLAUDE.md 有什么区别？
```

#### (1) Claude Code 给出的回答要点

Claude Code 对这个问题的回答是结构化的，核心两条：

- **Skill 是 `.claude/skills/` 目录下的 Markdown 文件**，定义一类特定任务的操作流程；创建就是在该目录下新建一个 `.md` 文件，使用时在对话里引用它的名字即可。
- **CLAUDE.md 与 Skill 的分工不同**：CLAUDE.md 是全局规范，每次对话自动加载，承载的是"所有任务都要遵守的约束"；Skill 是任务手册，引用时才生效，承载的是"做某一类任务的标准流程"。

#### (2) 两条机制的关键区别

把两条机制的差异讲清楚，后面才知道什么时候用哪个：

| 维度 | CLAUDE.md | Skill |
|---|---|---|
| 加载方式 | 每次对话自动加载 | 引用时才生效 |
| 承载内容 | 全局规范（命名、分层、返回结构等约束） | 任务手册（某类任务的标准流程） |
| 适用场景 | 任何对话都要遵守的底线 | 重复出现的固定流程（如模块交付） |

#### (3) 这一步的方法论含义

这一步本身就是咨询模式的样板：**不必翻文档，问 Claude Code 自己就教**。1.3-(6) 强调经验要"真的被调用"，而调用的前提是开发者知道有这个机制、知道它和 CLAUDE.md 的边界——这一认知用一次咨询就能建立，不需要先去读完一轮官方手册再动手。

### 6.2 第二步：把 Provider 模块的交付流程沉淀成 Skill

<img src="imgs/aicent-19-core-fea-recap/95a8ef609bac72320dbd24163234bd8f_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

机制理解清楚之后，把 Provider 模块走过的真实流程固化下来。沉淀时不只是"记一下步骤"，而是要把每一步的产出物、验证方式、关键决策点、踩过的坑一起写进去——这样下一份模块才能在不靠记忆的情况下复现这次的质量。

提示词（原样保留）：

```text
我刚完成了 Hify 项目 Provider 模块的开发，流程是这样的：
1. 先用咨询模式梳理供应商选型、数据模型设计、边界问题
2. 数据模型确定后更新 schema.sql
3. 后端按 MVC 分层拆解：Entity+Mapper → DTO → Service → Controller
4. 每步编译或 curl 验证通过再进下一步
5. 前端对接：创建 API 文件，把 mock 数据源换成真实 API
6. 完整验收：后端 curl + 浏览器全流程
帮我把这个流程沉淀成 .claude/skills/module-delivery.md。
要求：每一步有明确的产出物和验证方式，关键决策点标注"等待用户确认"，
把我踩过的坑写成注意事项。
```

#### (1) 打开生成的 module-delivery.md，先看结构

生成完成后，打开 `module-delivery.md` 检查它的整体结构。一份合格的模块交付 Skill 至少包含这几块：分步流程、每步产出物与验证方式、标注"等待用户确认"的决策点、踩坑注意事项。结构齐全，才能在下一个模块上稳定复现。

#### (2) review 三件事

Skill 生成完不等于可用，必须做一次针对性 review。这次 review 抓三件事：

##### ① 产出物是否明确

每一步的产出物必须落到具体可观测的产物上，而不是停在动词层面。判定标准是：把"产出物"那一栏遮住，能不能一眼看出这步做完会多出哪个文件、哪段配置。例如：

- 不合格写法："做需求分析"——产出物是什么不清楚。
- 合格写法："产出包含 DDL 的需求文档"——产出物是带 DDL 的文档，可评审、可 diff。

这一项对应 1.3-(2) 想清楚先行的可观测化要求：决策要落字成文，产出物要能被指认。

##### ② 决策点是否标注"等待用户确认"

关键决策点必须由人类拍板，Claude Code 只能在确认之后才继续。检查 Skill 里每个需要拍板的环节（供应商选型、数据模型定稿、前后端切换）是否显式标注了"等待用户确认"。这一项是 1.3-(6) 与 1.3-(2) 的交汇点——Skill 把"哪里该停下来等人"固化成流程的一部分，不让 AI 自作主张越过决策点。

##### ③ 坑是否写进去了

这次真实踩过的坑必须写进注意事项。这一项是 Skill 能否真正节省时间的关键：下一份模块会踩同样的坑，除非这次把它记下来。

#### (3) review 后手动补进去的一条注意事项

review 之后，有一条注意事项是手动补进去的——它是这次真实踩过的坑，Claude Code 不会主动知道：

> JSON 字段必须用 `@TableName(autoResultMap = true)` + `JacksonTypeHandler`，否则反序列化为 `null`。

这条坑的背景：Provider 模块里 `auth_config` 是 JSON 字段，如果 Entity 上漏了 `@TableName(autoResultMap = true)`、或字段上漏了 `@TableField(typeHandler = JacksonTypeHandler.class)`，查询回来的 JSON 字段会反序列化成 `null`，且不会报错——数据"消失"得很安静。这类坑不写进 Skill，下一个模块的 JSON 字段会原样再踩一遍。

手动补这一条的过程，本身就是 1.3-(6) 的核心动作：**每完成一个模块，把这次新踩的坑回填进 Skill**。Skill 不是一次性产物，而是活文档。

### 6.3 第三步：用 Skill 启动下一个模块，验证效果

Skill 写完不是终点，1.3-(6) 的后半句是"并真的调用"。这一步用 Agent 管理模块验证 Skill 的复用效果——预期是：一句话启动，Claude Code 自动按 Skill 的流程走，在数据模型设计完之后停下来等人拍板。

提示词（原样保留）：

```text
按模块交付 Skill 的流程，帮我做 Agent 管理模块。先从第一步开始。
```

#### (1) 对比演示：没有 Skill 时要手写一大段流程

为了看清 Skill 的价值，先做一个对比。没有 Skill 时，启动一个新模块要手写一大段流程描述，把每一步都讲给 Claude Code：

> "先帮我分析需求，然后设计数据模型，然后按 Entity、DTO、Service、Controller 的顺序拆解，每步编译通过再进下一步，前端再把 mock 换成 API，最后 curl 加浏览器全流程验收……"

这段描述每次都要写、每次都可能漏步或顺序走样。有了 Skill，同样的事一句话就能启动——`按模块交付 Skill 的流程，帮我做 Agent 管理模块。先从第一步开始。`流程细节全部由 Skill 承载，开发者只在关键决策点拍板。这是 1.3-(3) 模板复用在"流程层面"的体现：流程本身也被模板化了。

#### (2) Claude Code 自动按 Skill 的流程走

Claude Code 收到上面那句提示词后，自动按 `module-delivery.md` 的步骤推进。关键观察点：**它在数据模型设计完之后主动停下来**，明确标注"等待用户确认"，不多做也不漏做。

这个停顿点就是第二步 review 时盯住的"决策点标注"在真实运行里的兑现：Skill 把"数据模型定稿属于人类决策"固化进流程，Claude Code 不会替人类拍板往下冲。开发者确认数据模型之后，它才继续推进到 Entity、DTO、Service、Controller。整套节奏比 Provider 模块（系列第 13 篇，手动做）还要稳——因为流程不再依赖开发者当下记不记得，而是由 Skill 兜底。

#### (3) Skill 也需要迭代

做完 Agent 模块，会发现第一版 Skill 并不完美——这是正常的。一个具体的迭代点：Agent 模块依赖 Provider 和 MCP，而 Skill 里没写"跨模块依赖怎么处理"（这一条在系列第 13 篇做 Provider 时没遇到，所以第一版 Skill 里没有）。把这条补进 Skill，下一份带依赖的模块就不用再现场决策。

这就是 Skill 作为活文档的真实节奏：**每做一个模块，Skill 就更完善一点**。做到第四五个模块时，Skill 已经覆盖绝大多数场景，Claude Code 几乎不需要额外指导就能按标准交付。1.3-(6) 的完整闭环是"沉淀 → 调用 → 迭代 → 再沉淀"，三个动作缺一不可：只沉淀不调用，等于把经验存档了但从不读档；调用但不迭代，第二份模块就会踩第一份模块已经踩过的坑。

#### (4) 阶段性产出汇报

Agent 模块交付流程跑完后，Claude Code 给出一份结构化的阶段性总结，列出本次交付的产出物、验证结果与后续步骤。从执行到沉淀、再到下一模块复用，构成 Skill 的完整闭环。

### 6.4 方法论指认

本章三个步骤，每一步都对应第 1 章方法论的一条动作：

- **1.3-(6) 经验沉淀复用（核心）**：第二步把 Provider 模块的流程沉淀成 `module-delivery.md`，第三步用一句话启动 Agent 模块并真的调用它，做完之后再回填"跨模块依赖怎么处理"——沉淀、调用、迭代三件事在本章完整闭环。这是 1.3-(6) 的兑现章。
- **1.3-(2) 想清楚先行**：第二步 review 时盯住的"决策点是否标注'等待用户确认'"、第三步 Claude Code 在数据模型设计完后主动停下来，都是把"想清楚不可跳过"固化进 Skill 的体现——关键决策交还人类拍板，AI 不越界。
- **1.3-(3) 模板复用**：第三步的对比演示把价值讲得很直白——没有 Skill 时要手写一大段流程描述，有了 Skill 一句话启动。流程本身被模板化，是模板复用在"流程层"的延伸。

本章同时给出一个可操作的强制动作，对应 1.3-(6) 的"真的被调用"：**习惯养成之前，在 CLAUDE.md 顶部加一行提醒——"开始新模块前，先看 `.claude/skills/` 目录"**。Skill 写了不用，等于把经验存档了但从不读档；一行固化提醒，是让经验真的被读出来的最低成本机制。

## 7. 本篇收束与延伸思考

<img src="imgs/aicent-19-core-fea-recap/82e16fe8aa2951ec57229a161e189bf0_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

三个场景走完，本章收束第 1 章六步方法论中的 1.3-(2)、1.3-(4)、1.3-(5)、1.3-(6) 四条，并以此印证本篇的两条核心交付路径：模块从后端到前端的完整闭环、经验从执行到沉淀的 Skill 循环。

### 7.1 四条值得记住的动作

下面四条动作，每条都对应第 1 章方法论骨架中的一条编号。把它们固化到日常协作里，比记住任何一个具体技术细节都更值钱。

#### (1) 数据模型讨论不能跳

对应方法论 1.3-(2)：想清楚先行——数据模型与边界决策落字成文，再让 AI 动手。

`auth_config` 用 JSON 存、健康状态独立成表，这些决策在后面每个模块都有依赖。跳过讨论直接建表，前面省下的时间，后面要还三倍——这正是"想清楚不可跳过"的实证。

#### (2) 报错给完整日志

对应方法论 1.3-(4)：报错给全量——完整日志 + 配置喂给 AI。

只给最后一行，Claude Code 在猜；给完整栈和相关配置，它能定位根本原因，通常一轮解决。

#### (3) curl 通了不算完

对应方法论 1.3-(5)：完整验收——curl 不是终点，走完整链路。

浏览器走完整链路（浏览器操作 → 反向代理 → 后端处理 → 数据推流 → 前端渲染），才是真正的交付闭环，每一层都可能有自己的坑。

#### (4) Skill 写了要真的用

对应方法论 1.3-(6)：经验沉淀复用——编码成 Skill 并真的调用、Skill 迭代。

开始新模块前先看一眼 `.claude/skills/`；习惯养成之前，可以在 `CLAUDE.md` 顶部加一行提示："开始新模块前，先看 .claude/skills/ 目录。"

### 7.2 方法论指认

本章的四条动作，分别收束第 1 章六步方法论中的四条：(1) 数据模型讨论不能跳收束 1.3-(2) 想清楚先行；(2) 报错给完整日志收束 1.3-(4) 报错给全量；(3) curl 通了不算完收束 1.3-(5) 完整验收；(4) Skill 写了要真的用收束 1.3-(6) 经验沉淀复用。这四条贯穿本篇三个实战场景，是"模块从后端到前端的完整闭环"与"经验从执行到沉淀的 Skill 循环"两条交付路径的共同支撑。把它们固化进团队 Wiki 与项目规范，比记住任何单一技术细节更值钱。

下一篇（系列第 20 篇）进入高级篇，为智能客服接入产品文档（RAG 知识库），让它回答时能引用真实内容。

本章同时给出一个可操作的强制动作，对应 1.3-(6) 的"真的被调用"：**习惯养成之前，在 CLAUDE.md 顶部加一行提醒——"开始新模块前，先看 `.claude/skills/` 目录"**。Skill 写了不用，等于把经验存档了但从不读档；一行固化提醒，是让经验真的被读出来的最低成本机制。
