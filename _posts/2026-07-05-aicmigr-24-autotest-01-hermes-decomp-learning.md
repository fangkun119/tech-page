## 2. 第一部分 方法论手册：让 AI 帮你建立顶层认知

本部分是「接到一个基于开源项目做需求的任务」后，在动手之前应该完成的方法论，不涉及具体技术栈。后续在第二部分会结合 Hermes 实战逐条对照。

### 2.1 摸透开源项目的目标与边界

<img src="imgs/aicmigr-24-autotest-01-hermes-decomp-learning/c05415cdab1ea5d6ca0290620a0d88b6_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

#### (1) 摸透不是「读到能提 PR」

接到「基于开源项目做需求」的任务，第一件事不是动手，是先把工具摸透。但「摸透」有明确的工程定义：

##### ① 摸透的目标

建立一个**够用于下一步拆需求的顶层认知**。不是把项目源码读到能提 PR，不是把官方文档逐字读完，而是建立一份对项目「边界、能力、扩展形状、运行时」的最小完备认知。

##### ② 顶层认知的四层结构

最小完备认知由四层组成。少了任何一层，要么用不起来，要么用错：

| 层 | 含义 | 决定了什么 |
|---|------|----------|
| 它是什么 | 定位与边界，解决什么、不解决什么、与同类工具的关系 | 要不要继续往下看 |
| 它能做什么 | 原生能力清单，不写一行代码能白嫖多少 | 能省多少工作量 |
| 它怎么扩展 | 扩展机制（插件 / Skill / Hook 等）与开发范式 | 自己要写的部分长什么样 |
| 它内部怎么跑 | 进程模型、状态管理、对外接入面 | 能否预判生产环境表现 |

#### (2) 让 AI 帮你建立认知的两个前提

让 AI 帮你建立认知，前提是这两个条件同时成立，否则只能拿到一坨官方话术：

##### ① 前提一：自己有拆解框架

工程师对陌生开源项目要建立的顶层认知，不是 AI 自己能猜出来的，而是过去踩坑沉淀出来的判断维度。**这个框架是工程师的能力，不是 AI 的能力**。AI 的能力是：你给好框架之后，它能把框架填得又快又准。

反例：「Claude，告诉我 Hermes 是什么」→ 拿到一坨「它是 AI Agent 平台、支持多种扩展、有完整工具链」→ 依然不知道能拿它干什么。问题不在 AI，在于你没给 AI 一个拆解框架。

##### ② 前提二：带着任务去问

同样是问「Hermes 怎么扩展」，不带任务问和带着任务问，得到的答案完全不一样：

| 问法 | AI 给出的内容 | 工程师是否能直接用 |
|----|------------|--------------|
| 不带任务问 | 扩展机制大全：所有扩展点、所有 API、所有玩法 | 不能，看完仍不知道哪个对自己有用 |
| 带任务问 | 过滤掉与任务无关部分，告诉「对你这个场景该用哪个扩展点，因为……」 | 能，这是工程师真正需要的认知 |

带任务问的标准动作：每个提示词里都嵌入一个具体的「自动化测试 Agent」这类业务任务，让 AI 拿着这个任务去过滤资料、筛选回答、给出建议。

### 2.2 用三个提示词覆盖四个维度

<img src="imgs/aicmigr-24-autotest-01-hermes-decomp-learning/7065cf8fa992e3f53bac1d06f907d2ae_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

四个认知维度通过三个提示词拿到（「它是什么」与「它内部怎么跑」合并，因为这两块在多数项目里连着讲最自然）。三个提示词都丢给 Claude Code，让它在目标项目的仓库里跑。

#### (1) 三个提示词的职责分工

| 提示词 | 覆盖维度 | 输出要回答的问题 |
|-------|--------|-----------|
| 提示词一 | 是什么 + 内部机制 | 是否适合做这个任务、哪些核心抽象会被直接用、运行时模型对部署方案的影响 |
| 提示词二 | 能做什么 | 任务需要的每项能力分别属于「直接支持 / 半支持 / 不支持」，自己要写的工程量 |
| 提示词三 | 怎么扩展 | 扩展点的形状（目录结构、Tool 定义、状态共享、最小骨架），评估开发量 |

#### (2) 提示词措辞的两条经验

##### ① 列清单让 AI 对照，而不是让 AI 罗列

提示词二的关键设计是：把任务需要的能力一条一条列出来，让 AI 拿着这份清单去对照目标项目的能力。不带清单，得到的是「所有能力罗列」；带清单，得到的是「对自己有用的能力筛选」。

##### ② 给 AI 留反问的余地

带任务问还不够，带任务问 + 给 AI 留反问的余地，才是更高级的姿势。措辞示例：

```text
继续。如果我决定用 Hermes 做这个系统，我自己要写的核心是 Skill 吗？
如果是，请告诉我……
```

特地把「核心是 Skill」写成疑问句而不是肯定句。**AI 拿到一个肯定句会顺着你说，拿到一个疑问句才有空间纠正你的预设**。这种走回头路在工程上是可以避免的，前提是问 AI 时留出反问的余地。

### 2.3 AI 建立认知后，回到官方文档沉下去

<img src="imgs/aicmigr-24-autotest-01-hermes-decomp-learning/18448670d598cd855c8661d0f5eb9ed2_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

#### (1) 为什么不能停在 AI 的总结上

AI 给出的都是经过总结后的内容。如果一直只看这种被总结过的内容，没法真正学会一个东西——**总结的过程会丢东西，而丢掉的往往就是建立体系化认知最需要的部分**。

#### (2) 回看文档的标准

带着两个东西去过文档：

- 前面 AI 帮你建立的认知框架
- 「做一个 XX 系统」的具体任务

看的标准：**大概看懂、大概知道有什么东西就行**。不需要把每一篇都搞懂，因为下一步拆完需求后还会带着拆出来的需求再回头看一次，三轮叠加下来效果最好。

### 2.4 学习路径的分工

#### (1) AI 与系统化资料各有擅长

| 学习方式 | 适合的阶段 | 局限 |
|--------|---------|------|
| Claude Code 辅助学习 | 快速建立认知、总结核心要点、过滤与任务无关的部分 | 给的认知片段化，不够成体系；从头学一个大项目效果不好 |
| 官方系统化资料 | 把 AI 建立的认知沉下去、补成体系 | 不带框架去读会迷失在细节里 |

#### (2) 推荐的组合路径

```text
先用 Claude Code + 四维度框架 + 带任务问 → 快速建立顶层认知
↓
带着认知框架 + 任务 → 过一遍官方文档 → 沉成体系
↓
拆完需求后 → 回看认知框架与官方文档 → 三轮叠加效果最好
```

### 2.5 接到「基于开源项目做需求」任务的 Check List

<img src="imgs/aicmigr-24-autotest-01-hermes-decomp-learning/ab45a2cc078201eb7f44aecc0f79fe54_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

把本篇方法论裁剪成一份项目阶段速查表。每接到一个「用 XX 开源项目实现 YY 需求」的任务，照此清单逐条勾。

#### (1) 第 0 步：明确任务边界

- [ ] 用一句话写出需求（谁、要做什么、关键约束是什么）
- [ ] 列出任务需要的能力清单（每条独立，便于后续对照）

#### (2) 第 1 步：建立四维度认知

- [ ] 它是什么：定位、解决什么、不解决什么、与同类工具关系
- [ ] 它能做什么：原生能力清单、不写代码能白嫖多少
- [ ] 它怎么扩展：扩展机制、扩展边界、开发范式
- [ ] 它内部怎么跑：进程模型、状态管理、对外接入面
- [ ] 至少能回答「不解决什么」——这块比「能做什么」更值钱

#### (3) 第 2 步：让 AI 帮你建立认知

- [ ] 三个提示词分别覆盖「是什么+内部机制」「能做什么」「怎么扩展」
- [ ] 每个提示词都嵌入具体业务任务，不是裸问
- [ ] 提示词二把任务能力列成清单，让 AI 对照而不是罗列
- [ ] 提示词三措辞留反问余地（疑问句而非肯定句）

#### (4) 第 3 步：输出边界图与工程量判断

- [ ] 画一张边界图：白嫖区 vs 开发区一目了然
- [ ] 给出开发区工程量估算（行数级别 + 集中在哪几块）
- [ ] 区分「结构化产物用代码生成」与「叙述性产物交给 LLM」

#### (5) 第 4 步：回看官方文档沉成体系

- [ ] 带着 AI 建立的认知框架 + 任务，过一遍官方文档
- [ ] 标准是「大概看懂」即可，不必逐篇搞懂
- [ ] 拆完需求后，回看一次认知框架与文档，完成三轮叠加

#### (6) 反模式（避免）

- [ ] ❌ 第一件事就动手写代码
- [ ] ❌ 「Claude，告诉我 XX 是什么」裸问
- [ ] ❌ 把肯定句写死，不给 AI 留反问余地
- [ ] ❌ 停在 AI 总结层，不回看官方文档
- [ ] ❌ 把所有产物都交给 LLM 生成（包括结构化 JSON）

## 3. 第二部分 实战演示：拆解学习 Hermes Agent

本部分用第一部分的方法论，复现「基于 Hermes Agent 实现 7×24 跑混沌测试的 AI 系统」这一任务的认知建立过程。原文使用的开源项目是 Hermes Agent，任务背景来自一个真实的二次开发场景。

### 3.1 场景与任务起点

<img src="imgs/aicmigr-24-autotest-01-hermes-decomp-learning/ebe4af8dc084db527c077ddcf72570cf_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

#### (1) 第六部分的定位

系列前几部分改造的是公司内部维护的老项目——代码、bug、方向都是自己的。第六部分换一种同样常见的场景：**接到一个需求，要基于一个开源项目去实现**。这是工程师的标准动作。

| 角色 | 典型任务 | 先看哪个开源生态 |
|----|-------|-------------|
| 运维 | 内部巡检平台 | Prometheus 生态 |
| 后端 | 业务工作流引擎 | Temporal / Airflow |
| 算法 | RAG 应用 | LlamaIndex / Dify |

#### (2) 本系列的演示项目

第六部分用一个开源项目作为案例：给 **Hermes Agent**（一个开源的自托管 AI Agent 控制平面）添加「基于 Hermes 实现的 7×24 不间断跑混沌测试的 AI 系统」。这个自动化测试系统主要包括：

- 自动起集群
- 自动注入故障
- 自动调多语言 SDK 跑测试
- 自动收集日志
- 自动出报告

参考资料（项目方关于「基于 AI 的自动化测试」的两篇博客）：

```text
https://robustmq.com/zh/Blogs/95
https://robustmq.com/zh/Blogs/84
```

#### (3) 任务起点：Hermes 还很新

假设对 Hermes Agent 一点都不了解。Hermes 是开源的自托管 AI Agent 控制平面，作者之前听过但没真正用过，公开渠道也搜不到太多企业落地案例——这恰好是这一部分的价值：**在没有现成先例的时候，基于 AI 快速摸出一条能走通的路**。

### 3.2 学习前的两个想法

#### (1) 第一反应：Hermes Agent 是什么？怎么用它？

接到任务的第一反应是「Hermes Agent 是什么？我要怎么用它来实现这个 7×24 不间断跑混沌测试的 AI 系统？」。所以准备先去学习一下 Hermes Agent。

#### (2) 学习方式的两个选择

##### ① 选择一：通过 Claude Code 辅助学习

近期已经习惯让 Claude Code 帮做总结。但个人经验是：**从头学一个大的东西（比如 OpenClaw 原理、Hermes Agent 等），用 Claude Code 学下来并不是那么好**——这里说的「好」是系统性，AI 给的认知是片段化的，不够成体系。

##### ② 选择二：系统化学习

直接找官方系统化的资料来学。让 Claude Code 推荐系统化的学习资料，找到了 `hermesagent.org.cn/docs` 这一份，内容系统、合适。

#### (3) 两条个人经验

作者个人摸出来的两条经验：

- 要有自己的学习框架（即第一部分「四维度拆解框架」）
- 学习是带着任务去的、不是大而全的学习

所以个人习惯是：**Claude Code 帮忙快速建立认知、总结核心要点，然后通过系统化资料沉下去学**。

### 3.3 提示词一：它是什么 + 它的内部机制

<img src="imgs/aicmigr-24-autotest-01-hermes-decomp-learning/d3269c4b797a86e142ade79a5568c2d8_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

#### (1) 提示词全文

```text
你现在在 Hermes 这个开源项目的目录下。我刚接到一个公司内的任务:

要基于 Hermes 做一个 7×24 跑混沌测试的 AI Agent，功能包括:
自动起集群、注入故障、调多语言 SDK 跑测试、收集日志、生成报告、通过飞书通知团队。

请你读 README、docs/architecture.md 和 src 顶层目录结构，告诉我:

1. Hermes 的核心定位是什么。一句话说清楚。
2. Hermes 的核心抽象有哪些。每个抽象用一句话解释它的作用。
3. Hermes 不解决什么问题。它的设计边界在哪。
4. Hermes 是怎么跑起来的。进程模型、配置加载、状态管理大致是什么样。
5. 结合我接到的任务，Hermes 的定位是不是和我的需求方向一致。有没有明显的不匹配。

不要复述官方话术。用你判断后的结论来回答。
```

读完这个提示词的输出，应该能回答这几个问题：Hermes 是不是适合做这个任务、它的核心抽象有哪些会被直接用到、它的运行时模型对部署方案有什么影响。

#### (2) Claude Code 跑完输出的关键信息

##### ① 核心定位

有学习能力的通用 AI 工作台，把 LLM、工具调用、跨平台消息、定时任务封装成可以长期运行的 Agent 基础设施。

##### ② 核心抽象

| 抽象 | 作用 |
|----|----|
| AI Agent | 对话循环 |
| SessionDB | SQLite + FTS5 会话存储 |
| Tools | 自动发现的工具注册表 |
| Skills | Markdown 写的过程记忆 |
| Delegate | 子 Agent 隔离调度 |
| Cron | 定时触发 |
| Gateway | 多平台消息路由 |
| Terminal Backends | 本地 / Docker / SSH / Modal 等执行环境 |

##### ③ 运行时

```text
通常两个长期进程：hermes CLI 和 hermes gateway 网关
通过 SQLite WAL 共享状态
子 Agent 是同进程线程
配置走 ~/.hermes/config.yaml，API Key 单独放 .env
```

##### ④ 不解决什么问题（比「能做什么」更值钱）

- 不是测试框架（没有 assertion / fixture / test suite 概念）
- 不是可观测性平台（日志只到 agent.log，没有结构化指标和告警）
- 不是集群编排器（Terminal backend 只是一个 shell）
- 不是报告生成引擎（没有结构化报告模板）
- 没有任何故障注入能力（tools/ 里没有 chaos engineering 相关实现）
- 默认还是 human-in-the-loop 模式（7×24 无人值守需要大量 allowlist 配置）

#### (3) 输出的工程价值

「不解决什么问题」这一段，比「能做什么」更值钱。它直接告诉工程师**接下来要写的核心，正好填在这些「它不做」的位置上**：故障注入、结构化报告、跨会话状态、无人值守的 allowlist 配置。

结论：方向是对的，但 Hermes 是骨架，不是答案。要做的是在它给的扩展点上，把它不做的那些事补齐。

### 3.4 提示词二：它能做什么

<img src="imgs/aicmigr-24-autotest-01-hermes-decomp-learning/a23930af5900114d91571d9c7846c42a_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

#### (1) 提示词全文

```text
继续。我的任务需要这些能力，请对照 Hermes 现有功能，
告诉我每一项能不能直接用、用哪个特性、还需要我自己实现什么:

1. 定时触发(每 2 小时跑一次基础场景、每天凌晨跑全量场景)
2. 调用工具函数(起集群、注入故障、跑客户端、收日志、出报告)
3. 在多轮调用之间共享状态(集群信息、测试进度)
4. 生成结构化报告(JSON + Markdown)
5. 通过飞书发通知(测试失败、报告生成完毕)
6. 手动触发(开发期间针对性验证某个场景)
7. LLM 调用循环(多轮 tool use、错误重试、token 管理)

输出格式:每一项分别说，Hermes 直接支持哪些、半支持哪些(需要简单配置)、
完全不支持哪些(需要我自己写)。
最后给我一个总结，我自己要写的工程量大概有多大。
```

关键设计：把任务需要的能力一条一条列出来，让 AI 拿着这份清单去对照 Hermes 的能力。不带清单得到「能力罗列」，带清单得到「对自己有用的能力筛选」。

#### (3) 一条关键设计判断

跑完这一轮得到一条非常重要的判断：**结构化报告这件事，不能交给 LLM 写**。Claude Code 直接告诉作者，如果让 LLM 直接生成 JSON，字段名会飘、嵌套层级会乱，任何下游程序解析都会踩坑。

这条分工规则成为后续设计 Tool 时的一条硬规则：

| 产物类型 | 由谁生成 | 原因 |
|--------|--------|----|
| 结构化产物（JSON、配置、报告字段） | 代码生成 | 字段名 / 嵌套层级稳定，下游可解析 |
| 叙述性产物（人类可读说明） | LLM 生成 | 擅长自然语言组织 |

这个判断非常重要——它直接决定了第 25 篇拆需求时，**拆到的颗粒度只到工具函数级别就够了，不用从零规划一个 Agent 框架**。

### 3.5 提示词三：它怎么扩展

<img src="imgs/aicmigr-24-autotest-01-hermes-decomp-learning/c613850aff722f6127010568d00e95e9_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

#### (1) 提示词全文

```text
继续。如果我决定用 Hermes 做这个系统，我自己要写的核心是 Skill 吗？

如果是，请深入读 docs/skills.md 和 examples/ 下的几个示例 Skill，告诉我:

1. Skill 是什么形状。目录结构、关键文件、一个最小 Skill 包含哪几样。
2. Skill 里的 Tool 怎么定义。函数签名、参数类型、返回值约定。
3. Skill 之间怎么共享状态。Memory 系统的使用范式是什么。
4. SKILL.md 这个文件是给谁看的、写什么内容、有没有写法约定。
5. 给我一个最小可行的 Skill 骨架，假设这个 Skill 只做一件事：返回当前时间。让我能判断扩展一个 Skill 的工程量大概多大。

不要泛泛而谈，我要的是能直接用来评估开发量的具体认知。
```

核心是让 AI 给「扩展点的形状」，而不是讲「扩展机制有哪些」。注意第一句措辞「我自己要写的核心是 Skill 吗？如果是，请告诉我……」——特地把「核心是 Skill」写成疑问句而不是肯定句，给 AI 留出反问的余地。

#### (2) 反认知的纠正：核心是 Tool，不是 Skill

事实证明这个余地留对了。Claude Code 跑完后给出一个反认知的纠正：**以为要写的核心是 Skill，实际上要写的核心是 Tool**。

Skill 不能定义新的 Tool，Skill 只是 Tool 的「使用说明书」。三者的分工长这样：

```text
Tool 在 tools/*.py 用 Python 写，通过 registry.register() 注册。
Skill 在 skills/*/SKILL.md 用 Markdown 写，告诉 LLM 什么时候调哪个 Tool、按什么顺序调。
Skill 目录下的 scripts/*.py 是普通辅助脚本，不是 Tool，Agent 通过 Terminal 工具调用它们。
```

#### (3) 工程影响

这个纠正改变了对工程量的判断：

- Skill 写一份 Markdown，几小时就够
- 真正的工作量在 Tool 的 Python 实现

如果一开始就按「写一个大的 Skill」去拆需求，最后会发现 Skill 里装不下任何业务逻辑，绕一圈才回到 Tool 上——**这种走回头路在工程上是可以避免的，前提是问 AI 时留出反问的余地**。

### 3.7 回看官方文档：把认知沉成体系

#### (1) 为什么此时应该回看官方文档

在 Claude Code 的帮助下，对 Hermes Agent 已经有了一个基础的框架认识。按个人经验，这时候不应该再依赖 Claude Code 来学，应该回头去看官方资料，系统化地补一遍。原因：

- AI 给的内容都是经过总结后的。一直只看这种被总结过的内容，没法真正学会一个东西
- 总结的过程会丢东西，而那些被丢掉的，往往就是建立体系化认知最需要的部分

#### (2) 此时的看文档标准

带着两个东西去过文档：

- 前面建立起来的认知框架
- 「做一个自动化测试 Agent」的具体任务

看的标准是：**大概看懂、大概知道有什么东西就行**。不需要把每一篇都搞懂。

#### (3) 为什么这个标准就够了

因为下一讲会拆解「自动化测试 Agent」的需求，然后带着拆出来的需求，再回头看本节的认知框架和官方文档。**三轮叠加下来，效果是最好的**：

```text
第一轮：AI 建立顶层认知
第二轮：带着认知 + 任务回看官方文档
第三轮：拆完需求后，回看认知框架与官方文档
```

### 3.8 本篇小结

接到一个基于开源项目做需求的任务，第一件事不是动手，是先把工具摸透。摸透不是把项目读到能提 PR 的程度，而是建立一个够用于下一步拆需求的顶层认知。这个认知有四层：它是什么、它能做什么、它怎么扩展、它内部怎么跑。

让 AI 帮你建立认知的关键是两点：

- 要有自己的拆解框架
- 要带着任务去问

**框架决定问什么，任务决定怎么问**。两个都到位，认知建立的速度会比想象的快。

跑完本篇，手上已经有一份对 Hermes 够用的认知。但还差一步：还没拿这份认知去对照那句「做一个自动化测试 Agent」的需求。下一篇就来做这件事，把一句话需求拆成能交付的方案。

这一套打法不是 Hermes 专用的——下次接到任何「用某某开源项目实现某某需求」的任务，这套四维度框架和带任务问的姿势，都能直接套上去。
