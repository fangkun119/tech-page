## 2. 方法论：用 AI 做架构决策的四个思考维度

### 2.1 方法论闭环：AI 给方案 → 约束清单 → 追问否决 → 项目最优

<img src="imgs/aicent-04-arch-code-organize/7b7cfc22069c2d4c7354113a23a2c657_MD5.jpg" style="display: block; width: 100%;" alt="替换文字">

用 AI 做架构决策时，反复出现同一个模式：<span style="color: red; font-weight: bold;">AI 给通用最优方案 → 工程师用项目约束清单 → 追问否决 → 得到项目最优方案</span>。这不是偶发动作，而是可复用的方法论闭环。

要害在于：<span style="color: red; font-weight: bold;">AI 给的是“技术上最优”的方案，你要基于项目约束选“对你最优”的方案。这两者经常不一样。</span>AI 默认推荐微服务、WebFlux这类“通用最优”——技术上无可挑剔，但你的项目可能一人维护、不希望增加技术栈，于是这些方案被否决。<span style="color: red; font-weight: bold;">决定最终方案的不是 AI 的判断，而是你的项目约束。</span>

<span style="color: red; font-weight: bold;">闭环的四个动作必须连起来用，缺一环就会跑偏</span>：

<img src="imgs/aicent-04-arch-code-organize/d3ef08b40309100d0a36f0b382a7d1f4_MD5.jpg" style="display: block; width: 800px;" alt="替换文字">

<!--

    B --\> C{三类追问<br/>成本/兼容/参数}
    C --\>|否决或需调整| D[否决或调整]
    D --\>|未通过| A

下面四节展开四个思考维度。维度一是约束清单（输入端），维度二是追问否决（中段动作），维度三是规范即合约（沉淀方式），维度四是为未来留口（验证动作）。

### 2.2 维度一：约束驱动——用项目真实约束清单 AI 方案

<img src="imgs/aicent-04-arch-code-organize/33d695bb79a6e16e3082b45ce838f4e7_MD5.jpg" style="display: block; width: 100%;" alt="替换文字">

<span style="color: red; font-weight: bold;">让 AI 给方案前，先把项目约束清单列出来——写下来</span>，不是脑子里想。否则 AI 的第一版方案默认按"通用最优"给，你很容易被它说服，接受了"技术上最优但对项目灾难"的方案。

#### (1) 项目约束清单模板

| 维度 | 示例值 | 作用 |
|---|---|---|
| 人力 | 1 人开发 / 3 人小组 / 10 人团队 | 决定能否承担多服务运维 |
| 当前规模 | 50 人使用 / 500 人 / 5000 人 | 决定架构复杂度上限 |
| 峰值规模 | 50 并发 / 500 并发 | 决定线程池、缓存设计 |
| 1 年增长预期 | 翻倍 / 10 倍 / 100 倍 | 决定是否预留拆分口子 |
| 技术栈边界 | 不引入 WebFlux / 不引入消息队列 | 否决"技术优雅但超界"的方案 |
| 未来扩展点 | 可能拆微服务 / 可能上多区域 | 决定接口设计颗粒度 |

#### (2) 操作要领

- 每次让 **AI 给方案前，先把约束清单贴在对话开头**，让 AI 在约束下重给方案，而不是直接接受第一版
- 约束清单要分两类写明：**硬边界**（绝不引入 X）和**软约束**（1 年内可能 Y）
- 约束变了要重新过一遍清单：人员增加、规模变化、新需求引入都属于约束变更
- 把约束写进项目说明文件（如 CLAUDE.md），让 AI 后续每次生成都自动遵守

### 2.3 维度二：追问否决——三类必问问题

<span style="color: red; font-weight: bold;">AI 推荐方案后，必问三类问题：成本、兼容、参数合理性</span>。<span style="color: red; font-weight: bold;">这不是抬杠，是逼 AI 把“通用最优”在“你的场景”下验算一遍。AI 不会主动暴露短板，要靠追问挖出来。</span>

#### (1) 成本类追问

- "这套方案的人力 / 运维成本多大？"
- "独立部署、独立配置、独立监控的精力消耗我能承担吗？"
- "上线后出问题的排查难度相比现在增加多少？"

<span style="color: red; font-weight: bold;">成本类追问专治“看起来漂亮、扛起来要命”的方案</span>——微服务、独立网关、消息总线这类典型陷阱。

#### (2) 兼容类追问

- "这与现有技术栈冲突吗？"
- "引入它会带来哪些连带依赖？"
- "团队 / 我自己熟悉吗？学习曲线多陡？"

<span style="color: red; font-weight: bold;">兼容类追问专治“为了技术优雅引入新编程模型”的诱惑</span>——例如项目用同步 MVC 已经够用，AI 建议引入响应式框架处理流式响应。

#### (3) 参数合理性追问

- "在我实际场景下（如流式响应、长连接）这个参数合理吗？"
- "边界情况（高峰、超时、限流）下行为是什么？"
- "有没有 AI 没主动提的隐患？"

<span style="color: red; font-weight: bold;">参数合理性追问专治“AI 给默认值、你照单全收”的懒惰</span>——AI 给 60 秒超时，但你做的是 SSE 流式响应，60 秒根本不够用。

### 2.4 维度三：规范即合约——规范是写给 AI 看的

<img src="imgs/aicent-04-arch-code-organize/4dc960eaa756c8604de900cc545f67e3_MD5.jpg" style="display: block; width: 100%;" alt="替换文字">

<span style="color: red; font-weight: bold;">代码组织规范在 AI 编程里是刚需，不是可选项。这些规范不是写给人看的，是写给 AI 看的。</span>

#### (1) 颗粒度跟着"AI 容易跑偏处"走

准则：<span style="color: red; font-weight: bold;">AI 反复跑偏处写细，不跑偏处不写</span>。

- **写细的例子**：AI 喜欢"顺手"在 Controller 加业务逻辑——这条规范要明确写"Controller 只做参数校验和调用 Service，禁止业务逻辑"
- **不写的例子**：AI 不会把事务写在 Mapper 里——这条不用写，写了反而稀释关键信息
- **写细的例子**：AI 不知道跨模块要走 Service 接口——这条要明确写"跨模块只能调对方 Service 接口，禁止直接引用 Mapper 或 Entity"

<span style="color: red; font-weight: bold;">颗粒度的判断标准是“AI 在这块是否反复跑偏”，不是“这条规则多基础”。</span>

#### (2) 为什么这条颠覆传统认知

<span style="color: red; font-weight: bold;">传统代码规范的受众是“团队工程师”——颗粒度粗，大家凭习惯就能补全。AI 编程规范的受众是“AI 代码生成器”——每次生成都是全新一次理解，颗粒度细才能保证一致性。</span>

这条颠覆体现在三处：

- 受众变了：从人变 AI，要求从"约定俗成"变"白纸黑字"
- 投入产出变了：现在花半小时定好，后面几百次代码生成都受益
- 价值衡量变了：规范的衡量标准不是"工程师读着舒服"，而是"AI 生成的代码符合预期"

### 2.5 维度四：为未来留口——每个决策问"未来 X 改动多大"

<span style="color: red; font-weight: bold;">架构决策的真正价值不是“现在好用”，而是“未来要改的时候改动可控”。判断一个决策好坏的标准，是它有没有堵未来的路。</span>

#### (1) 未来扩展验证 Prompt 模板

每个关键架构决策定下来后，用下面这个通用骨架让 AI 量化评估：

```
假设未来要把【某个模块 / 某个能力】独立出来（如拆成独立服务、上多区域、换实现），
按照当前的架构和规范，改动量大概多大？哪些点会卡住？
```

这个 Prompt 的作用是把抽象的"扩展性"变成具体的"改动量清单"。如果 AI 回答"改动很大，要重写一半"，说明当前决策堵了路，得回去调整。

#### (2) 架构功力的体现

<span style="color: red; font-weight: bold;">“架构功力”不在于现在做多复杂，在于不堵未来的路。</span>判断方法是决策时多问一句：“未来扩展时这会变成障碍吗？”

- 跨模块走 Service 接口而非直接引用 Mapper——当下看不出区别，未来拆服务时这条决定了改动量
- 模块边界单向依赖——当下是约定，未来拆分时是地基
- 接口设计颗粒度跟着未来扩展点走——当下是规范，未来演进时是护栏

所有"为未来留口"的规范，当下都不会带来收益、只带来约束成本，但它们的价值在未来某次拆分 / 替换 / 扩展时一次性兑现。<span style="color: red; font-weight: bold;">这是“做架构”和“写代码”的本质区别。</span>


## 3. 三大主题方法论展开

### 3.1 应用架构方法论

<img src="imgs/aicent-04-arch-code-organize/f430d2e8bfbc30addbd9bbf066fe28cc_MD5.jpg" style="display: block; width: 100%;" alt="替换文字">

#### (1) 目标与思考框架

应用架构方法论的目标是：**在动手写任何一行代码前，先把应用内部如何组织确定下来**，避免后期模块互相耦合、风格逐渐混乱、改一处牵一片。

思考框架按四步逐层细化：**形态选型 → 模块划分 → 依赖关系 → 边界规则**。先选整体形态（单体还是分布式），再决定模块怎么切，再约定模块之间的依赖走向，最后用边界规则约束跨模块调用。每一步的输出是下一步的输入，不能跳步。

#### (2) 关注点清单

| 关注点 | 关键问题 |
|---|---|
| 形态选型 | 单体 / 模块化单体 / 微服务？依据是人力和规模，不是技术先进度 |
| 模块划分原则 | 按业务功能分（推荐）还是按技术层分？按业务分利于演进和未来拆分 |
| 依赖方向 | 依赖是单向的，不能循环。循环依赖说明模块边界划错了 |
| 共用逻辑下沉 | 多模块共用的工具、常量、异常下沉到 common 模块，避免散落各处 |

#### (3) 操作要领

- 让 AI 给 2-3 个方案做对比——不要只问"怎么做"，要明确要求"给我方案对比"，AI 才会摆出取舍而不是单点推荐
- 用项目约束清单（参考 §2.2）筛方案——把人力、规模、技术栈边界贴在对话开头，让 AI 在约束下重给方案
- 让 AI 主动梳理模块依赖关系图，并要求它**主动识别循环依赖风险**，而不是等出问题才发现
- 用未来拆分场景验证当前划分是否合理（参考 §2.5.1 的 Prompt 模板），把"扩展性"量化成"改动量"

#### (4) 注意点

- 单机部署 ≠ 所有代码堆在一个 package——单机部署是运行形态，代码内部仍要按模块清晰分层
- 微服务对一人或小团队是负担——独立部署、独立配置、独立监控的精力消耗远超技术收益，要敢于否决
- 出现循环依赖，说明模块边界划错了——解决办法是把共用部分下沉到 common 模块，而不是在两个模块之间互相 import

### 3.2 代码组织方法论

<img src="imgs/aicent-04-arch-code-organize/986f7e41d3fe17edd61e2e700b5384ee_MD5.jpg" style="display: block; width: 100%;" alt="替换文字">

#### (1) 目标与思考框架

代码组织方法论的目标是：**定义一套代码分层规范，约束 AI 生成代码的行为**。注意——这套规范不是写给人看的，是写给 AI 看的，这一条会颠覆传统认知。

思考框架按四步走：**分层结构 → 职责边界 → 跨模块调用 → 转换规则**。先约定分几层、每层叫什么，再写清每层能做什么不能做什么，再约束跨模块怎么调，最后规定数据对象在层与层之间怎么转换。

#### (2) 关注点清单

| 关注点 | 关键规则 |
|---|---|
| 分层结构 | Controller / Service / Mapper / Entity / DTO 五层，缺一不可 |
| 每层职责边界 | Controller 只做参数校验和调用 Service；Service 处理业务逻辑；Mapper 只做数据库操作 |
| 跨模块调用 | 只能通过 Service 接口调用，禁止直接引用其他模块的 Mapper、Entity 或内部类 |
| Entity / DTO 转换 | Entity 不直接返回前端（含敏感字段如 API Key、内部 ID），必须用 DTO 转换 |

#### (3) 操作要领

- 让 AI 基于你的技术栈给规范——告诉它用的框架版本、ORM、依赖注入方式，它给的规范才贴合实际
- 明确要求"**具体到 AI 能直接执行，不要模糊描述**"——把"Controller 要简洁"换成"Controller 只做参数校验和调用 Service，禁止业务逻辑"
- 自己调整后定稿，不要全盘接受 AI 的版本——AI 给的初版往往偏理想化，要按项目约束裁剪
- 用未来拆分场景验证规范的合理性（参考 §2.5.1），特别是跨模块调用规则是否足以支撑未来拆服务

#### (4) 注意点

- AI 喜欢"顺手"在 Controller 里加业务逻辑——<span style="color: red; font-weight: bold;">必须明文禁止，这是 AI 反复跑偏的高发点</span>
- Service 之间只能调接口（interface），不能直接 new 实现类——否则未来替换实现或拆服务时改动量爆炸
- 不要把 Entity 直接返回给前端——敏感字段（API Key、内部状态码）会顺着接口泄漏出去
- <span style="color: red; font-weight: bold;">规范不是写给人看的，是写给 AI 看的</span>——这条颠覆传统认知，决定了规范的颗粒度和写法

### 3.3 外部调用方法论

<img src="imgs/aicent-04-arch-code-organize/ea738cbe9d4fb5c162f43a92a1822cb5_MD5.jpg" style="display: block; width: 100%;" alt="替换文字">

#### (1) 目标与思考框架

外部调用方法论的目标是：**为外部 API 调用设计完整方案**。外部调用有三个先天特征——慢、不稳定、多供应商——任何一个维度漏掉设计，都会在生产环境被现实打脸。

思考框架从四个维度切入：**线程管理 → 容错 → 超时 → 重试**。这四个维度互相独立又互相影响，<span style="color: red; font-weight: bold;">必须一次性整体设计，不能东补一块西补一块。</span>

#### (2) 关注点清单

| 维度   | 关键问题                                 |
| ---- | ------------------------------------ |
| 线程管理 | 外部调用用独立线程池，不与业务请求线程池共用，避免外部慢拖垮主线程    |
| 容错   | 熔断机制必备；多供应商时每个供应商独立熔断器，一家挂不影响其他      |
| 超时   | 按场景分层：同步调用、流式响应、连通性测试分别设超时，不能用一个值套所有 |
| 重试   | 按异常类型区分：网络抖动重试、认证失败不重试、限流按退避策略重试     |

#### (3) 操作要领

- 让 AI 系统性给方案——一次覆盖全部四个维度，不要东问一句线程池、西问一句超时，AI 才能给出连贯设计
- 逐维度判断（同意 / 不同意 / 要调整），而不是整体接受或整体否决——四个维度独立性强，混在一起判断会漏
- 追问细节——例如"这个超时对 SSE 流式响应够用吗""熔断打开后正在执行的请求怎么处理"
- **追问 AI 没主动提的问题**——例如"流式响应该用什么技术处理"——这是发现盲区的关键，AI 不会主动暴露短板

#### (4) 注意点

- **不为了"技术上更优雅"引入一套新的编程模型**——能用 SSE 就别引入 WebFlux，能用同步 MVC 就别上响应式框架
- 不同异常的重试价值不同——认证失败重试一百次还是认证失败，盲目重试只会加剧限流
- 超时必须分场景——流式响应的合理超时远大于同步调用，套同一个值要么误杀长任务要么放任卡死

### 3.4 方法论总结：三大主题的共性

三大主题表面看完全不同——应用架构问“形态怎么选”，代码组织问“规范怎么定”，外部调用问“方案怎么完善”——但底层用的是同一套方法论：让 AI 给方案、用项目约束清单、追问细节挖盲区、否决不合理推荐、为未来留口子。<span style="color: red; font-weight: bold;">三个问题答案各不相同，求解路径却完全一样。</span>

这套方法论是本系列反复出现的核心命题：<span style="color: red; font-weight: bold;">决定最终方案的从来不是 AI 的判断，而是你的项目约束。</span>应用架构被否决微服务，是因为人力扛不住；代码组织写得这么细，是因为受众是 AI 不是人；外部调用拒绝 WebFlux，是因为不为了技术优雅引入新编程模型。第二部分（章 5）会用 Hify 实战把这套方法论逐一印证——从模块化单体的选型，到写给 AI 看的代码规范，再到外部调用的四维设计，每一步都是这套方法论的具象化。

## 5. 实战复现：Hify 架构决策全过程

### 5.1 应用架构决策：从三方案到模块化单体

#### (1) 方法论锚点

本节呼应**§3.1 应用架构方法论**。方法论里强调的核心动作是"约束驱动决策"——让 AI 给方案、用项目约束清单、追问否决、为未来留口。这一节要把这套动作在 Hify 上完整跑一遍，最终落点在**模块化单体**：用一人维护这一硬约束否决了 Claude Code 推荐的微服务，再用代码层面的 Maven 多模块兼顾隔离和成本。

#### (2) 让 Claude Code 给方案对比

第 3 篇已经确定一期 50 人单机部署。但单机不等于所有代码堆在一个 package 里——单机部署是运行形态，代码内部仍要按模块清晰分层（这条在 §3.1.4 已强调过）。我先让 Claude Code 给方案对比，明确要求"给我方案对比"，它才会摆出取舍而不是单点推荐：

```
Hify 是一个 Spring Boot 单体应用，功能包括模型提供商管理、Agent 配置、对话引擎、知识库 RAG、简版工作流、MCP 工具接入。一个人开发，一期 50 人使用，但后续可能要扩到几千人。代码内部怎么组织？给我方案对比。
```

Claude Code 给了三个方案，整理如下：

| 方案 | 做法 | 优点 | 缺点 |
|---|---|---|---|
| A 按层分 package | 所有功能放一个模块，按 controller/service/mapper 分 package | 最简单 | 功能混在一起，代码量大后找文件费劲，后期难拆 |
| B Maven 多模块 | 一个 Spring Boot 应用，按功能拆 Maven 模块，模块间通过接口调用 | 代码层面严格隔离，后续拆分成本低 | 比方案 A 复杂一点 |
| C 微服务 | 每个功能独立服务，网关路由，服务间 Feign 调用 | 方便独立扩展 | 一人维护成本极高 |

Claude Code 推荐了方案 C，理由是方便独立扩展——这正是 §2.1 方法论闭环里说的"AI 给的通用最优方案"。

#### (3) 用约束否决微服务

按 §2.3.1 的成本类追问模板，我追问了运维精力：

```
一个人维护六七个微服务，每个要独立部署、独立配置、独立监控，精力消耗会不会太大？
```

Claude Code 承认了，说一人开发建议方案 B。<span style="color: red; font-weight: bold;">我的判断：选方案 B，模块化单体。</span>三条 why：

- **否决 C**：一人维护六七个微服务，每个要独立部署、独立配置、独立监控，精力消耗太大——这是 §3.1.4 列的典型陷阱
- **否决 A**：后期代码量大后找文件费劲，想拆也拆不动，堵死了未来扩展的路
- **选 B**：兼顾模块隔离（代码不混乱）和运维成本（只维护一个应用），后续拆分成本可控

#### (4) 模块划分与依赖关系

方案定下后是模块划分，按业务功能分（不是按技术层分）：

```text
hify/
├── hify-app/
├── hify-provider/
├── hify-agent/
├── hify-chat/
├── hify-mcp/
├── hify-workflow/
├── hify-knowledge/
├── hify-common/
├── hify-web/
└── deploy/
```

模块之间的依赖关系要提前想清楚，不能等出问题再发现。我让 Claude Code 主动梳理，并要求它**主动识别循环依赖风险**：

```
基于 Hify 的功能，帮我梳理这些模块之间的依赖关系。谁依赖谁？有没有循环依赖的风险？
```

它的分析（保留原意）：hify-chat 是依赖最多的模块——对话时要读 Agent 配置（依赖 hify-agent）、调模型（依赖 hify-provider）、可能走工作流（依赖 hify-workflow）、可能做 RAG 检索（依赖 hify-knowledge）、可能调工具（依赖 hify-mcp）。hify-agent 依赖 hify-mcp（Agent 要绑工具）和 hify-knowledge（Agent 要绑知识库）。所有模块依赖 hify-common。

#### (5) 单向依赖原则

模块划分的硬边界是单向依赖。三条关键命题：

- **"依赖是单向的，不能循环。"**
- "hify-chat 可以依赖 hify-agent，但 hify-agent 不能反过来依赖 hify-chat。"
- <span style="color: red; font-weight: bold;">“如果出现循环依赖，说明模块边界划错了，需要把共用的部分下沉到 hify-common。”</span>

第三条对应 §3.1.4 注意点：循环依赖的解决办法不是在两个模块之间互相 import，而是把共用部分下沉到 common 模块。

### 5.2 Spring 代码组织规范：写给 AI 看的合约

<img src="imgs/aicent-04-arch-code-organize/bad2b46acd19f6032f15a9053f3816ed_MD5.jpg" style="display: block; width: 100%;" alt="替换文字">

#### (1) 方法论锚点

本节呼应**§3.2 代码组织方法论**。方法论里强调这条颠覆传统认知的命题：**规范不是写给人看的，是写给 AI 看的**。颗粒度要跟着"AI 容易跑偏处"走，并且每个决策都要用"未来 X 改动多大"来验证（呼应 §2.5）。

#### (2) 让 Claude Code 定义规范

先说为什么这一节必须讲。原文的三句关键论断：

> 这一节很多人觉得没必要讲，Controller、Service、Mapper 三层架构，谁不知道？
>
> 人写代码确实不需要讲。你凭经验和习惯，自然知道业务逻辑放 Service 里、Controller 只做参数校验。但 AI 不知道。
>
> <span style="color: red; font-weight: bold;">这些规范不是写给人看的，是写给 Claude Code 看的。后面每一行代码都是它生成的，这些规矩定不好，生成出来的代码越多越乱。</span>

人不告诉 AI 规矩，它就会把事务写在 Controller 里、把数据库查询写在 DTO 里、直接引用另一个模块的 Mapper 把两个模块焊死。这就是 §2.4.2 说的"受众变了、价值衡量变了"。我让 Claude Code 基于具体技术栈给规范，并明确要求"具体到 AI 能直接执行"：

```
Hify 是模块化单体，用 Spring Boot + MyBatis-Plus。帮我定义代码组织规范，覆盖：每个模块内部的分层结构、每一层的职责边界、跨模块调用的规则。要求具体到 AI 能直接执行，不要模糊的描述。
```

Claude Code 给了一版，我调整后定稿——按 §3.2.3 操作要领，不全盘接受 AI 的版本，要按项目约束裁剪。

#### (3) 模块内部统一结构

每个业务模块内部统一目录结构：

```text
src/main/java/com/hify/{module}/
├── controller/
├── service/
├── service/impl/
├── mapper/
├── entity/
├── dto/
├── config/
├── exception/
└── constant/
```

#### (4) 每层职责边界

每一层的职责边界、严禁项、原因——必须明文写清，AI 才不会跑偏：

| 层 | 职责 | 严禁项 | 为什么 |
|---|---|---|---|
| Controller | 只做参数校验和调用 Service | 业务逻辑、数据查询、事务处理 | Claude Code 喜欢"顺手"在 Controller 加逻辑，后续测试、重构、拆分全部受影响 |
| Service | 处理所有业务逻辑（事务管理、数据校验、业务规则） | Service 间直接 new 实现类 | 只能调接口（interface），便于替换实现 |
| Mapper | 只做数据库操作 | 在 XML 里写业务逻辑（复杂条件判断） | 业务逻辑是 Service 的事 |
| Entity | 与数据库表一一对应 | 直接返回给前端 | 可能有敏感字段（API Key） |
| DTO | 接口请求 / 响应对象 | — | Entity 和 DTO 必须转换，控制暴露字段 |

Controller 那一行的"严禁项"对应 §3.2.4 列的 AI 反复跑偏高发点——这条规范颗粒度必须写到"禁止业务逻辑"这一层。

#### (5) 跨模块调用规则与为未来留口

**规则**：模块之间只能通过 Service 接口调用，不能直接引用另一个模块的 Mapper、Entity 或内部类。

这条是模块化单体最关键的规范，原因来自选模块化单体的初衷——后续能拆分：

- 直接引用 Mapper → 两个模块焊死，拆分时要找出所有引用改成远程调用
- 只走 Service 接口 → 拆分时只需把本地调用改成 HTTP/RPC 调用，<span style="color: red; font-weight: bold;">接口签名都不用变</span>

<span style="color: red; font-weight: bold;">这条规范本身没有当下收益，只带来约束成本，但它在未来某次拆分时一次性兑现</span>——这是 §2.5.2 说的"为未来留口"。

#### (6) 用未来拆分场景验证

按 §2.5.1 的未来扩展验证 Prompt 模板，我追问了拆分改动量：

```
如果有一天要把 hify-chat 拆成独立服务，按照这套规范，改动量大概多大？
```

Claude Code 的分析（保留原意）：Service 接口不变，实现类里的本地调用改成 Feign 调用，加上网络异常处理；每个跨模块调用点改两三行代码；**改动量可控**——这就验证了这套规范的合理性。作者原文标注（逐字保留）：

> 这里我标注下，能考虑到未来的拆分主要靠的是自己的项目经验，我希望你可以记下这个，就当积累一个经验。

这句话点出了方法论背后的功力来源：能不能想到"未来拆分"这一问，靠的是项目经验；没有经验时，记下来、照着问，就是积累经验。

### 5.3 外部调用设计：四个维度逐个判断

<img src="imgs/aicent-04-arch-code-organize/d482e9c29a45ae054101348ce0cedf54_MD5.jpg" style="display: block; width: 100%;" alt="替换文字">

#### (1) 方法论锚点

**呼应 §3.3 外部调用方法论。** 这一节演示的要点是：用追问细节否决 WebFlux、用追问调整超时阈值，把"慢 / 不稳定 / 多供应商"的 LLM 调用从四个维度（线程 / 容错 / 超时 / 重试）逐一落地。

Hify 最大的技术挑战不在 CRUD，而在外部 LLM 调用。第 3 篇运维预期已指出：瓶颈不在并发，而在 LLM 长连接占用线程。这一节就把那个预判的解决方案定下来。

#### (2) LLM 调用的三大特点

LLM API 调用有三个特点：

- **慢**：一次对话 3-30 秒
- **不稳定**：随时可能超时、限流、报错
- **多供应商**：每家行为不一样（OpenAI、Claude、Gemini、Ollama）

一句话总结：不提前设计好，上线就炸。

我用这段 Prompt 让 Claude Code 系统性给方案：

```
Hify 要调用多个外部 LLM API（OpenAI、Claude、Gemini、Ollama），这些调用慢且不稳定。从线程管理、容错、超时、重试四个维度，给出完整的技术方案。
```

**四维度判断总览（5.3.3-5.3.6）**

它给的方案很全面，我逐个判断。为了对比清晰，把四个维度的判断汇总成一张表：

| 维度 | Claude Code 建议 | 作者判断 | 为什么 |
|---|---|---|---|
| 线程池隔离 | LLM 调用用独立线程池，不与 Tomcat 业务线程池共用 | 同意 | 50 个并发 SSE 连接若共用池，管理页面请求会卡死（第 3 篇已分析）；独立池让对话请求和管理请求互不影响 |
| 熔断机制 | Resilience4j，每个提供商独立熔断器 | 同意 | 连续失败时快速熔断，后续请求直接返回错误，不浪费时间等超时；OpenAI 挂了不影响 Claude |
| 超时控制 | 对话接口 60 秒，连通性测试 10 秒 | 同意但**追问调整** | 见下方 §5.3.5 追问细节 |
| 重试策略 | 按异常类型区分 | 同意 | 见下方 §5.3.6 追问细节 |

下面四节补充各维度的关键追问细节。

#### (3) 线程池隔离（细节）

同意。补充：第 3 篇分析过 50 个并发 SSE 连接，每个持续 10-30 秒，如果共用 Tomcat 业务线程池，管理页面的请求会排在队列里等，用户看到的就是页面一直转圈。独立线程池让对话请求用自己的池子，管理请求用 Tomcat 默认池子，两者互不影响。

#### (4) 熔断机制（Resilience4j）

同意。补充：Claude Code 建议 Resilience4j，每个提供商独立熔断器——OpenAI 挂了不影响 Claude。某个提供商连续失败时快速熔断，后续请求直接返回错误，不浪费时间等超时。

#### (5) 超时控制（追问调整 SSE 120s）

同意，但**追问了细节**。Claude Code 建议对话接口 60 秒超时，连通性测试 10 秒。我追问：

```
流式响应场景下，一个 SSE 连接可能持续一两分钟，60 秒会不会把正常对话切断？
```

Claude Code 调整：<span style="color: red; font-weight: bold;">SSE 连接超时设 120 秒</span>，普通同步调用 60 秒。**合理**。

#### (6) 重试策略（按异常类型区分）

同意，但细节有价值。不是所有失败都值得重试：

- **网络抖动**：重试有意义
- **认证失败**：重试没用（重试一百次还是认证失败）
- **限流**：退避重试（等一等再试）

按异常类型区分重试策略，这个细节 Claude Code 给得很好。

#### (7) 追问 SSE 处理：否决 WebFlux

我又追问了一个 Claude Code 没主动提到的问题：

```
流式响应用 SSE，Spring MVC 怎么处理？需不需要引入 WebFlux？
```

它建议用 WebFlux 做异步非阻塞。我不同意，理由有三条：

1. 第 3 篇已经确定不引入额外技术栈
2. SSE 场景用 Spring MVC 的 SseEmitter 就够
3. 配合独立线程池和 120 秒超时，50 人完全能扛

关键论断（逐字保留）：<span style="color: red; font-weight: bold;">不为了“技术上更优雅”引入一套新的编程模型。</span>

### 6.1 本篇核心收获

- **三件实事落地**：确定了模块化单体的应用架构、定义了 Spring 代码组织规范、设计了外部 LLM 调用方案，三件事分别对应"模块怎么分、代码怎么组织、外部调用怎么处理"。
- **核心方法论**：最有价值的不是最终方案，而是和 Claude Code 的追问环节。每一次追问的背后，都是你在用项目的真实约束来修正 AI 的通用建议。AI 给的是技术上最优的方案，你要基于约束选对你最优的方案，这两者经常不一样。
- **规范即合约**：代码组织规范看起来是常识，但对 AI 写代码来说是刚需。人凭习惯知道 Controller 不写业务逻辑，AI 不知道——你不告诉它，它就按自己的理解来，而且每次理解都不一样。现在花半小时定好，后面几百次代码生成都受益。
- **经验可以积累**：你以前没有架构知识，不知道要问什么问题，当看完这篇不就有了吗？把本篇当作一个实践的架构示范，记下我们怎么思考的，以后就知道怎么问问题了。
- **方法论可复用**：<span style="color: red; font-weight: bold;">本篇的"让 AI 给方案 → 用约束清单 → 追问否决 → 留口子"四步法，适用于任何架构决策场景</span>——它推荐微服务被否决，它建议 WebFlux 被否决，它给的超时被追问后调整，都是同一套打法。

### 6.2 思考

结合这两个问题让 Claude Code 帮你分析当前项目的情况：

**问题一**：你的模块之间是通过接口调用还是直接引用内部类？如果要把某个模块拆成独立服务，改动量大概多大？

**问题二**：你的外部调用（数据库、第三方 API、消息队列）有没有做线程池隔离？如果某个外部服务卡了 30 秒，你的系统会怎样？
